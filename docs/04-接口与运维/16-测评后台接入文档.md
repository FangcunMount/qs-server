# 测评后台接入文档

**适用对象**：测评运营后台、模型配置后台、管理端前端  
**接入服务**：qs-apiserver  
**契约基线**：`befd73c53`（2026-07-12）

本文指导后台在 ModelCatalog、Evaluation、Interpretation 重构后更新 API、页面状态和 TypeScript 模型。

---

## 1. 接入结论

后台前端需要把三个模块作为三个独立事实面接入：

| 模块 | 前端职责 | 推荐接口 |
| ---- | -------- | -------- |
| ModelCatalog | 模型元数据、DefinitionV2、问卷绑定、校验、预览、发布 | `/api/v1/assessment-models*` |
| Evaluation | Assessment、EvaluationOutcome、Run、分数与重试 | 新列表/详情优先 `/api/v2/evaluations/assessments*`；运行与运维动作仍用 `/api/v1/evaluations*` |
| Interpretation | 报告正文和报告生成生命周期 | 正文优先 `/api/v2/evaluations/*/report`；审计生命周期用 `/internal/v1/interpretation*` |

最重要的状态边界：

```text
Assessment evaluated
  ≠ 报告已生成

EvaluationRun succeeded
  ≠ Interpretation succeeded
```

前端不得再用一个 `interpreted` 状态覆盖整个后台领域模型：

- Evaluation 的成功终态是 `evaluated`。
- Interpretation 独立维护 generation/run/report。
- `/api/v1/evaluations/assessments` 为兼容旧页面，会把“evaluated 且已有报告”投影为 `interpreted`。
- `/api/v2/evaluations/assessments` 返回 Evaluation 原始事实，适合新页面。

## 2. 基础接入约定

### 2.1 地址与鉴权

| 环境 | 示例 |
| ---- | ---- |
| 本地 | `http://localhost:18082` |
| 生产 | `https://qs.fangcunmount.cn` |
| OpenAPI | `GET /api/rest/apiserver.yaml` |
| Swagger UI | `/swagger-ui/` |

所有本文接口都需要 IAM JWT：

```http
Authorization: Bearer <IAM JWT>
```

apiserver 会从当前 Operator 解析组织范围。前端不要自行提交 `org_id` 覆盖登录态组织。

### 2.2 响应与 ID

统一响应：

```ts
interface ApiResponse<T> {
  code: number;
  message: string;
  data?: T;
  reference?: string;
}

type Id = string;
```

后台 DTO 中仍有少量数字 ID，但页面状态、路由参数、缓存 key 和比较逻辑统一使用字符串，避免 JavaScript 大整数精度问题。

### 2.3 权限能力

模型后台按能力拆分按钮与路由：

| capability | IAM resource/action | 页面能力 |
| ---------- | ------------------- | -------- |
| `read_assessment_models` | `qs:assessment_models` 的 `read/list` | 列表、详情、已发布快照、问卷绑定读取 |
| `manage_assessment_models` | `qs:assessment_models` 的 `create/update/delete/archive` | 创建、基础信息、绑定、归档、删除 |
| `edit_assessment_model_definitions` | `qs:assessment_model_definitions` 的 `read/update/validate/preview/apply_codes` | Definition 编辑、校验、预览、编码申请 |
| `publish_assessment_models` | `qs:assessment_models` 的 `publish/unpublish` | 发布、下架 |
| `evaluate_assessments` | `qs:assessments` 的 `retry/batch_evaluate` | 重试、批量执行 |
| `audit_interpretation` | Interpretation 审计授权 | 查看 generation/run/report 生命周期 |

前端按钮隐藏只是体验优化；服务端仍会执行最终鉴权。

## 3. ModelCatalog 后台接入

### 3.1 页面/API 映射

| 页面动作 | API |
| -------- | --- |
| 筛选选项 | `GET /api/v1/assessment-models/options?kind=` |
| 草稿/生命周期列表 | `GET /api/v1/assessment-models` |
| 创建 | `POST /api/v1/assessment-models` |
| 详情 | `GET /api/v1/assessment-models/{code}` |
| 更新基础信息 | `PUT /api/v1/assessment-models/{code}/basic-info` |
| 获取/绑定问卷 | `GET/PUT /api/v1/assessment-models/{code}/questionnaire` |
| 获取/整体保存 DefinitionV2 | `GET/PUT /api/v1/assessment-models/{code}/definition` |
| 申请内部编码 | `POST /api/v1/assessment-models/{code}/codes/apply` |
| 校验 | `POST /api/v1/assessment-models/{code}/validate` |
| 报告预览 | `POST /api/v1/assessment-models/{code}/preview-report` |
| 发布/下架 | `POST /api/v1/assessment-models/{code}/publish`、`unpublish` |
| 已发布快照 | `GET /api/v1/assessment-models/published/{code}?version=` |
| 归档/删除 | `POST /api/v1/assessment-models/{code}/archive`、`DELETE /api/v1/assessment-models/{code}` |

不要再按医学量表、人格模型维护两套管理 API。统一模型族：

```ts
type ModelKind = 'scale' | 'typology' | 'behavioral_rating' | 'cognitive';
```

`personality` 只作为旧数据的读兼容值；新建人格模型提交 `kind: 'typology'`、`sub_kind: 'typology'`、`product_channel: 'typology'`。

### 3.2 推荐编辑流程

```text
GET options
  → POST 创建模型
  → PUT questionnaire 绑定已发布题版
  → GET definition
  → PUT definition 整体保存
  → POST validate
  → POST preview-report（当前主要用于 typology）
  → POST publish
  → GET published/{code} 验证发布快照
```

创建示例：

```json
{
  "code": "MBTI_OEJTS",
  "kind": "typology",
  "sub_kind": "typology",
  "algorithm": "mbti",
  "product_channel": "typology",
  "title": "MBTI 人格类型测评",
  "description": "...",
  "category": "personality",
  "tags": ["人格", "MBTI"],
  "questionnaire_code": "MBTI_OEJTS",
  "questionnaire_version": "1.0.0"
}
```

`code` 是后续 URL、发布快照和小程序 `model_code` 的稳定标识；创建后不要把标题或 algorithm 当主键。

### 3.3 DefinitionV2 保存约束

`PUT /definition` 是完整替换，不是 JSON Merge Patch。前端应先 GET，再编辑完整对象并整体 PUT。

当前 JSON 结构的顶层键为：

```json
{
  "Measure": {
    "Factors": [],
    "FactorGraph": {},
    "Scoring": []
  },
  "Calibration": {
    "NormRefs": []
  },
  "Conclusions": [],
  "Outcomes": [],
  "ReportMap": {
    "Sections": []
  }
}
```

注意大小写：当前 wire contract 使用 `Measure / Conclusions / ReportMap` 等大驼峰字段。前端不得自行改成 `measure`，应以 `GET /definition` 返回对象和 OpenAPI schema 为准。

`Conclusions[]` 是带 `Kind` 的多态结构，支持：

```text
risk | type | norm | ability
```

不要根据字段重叠猜测类型；保存时必须保留 `Kind`。

建议编辑器分两层：

- 表单模式维护常用结构。
- JSON 高级模式持有完整 DefinitionV2。
- 两种模式共享同一份对象；切换时不丢未知字段。

### 3.4 校验与错误定位

校验响应：

```json
{
  "passed": false,
  "valid": false,
  "issues": [
    {
      "field": "definition_v2.conclusions",
      "code": "definition_v2.decision.invalid",
      "message": "...",
      "level": "error"
    }
  ],
  "errors": ["..."]
}
```

前端新实现只依赖 `passed + issues[]`；`valid + errors[]` 是兼容字段。

推荐定位策略：

```ts
const tabByField = (field: string) => {
  if (field.startsWith('binding.')) return 'questionnaire';
  if (field.includes('measure') || field.includes('factor')) return 'measure';
  if (field.includes('conclusion') || field.includes('outcome')) return 'conclusion';
  if (field.includes('report')) return 'report';
  return 'json';
};
```

发布会再次执行绑定和 Definition 校验。不要只靠前端校验启用发布按钮；以服务端发布结果为准。

### 3.5 报告预览

当前 typology 预览请求：

> 当前 Swagger 生成器不能完整展开 `json.RawMessage` 类型的 `answers`，因此 OpenAPI 中该请求体 schema 可能不完整；联调以本节结构和 `request.PreviewAssessmentModelReportRequest`、`definition/typology_handler.go` 为准。

```json
{
  "answers": [
    { "question_code": "Q1", "value": "A" },
    { "question_code": "Q2", "score": 1 }
  ]
}
```

每个答案至少提供 `value` 或 `score`；题目必须存在于绑定的已发布问卷，不能重复。

预览响应统一读取：

```ts
interface PreviewReport {
  outcome: { code?: string; title?: string };
  score_detail?: Record<string, number>;
  report_sections: Array<{ title: string; content?: string; kind?: string }>;
  issues?: ValidationIssue[];
  raw_report?: unknown;
}
```

目前不要假设每个 ModelKind 都支持预览；UI 应根据接口结果显示“不支持预览”，而不是阻塞保存。

### 3.6 生命周期

```text
draft → published → draft（unpublish）
   └──────────────→ archived → deleted
```

约束：

- `archived` 不可再编辑、发布或下架。
- 只有已归档模型可以物理删除。
- 发布生成不可变运行时快照；运行时 Evaluation 不读取正在编辑的 draft。
- 下架会删除当前发布快照并回到 draft。
- 归档已发布模型时也会移除发布快照。
- 已发布模型仍可形成新的工作修订，但不要在页面上把草稿对象等同于已发布快照；预览/执行历史按快照版本追溯。

## 4. Evaluation 后台接入

### 4.1 新页面优先使用 v2 outcome 读面

| 用途 | API |
| ---- | --- |
| Assessment 列表 | `GET /api/v2/evaluations/assessments?page=&page_size=&status=&testee_id=` |
| Assessment 详情 | `GET /api/v2/evaluations/assessments/{id}` |
| 报告详情 | `GET /api/v2/evaluations/assessments/{id}/report` |
| 报告列表 | `GET /api/v2/evaluations/reports?testee_id=&page=&page_size=` |

v2 统一使用：

```ts
interface ModelIdentity {
  kind: string;
  sub_kind?: string;
  algorithm?: string;
  code: string;
  version?: string;
  title?: string;
  product_channel?: string;
  algorithm_family?: string;
}

interface ScoreValue {
  kind: string;
  value: number;
  label?: string;
  max?: number;
}

interface ResultLevel {
  code: string;
  label: string;
  severity?: string;
}
```

旧字段迁移：

| 旧 v1 字段 | v2 字段 |
| ---------- | ------- |
| `model_kind/model_code/model_version/model_title` | `model.kind/code/version/title` |
| `total_score` | `primary_score.value` |
| `risk_level` | `level.code`；展示使用 `level.label` |
| `scale_code/scale_name` | `model.code/model.title` |

`primary_score` 和 `level` 都可为空：类型学模型可能只有分类 level，某些任务型模型可能没有传统总分。

### 4.2 Assessment 状态

Evaluation 当前状态语义：

| 状态 | 含义 |
| ---- | ---- |
| `pending` | 已创建，尚未提交执行 |
| `submitted` | 已提交 Evaluation |
| `evaluated` | EvaluationOutcome 已可靠提交，Evaluation 成功终态 |
| `failed` | Evaluation 失败 |

新页面不要等待 `interpreted`。报告是否存在通过报告接口或 Interpretation 生命周期判断。

若旧页面暂时继续使用 `/api/v1/evaluations/assessments`，需明确它是 Journey 兼容投影：`evaluated + report generated` 会显示为 `interpreted`。

### 4.3 Run、分数与运维动作继续使用 v1

| 用途 | API |
| ---- | --- |
| Run 列表 | `GET /api/v1/evaluations/assessments/{id}/runs?limit=20` |
| 最新 Run | `GET /api/v1/evaluations/assessments/{id}/runs/latest` |
| 分数事实 | `GET /api/v1/evaluations/assessments/{id}/scores` |
| 高风险因子 | `GET /api/v1/evaluations/assessments/{id}/high-risk-factors` |
| 失败重试 | `POST /api/v1/evaluations/assessments/{id}/retry` |
| 批量执行 | `POST /api/v1/evaluations/batch-evaluate` |

Run 展示字段除原有字段外，还包括 `attempt_origin`、`retry_disposition`、`max_automatic_attempts`、`remaining_automatic_attempts`、`next_attempt_at`、`retry_event_id` 和 `action_request_id`。

重试按钮应满足：

- Assessment 与最新 Run 均为 `failed`。
- 普通 Retry 仅在最新 Run 为 `manual_required` 时可用；`terminal` 只能走 System Governance Force Retry。
- 当前用户具有 `evaluate_assessments`。
- 点击后重新拉取 Assessment、Run 和报告状态，不在前端乐观改为完成。

## 5. Interpretation 后台接入

### 5.1 普通报告查看

新页面使用：

```text
GET /api/v2/evaluations/assessments/{assessment_id}/report
GET /api/v2/evaluations/reports?testee_id=...
```

报告 DTO 与小程序通用结构一致：`model`、`primary_score`、`level`、`conclusion`、`dimensions`、`suggestions`、`model_extra`、`created_at`。

报告 404 不代表 Evaluation 失败，可能只是 Interpretation 尚未生成。页面应回到 Assessment/Interpretation 状态判断，不要自动重跑 Evaluation。

### 5.2 运维与审计生命周期

以下接口是内部审计面，不用于普通报告详情页：

| 用途 | API |
| ---- | --- |
| Assessment 的 Interpretation 生命周期 | `GET /internal/v1/interpretation/assessments/{assessment_id}/lifecycle` |
| Assessment 历史报告 | `GET /internal/v1/interpretation/assessments/{assessment_id}/reports` |
| Outcome 的 generation 历史 | `GET /internal/v1/interpretation/outcomes/{outcome_id}/generations` |
| Report 元数据 | `GET /internal/v1/interpretation/reports/{report_id}` |

需要 `audit_interpretation`。生命周期响应应按三层展示：

```text
Generation
  ├── latest_run
  │     └── failure(kind/code/safe_message/retryable)
  └── report
```

Generation 典型状态：`pending / generating / generated / failed`。报告失败只影响 Interpretation，不应把 Assessment 从 `evaluated` 改为失败。

## 6. 前端迁移建议

建议分四步上线：

1. 新增统一 `ModelIdentity / ScoreValue / ResultLevel / Report` 类型，兼容读取旧 v1 DTO。
2. 模型后台 API 全部切到 `/assessment-models`，移除按模型族分裂的 service。
3. 测评列表和详情切到 v2 outcome 读面；Run、score、retry 暂留 v1。
4. 报告详情切 v2；另建仅授权用户可见的 Interpretation 生命周期抽屉。

兼容适配器示例：

```ts
function normalizeLegacyAssessment(x: any) {
  return {
    ...x,
    model: x.model ?? {
      kind: x.model_kind,
      sub_kind: x.model_sub_kind,
      algorithm: x.model_algorithm,
      code: x.model_code,
      version: x.model_version,
      title: x.model_title,
    },
    primary_score: x.primary_score ?? (
      x.total_score == null ? undefined : { kind: 'raw_total', value: x.total_score }
    ),
    level: x.level ?? (
      x.risk_level == null ? undefined : { code: x.risk_level, label: x.risk_level_label ?? x.risk_level }
    ),
  };
}
```

适配器只用于迁移期；新组件内部只使用新结构。
