# 事故复盘：2026-04-16 `qs.evaluation.lifecycle` 积压与 `CreateAssessmentFromAnswerSheet` 30 秒尾延迟

**本文回答**：这篇复盘记录 `2026-04-16` 生产环境 `qs.evaluation.lifecycle` 主题持续积压、`qs-worker` 大量 `DeadlineExceeded`、`answersheet.submitted -> CreateAssessmentFromAnswerSheet` 链路吞吐骤降的排障过程、根因判断、修复动作与后续治理项；本文先给结论和速查，再展开现象、证据、改动与恢复结果。

本文对应一次真实生产排障，不是设计预案。文中结论以当时的日志、监控、配置与代码为依据。

---

## 30 秒结论

如果只看一屏，先看下面这张表：

| 维度 | 结论 |
| ---- | ---- |
| 直接现象 | `qs-worker` 大量报 `rpc error: code = DeadlineExceeded desc = context deadline exceeded`，`qs.evaluation.lifecycle` 持续积压，`Requeued` 快速增长 |
| 关键误导项 | 云 MySQL CPU / IOPS / 连接数都很低，Mongo 查询 explain 也只有 `1ms`；因此“数据库资源吃满”不是这次事故的直接解释 |
| 主根因 | `CreateAssessmentFromAnswerSheet` 成功路径尾部同步执行了“我的测评列表缓存失效”；该失效通过 Redis `SCAN + DEL` 按前缀删键，拖长请求尾延迟，使 gRPC 在接近 `30s` deadline 时才返回 |
| 放大因素 | worker 原生产配置每实例 `concurrency=256`；3 副本理论入口约 `768`，明显高于 apiserver 下游保护预算，导致重试、backoff 和积压被放大 |
| 修复动作 | 先把 worker 并发降到 `48` 止血；随后在装配层直接禁用 `my_assessment_list_cache`，使创建/提交流程不再触发这层同步缓存失效 |
| 修复结果 | 3 个 worker 稳定跑满 `48`，总 `In-Flight=144`，单连接 `Requeued=0`，`Depth` 持续下降，链路恢复到稳定消峰状态 |

## 重点速查

1. **这次不是“数据库打满”事故**：MySQL 低 CPU / 低 IOPS、Mongo explain `1ms`，都和 `30s` 尾延迟不匹配。  
2. **最关键证据不是失败日志，而是成功日志**：`CreateAssessmentFromAnswerSheet` 多次 `status_code=OK`，但 `duration_ms=29995~30001`，说明请求不是失败在主逻辑，而是成功前被同步尾步骤拖住。  
3. **worker 并发过大是放大器，不是唯一根因**：`256 x 3` 的入口预算会把任何尾延迟迅速放大成 MQ 积压。  
4. **根因在应用层同步缓存失效**：`Submit()` 打完“提交测评成功”日志后，仍会同步调用 `invalidateMyListCache(...)`；缓存失效内部使用 Redis `DeletePattern -> SCAN + DEL`。  
5. **最终治理原则**：对“只在用户访问时按需构建、读路径可回退数据库”的缓存，不允许把同步失效放在主写链路上。  

## 影响范围

本次事故主要影响 **答卷提交后的异步测评创建链路**：

- Topic：`qs.evaluation.lifecycle`
- 事件：`answersheet.submitted`
- worker handler：`qs-worker` 对应 evaluation lifecycle handler
- gRPC：`/internalapi.InternalService/CreateAssessmentFromAnswerSheet`

从业务效果上看，表现为：

- 答卷已提交，但测评创建与自动提交明显滞后；
- `qs.evaluation.lifecycle` 主题积压持续增长；
- worker 连续出现 `DeadlineExceeded`、`RST_STREAM`、`backing off`；
- 消费吞吐下降，重试与积压相互放大。

## 现象

### 1. worker 错误与 backoff 显著放大

生产环境中，`qs-worker` 持续出现如下错误：

```text
failed to create assessment from answersheet:
rpc error: code = DeadlineExceeded desc = context deadline exceeded
```

在排障中采样最近 `5m` 日志，三台 worker 的错误计数分别为：

- `qs-worker-qs-worker-1`: `776`
- `qs-worker-qs-worker-2`: `120`
- `qs-worker-qs-worker-3`: `115`

同一阶段 NSQ 面板表现为：

- `Depth` 高位积压；
- `Requeued` 快速增长；
- 某些 worker 反复退避，`In-Flight` 无法稳定跑满。

### 2. MySQL 与 Mongo 资源图不支持“实例吃满”结论

用户当时提供的云 MySQL 报表显示：

- CPU 使用率约 `2.16%`
- IOPS 使用率约 `3.65%`
- 连接使用率约 `3.08%`

Mongo 排查结果显示 `scales` 集合查询：

- 命中索引 `idx_scales_questionnaire_deleted_created`
- `executionTimeMillis = 1`
- `totalKeysExamined = 2`
- `totalDocsExamined = 1`

这两组事实共同说明：

- 不能把这次事故直接归因为“RDS 规格不够”
- 也不能把 Mongo 的量表查询当作主瓶颈

### 3. 真正反常的是“成功请求也耗满 30 秒”

apiserver 日志里最关键的一组事实不是失败，而是成功：

- `创建测评成功`
- `提交测评成功 duration_ms=20~28`
- 但同一个 `/internalapi.InternalService/CreateAssessmentFromAnswerSheet` 最终 `status_code=OK`
- 同时 `duration_ms` 却稳定落在 `29995~30001`

这说明：

- 主逻辑里的 MySQL / Mongo 查询并不慢
- gRPC 不是“做不完业务”
- 而是在**成功返回前的尾部同步步骤**里被拖住

### 4. 噪音日志一度误导排障

同一时期日志里还有大量类似：

```text
SELECT * FROM assessment_episode ... record not found
```

这类日志在当次事故里是噪音：

- SQL 耗时只有 `2ms~6ms`
- `record not found` 是业务上允许出现的查询结果
- 它们并不是导致 30 秒尾延迟的主因

## 时间线（摘要）

> 以下时间为 `2026-04-16` 亚洲上海时区。

1. **19:58 左右**：`qs-worker` 开始大量报 `CreateAssessmentFromAnswerSheet` 超时，`qs.evaluation.lifecycle` 积压持续增长。  
2. **初步止血阶段**：确认 worker 扩副本真正生效，并统一 compose 项目名与清理逻辑。  
3. **并发预算修正阶段**：发现 worker 生产并发 `256` 与 apiserver 下游预算严重失配，将 [configs/worker.prod.yaml](../../configs/worker.prod.yaml) 中 `concurrency` 降为 `48`。  
4. **证据收敛阶段**：通过 apiserver 成功日志发现大量 `CreateAssessmentFromAnswerSheet` 请求在 `30s` 边缘才成功返回，排障方向从“数据库慢”转向“应用层尾延迟”。  
5. **根因定位阶段**：沿 `CreateAssessmentFromAnswerSheet -> Submit -> invalidateMyListCache` 代码链追踪，确认同步缓存失效是最可疑的尾步骤。  
6. **根因修复阶段**：先增加缓存失效短超时与可观测性，再根据业务判断在装配层直接禁用 `my_assessment_list_cache`。  
7. **恢复确认阶段**：3 个 worker 都稳定跑满 `48`，总 `In-Flight=144`，单连接 `Requeued=0`，`Depth` 开始稳定下降。  

## 根因

## 主根因：同步前缀删缓存把主写链路拖到 30 秒边缘

### 代码路径

`answersheet.submitted` 进入 `qs-worker` 后，会调用内部 gRPC：

- worker 超时设置位于 [internal/worker/process/runtime_bootstrap.go](../../internal/worker/process/runtime_bootstrap.go) 与 [internal/worker/infra/grpcclient/internal_client.go](../../internal/worker/infra/grpcclient/internal_client.go)
- gRPC 方法实现位于 [internal/apiserver/interface/grpc/service/internal.go](../../internal/apiserver/interface/grpc/service/internal.go)

`CreateAssessmentFromAnswerSheet` 的主路径是：

1. 根据 `questionnaire_code` 查量表  
2. 按 `answersheet_id` 查是否已有测评  
3. 创建测评  
4. 若有关联量表，则自动提交测评  
5. 如命中计划任务，则回写计划任务完成状态  
6. 返回 gRPC 响应

这条链会涉及：

- Mongo 查量表：`scaleRepo.FindByQuestionnaireCode(...)`
- MySQL 查测评 / 保存测评 / 提交测评
- 读侧缓存失效：`invalidateMyListCache(...)`

### 为什么成功日志比失败日志更关键

在 [internal/apiserver/interface/grpc/service/internal.go](../../internal/apiserver/interface/grpc/service/internal.go) 中：

- `CreateAssessmentFromAnswerSheet` 会调用 `submissionService.Submit(...)`
- `Submit(...)` 返回后，handler 才会打印“自动提交测评成功”
- 然后才会继续执行 `completeMatchedTask(...)`
- 最后返回 gRPC 响应

而在 [internal/apiserver/application/evaluation/assessment/submission_service.go](../../internal/apiserver/application/evaluation/assessment/submission_service.go) 中：

- `Submit(...)` 自己会先打印“提交测评成功”
- 然后调用 `invalidateMyListCache(...)`
- 最后才真正返回到 handler

这就形成了一个非常关键的判断窗口：

- 如果“提交测评成功”很快，但 handler 很久之后才打印“自动提交测评成功”
- 那么卡点极可能在 `Submit(...)` 的收尾阶段，而不是数据库提交本身

日志正好符合这个模式。

### `my_assessment_list_cache` 为什么会拖长尾延迟

“我的测评列表缓存”位于：

- [internal/apiserver/infra/cachequery/my_assessment_list_cache.go](../../internal/apiserver/infra/cachequery/my_assessment_list_cache.go)

其失效逻辑原本是：

1. 根据 `userID` 组装缓存 key 前缀
2. 调用缓存接口 `DeletePattern(ctx, pattern)`
3. Redis 层在 [internal/apiserver/infra/cacheentry/redis_cache.go](../../internal/apiserver/infra/cacheentry/redis_cache.go) 中通过 `SCAN + DEL` 按模式删除

这意味着缓存失效不是 O(1) 的单 key 删除，而是：

- 依赖 Redis 遍历键空间
- 同步阻塞在业务请求里
- 删除成本与前缀下键数量、Redis 时延和实例状态相关

从业务上看，这层缓存又不具备必须侵入主写链路的价值：

- 只有用户访问“我的测评列表”时才会按需构建
- 用户如果没访问过该列表，本来就不需要维护这层缓存
- 即使完全禁用，读路径也能直接回退数据库

因此，这次事故的主根因可以归纳为：

> 一个低收益、可选的读侧缓存，被以同步模式失效的方式放进了核心写链路；在 Redis 模式删键成本不可预测的情况下，它把 `CreateAssessmentFromAnswerSheet` 拖到了 30 秒 deadline 边缘。

## 放大因素：worker 并发预算显著高于下游保护预算

虽然根因在缓存，但 worker 并发配置明显放大了事故：

- 原配置每实例 `concurrency=256`，见 [configs/worker.prod.yaml](../../configs/worker.prod.yaml)
- 3 副本理论总入口约 `768`
- apiserver MySQL 连接池 `max-open-connections=180`，见 [configs/apiserver.prod.yaml](../../configs/apiserver.prod.yaml)
- apiserver MySQL 背压 `max_inflight=160`，同样位于 [configs/apiserver.prod.yaml](../../configs/apiserver.prod.yaml)

结果是：

- 一旦单请求尾延迟升高，worker 很快堆积大量 in-flight
- 超时触发 requeue
- requeue 再放大队列深度与 backoff
- 最终形成“成功率下降、吞吐下降、积压上升”的正反馈

因此本次事故应当拆成两个层次：

- **主根因**：同步缓存失效拖长 `CreateAssessmentFromAnswerSheet` 尾延迟
- **放大因素**：worker 并发配置超过下游保护预算，导致超时放大成 MQ 积压

## 修复

## 止血动作：先把 worker 总入口并发降到安全区间

直接修改：

- [configs/worker.prod.yaml](../../configs/worker.prod.yaml)

将每实例 `concurrency` 从 `256` 下调到 `48`，让 3 副本总 `In-Flight` 预算约为 `144`。这一步的目标不是“彻底修复”，而是：

- 让 worker 停止无意义地超量施压
- 减少 `DeadlineExceeded -> requeue -> backoff` 的正反馈
- 给 apiserver 争取稳定排障窗口

## 过渡修复：给缓存失效增加短超时与日志

在根因完全确认前，先对以下文件增加了防守性修改：

- [internal/apiserver/application/evaluation/assessment/submission_service.go](../../internal/apiserver/application/evaluation/assessment/submission_service.go)
- [internal/apiserver/infra/cachequery/my_assessment_list_cache.go](../../internal/apiserver/infra/cachequery/my_assessment_list_cache.go)

这一步做了两件事：

- 让 `invalidateMyListCache(...)` 变成 best-effort，并加短超时
- 增加 `invalidate_my_assessment_list_cache` 的慢日志与失败日志

目的不是长期保留，而是：

- 即使缓存继续存在，也不允许它无限制占满 gRPC deadline
- 为根因验证补足观测证据

## 最终修复：在装配层直接禁用 `my_assessment_list_cache`

根据业务判断，最终改动落在：

- [internal/apiserver/container/assembler/evaluation.go](../../internal/apiserver/container/assembler/evaluation.go)

修改内容是：

- `SubmissionService` 不再走 `NewSubmissionServiceWithListCache(...)`
- 直接退回 `NewSubmissionService(...)`
- 即 `my_assessment_list_cache` 不再注入到测评提交服务里

这样做的结果是：

- 创建测评 / 提交测评主链路不再触发这层缓存失效
- 读路径回退数据库
- 缓存实现虽然仍保留在代码里，但当前运行时已不再装配到这条主链路中

## 恢复验证

修复后的 NSQ 面板显示：

- 3 个 worker 都稳定 `In-Flight=48`
- 总 `In-Flight=144`
- 每个连接的 `Requeued=0`
- `Finished` 持续增长
- `Depth` 从 `233,012` 降到 `213,535`

这组信号说明：

- worker 不再反复进入 backoff
- `CreateAssessmentFromAnswerSheet` 已不再普遍被拖到 30 秒边缘
- 消费系统重新进入“稳定消峰”状态
- 修复动作与恢复结果相互印证

本次相关改动的最小验证命令包括：

```bash
go test ./internal/apiserver/container/assembler \
  ./internal/apiserver/application/evaluation/assessment \
  ./internal/apiserver/infra/cache
```

## 后续动作

## 已落地

- 将生产 worker 并发下调为 `48`
- 禁用 `my_assessment_list_cache` 注入
- 为缓存失效路径补充短超时与结构化日志
- 通过最小测试验证评估模块与缓存相关包编译 / 单测正常

## 建议继续推进

### 1. 明确把“可选缓存”从主写链路设计准入里排除

后续设计上建议明确一条规则：

- 对于“只在读路径按需构建、关闭后可回退数据库”的缓存
- 不允许把其同步失效放到主写链路里

如果未来必须恢复类似缓存，优先方案应是：

- 版本号或命名空间 bump
- 单 key 失效
- 异步重建 / 延迟淘汰

而不是 `SCAN + DEL` 按前缀删。

### 2. 继续保守维持 worker 并发预算

当前 `48 x 3 = 144` 与下游预算对齐较好。后续如果要提升吞吐，应满足两个前提：

- 有压测数据证明 `CreateAssessmentFromAnswerSheet` p95 / p99 可接受
- 下游预算（MySQL 背压、连接池、Redis、gRPC deadline）同步调整并验证

没有证据前，不应再把 worker 并发直接抬回 `256`。

### 3. 增加分步骤耗时观测

本次事故里，真正有价值的是“成功日志里的 30 秒”。后续建议在以下步骤增加清晰耗时：

- 查量表
- 查已有测评
- 创建测评
- 提交测评
- 计划任务回写
- 缓存失效 / 其他尾步骤

这样以后再出现尾延迟时，不需要靠人工推断日志间隙来定位。

### 4. 降低误导性噪音日志

本次排障中，`assessment_episode record not found` 的 GORM trace 大量刷屏，但它并不是事故根因。后续可考虑：

- 降低这类“预期未找到”日志级别
- 或在日志中区分“真正 SQL 失败”和“业务上允许的 not found”

这能显著降低排障时的误导成本。

## 经验教训

### 1. 低数据库利用率不等于链路没有数据库相关等待

这次事故里，MySQL 指标很低，但请求仍然在 30 秒边缘才返回。原因不是数据库不相关，而是：

- 请求主要耗在应用层同步等待
- 这种等待未必转化为数据库 CPU / IOPS
- 因此“资源图很低”只能说明“不是粗粒度资源吃满”，不能说明“链路没有存储等待”

### 2. 成功路径的长尾比失败更值得优先看

大量失败日志只会告诉你“超时了”；而“为什么成功也要 30 秒”才真正暴露了主矛盾。以后排查类似问题时，应优先找：

- 成功请求的总耗时
- 关键步骤之间的日志间隙
- 哪一步在成功前最后执行

### 3. 缓存的设计目标必须服从主链路预算

缓存是读优化手段，不是可以无条件侵入写路径的默认部件。特别是：

- 收益不明确的缓存
- 需要模式删 / 扫描删的缓存
- 与业务正确性无关的缓存

都不应被允许同步阻塞核心写链路。
