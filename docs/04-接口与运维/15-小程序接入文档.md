# 小程序接入文档

**适用对象**：小程序、H5、Taro 等受试者端前端  
**接入服务**：collection-server  
**契约基线**：`befd73c53`（2026-07-12）

本文是小程序接入的现行入口。报告等待机制的容量、WebSocket 和降级细节见 [12-小程序报告等待接入指南.md](./12-小程序报告等待接入指南.md)。

---

## 1. 接入结论

小程序主流程固定为：

```text
开始会话
  → 提交答卷（HTTP 202）
  → 查询提交状态
  → status=done 时同时取得 answersheet_id 和 assessment_id
  → 查询报告状态
  → status=interpreted 后读取报告
```

前端必须按以下契约更新：

| 项目 | 当前契约 |
| ---- | -------- |
| HTTP 服务 | collection-server，业务前缀 `/api/v1` |
| 人格产品 URL | 使用 `/typology-*`，不再使用 `/personality-*` |
| 人格目录身份 | `kind=typology`、`product_channel=typology` |
| 测评/报告兼容身份 | `model.kind=personality`；WebSocket `subscribe.kind=personality` |
| 提交结果 | `POST /answersheets` 只表示已排队；业务 ID 从 `submit-status` 获取 |
| 提交完成不变量 | `status=done` 时同时存在 `answersheet_id`、`assessment_id` |
| 报告完成判断 | 以 `report-status.status=interpreted` 为准，不以 Assessment 的内部状态为准 |
| ID 类型 | 所有 64 位 ID 在 TypeScript 中按 `string` 保存和比较 |

> `typology-assessment-sessions.endpoints.assessment_by_answer_sheet` 是已下线端点的遗留字段，前端必须忽略，不得调用。

## 2. 服务、鉴权与响应包络

### 2.1 地址

| 环境 | 示例 |
| ---- | ---- |
| 本地 | `http://localhost:18083/api/v1` |
| 生产 | `https://collect.fangcunmount.cn/api/v1` |
| OpenAPI | `GET /api/rest/collection.yaml` |
| Swagger UI | `/swagger-ui/` |

除下列目录接口外，业务接口都需要：

```http
Authorization: Bearer <IAM JWT>
```

可匿名访问的只读接口：

```text
GET /api/v1/assessment-models
GET /api/v1/assessment-models/hot
GET /api/v1/assessment-models/options
GET /api/v1/typology-models
GET /api/v1/typology-models/categories
```

`GET /assessment-models/{code}`、`GET /typology-models/{code}`、会话、答卷、测评和报告接口仍需登录。

### 2.2 通用响应

成功：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

失败：

```json
{
  "code": 112001,
  "message": "Assessment not found",
  "reference": ""
}
```

前端应同时判断 HTTP 状态码与 `code`；只有 `2xx && code === 0` 才进入成功分支。

## 3. 人格测评完整流程

### 3.1 开始会话

```http
POST /api/v1/typology-assessment-sessions
Content-Type: application/json
```

```json
{
  "model_code": "MBTI_OEJTS",
  "testee_id": "618855887087350318"
}
```

响应 `data`：

| 字段 | 用法 |
| ---- | ---- |
| `model` | 当前已发布模型摘要 |
| `questionnaire` | 与模型绑定的精确题版和题目 |
| `submit_contract` | 提交答卷时必须原样带回的问卷编码、版本、受试者 ID |
| `endpoints.submit_answer_sheet` | 提交地址 |
| `endpoints.wait_report` | legacy 长轮询模板 |
| `endpoints.report` | 报告地址模板 |
| `endpoints.assessment_by_answer_sheet` | 遗留字段，忽略 |

会话只聚合读取已发布模型与问卷，不创建 Assessment。不要缓存“最新问卷版本”替换 `submit_contract.questionnaire_version`。

### 3.2 提交答卷

```http
POST /api/v1/answersheets
Authorization: Bearer <token>
X-Request-Id: <uuid>
Content-Type: application/json
```

```json
{
  "questionnaire_code": "MBTI_OEJTS",
  "questionnaire_version": "1.0.0",
  "testee_id": "618855887087350318",
  "idempotency_key": "mp-MBTI_OEJTS-618855887087350318-20260712-001",
  "answers": [
    {
      "question_code": "Q1",
      "question_type": "Radio",
      "score": 1,
      "value": "{\"option\":\"A\"}"
    }
  ]
}
```

注意：

- `testee_id` 请求端可传字符串或数字，建议始终传字符串。
- `value` 是 JSON 字符串，不是任意嵌套对象。
- `questionnaire_code`、`questionnaire_version`、`testee_id` 取自 session 的 `submit_contract`。
- 重试同一次业务提交时复用同一个 `idempotency_key`；一次新的作答使用新 key。
- 建议显式传 `X-Request-Id`，否则服务端会生成。

成功受理返回 HTTP 202：

```json
{
  "code": 0,
  "message": "accepted",
  "data": {
    "status": "queued",
    "request_id": "8bde6a9f-2c3e-4ee8-bb8d-64d48e5b6762"
  }
}
```

HTTP 202 不是业务完成，不能在这里跳转报告页。

### 3.3 查询提交状态并取得两个 ID

```http
GET /api/v1/answersheets/submit-status?request_id=<request_id>
```

状态：

| `status` | 前端动作 |
| -------- | -------- |
| `queued` | 等待后重试 |
| `processing` | 等待后重试 |
| `done` | 保存 `answersheet_id` 和 `assessment_id`，进入报告等待 |
| `failed` | 停止轮询，展示失败并允许用户发起一次新的提交 |

完成响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "status": "done",
    "answersheet_id": "71001",
    "assessment_id": "80001",
    "updated_at": 1783785600
  }
}
```

当前实现先持久化 AnswerSheet，再同步确保 Assessment 已持久化，最后把队列状态推进为 `done`。因此前端若收到缺少任一 ID 的 `done`，应视为契约异常并上报，而不是自行拼接或猜测 ID。

提交状态保存在 collection-server 进程内且有 TTL；小程序应在拿到 202 后立即轮询并本地保存成功取得的两个 ID。

### 3.4 查询报告状态

推荐短轮询：

```http
GET /api/v1/typology-assessments/{assessment_id}/report-status?testee_id={testee_id}
```

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "status": "processing",
    "stage": "interpreting",
    "message": "报告生成中",
    "next_poll_after_ms": 3000,
    "updated_at": 1783785600
  }
}
```

前端状态机：

| `status` | 行为 |
| -------- | ---- |
| `processing` | 按 `next_poll_after_ms` 继续轮询；具体阶段看 `stage` |
| `interpreted` | 停止轮询，读取报告 |
| `failed` | 停止轮询，展示 `reason` |

`stage` 用于提示 `queued / processing / scoring / interpreting` 等内部阶段，不作为终态判断。服务端会把内部兼容值 `completed` 对外归一化为 `interpreted`。

可选增强：连接 `WSS /api/v1/report-events`，订阅帧仍使用 `kind=personality`：

```json
{
  "op": "subscribe",
  "assessment_id": "80001",
  "kind": "personality",
  "testee_id": "618855887087350318"
}
```

WebSocket 只有在服务端启用 `report_events` 时存在。断线、超时或未启用时必须降级到 `report-status`。

### 3.5 读取报告

仅在状态为 `interpreted` 后调用：

```http
GET /api/v1/typology-assessments/{assessment_id}/report?testee_id={testee_id}
```

核心响应：

```json
{
  "assessment_id": "80001",
  "model": {
    "kind": "personality",
    "sub_kind": "typology",
    "code": "MBTI_OEJTS",
    "version": "1.0.0",
    "title": "MBTI 人格类型测评",
    "product_channel": "typology",
    "algorithm_family": "factor_classification"
  },
  "primary_score": null,
  "level": { "code": "INTJ", "label": "建筑师" },
  "conclusion": "...",
  "dimensions": [],
  "suggestions": [],
  "model_extra": {
    "type_code": "INTJ",
    "type_name": "建筑师",
    "one_liner": "...",
    "image_url": "https://...",
    "rarity": { "percent": 2.1, "label": "较少见", "one_in_x": 48 }
  },
  "created_at": "2026-07-12 12:00:00"
}
```

展示建议：

- 人格主结果优先使用 `model_extra.type_code/type_name/one_liner`，再回退到 `level`。
- 通用结果使用 `primary_score`、`level`、`dimensions`、`suggestions`。
- 不要用 `algorithm` 判断页面类型；产品分流使用 URL 上下文或 `model.product_channel`，具体模型使用 `model.code`。
- `model_extra`、`primary_score`、`level` 都可能为空，类型定义必须可选。

## 4. TypeScript 最小类型与状态机

```ts
type Id = string;

interface SubmitStatus {
  status: 'queued' | 'processing' | 'done' | 'failed';
  answersheet_id?: Id;
  assessment_id?: Id;
  updated_at: number;
}

interface ReportStatus {
  status: 'processing' | 'interpreted' | 'failed';
  stage?: string;
  message?: string;
  reason?: string;
  next_poll_after_ms?: number;
  updated_at: number;
}

function assertCompletedSubmission(value: SubmitStatus) {
  if (value.status === 'done' && (!value.answersheet_id || !value.assessment_id)) {
    throw new Error('submit-status contract violation');
  }
}
```

建议页面状态：

```text
idle
  → loading_session
  → answering
  → submit_queued
  → submit_processing
  → assessment_ready
  → report_processing
  → report_ready | failed
```

本地恢复至少保存：`model_code`、`testee_id`、`request_id`、`answersheet_id`、`assessment_id`。

## 5. 禁止继续调用的旧契约

```text
/api/v1/personality-*
/api/v1/answersheets/{id}/assessment
/api/v2/assessments*
```

不要从旧的 `endpoints.assessment_by_answer_sheet` 调用已删除接口。正常路径直接使用 `submit-status.assessment_id`；`GET /typology-assessments?testee_id=...` 按 `answer_sheet_id` 匹配只作为历史客户端恢复兜底。

## 6. 联调验收清单

- [ ] 所有 64 位 ID 在 TypeScript 中使用 `string`
- [ ] session 返回的题版与 submit 使用的版本完全一致
- [ ] POST 答卷后按 HTTP 202 处理，不读取不存在的业务 ID
- [ ] `submit-status=done` 后同时保存两个 ID
- [ ] 不调用 `assessment_by_answer_sheet` 遗留 URL
- [ ] 报告轮询按 `next_poll_after_ms` 退避
- [ ] 仅在 `interpreted` 后读取报告
- [ ] 所有报告请求携带同一个 `testee_id`
- [ ] WebSocket 使用 `kind=personality`，且具备 HTTP 降级
- [ ] 404 不重试读取他人报告；429 采用指数退避
