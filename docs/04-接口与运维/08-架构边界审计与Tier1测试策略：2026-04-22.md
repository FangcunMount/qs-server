## 审计结论

### 1. `domain` 边界相对稳定

对 `internal/apiserver/domain` 的导入扫描没有发现直接依赖 `internal/apiserver/application`、`internal/apiserver/infra` 或 `internal/apiserver/interface` 的情况。现状符合“领域层只依赖领域对象和值对象/共享内核”的预期。

这意味着第一阶段不需要先做 `domain` 层的大拆分，重点应转向外围层的依赖收口。

### 2. `application -> infra` 仍有直接具体依赖，但主链已开始收口

本轮前后对比，评估主链里已经落下三类 package-local port：

- [`internal/apiserver/application/evaluation/assessment/submission_service.go`](../../internal/apiserver/application/evaluation/assessment/submission_service.go) 把“我的测评列表缓存”收成了 `assessmentListCache` 接口。
- [`internal/apiserver/application/evaluation/engine/service.go`](../../internal/apiserver/application/evaluation/engine/service.go) 把 waiter 注册表字段收成了 `waiterNotifier` 接口。
- [`internal/collection-server/application/answersheet/submission_service.go`](../../internal/collection-server/application/answersheet/submission_service.go) 把答卷 gRPC 网关和监护关系能力分别收成了 `answerSheetGateway`、`guardianshipChecker`。

但主链之外，`application -> infra` 仍不是全面隔离状态，后续还需要按热点包分批改成“应用依赖端口，装配层绑定实现”。

### 3. `infra -> application` 已从“明确存在”收口到“优先例外已清零”

第一波审计时，已登记的代表性穿透点主要集中在三类：

- IAM 快照同步：[`internal/apiserver/infra/iam/operator_roles_sync.go`](../../internal/apiserver/infra/iam/operator_roles_sync.go)

本轮继续推进后，这三类优先例外已经全部完成收口：

- 新增共享契约 [`internal/apiserver/port/planentry/generator.go`](../../internal/apiserver/port/planentry/generator.go) 和 [`internal/apiserver/port/outbox/outbox.go`](../../internal/apiserver/port/outbox/outbox.go)

`interface` 层本来就要做协议适配，因此依赖 `application` 是正常的；但当前还存在两类值得警惕的混用：

## 第一阶段规则清单

### Domain

- `domain` 只能依赖同层领域对象、值对象、共享内核与中性工具包。
- 禁止新增 `domain -> application / infra / interface` 导入。
- 领域事件允许使用共享事件抽象，但不得回调应用服务。

### Application

- `application` 优先依赖 `domain` 与 package-local port。
- 新增远程调用、缓存、waiter、IAM、消息发送等能力时，先在包内定义接口，再由装配层注入具体实现。
- 不再允许在评估主链新增新的具体 infra 字段类型。

### Infra

- `infra` 可以实现仓储、网关和出站端口，但不得新增对 `interface` 的依赖。
- 不得继续引入新的 `application DTO` 到基础设施实现里。
- 对已有 `infra -> application` 穿透点，短期保留例外，后续通过端口或共享契约下沉来消除。

### Interface

- `interface` 负责协议适配、参数校验、DTO 转换、错误码映射。
- 禁止新增直接业务持久化逻辑。
- 若一个 handler / service 需要直接使用 infra 能力，优先评估是否应下沉到 `application` 的 use case 或治理端口。
