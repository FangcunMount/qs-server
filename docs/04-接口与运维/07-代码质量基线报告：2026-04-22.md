## 当前主要风险

### 1. Go toolchain 与依赖漏洞债务仍偏重

`govulncheck` 当前的 74 条 module 级漏洞里，69 条来自 Go 标准库 / toolchain 条目。这说明最优先的安全动作不是继续堆规则，而是升级 Go 版本与关键依赖。

目前最值得优先处理的第三方模块是：

1. `google.golang.org/grpc`
2. `github.com/quic-go/quic-go`
3. `github.com/go-viper/mapstructure/v2`
4. `filippo.io/edwards25519`

### 2. `gosec` 剩余噪音已经收敛，但业务层转换问题仍然集中

过滤 generated/docs/proto 后，`gosec` 结果没有再散成几十种规则，当前几乎全部集中在 `G115`。这不是“扫描器太吵”，而是说明 gRPC / DTO 边界里的整数缩窄转换确实需要一次系统化梳理。

### 3. 测试覆盖率仍不足以支撑强门禁

虽然 53 个 package 已有测试，但整体语句覆盖率只有 11.0%，说明测试主要集中在部分热点模块，很多应用层、infra 适配层和进程装配层仍然没有有效保护。当前不适合直接上全仓覆盖率硬阈值。

---

## 建议的整改顺序

建议按下面顺序推进：

1. 升级 Go toolchain 到受支持且带安全补丁的版本，并同步处理 `grpc` / `quic-go` / `mapstructure` / `edwards25519`
2. 集中清理 `G115`，优先处理 gRPC service / DTO 边界上的窄化转换
3. 针对 `G304` / `G404` 做逐点审查，区分真实风险与可接受场景
4. 把测试建设聚焦到核心链路，而不是先上全仓覆盖率阈值

如果要把第三阶段做成一轮有节奏的治理，建议先锁定三个热点区域：

1. `internal/apiserver/transport/grpc/service`
2. `internal/collection-server`
3. `internal/worker`
