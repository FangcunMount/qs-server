# 常见排障

## 1. 先判断失败层

```text
客户端渲染/请求
  -> collection 入口保护
  -> gRPC/认证/上下文
  -> apiserver 用例与持久化
  -> Outbox/MQ/worker
  -> Evaluation Outcome
  -> Interpretation Report
  -> 查询投影/缓存/信令
```

不要看到最终“报告未完成”就直接归因于 WebSocket、MQ 或缓存。

## 2. 常见症状

| 症状 | 首查 |
| --- | --- |
| 429/排队超时 | collection 限流、队列、背压状态和配置 |
| 重复提交 | coalescer owner/contender/timeout 指标、request ID、Mongo durable lookup/transaction 指标 |
| 答卷已收但无评估 | Outbox 记录、relay、MQ、worker handler |
| Outcome 已有但无报告 | `evaluation.outcome.committed`、Interpretation run/失败状态 |
| 等待不唤醒 | Redis signal 与订阅；随后回查报告事实 |
| 统计不一致 | projector/checkpoint/pending/reconcile，不直接改读表 |
| 治理操作无结果 | request ID、操作审计主/备用存储与恢复 runner |

## 3. 证据原则

从一次具体 request/event/assessment/report 标识沿链路追踪；把配置、运行状态、持久化事实和日志相互印证。
