## 主线之间的关系

| 问题 | 缓存模块 | 事件模块 | 高并发保护 |
| ---- | -------- | -------- | ---------- |
| 答卷提交后不阻塞 | 不参与主写入 | Outbox + MQ + worker 推动异步测评 | SubmitQueue 削峰，SubmitGuard 抑制重复提交 |
| 报告生成后通知前端 | report_status 可缓存当前状态 | report.generated 是可靠业务事实 | report-status / wait-report / WS 控制查询放大 |
| 模型或问卷发布后读侧刷新 | L1/L2 失效与预热 | 变更事件或信令触发刷新 | 防止发布后冷启动击穿 |
| 高并发查询目录 | L1 命中省 gRPC，L2 命中省 DB | 不承担目录读取 | rate limit、catalog 并发池保护入口 |
| worker 处理能力有限 | 不直接消费 | MQ 订阅、Ack/Nack、重试 | MaxInFlight、duplicate suppression、防重复副作用 |

## 关键边界速查

| 容易混淆的问题 | 正确边界 |
| -------------- | -------- |
| 既然有 MQ，为什么还要 Outbox？ | MQ 负责传输，Outbox 负责业务写入与事件发布的一致性。 |
| 既然有 Redis Pub/Sub，为什么还要 MQ？ | Redis signaling 是 best-effort 临时唤醒；MQ 承载跨进程异步消费。 |
| 既然报告完成会写 Redis，为什么还要事件？ | Redis 状态服务查询体验；业务事实仍以数据库状态、Outbox 事件和报告状态持久化为准。 |
| SubmitQueue 是不是 MQ？ | 不是。它是 collection-server 进程内有界队列，用于提交削峰，不提供 durable queue 语义。 |
| SubmitGuard 是不是限流？ | 不是。它解决同一业务动作的短期幂等和进行中抑制。 |
| Redis lock 能不能保证 exactly-once？ | 不能。它只是短期租约 primitive，业务正确性仍靠 DB 约束、状态机和幂等处理。 |
| 缓存 miss 能不能直接打 DB？ | 可以回源，但高并发下必须配合 singleflight、backpressure、限流或队列。 |
