# 基础设施设计原则

这些原则约束 qs-server 的基础设施取舍。它们优先服务高并发测评链路，而不是追求组件覆盖面。

## 1. 热路径优先保护

高频读、答卷提交、报告查询是优先保护对象。缓存、限流、队列、背压和报告状态治理都先覆盖这些路径。

## 2. 写入链路短事务

提交接口只做必要校验、AnswerSheet / Assessment 写入和事件出站准备，不同步执行测评计算和报告生成。耗时不可控的工作进入事件链路和 worker。

## 3. 业务事实可靠，临时通知可丢

Outbox 里的业务事件不能丢；一次性信令只负责唤醒正在等待的请求，可以丢。丢信令后客户端仍通过状态查询和 `next_poll_after_ms` 继续推进。

## 4. 缓存是读侧治理，不是事实源

MongoDB / MySQL 是业务事实源。L1、本地 TTL、L2 Redis、report status cache 都只能承接读侧压力，不能替代事实写入。

## 5. 队列必须有边界

SubmitQueue、worker pool、MQ 消费并发、下游 max inflight 都必须有上限。超过边界时优先快速失败或退避，而不是无限排队。

## 6. 查询必须告诉客户端下一次什么时候再来

报告生成是异步过程。未完成时接口必须返回 `next_poll_after_ms` 或等价退避语义，避免客户端固定频率无脑轮询。

## 7. 异步链路必须可观测、可重试、可补偿

Outbox 积压、MQ depth、worker 消费延迟、报告生成失败、缓存失效失败都必须有指标或排障入口。没有观测的异步链路不能作为可靠方案。
