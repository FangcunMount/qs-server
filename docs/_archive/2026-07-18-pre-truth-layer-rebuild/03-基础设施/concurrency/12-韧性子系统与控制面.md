# 韧性子系统与控制面

## 1. 结论

限流、背压、并发闸门和租约仍由各自的小型共享内核实现；apiserver、collection-server、worker 分别用一个进程级 `resilience/subsystem` 持有实例、生命周期和快照。业务 Repository、队列和 transport 只依赖窄接口，不依赖 Redis 控制实现。

## 2. 所有权

| 进程 | 子系统所有权 |
| --- | --- |
| apiserver | `query`、`submit`、`admin_submit`、`wait_report` 共享预算；MySQL/Mongo/IAM 背压；锁租约；治理执行入口 |
| collection-server | `query`、`submit`、`wait_report`、`report_events` 共享预算；query/catalog/submit/wait-report 闸门；答卷队列注册；锁与幂等快照 |
| worker | 锁和重复抑制快照；不创建不存在的入口限流或背压能力 |

## 3. 动态限流

路由拿到的是稳定的 budget 代理。策略切换时代理保持不变，底层 global/user limiter 成对替换；本地桶在短过渡窗口同时受旧、新策略约束，避免重建桶带来额外 burst。Redis 分布式 limiter 保留原 key，只更新策略。覆盖由 Redis 版本和 TTL 管理，过期、删除或 reset 后回到启动配置。

## 4. 队列状态机

```text
active -> draining -> paused -> active
```

进入 `draining` 后，新 request_id 返回 `503` 和 `Retry-After`，已有 request_id 仍复用原状态；普通队列满继续返回 `429`。只有 queued 与 processing 全部归零才进入 `paused`。超时保持 `draining`，任务不会被丢弃；只有完全排空的 `paused` 可以 resume。

治理发布使用 command-first/state-commit 握手：apiserver 先写 command，再 CAS desired state；command 携带 `request_id + state_version`。collection 领取命令后，只有观察到 Redis desired state 的外层版本、request_id 和 desired_state 全部匹配才执行 QueueController。CAS 失败留下的未提交 command 只等待到期，不会触发 drain/resume。历史 payload 没有握手字段时按已提交状态兼容读取。

## 5. Leader 安全让权

`resilience.release_lock` 不接收原始 Redis key，也不直接删锁。子系统只允许 leader workload：先设置本实例重获冷却期，再取消受保护 body，等待 body 退出，最后由原 `Run` defer 按原 token 释放。cooldown 发布与 acquire 后的 active run 准入使用同一把状态锁形成线性顺序：准入先发生时让权动作一定能看到并取消该 run，让权先发生时新获取的租约会立即安全释放且不启动 body。超时时不提前释放。task lock、幂等锁和重复抑制锁均拒绝该动作。

## 6. 控制与审计

控制状态使用 ops Redis family，采用版本化持久状态、信号唤醒入口和定时对账；实例以 `component + instance_id + generation` 心跳。控制面不可用时拒绝新治理动作，数据面继续使用当前有效策略。

动作要求 org admin、`confirm=true`，并支持 `request_id`。MySQL 表 `system_governance_action_runs` 先以 `(org_id, request_id)` 原子 claim，再记录脱敏输入和终态结果，防止安全重试重复执行。若 MySQL 终态回填在有界重试后仍失败，ops Redis 以 `governance:audit-replay:<org_id>:<request_id>` 无 TTL 保存不含输入和凭据的终态；重试优先从该记录精确重放，后台 runner 每 30 秒最多回填 100 条，成功后删除 Redis fallback。MySQL 与 ops Redis 同时无法保存时返回固定 500，保留原 claim，绝不再次执行动作。

collection 的 `resilience.control.enabled` 默认且在 dev/prod 样例中均为 `true`。首次控制状态读取并实际应用前 `/readyz` 返回 503，存活检查不受影响；首次成功后的短暂 Redis 故障不撤销 Ready。显式配置为 `false` 时不启动心跳、命令消费或动态状态对账，只允许用于所有韧性动作均关闭且不存在维护暂停状态的旧部署。

四个韧性动作均由 `system_governance.resilience` 的独立开关控制，默认关闭。
