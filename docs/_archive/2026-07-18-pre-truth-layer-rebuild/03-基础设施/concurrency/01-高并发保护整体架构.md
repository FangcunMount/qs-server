# 高并发保护整体架构

## 1. 解决什么问题

高并发保护解决突发流量、重复提交、下游处理慢、worker 积压和报告查询风暴的问题。

## 2. 所在位置

保护层横跨 Nginx / HTTP handler、collection-server submit service、apiserver、Outbox、MQ、worker、Redis 和 DB。

## 3. 设计目标

入口快速失败；提交有界排队；业务动作幂等；下游有背压；报告查询退避；压测可验证容量边界。

## 4. 整体流程

客户端请求先经过入口限流；提交请求进入 SubmitQueue；重复提交由 SubmitGuard 抑制；业务服务写入事实和 Outbox；worker 按受控并发消费；report 查询优先查状态缓存并返回退避建议。

## 5. 核心数据结构

rate limit key、queue item、request_id、guard key、lock lease、backpressure snapshot、report status、next_poll_after_ms。

## 6. 正常流程

流量低于容量时请求正常通过；提交请求被排队后异步处理；报告未完成时返回 pending 和下一次查询时间。

## 7. 异常流程

超过限流直接失败；队列满返回 queue full；重复提交复用或拒绝；下游慢时触发超时、退避或熔断；report 等待超时返回轮询建议。

## 8. 幂等 / 降级 / 背压

限流解决太多请求；SubmitQueue 解决瞬时洪峰；SubmitGuard 解决同一业务动作重复执行；Backpressure 解决下游处理能力不足。

## 9. 可选方案

单靠 Nginx 限流无法按业务语义保护；单靠 DB 唯一索引无法阻止重复计算；直接 WebSocket 需要连接治理。

## 10. 当前方案取舍

采用分层保护。每一层只解决自己的问题，并用观测指标证明是否有效。

## 11. 观测指标

QPS、P95/P99、错误率、reject count、queue length、worker lag、DB slow query、Redis latency、report timeout count。
