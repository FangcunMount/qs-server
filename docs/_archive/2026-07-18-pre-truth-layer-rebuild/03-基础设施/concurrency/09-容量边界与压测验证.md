# 容量边界与压测验证

## 1. 解决什么问题

容量验证解决“高并发保护是否真的有效”的问题。没有压测指标，限流、SubmitQueue、Outbox、MQ、worker 和 report 查询治理都只是设计描述。

## 2. 所在位置

压测覆盖 collection-server、qs-apiserver、worker、Redis、MongoDB、MySQL、MQ 和 IAM 依赖，重点看混合场景。

## 3. 设计目标

用 smoke 到目标压测逐步推进；记录 P95/P99、错误率、队列深度、Outbox 积压、MQ depth、worker 延迟、Redis 命中率、DB 慢查询；压测失败能定位是入口、队列、事件、worker 还是 DB。

## 4. 整体流程

先做 token / fixture / health preflight；跑 smoke；再跑 60 / 120 / target 阶段；每阶段收集服务日志、k6 / ghz 指标、Outbox / MQ / DB / Redis 状态；达到错误率或延迟阈值就停止扩大。

## 5. 核心数据结构

scenario、target QPS、duration、threshold、submit_queue snapshot、outbox pending、MQ depth、worker lag、report latency、error classification。

## 6. 正常流程

在目标 QPS 下，提交入口错误率可控，SubmitQueue 不持续增长，Outbox 可在窗口内 drain，MQ depth 不持续抬升，report 平均生成耗时和 P95 可解释。

## 7. 异常流程

如果所有场景同时出现 EOF / reset / timeout，先判断 upstream disruption 或压测环境问题；如果只有 submit 失败，检查 SubmitQueue、SubmitGuard、DB 写入和 Outbox；如果 report 查询失败，检查 status cache、wait timeout 和信令。

## 8. 幂等 / 降级 / 背压

压测要验证边界行为：队列满能快速失败，重复提交不生成多份报告，下游慢时上游降速，客户端按 `next_poll_after_ms` 退避。

## 9. 可选方案

只看平均 QPS 会掩盖尾延迟和积压；只跑单接口无法覆盖事件链路；只看 k6 不看后端队列和 Outbox 无法解释瓶颈。

## 10. 当前方案取舍

采用混合场景和分阶段压测。容量结论必须同时满足接口指标和后台积压指标，而不是只看压测工具成功率。

## 11. 观测指标

QPS、P95、P99、错误率、SubmitQueue length、SubmitQueue reject、Outbox pending / oldest age、MQ depth、worker consume latency、Redis hit rate、DB slow query、report generation duration。
