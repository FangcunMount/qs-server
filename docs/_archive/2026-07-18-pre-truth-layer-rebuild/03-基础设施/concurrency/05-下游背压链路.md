# 下游背压链路

## 1. 解决什么问题

下游背压解决 worker、MQ、DB、Redis、gRPC 或 report 查询处理能力不足时，上游继续无限施压的问题。

## 2. 所在位置

背压位于 apiserver、collection-server、worker、DB 访问、Redis 访问、MQ 消费和 IAM gRPC 调用周围。

## 3. 设计目标

下游慢时让上游降速；所有队列有边界；超时和重试有上限；失败可观测；避免局部故障扩散。

## 4. 整体流程

调用下游前检查 max inflight 或令牌；超限快速失败或排队；调用使用 timeout；失败使用 backoff；持续失败时熔断或降级。

## 5. 核心数据结构

resource、max_inflight、timeout、queue depth、retry policy、backoff、circuit state、snapshot。

## 6. 正常流程

低负载时请求通过；轻微抖动时短暂排队或重试；指标显示资源利用率。

## 7. 异常流程

DB 慢查询上升时降低回源；MQ 积压时降低消费并发或扩 worker；Redis 慢时限制缓存回源；IAM gRPC 慢时超时失败而非阻塞。

## 8. 幂等 / 降级 / 背压

重试只用于幂等或可补偿动作；非幂等写入先保证状态机和 Outbox；背压结果需要向上游暴露，让入口和队列停止加压。

## 9. 可选方案

无限重试会放大故障；无限排队会拖垮进程内存和连接；只加机器不解决业务热路径放大。

## 10. 当前方案取舍

采用 max inflight、bounded queue、timeout、retry with backoff、worker pool、circuit breaker 组合。背压优先保护事实源和 worker。

- apiserver resilience subsystem 根据启动配置唯一创建 MySQL、Mongo、IAM limiter；Repository、event outbox 和 IAM adapter 获取同一实例。
- collection resilience subsystem 唯一创建 `grpc_downstream` gate；gRPC manager 缺少该依赖时拒绝构造，不再静默生成第二份并发预算。
- `/governance/resilience` 中的 `grpc_downstream` 行直接读取共享 gate 的 capacity、in-flight 和等待超时。

## 11. 观测指标

max inflight usage、timeout count、retry count、circuit open、queue depth、DB slow query、Redis latency、MQ depth、worker lag。
