# 高并发保护方案取舍

## 1. 为什么需要 SubmitQueue，而不是只做限流

限流解决“太多请求”，SubmitQueue 解决“短时间涌入但仍可处理的提交洪峰”。队列把洪峰转换成有界处理能力，队列满时才快速失败。

## 2. 为什么需要 SubmitGuard，而不是只靠数据库唯一索引

数据库唯一索引只能挡住部分重复写入，不能阻止重复校验、重复 Outbox、重复计算和重复报告生成。SubmitGuard 在业务动作进入写入前抑制重复。

## 3. 为什么先做长轮询，而不是直接 WebSocket

长轮询兼容 HTTP，能减少无效短轮询，连接治理成本低于 WebSocket。WebSocket / SSE 适合后续对实时性和交互体验要求更高的场景。

## 4. 为什么查询接口要返回 next_poll_after_ms

异步报告生成耗时不可控。服务端返回下一次查询建议，才能让客户端退避，避免固定间隔轮询把在线用户数和生成耗时线性放大成请求风暴。

## 5. 为什么高并发保护不能只放在 Nginx

Nginx 不知道 testee、questionnaire、assessment task、request_id、report status 等业务语义。业务层仍然需要 SubmitQueue、SubmitGuard 和 report 查询治理。

## 6. 观测指标

limit reject、queue length、duplicate submit、worker lag、DB slow query、report poll QPS、wait timeout、WebSocket active connection。
