# 数据访问整体架构

## 1. 解决什么问题

数据访问整体架构解决业务模块如何依赖持久化而不被具体 Mongo / MySQL 实现绑死的问题。

## 2. 所在位置

它位于应用服务、领域模型和 infra/mongo、infra/mysql 之间。

## 3. 设计目标

端口先行；基础设施实现可替换；事务边界清晰；读模型和写模型分离；Outbox 与业务写入边界明确。

## 4. 整体流程

应用服务调用 repository / port；infra 实现数据库访问；写入成功后通过事件或读模型投影服务同步后续状态。

## 5. 核心数据结构

Repository interface、DAO / Store、Unit of Work、transaction boundary、read model、projection event。

## 6. 异常流程

跨库写入不能假设强事务；一边成功一边失败时通过 Outbox、状态机和补偿恢复。

## 7. 观测指标

DB latency、slow query、transaction duration、repository error、projection lag、read model freshness。
