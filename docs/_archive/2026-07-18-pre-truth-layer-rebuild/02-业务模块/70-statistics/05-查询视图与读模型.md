# 查询视图与读模型

## 保留的查询族

| 查询族 | 说明 |
| ------ | ---- |
| Organization Overview | 机构规模、内容、答卷、接入、测评、计划概览 |
| Clinician | 从业者快照、窗口与入口漏斗 |
| Entry | 入口累计、窗口与最后行为时间 |
| Periodic | 受试者周期统计 |
| Content Batch | questionnaire/scale 统一批量统计 |

统一内容接口是 `POST /api/v1/statistics/contents/batch`。输入按 `(type, code)` 去重并保留首次出现顺序；累计形成数和完成数始终从当前机构的 `assessment` 事实聚合，不与日投影合并。完成数只认 `status=evaluated`。

Overview 的机构规模等指标优先读取 snapshot，未就绪时查询主事实；累计内容数、累计答卷提交数和今日答卷提交数始终读取当前机构的 `assessment` 事实。SQL 错误必须向上返回，不能伪装成零统计。

内容接口按输入类型鉴权：`questionnaire` 需要 `manage_questionnaires`，`scale` 需要 `manage_assessment_models`；混合请求必须同时具备两项能力，缺失任一能力时整个请求返回 403。

Plan 响应只包含：

- `activity`：计划活动窗口与趋势。
- `fulfillment`：履约窗口与趋势。

不再提供顶层 `plan.window`、`plan.trend` 兼容字段。

## 应用层读端口

- `OverviewReader`
- `ClinicianStatisticsReader`
- `EntryStatisticsReader`
- `ContentStatisticsReader`

`ReadServiceDeps` 分别接收四个 Reader，应用服务不再依赖组合型总读模型。MySQL adapter 通过 Go 隐式满足这些窄接口。clinician 和 entry 列表按整页 ID 批量聚合，SQL 数量不随分页条数线性增长；单条详情复用相同批量接口并传入一个 ID。

## 缓存边界

现役查询缓存只保留 `query.stats_overview`。已删除 system、questionnaire、plan 的旧 warmup kind；旧 Redis key 不主动扫描，按 TTL 自然过期。
