# 领域事件总览

**本文回答**：当前业务模块通过哪些事件协作，哪些事件是可靠出站，哪些事件只做尽力投递或读侧投影。

事件契约以 [`configs/events.yaml`](../../configs/events.yaml) 为准。本文只解释业务含义和模块边界。

---

## 1. 事件分组

| Topic | 事件组 | 主要模块 | 语义 |
| ----- | ------ | -------- | ---- |
| `qs.survey.lifecycle` | `questionnaire-lifecycle` | `survey`、`model-catalog` | 问卷和量表生命周期变化 |
| `qs.evaluation.lifecycle` | `assessment-lifecycle` | `survey`、`evaluation`、`interpretation` | 答卷、测评和报告生命周期 |
| `qs.plan.task` | `task-lifecycle` | `plan` | 任务开放、完成、过期、取消 |

---

## 2. 主链路事件

| 事件 | 产生模块 | 消费方向 | Delivery | 说明 |
| ---- | -------- | -------- | -------- | ---- |
| `questionnaire.changed` | `survey` | 缓存、目录、读侧刷新 | `best_effort` | 问卷生命周期变化 |
| `assessment_model.changed` | `model-catalog` | 目录、缓存、模型资产刷新 | `best_effort` | 测评模型发布、下架或归档后的资产变化 |
| `answersheet.submitted` | `survey` | `evaluation` | `durable_outbox` | 答卷提交后的可靠执行入口 |
| `evaluation.requested` | `evaluation` | Worker / 执行处理 | `durable_outbox` | Assessment 已具备执行条件 |
| `evaluation.outcome.committed` | `evaluation` | Worker / Interpretation | `durable_outbox` | Outcome、Run 与评分事实已可靠提交 |
| `evaluation.failed` | `evaluation` | 补偿、告警、统计 | `durable_outbox` | Evaluation 执行失败事实 |
| `interpretation.report.generated` | `interpretation` | 查询、重点关注同步、统计 | `durable_outbox` | 最终报告成品已生成 |
| `interpretation.report.failed` | `interpretation` | 查询、告警、重试观测 | `durable_outbox` | 一次报告生成尝试失败 |

---

## 3. 行为旅程扫描投影

`statistics` 不再消费 `footprint.*` 事件。`behavior_journey_scan` 轮询入口解析/接入日志、答卷、测评和报告事实，重建行为旅程读侧投影；这些投影不反向驱动核心写模型。

---

## 4. 任务生命周期事件

| 事件 | 产生模块 | Delivery | 说明 |
| ---- | -------- | -------- | ---- |
| `task.opened` | `plan` | `best_effort` | 任务入口已开放 |
| `task.completed` | `plan` | `best_effort` | 任务已完成 |
| `task.expired` | `plan` | `best_effort` | 任务已过期 |
| `task.canceled` | `plan` | `best_effort` | 任务已取消 |

任务事件说明计划层状态，不代表测评已经计分或报告已经生成。

---

## 5. 事件边界规则

1. `survey` 发布作答事实，不调用执行细节。
2. `evaluation` 发布 `evaluation.outcome.committed / evaluation.failed` 执行事实，不依赖统计结果。
3. `interpretation` 发布 `interpretation.report.generated / interpretation.report.failed`，不修改 Assessment 或统计读模型。
4. `statistics` 消费事件和读模型，不反向改变主业务状态。
5. 新增事件必须先更新 `configs/events.yaml`，再更新本文和相关模块文档。

---

## 6. 事件命名原则

事件类型体现产生事实的限界上下文，而不泄漏内部聚合兼容名。客户端完成态由 Assessment `evaluated` 与 ReportGeneration `generated` 的 Journey / ReadModel 投影得到，不再通过 Assessment 的 `interpreted` 状态或事件表示。
