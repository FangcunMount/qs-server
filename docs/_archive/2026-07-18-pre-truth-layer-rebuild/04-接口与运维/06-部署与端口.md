# 部署与端口

**本文回答**：qs-server 三个进程在 dev/prod 中分别监听哪些端口；配置监听、容器内端口、宿主机映射有什么区别；TLS/mTLS 和 pprof 端口如何看；端口不通时如何排障。

---

## 30 秒结论

| 进程 | HTTP/HTTPS | gRPC | 说明 |
| ---- | ---------- | ---- | ---- |
| qs-apiserver | 监听 HTTP(S) | 监听 gRPC | 后台 REST + 内部 gRPC server |
| collection-server | 监听 HTTP(S) | 不监听 gRPC | 前台 BFF，作为 gRPC client 连接 apiserver |
| qs-worker | 不暴露业务 HTTP/gRPC | 不监听 gRPC | MQ consumer + gRPC client |

一句话概括：

> **排障端口时要分清：配置监听地址、容器内端口、宿主机映射。**

---

## 1. 三层端口事实

| 层 | 说明 | 真值 |
| -- | ---- | ---- |
| 配置监听 | 进程实际 bind 的 address/port | `configs/*.yaml` |
| 容器内端口 | 容器内部监听端口 | image + config |
| 宿主机映射 | Docker/Compose 暴露给宿主机的端口 | `build/docker/*.yml` |

不要把宿主机端口误写成服务内部配置端口。

---

## 2. 生产典型端口

| 服务 | 容器内 | 宿主机映射示例 |
| ---- | ------ | -------------- |
| apiserver HTTP | 8080 | 8081 -> 8080 |
| apiserver HTTPS | 8443 | 9445 -> 8443 |
| apiserver gRPC | 9090 | 通常不映射宿主机 |
| collection HTTP | 8080 | 8082 -> 8080 |
| collection HTTPS | 8443 | 9446 -> 8443 |
| collection pprof | 6060 | 6060 -> 6060，如启用 |
| worker | 无业务端口 | 无 |

---

## 3. gRPC 网络

生产通常：

```text
collection-server -> qs-apiserver:9090
worker -> qs-apiserver:9090
```

通过容器网络访问，不一定暴露到宿主机。

如果宿主机 curl/nc 9090 不通，不代表容器间 gRPC 不通。

---

## 4. TLS / mTLS

| 类型 | 用途 |
| ---- | ---- |
| REST TLS | HTTPS 访问 |
| gRPC TLS | gRPC 加密 |
| gRPC mTLS | service identity / client cert |
| ACL/Audit | gRPC server interceptor |

证书路径由 yaml 与 compose volume 共同决定。

---

## 5. 健康检查

常见检查：

```bash
curl -s http://127.0.0.1:<host-port>/healthz
curl -s http://127.0.0.1:<host-port>/health
curl -s http://127.0.0.1:<host-port>/readyz
```

注意：`/healthz` 是 GenericAPIServer 通用健康入口；`/health`、`/readyz` 是业务路由实现。

---

## 6. 端口不通排障

1. 查配置监听地址。
2. 查容器是否启动。
3. 查容器内端口。
4. 查 compose port mapping。
5. 查防火墙/安全组。
6. 查 TLS 是否需要 HTTPS。
7. 查是否访问了容器网络内地址。
8. 查进程日志。

---

## 7. 常见误区

### 7.1 “容器内 8080 就是宿主机 8080”

不一定。看 compose mapping。

### 7.2 “gRPC 不映射宿主机就是没启动”

不一定。内部服务可能只在容器网络访问。

### 7.3 “6060 是业务端口”

不是。6060 是诊断/pprof 类端口，不能作为业务依赖。

### 7.4 “HTTPS 端口能用 HTTP 访问”

不能。协议要匹配。
