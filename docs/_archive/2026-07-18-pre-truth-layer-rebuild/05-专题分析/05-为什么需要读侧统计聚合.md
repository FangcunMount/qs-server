# 为什么需要读侧统计聚合

## 结论

Statistics 不是业务事实的第二份权威存储，而是面向查询的编排、窗口投影和缓存层。是否使用投影取决于指标语义：累计内容和答卷指标直接读取 `assessment` 事实；行为旅程、计划趋势等明确时间窗口指标读取可重建的 daily 投影。

这种边界同时解决三件事：跨模块查询口径集中、趋势查询成本可控、投影失败后可以补偿重放。代价是必须明确事实源、投影水位和事务边界，不能把“有一行投影”误当成“历史已完整覆盖”。

## 当前查询能力

公共查询由两类服务组成：

| 能力 | 职责 |
| ---- | ---- |
| `ReadService` | Organization Overview、Clinician、Entry、Content Batch |
| `PeriodicStatsService` | 受试者周期项目统计 |

`ReadService` 通过 `ReadServiceDeps` 分别依赖：

- `OverviewReader`
- `ClinicianStatisticsReader`
- `EntryStatisticsReader`
- `ContentStatisticsReader`

应用层没有组合型总读模型。MySQL adapter 可以由同一个具体实现隐式满足这些窄接口，但每个查询用例只持有自己需要的能力。

已删除的 System、单 Questionnaire、旧 Testee、单 Plan 服务与接口不属于当前模块。Plan 仍是 Overview 的一个查询分区，响应只包含 `activity` 和 `fulfillment`。

## 指标事实源

### Assessment 直接负责的累计指标

以下指标始终按机构条件从 `assessment` 事实计算：

- 唯一 `(content_type, content_code)` 数量
- 内容 `total_submissions`
- 内容 `total_completions`，仅统计 `status=evaluated`
- Overview 累计答卷提交数
- Overview 本地日界线内的今日答卷提交数

内容批量查询按类型拆成最多两条聚合 SQL。questionnaire 和 scale 即使 code 相同也分别统计；应用层负责去重、恢复首次输入顺序、补零和完成率计算。

这条路径不读取 Mongo 答卷数量，不把提交数推断为完成数，也不与 `statistics_content_daily` 合并。

### Daily projection 负责的窗口指标

`statistics_journey_daily` 承载机构、从业者和入口的行为窗口统计，包括入口打开、intake、关系建立、测评形成、报告生成和失败等事件。`statistics_plan_daily` 承载计划活动趋势。

Daily projection 可以按机构和日期窗口重建，因此适合日、周、月趋势；它不能替代没有覆盖水位的累计事实查询。

### Snapshot 负责的机构快照

`statistics_org_snapshot` 为机构规模类指标提供快速快照。快照不存在时，读模型回到对应主事实；查询错误必须向上返回，不能吞成固定零值。

累计内容与答卷提交指标即使快照存在，仍从 `assessment` 读取。

## 行为投影与补偿

行为事件投影在事务内完成 Episode、Footprint、daily mutation 和 checkpoint settlement。任一步失败，事务返回原错误并回滚；pending 的错误和退避时间在第二个独立事务更新。

pending 处理结果分为：

- `ok`
- `rescheduled_pending`
- `rescheduled_error`
- `error`

Prometheus 标签不包含 org ID。结构化日志可以携带 org/source 用于定位部分失败。

补偿扫描按 source/org 保存 watermark：只有一个 source batch 全部成功才推进 last-seen 水位；部分失败保留旧水位以便重放。dry-run 不写水位、不投影、不重建窗口。

## 缓存与治理

现役公共查询缓存只服务 Overview。缓存 key 使用当前版本前缀；旧版本 key 不主动扫描删除，按 TTL 自然过期。

缓存不是事实源。重建完成后刷新 org snapshot 并预热 Overview，只是降低发布后的冷启动查询成本。

## 存储边界

运行时使用的 Statistics 存储包括：

| 表 | 用途 |
| -- | ---- |
| `behavior_footprint` | 幂等行为足迹 |
| `assessment_episode` | 答卷到测评、报告的旅程状态 |
| `statistics_journey_daily` | 行为旅程窗口投影 |
| `statistics_plan_daily` | 计划窗口投影 |
| `statistics_org_snapshot` | 机构规模快照 |
| `analytics_scan_watermarks` | 补偿扫描水位 |
| `analytics_pending_event` | 待重试行为事件 |

`statistics_content_daily` 已退出运行时读写。历史 migration 和物理表保留，不执行 DROP；归档文档中的旧模型仅用于追溯。

## 权限和机构隔离

所有统计 SQL 都必须包含 org 条件。

`POST /api/v1/statistics/contents/batch` 还按 item 类型鉴权：

- questionnaire 需要 `manage_questionnaires`
- scale 需要 `manage_assessment_models`
- 混合请求需要同时具备两项能力

非法 type 或空 code 先返回 400；权限不足时整个请求返回 403，不返回部分结果。

## 性能和失败语义

Clinician 和 Entry 列表按整页 ID 批量读取详情，SQL 数量不随分页条数线性增长；单条详情复用同一个批量接口并传入单个 ID。

Statistics repository 和 read model 参与共享 MySQL limiter。导出操作在入口获取许可并在成功、SQL 错误和事务失败路径释放；私有 SQL helper 不重复获取。

统计查询不能：

- 吞掉 SQL 错误并返回空趋势或零统计
- 用 Mongo 答卷数制造完成数
- 使用已废弃的 `interpreted` assessment 状态
- 在查询请求里触发无界全量重建
- 绕过机构隔离读取其他机构事实

## 发布与验证

发布后 Journey daily 按机构、按不超过 31 天的窗口重建，再刷新 org snapshot 和 Overview cache。单个机构或窗口失败只重试该范围，不回滚已完成窗口。

服务端回滚不需要数据库回滚：历史表仍保留，新旧缓存 key 相互隔离，事实表没有 schema 变化。

最终验证包括 Statistics 窄测试、`go test ./internal/apiserver/...`、docs hygiene、`git diff --check`，以及前端 Node 16.20.2 下的目标测试和生产构建。
