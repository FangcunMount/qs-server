# qs-server 文档写作约定

本文约定 `docs/` 下的现行文档如何与代码对齐。读者入口见 [README.md](./README.md)。

---

## 适用范围

- 本文适用于 `docs/00-总览` 到 `docs/06-宣讲` 的现行文档。
- `docs/_archive/` 是本轮重建的临时迁移输入，不适用现行结构要求。
- 归档文档只用于寻找待重新核对的线索，不能直接视为当前事实。全部模块重建完成后删除该目录。

---

## 事实来源与优先级

判断文档事实时，优先级如下：

1. 源码与运行时行为：`cmd/`、`internal/`、`pkg/`。
2. 机器可读契约与配置：`api/rest/`、`api/grpc/`、`configs/events.yaml`、`configs/*.yaml`、migration、`Makefile`。
3. `docs/00-05` 现行维护文档。
4. `docs/06-宣讲`。
5. `docs/_archive` 和其它历史材料。

如果 prose 文档与代码或机器契约冲突，以代码和机器契约为准。

## 写作规则

- 先结论，再展开。
- 先写当前事实，再写历史背景或规划。
- 模块入口只做阅读地图和边界，不重复维护深讲细节。
- 业务模块文档先讲领域模型和关键链路，再讲接口、事件和存储。
- 长文如果引用旧设计，必须标明 `历史资料`、`待补证据` 或 `规划改造`。
- 一个事实只在一个 canonical 文档讲透，其它文档摘要并回链。

---

## 业务模块文档模板

`docs/02-业务模块` 按业务理解路径组织，不按代码包、接口清单或表结构组织。

每个模块 README 只做边界声明和阅读地图，必须回答：

1. 这个模块负责什么。
2. 这个模块不负责什么。
3. 这个模块有哪些核心领域模型。
4. 这个模块参与哪些关键业务链路。
5. 它依赖哪些模块，又被哪些模块依赖。

深度文档按语义编号：

| 编号段 | 文档类型 | 最低内容要求 |
| ------ | ---------- | ------------ |
| `10-19` | 领域模型 | 模型图、聚合根、实体、值对象、不变式、状态迁移、领域事件与反例边界 |
| `20-29` | 领域规格、服务与扩展协议 | 存在原因、输入输出、不变式、实现选择和 application service 边界 |
| `30-69` | 关键路径 | 入口到落库/出站的完整责任链、事务边界、失败语义、幂等、并发策略和测试证据 |
| `80-89` | 模块边界 | 允许/禁止的依赖、上下游契约和跨模块编排归属 |
| `90-99` | 代码索引与验证 | 分层代码路径、组合根、机器契约、配置与可执行的定向验证命令 |

领域服务不等于 application service：前者封装无法自然放入单一实体的领域规则，后者负责事务、repository、外部 port 和用例步骤编排。文档必须指出两者的具体代码落点。

接口、事件、存储和代码路径是结论的证据，不是内容的替代品。不批量创建空模板；完成一个模块的源码审核与深度文档后，再进入下一个模块。

## 重建期的归档与删除规则

文档满足任一条件时，应从现行树删除：

- 指向不存在的代码路径或旧包名，且已有新入口替代。
- 描述的模块边界已被 `registry.go` 或当前代码事实取代。
- 只剩历史设计价值，不应参与现行阅读路径。
- 内容重复且没有独立维护价值。

处理方式：

- 先从旧文档提取仍有价值的问题、线索和决策背景，再与当前源码核对后写入 canonical 文档。
- 被 canonical 文档取代的 active 占位文档直接删除，不再制造新的归档副本。
- 现行文档不得链接 `_archive`。
- 所有模块完成重建、active docs 无 archive 依赖、文档检查通过后，删除整个 `docs/_archive/`。历史追溯交给 Git。
