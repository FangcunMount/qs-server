# ModelCatalog 目标领域模型设计草案

**本文回答**：ModelCatalog 层负责什么、输入是什么、输出是什么、前置和后置模块是谁；以及 ModelCatalog 应如何表达 `AssessmentModel`、`AssessmentSnapshot`、`Factor`、`Norm`、`Conclusion` 等核心模型。

本文是设计草案，不等同于当前代码已经全部完成。当前代码事实仍需以 `internal/apiserver/domain/modelcatalog`、`internal/apiserver/port/modelcatalog`、`internal/apiserver/infra/mongo/modelcatalog`、`internal/apiserver/infra/mongo/scale` 为准。

---

## 1. 一句话定位

ModelCatalog 是**测评模型资产目录**。

它负责回答：

```text
针对某个已发布问卷版本，应该使用哪套测评模型定义来解释答卷？
```

核心映射是：

```text
questionnaire_code + questionnaire_version -> AssessmentSnapshot
```

ModelCatalog 管的是“如何解释答卷的模型定义”。它不保存用户答卷，也不执行一次测评，更不生成用户的最终报告。

---

## 2. 职责边界

### 2.1 负责什么

ModelCatalog 负责以下能力：

| 职责 | 说明 |
| ---- | ---- |
| 管理测评模型配置 | 创建、编辑、归档 `AssessmentModel` |
| 管理模型定义 | 维护 `Definition`，包括测量、校准、结论和报告映射 |
| 绑定问卷版本 | 将 `AssessmentModel` 绑定到 `QuestionnaireCode + QuestionnaireVersion` |
| 管理配置生命周期 | 草稿、发布、归档；这是模型配置生命周期，不是测评执行生命周期 |
| 发布不可变快照 | `AssessmentModel --publish--> AssessmentSnapshot` |
| 提供查询和解析 | 按 model code、问卷版本、产品分类查询已发布快照 |
| 守住配置合法性 | 校验 Factor、NormRef、Conclusion、Outcome、ReportMap 的引用完整性 |

最重要的输出是 `AssessmentSnapshot`。查询、缓存和 evaluation 都应该消费 snapshot，而不是消费草稿态 `AssessmentModel`。

### 2.2 不负责什么

| 不负责 | 归属模块 |
| ------ | -------- |
| 设计问卷题目、选项、跳题 | questionnaire / survey |
| 保存用户答卷 | survey / answersheet |
| 执行一次测评 | evaluation |
| 计算某个人的 `FactorScore` | evaluation |
| 生成某个人的 `ConclusionResult` | evaluation |
| 生成最终报告实例 | interpretation / report |
| 统计聚合 | statistics |
| C 端入口策略和展示编排 | collection-server / application query |

### 2.3 输入

ModelCatalog 的输入分两类。

配置命令输入：

```text
CreateAssessmentModel
UpdateDefinition
BindQuestionnaire
PublishAssessmentModel
ArchiveAssessmentModel
ImportNorm
```

依赖资料输入：

```text
QuestionnaireCode
QuestionnaireVersion
Question definitions
Option score definitions
Norm data
Product / Identity options
```

其中最关键的前置输入是 `QuestionnaireVersion`，因为测评快照的业务版本锚点跟随问卷版本。

### 2.4 输出

核心输出：

```text
AssessmentSnapshot
```

辅助输出：

```text
Published model list
Product options
Identity options
Questionnaire binding resolution
Cache invalidation events
Model published / archived events
```

### 2.5 前置和后置模块

前置模块：

| 前置模块 | 提供什么 |
| -------- | -------- |
| questionnaire / survey | 问卷结构、题目、选项、问卷版本 |
| admin / operating system | 模型配置命令 |
| norm source | 常模资料 |
| auth / IAM | 操作人和权限上下文 |

后置模块：

| 后置模块 | 消费什么 |
| -------- | -------- |
| evaluation | 消费 `AssessmentSnapshot`，执行 `FactorScore` / `NormScore` / `ConclusionResult` |
| interpretation / report | 消费 `ReportMap` / `Outcome` 定义和 evaluation 结果，生成报告 |
| collection-server | 查询已发布测评目录，启动测评时携带模型引用 |
| cache | 缓存 `AssessmentSnapshot` |
| statistics | 使用模型元信息做聚合维度 |

边界图：

```text
Questionnaire
  -> provides questionnaire_code + questionnaire_version + questions

ModelCatalog
  -> owns AssessmentModel
  -> publishes AssessmentSnapshot

Evaluation
  -> consumes AssessmentSnapshot + AnswerSheet
  -> produces FactorScore / NormScore / ConclusionResult

Interpretation
  -> consumes EvaluationResult + ReportMap
  -> produces Report
```

## 3. 设计结论

ModelCatalog 的核心不是“医学量表包、人格包、行为包”并列，也不是“scoring、norming、typology”按历史包名并列。

ModelCatalog 应围绕四层配置建模：

1. **测评模型**：由 `AssessmentModel` 表达。
2. **发布快照**：由 `AssessmentSnapshot` 表达。
3. **测量层**：由 `MeasureSpec` / `Factor` 表达。
4. **校准、结论、报告层**：由 `Calibration` / `Norm` / `Conclusion` / `ReportMap` 表达。

产品概念、算法身份、模型定义和运行结果必须分开：

| 轴 | 回答的问题 | 示例 |
| -- | ---------- | ---- |
| 产品概念 | 业务上这是什么产品 | 医学量表、人格测评、行为能力 |
| 算法身份 | 这个模型使用哪类算法配置 | scoring、norm、classification、ability |
| 模型定义 | 这个模型如何测量、校准、生成结论 | Factor、NormRef、Conclusion、Outcome |
| 运行结果 | 某一次答卷算出了什么 | FactorScore、NormScore、ConclusionResult |

ModelCatalog 只建模配置侧，不建模某一次测评结果。

## 4. 当前代码的主要问题

当前 `domain/modelcatalog` 内部已经有一些正确方向，但包语义混在一起：

| 当前包 | 当前实际职责 | 问题 |
| ------ | ------------ | ---- |
| `publishing` | 承载 `AssessmentModel` 聚合、生命周期、算法家族映射 | 包名像“发布流程”，但实际是核心测评模型配置聚合 |
| `factor` | 跨算法的 Factor snapshot、层级、解释、常模引用 | 方向正确，但还没有成为所有算法 payload 的唯一结构语言 |
| `scoring/definition` | 旧医学量表完整聚合：`MedicalScale`、Factor、lifecycle、version、repository | 不是纯 scoring 算法模型，而是 legacy 医学量表产品模型 |
| `scoring/snapshot` | scale 执行载荷 | 与 `factor.FactorSnapshot` 有桥接，但仍重复定义 Factor 执行视图 |
| `norming` | behavioral rating / Brief-2 的 metadata adapter 和运行载荷解析 | 不是独立 Norm 领域模型 |
| `taskperformance` | cognitive / SPM 的 metadata adapter 和运行载荷解析 | 不是完整 ability / task performance 领域模型 |
| `typology` | 人格测评 payload、分类 runtime spec、报告 spec、legacy MBTI/SBTI 转换 | 不是纯结论配置模型 |

复杂度爆炸的根因是：**产品模型、算法配置、运行 payload、legacy aggregate 被放进同一层级包结构里**。

## 5. 目标模型清单

### 5.1 核心模型

| 模型 | 类型 | 含义 |
| ---- | ---- | ---- |
| `AssessmentModel` | 聚合根 | 后台测评模型配置。承载产品、算法身份、问卷绑定、生命周期、内部修订号 |
| `AssessmentSnapshot` | 发布态快照 | `AssessmentModel` 针对某个问卷版本发布后的不可变快照，用于查询、缓存、执行 |
| `Definition` | 值对象 | 测评模型定义主体，组合测量、校准、结论、报告映射 |

### 5.2 测量层

| 模型 | 类型 | 含义 |
| ---- | ---- | ---- |
| `MeasureSpec` | 值对象 | 测量配置。回答“测哪些维度，以及如何从题目得到维度分” |
| `Factor` | 实体 / 值对象 | 广义测量维度或测量节点。医学量表、人格、行为、认知任务都统一用它表达 |
| `FactorRole` | 枚举 | Factor 的语义角色：`dimension`、`total`、`index`、`validity`、`task_set`、`ability_domain`、`report_group` |
| `Scoring` | 值对象 | Factor 的计分规则：sum、avg、weighted_sum、count、accuracy、reaction_time 等 |
| `FactorGraph` | 值对象 | 可选。描述 Factor 父子层级、复合关系、根节点、叶子节点 |
| `ChildrenPolicy` | 值对象 | 复合 Factor 如何由子 Factor 聚合 |

`Factor` 是配置侧对象，表示“测什么、如何测”。`FactorScore` 是 evaluation 结果侧对象，表示“某一次测出了多少分”。二者不能混用。

### 5.3 校准层

| 模型 | 类型 | 含义 |
| ---- | ---- | ---- |
| `Calibration` | 值对象 | 校准配置。回答“测量结果是否需要常模或其他资料转换” |
| `Norm` | 领域对象 | 常模表，表示某套人群常模资料，可独立版本化 |
| `NormRef` | 值对象 | Factor 对 Norm 的引用，说明某个 Factor 使用哪套常模 |

`Norm` 是常模资料，不等同于当前 `norming` adapter，也不等同于 `factor_norm` 运行机制。

### 5.4 结论层

| 模型 | 类型 | 含义 |
| ---- | ---- | ---- |
| `Conclusion` | 值对象 / 策略配置 | 结论配置。回答“这些测量结果意味着什么” |
| `RiskConclusion` | `Conclusion` 子类型 | 医学量表常用。基于分数区间生成风险等级、解释、建议 |
| `TypeConclusion` | `Conclusion` 子类型 | 人格测评常用。基于 Factor scores / vector 做类型、模式或 trait profile 判定 |
| `NormConclusion` | `Conclusion` 子类型 | 行为能力常用。基于常模结果生成等级或解释 |
| `AbilityConclusion` | `Conclusion` 子类型 | 认知任务常用。基于任务集 / 能力域 Factor 生成能力等级 |
| `Outcome` | 值对象 | 结论候选项定义，如 ENFJ、moderate_risk、high_ability |

`TypeConclusion` 取代之前讨论中的 `FactorClassificationModel`。分类不是独立核心模型，而是 `Conclusion` 的一种策略配置。

### 5.5 报告映射层

| 模型 | 类型 | 含义 |
| ---- | ---- | ---- |
| `ReportMap` | 值对象 | 把 Factor、Norm、Conclusion、Outcome 映射到报告结构和文案 |

`ReportMap` 只描述报告展示映射，不生成某个人的报告实例。

### 5.6 身份与路由

| 模型 | 类型 | 含义 |
| ---- | ---- | ---- |
| `Product` | 枚举 / 值对象 | 产品概念：医学量表、人格测评、行为能力 |
| `Identity` | 值对象 | 算法身份：kind、sub_kind、algorithm |
| `Family` | 派生值对象 | 运行机制：factor_scoring、factor_norm、factor_classification、task_performance |

`Product` 只表达产品归属，不应作为 evaluation 运行时路由依据。运行时路由由 `Identity` 派生出的 `Family` 和 snapshot 中的执行语义决定。

## 6. 目标模型结构

```text
AssessmentModel
└── Definition
    ├── MeasureSpec
    │   ├── Factor[]
    │   ├── FactorGraph
    │   └── Scoring
    ├── Calibration
    │   └── NormRef[]
    ├── Conclusion
    │   ├── RiskConclusion
    │   ├── TypeConclusion
    │   ├── NormConclusion
    │   └── AbilityConclusion
    ├── Outcome[]
    └── ReportMap

AssessmentModel --publish--> AssessmentSnapshot
AssessmentSnapshot --uses--> Definition
```

典型测评映射：

| 产品 | 测量层 | 校准层 | 结论层 |
| ---- | ------ | ------ | ------ |
| 医学量表 | `Factor` / `Scoring` | 可选 `NormRef` | `RiskConclusion` |
| 人格测评 | `Factor` / `FactorGraph` | 通常无 | `TypeConclusion` |
| 行为能力 | `Factor` / `Scoring` | `NormRef` | `NormConclusion` |
| 认知任务 | `Factor`，role 可为 `task_set` / `ability_domain` | 可选 `NormRef` | `AbilityConclusion` |

核心边界：

```text
Factor 负责“测量”
Calibration 负责“校准”
Conclusion 负责“解释”
ReportMap 负责“展示映射”
EvaluationResult 负责“某一次测评的结果”
```

## 7. 关键模型语义

### 7.1 AssessmentModel

`AssessmentModel` 是 ModelCatalog 的核心聚合根。

目标字段语义：

| 字段组 | 字段 | 语义 |
| ------ | ---- | ---- |
| 模型身份 | `Code` | 测评模型稳定编码 |
| 产品概念 | `Product` | 医学量表、人格测评、行为能力 |
| 算法身份 | `Identity` | 指向具体算法配置类型 |
| 业务版本锚点 | `QuestionnaireCode`、`QuestionnaireVersion` | 绑定采集问卷；测评发布版本跟随问卷版本 |
| 配置内容 | `Definition` | 测评模型定义 |
| 生命周期 | `Status`、`PublishedAt`、`ArchivedAt` | 草稿、发布、归档 |
| 内部修订 | `Revision` | 内部并发控制、审计和缓存刷新；不是业务版本 |

约束：

- `AssessmentModel` 是唯一后台测评模型配置聚合。
- `AssessmentModel` 不拥有独立业务版本号；业务版本由 `QuestionnaireVersion` 承担。
- 同一个 `QuestionnaireVersion` 不应存在多个常规业务测评版本。
- 发布结果是 `AssessmentSnapshot`，但 `AssessmentSnapshot` 不是新的聚合根。

### 7.2 AssessmentSnapshot

`AssessmentSnapshot` 是发布态不可变快照，用于查询、缓存和执行引用。

目标字段语义：

| 字段组 | 字段 | 语义 |
| ------ | ---- | ---- |
| 快照身份 | `ModelCode`、`QuestionnaireCode`、`QuestionnaireVersion` | 指向 AssessmentModel 与问卷版本的唯一发布快照 |
| 产品概念 | `Product` | 发布时冻结的产品归属 |
| 算法身份 | `Identity` | 发布时冻结的算法身份 |
| 执行语义 | `Family`、`PayloadFormat` | 执行路由和 payload 解码所需信息 |
| 定义内容 | `Definition` / `Payload` | 发布时冻结的模型定义 |
| 展示信息 | `Title`、`Status` | 查询和运行诊断所需的稳定信息 |
| 来源修订 | `SourceRevision` | 可选，仅用于审计和缓存诊断，不参与业务身份 |

约束：

- `AssessmentSnapshot` 是发布态读模型 / 值对象，不拥有独立 lifecycle。
- Snapshot 可以有独立 reader / repository / cache port，但不能拥有独立业务变更方法。
- Snapshot 由 `AssessmentModel` 发布产生，不允许绕过 `AssessmentModel` 手写业务状态。
- 查询、执行、一级缓存、二级缓存优先使用 Snapshot，而不是草稿态 `AssessmentModel`。
- Snapshot 的业务唯一性应由 `model_code + questionnaire_code + questionnaire_version` 约束。
- 当前存储中的 `model_version` 短期保留为兼容字段；目标语义上它不再表示独立测评业务版本。

### 7.3 版本语义

测评的前置是问卷。问卷已经有多版本设计，因此测评模型不再引入独立业务版本号。

目标规则：

```text
questionnaire_code + questionnaire_version -> AssessmentSnapshot
```

也就是说，业务上只要求每个已发布问卷版本有一个对应测评快照。问卷发布多个版本时，可以根据历史问卷版本找到对应的测评快照。

不支持常规业务形态下的：

```text
同一个 questionnaire_version -> 多个 assessment_model_version
```

原因是这会让同一批基于同一问卷版本提交的答卷出现多个解释版本，历史报告追溯和执行一致性都会变复杂。

因此需要区分：

| 概念 | 建议命名 | 是否业务版本 | 用途 |
| ---- | -------- | ------------ | ---- |
| 问卷版本 | `QuestionnaireVersion` | 是 | 测评快照的业务版本锚点 |
| 模型修订号 | `Revision` / `ConfigRevision` | 否 | 后台编辑并发控制、审计、缓存刷新 |
| 兼容字段 | `model_version` | 否，短期兼容 | 现有存储和读取链路兼容；可等同或派生自 `QuestionnaireVersion` |

如果已发布测评规则需要修正，优先方式是发布新问卷版本并生成新快照。只有历史数据修复场景才允许 correction 流程；correction 必须显式标记，不能伪装成常规“同问卷版本多测评版本”能力。

### 7.4 Factor

`Factor` 是 ModelCatalog 的核心领域对象，表示广义测量维度或测量节点。

Factor 可以表达：

- 维度编码和名称；
- 题目归属；
- 分数聚合策略；
- 父子层级；
- 总分、指数、效度、任务集、能力域等 role；
- 解释规则引用；
- 分类规则引用；
- 常模引用。

目标语义：

```text
Factor = 测评维度 / trait / score node / task set / ability domain
```

认知任务也先统一建模为 Factor，不单独抽象 TaskMeasurement。只有未来出现 Factor 无法表达的复杂任务执行数据时，再考虑新增独立抽象。

## 8. 存储设计边界

### 8.1 后台配置存储

当前事实：

| Collection | 用途 |
| ---------- | ---- |
| `assessment_models` | 新后台测评模型配置 |

目标：

- `assessment_models` 存储 `AssessmentModel` 聚合状态。
- payload 应逐步从裸 JSON bytes 收口为 `Definition`。
- 不为每个算法配置复制生命周期、版本号和 repository。

### 8.2 已发布快照存储

当前事实：

| Collection | 用途 |
| ---------- | ---- |
| `published_assessment_models` | 发布后的 `AssessmentSnapshot` 存储 |

目标：

- 已发布记录是 `AssessmentSnapshot` 的持久化形态，不是第二个领域聚合。
- 它保留 `model_code`、`questionnaire_code`、`questionnaire_version`、`model_kind`、`model_algorithm`、`payload_format`、`payload` 等执行所需字段。
- `model_version` 短期保留为兼容字段，可等同或派生自 `questionnaire_version`，但不再表达独立测评业务版本。
- application assembler 负责从 `AssessmentModel` 生成 Snapshot。
- query、evaluation、一级缓存、二级缓存都可以围绕 Snapshot 建模。
- evaluation 只读取已发布 Snapshot，不读取草稿模型。

### 8.3 旧医学量表存储

当前事实：

| Collection | 用途 |
| ---------- | ---- |
| `scales` | 旧医学量表 head / published snapshot 双记录 |

目标：

- `scales` 应被视为 legacy medical-scale authoring store。
- 它不应继续决定目标领域模型。
- 后续迁移方向是将医学量表配置收口到 `AssessmentModel + Definition + MeasureSpec + RiskConclusion`。

## 9. 目标包结构

建议目标结构如下：

```text
internal/apiserver/domain/modelcatalog
├── assessmentmodel
│   ├── aggregate.go
│   ├── snapshot.go
│   ├── lifecycle.go
│   ├── definition.go
│   └── validation.go
├── factor
│   ├── factor.go
│   ├── graph.go
│   ├── scoring.go
│   ├── role.go
│   └── children_policy.go
├── norm
│   ├── norm.go
│   ├── lookup.go
│   └── ref.go
├── conclusion
│   ├── conclusion.go
│   ├── risk.go
│   ├── type.go
│   ├── norm.go
│   ├── ability.go
│   └── outcome.go
├── reportmap
│   └── map.go
└── identity
    ├── product.go
    ├── identity.go
    └── family.go
```

迁移时不要求一步到位。更重要的是先明确语义：

- `assessmentmodel`：核心聚合、定义和发布快照。
- `factor`：通用测量节点。
- `norm`：常模资料和引用。
- `conclusion`：结论配置，不再单独建 `FactorClassificationModel`。
- `reportmap`：报告展示映射。
- `identity`：产品概念、算法身份和运行机制映射。

| `norming` | 拆分为 `norm` 领域对象 + `NormConclusion` / compatibility adapter |
| `taskperformance` | 拆分为 `AbilityConclusion` / cognitive compatibility adapter |
| `typology` | 拆分为 `TypeConclusion` + 人格测评 profile / legacy adapter |

## 11. 设计不变量

后续代码重构必须守住以下不变量：

1. `AssessmentModel` 是唯一后台测评模型配置聚合。
2. `AssessmentSnapshot` 存在，但它是不可变发布态读模型 / 值对象，不是第二个 domain aggregate。
3. `QuestionnaireVersion` 是测评快照的业务版本锚点。
4. 同一个 `model_code + questionnaire_code + questionnaire_version` 只能有一个常规有效 Snapshot。
5. `Definition` 分为测量、校准、结论和报告映射四层。
6. `Factor` 是跨产品、跨算法配置复用的测量节点。
7. 认知任务也先用 `Factor` 表达，不单独建 TaskMeasurement。
8. `Norm` 是常模资料，不等于 `norming` adapter。
9. `TypeConclusion` 取代 `FactorClassificationModel` 作为人格分类配置。
10. 产品概念只有三种：医学量表、人格测评、行为能力。
11. 产品概念不等于算法身份，也不等于运行机制。
12. 不为每个算法配置复制 lifecycle、versioning、repository。
13. 旧 `scales` 存储只作为兼容事实，不作为目标设计来源。
14. 查询、缓存和 evaluation 只消费已发布 Snapshot，不消费草稿配置。
15. ModelCatalog 不消费用户答卷，不产出用户测评结果。
