# qs-server 文档写作约定

本文约定 **`docs/` 下业务模块等文档写什么、写到什么粒度、如何与代码对齐**。读者入口见 [README.md](./README.md)。

---

## 事实来源与优先级

1. **源码**（`internal/`、`cmd/`、`pkg/`）——行为与边界的最终依据。  
2. **机器可读契约**——[`api/rest/`](../api/rest/)、[`internal/apiserver/interface/grpc/proto/`](../internal/apiserver/interface/grpc/proto/)、[`configs/events.yaml`](../configs/events.yaml)、各进程 [`configs/`](../configs/)。  
3. **`docs/` 现行正文**——在 1、2 未变更的前提下保持连贯。  
4. **`docs/_archive/`**——仅作历史补充；迁入正文前须**逐条对照源码**，并在文内或变更说明标注核对依据；无法核对则标「待核对」，勿写死行为细节。

---

## 表达习惯

- **金字塔**：先结论或要点，再展开。  
- **动机 → 结论 → 做法**：展开时尽量覆盖动机/约束、结论、步骤或配置。  
- **图 + 描述**：优先用 **mermaid**（`flowchart`、`sequenceDiagram`、`classDiagram` 等）表达模型关系、进程协作、时序；配图下用短段落或表格说明参与者、触发条件与代码锚点。一图一事，复杂流程拆多图。  
- **标题层级**：在 `###` 三级标题下仍需细分时，使用 **`####` 第四级标题**（例如「限界上下文」「REST」「处理器顺序」），少用仅加粗、不加标题的「假小节」。  
- **少贴代码**：用「路径 + 类型/函数名」指路；示例只保留关键片段。  
- **安全**：不在文档中写入真实密钥、Token、密码；配置只写键名或小节名。

## 讲解维度：Why / What / Where / Verify

每个业务模块（及需要讲清边界的专题、基础设施长文）建议**同时覆盖**下列四个维度，避免只写原则没有锚点，或只列路径没有动机。

| 维度 | 回答的问题 | 典型内容 |
| ---- | ---------- | -------- |
| **Why** | 解决什么业务问题、与其它方案或反模式相比的取舍、模块/机制存在的理由 | 限界与职责、与相邻模块的分工、对比表或要点列表 |
| **What** | 具体叫什么（业务与工程命名一致） | 领域概念、事件类型字符串、REST 路径或操作、RPC 名、配置小节名 |
| **Where** | 在仓库哪里实现、依赖哪些契约 | **代码锚点**（路径 + 类型/函数名）、`configs/*.yaml` 键、OpenAPI / `.proto` / `events.yaml` 中的条目 |
| **Verify** | 读者如何自检理解是否与代码一致 | 与 `events.yaml` 对表、主链路从入口跟到实现、变更后应同步检查哪些契约行 |

撰写**正文**时应用**贴切的中文小标题与段落**落实上表四方面，不必在篇内反复标注 `Why` / `What` / `Where` / `Verify` 等英文字样；上表供自检清单使用。

**与下文结构的大致对应**（可交叉，不必机械一一对应）：

| 推荐结构中的小节 | 侧重维度 |
| ---------------- | -------- |
| `30 秒了解系统` / `模块边界` | 以 **Why** 为主，辅以 **What**（模块叫什么、在系统中的位置） |
| `运行时示意图`、模型图 | **Why**（协作关系）+ **What**（参与者名称） |
| `领域模型与领域服务`、`应用服务设计` | **What**（概念与职责）+ **Where**（包与类型锚点） |
| `核心设计` 各节 | **Why**（设计动机）+ **What/Where**（事件、接口、流水线）+ **Verify**（与 yaml/proto 对齐说明） |
| `边界与注意事项` | **Why**（易踩坑的成因）+ **Verify**（何时需重读契约） |

## 业务模块文档推荐结构（`docs/02-业务模块/`）

单篇业务模块文建议采用以下层级（不适用的小节可写 **N/A** 并一句话说明原因）。撰写时对照上一节 **Why / What / Where / Verify**，避免整篇只覆盖其中一两维。

### 30 秒了解系统

用极短篇幅让读者建立心智模型。

#### 模块边界

- 本模块**负责**什么、**不负责**什么（列表或表）。  
- 与其它模块的依赖关系（可链向相邻 `02-业务模块/*.md`）。  
- 若涉及进程，与 [01-运行时](./01-运行时/) 中的命名一致。

#### 运行时示意图

- 至少一张 **mermaid 图**：模块在 `qs-apiserver` / `collection-server` / `qs-worker` 中的位置，或与上下游的数据/事件流向。  
- 图下用 2～4 句说明主入口（REST、gRPC、事件）即可。

---

### 模型与服务

可视需要拆成独立小节，避免一张图承载两种视角：

- **模型 ER 图**（`erDiagram`）：实体、关系、对外域引用；与存储/概念对齐。  
- **应用服务 + 领域服务 + 领域模型**（`flowchart` 分层）：interface → application → domain 服务与聚合的依赖方向，与 ER 正交。

#### 领域模型与领域服务

- **限界上下文**：本模块解决的问题域与不解决的域。  
- **核心概念表**：`概念名 | 职责 | 与相邻概念的关系`（业务语义为主，可与代码类型名对应）。  
- **聚合 / 实体 / 值对象**：不变量、生命周期、创建与变更的权威入口。  
- **领域服务**（若存在）：职责、与聚合的分工。  
- **建议配图**：除 ER 外，可用 **flowchart / classDiagram** 表达聚合关系（避免与源码字段 1:1 罗列）。

#### 应用服务设计

- 用例级服务划分（如提交、查询、引擎编排等）：**职责一句 + 锚点**。  
- 推荐另画 **分层示意图**：interface → application → domain 服务 → 领域模型（与「模型 ER 图」分工：前者运行时依赖，后者数据关系）。

---

### 核心设计

按主题拆成多个三级标题，命名统一为：

```text
### 核心<主题>：<简短标题>
```

示例：`### 核心数据流：从事件到内部 gRPC`、`### 核心模式：评估流水线`、`### 核心契约：领域事件与 Topic`。

每个三级标题下建议包含：

- **结论**（1～2 句）。  
- **图**：时序图（跨进程/跨服务）、流程图或模型片段，按需选用。  
- **表**（可选）：步骤与代码路径、或事件类型与 `configs/events.yaml` 的 handler/Topic 对齐表。  
- 涉及领域事件时，**事件名字符串、Topic、handler** 须与 [`configs/events.yaml`](../configs/events.yaml) 一致。

子主题较多时，在 `### 核心…` 下用 **`####` 分节**（如「对内 gRPC」「领域事件表」「与 scale 的职责边界」），便于目录导航与交叉引用。

常见可写主题（按需选用，非每篇写全）：

| 方向 | 可写内容 |
| ---- | -------- |
| 数据与存储 | 表/文档拆分、缓存位置、读写路径 |  
| 异步与事件 | 发布点、消费者、与 worker 的衔接 |  
| 集成 | 调用的其它模块、gRPC/REST 边界 |  
| 模式与扩展 | 真实使用的模式（管道、策略、仓储等）：意图 + 锚点 |  
| 配置 | 影响本模块的 `configs/*.yaml` 键（不写密钥） |  

**代码锚点索引**可单独成段落在「核心设计」末或文后，表头建议：`关注点 | 路径（相对仓库根）| 说明`，覆盖装配、主要 application、domain、infra。

---

### 边界与注意事项

- 易误解点、与历史文档不一致处、已知限制。  
- 与其它文档（如 [05-专题分析](./05-专题分析/)）的职责分工：此处只交叉引用一句，避免三处重复维护同一细节。

---

## 专题文档（`docs/05-专题分析/`）

专题同样建议用 **Why / What / Where / Verify** 组织：为何拆界、概念与事件叫什么、链路落点在哪、如何与 `events.yaml`/代码核对。

- **01-测评业务模型**：三界分离的问题—方案—反模式；标识符与引用与业务模块术语对齐。  
- **02-异步评估链路**：事件表 + 分段子时序 + 内部 gRPC/流水线锚点；幂等、重试等与实现对齐时再写。  
- **03-保护层与读侧**：与 [03-基础设施](./03-基础设施/) 联动，侧重限流、排队、背压、缓存、预聚合；与 02 分工为「资源与读路径」vs「业务流程与事件」。

---

## 基础设施与 `_archive` 扶正

横切主题（如高并发、缓存）正文落在 [03-基础设施](./03-基础设施/) 及必要时的专题 03；从 `_archive` 抽取时须 **结论 + 机制 + 代码锚点 + 已知限制**，并配图/表。

---

## 总览与运行时文档

- [00-总览](./00-总览/)：系统级地图，进程与存储关系图与列表随代码变更更新。  
- [01-运行时](./01-运行时/)：每进程对外暴露面（HTTP/gRPC/MQ）、依赖、明确不承担的事项。

---

## 维护节奏

- 变更行为或契约的 PR：**同一 PR 或跟进 PR** 更新对应文档（至少：事件/接口表、锚点或配置键之一）。  
- 新增业务模块文档：先按**业务模块推荐结构**搭全骨架，再加深；优先保证 **运行时示意图、主路径时序、事件与 `events.yaml` 一致、代码锚点**，并自检 **Why / What / Where / Verify** 是否都能从文中找到落点。
- 维护现行文档时，提交前运行 `make docs-hygiene`；该检查默认只覆盖**当前正文**，不包含 `docs/_archive/`。
