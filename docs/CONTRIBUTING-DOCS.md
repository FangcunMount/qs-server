# qs-server 文档维护约定

**本文回答**：这份文档只规定 `docs/` 的维护规则，包括事实优先级、目录分工、长文结构、archive 政策和提交流程；它不承担项目介绍，也不替代各目录 README。

---

## 适用范围

- 本文适用于 `docs/00-总览` 到 `docs/06-宣讲` 的现行文档。
- `docs/_archive` 不适用本文的结构要求；archive 只受它自己的归档政策约束。

---

## 事实来源优先级

维护文档时，必须按下面的顺序判断真值：

1. **源码**
   - `internal/`
   - `cmd/`
   - `pkg/`
2. **机器契约与配置**
   - `api/rest/`
   - `internal/apiserver/interface/grpc/proto/`
   - `configs/events.yaml`
   - `configs/*.yaml`
3. **现行正文**
   - `docs/00-05`
4. **宣讲层**
   - `docs/06-宣讲`
5. **历史层**
   - `docs/_archive`

文档与代码、契约冲突时，以代码和契约为准。

## 长文默认结构

现行长文默认按下面的顺序组织：

1. `本文回答`
2. `30 秒结论`
3. 主图或主表
4. 详细展开
5. 代码锚点、配置锚点或回链

额外规则：

- 目录级 `README.md` 可以更短，但至少要先给阅读入口和边界。
- 如果需要目录，目录应放在 `30 秒结论` 之后。
- 先结论，再展开；不要把核心结论埋在长背景后面。

---

## 表达规则

### 先结论，后细节

- 每个长文开头必须让读者在一屏内知道“这篇文档解决什么问题”。
- 每个重要三级标题下，优先给 1～2 句结论，再展开解释。

### 优先用图和表

以下内容优先使用图或表表达：

- 进程关系
- 模块边界
- 对比关系
- 流程步骤
- 配置项映射

正文负责解释图表不能直接说明的部分，不要让整页只剩长段落。

### 一处讲透，其它地方回链

- 同一事实不要在引言、图注、正文和 FAQ 里完整重复。
- 某处已经讲透后，其它文档只保留摘要和链接。

### 少抄代码，多给锚点

- 优先写“路径 + 类型 / 函数 / 配置项名”
- 不要在文档里复制大段源码
- 必要示例只保留最小关键片段

---

## 状态标签

涉及能力成熟度、证据完整度或未来规划时，统一使用下面的状态：

| 标签 | 含义 |
| ---- | ---- |
| `已实现` | 代码和配置都能证明，能直接写成当前能力 |
| `待补证据` | 方向存在，但仓内证据还不够完整，不能讲成标准能力 |
| `规划改造` | 当前仓库未完成，只能作为后续方案或目标 |
| `历史资料` | 只存在于 `_archive` 或阶段记录中，不属于现行真值层 |

不要把 `规划改造` 写成现状，也不要把 `历史资料` 写成当前设计。

---

## archive 政策

`docs/_archive` 是长期保留的历史层，规则如下：

- 不进入现行阅读主路径
- 不作为现行真值来源
- 默认不参与 `make docs-hygiene`
- 从 archive 回迁内容前，必须重新核对源码和契约

archive 目录本身的说明见 [docs/_archive/README.md](./_archive/README.md)。

---

## 链接与锚点规则

- 文档内优先使用相对路径链接到仓内其他文档或代码。
- 能直接指向代码文件时，不写模糊描述。
- 跨仓库路径可以写成普通代码路径说明，但不要把它们当成仓内可校验链接。
- 历史文档不要作为现行主路径的必经链接。

---

## 提交前检查

修改现行文档后，至少执行：

```bash
make docs-hygiene
git diff --check
```

如果改动涉及代码事实，额外确认对应锚点和契约没有漂移。

---

## 维护清单

提交文档前，至少自检下面几项：

- 这篇文档的边界是否清楚，没有和相邻文档重复维护同一事实
- 关键结论是否能在一屏内读到
- 是否给了足够的代码、配置或契约锚点
- 是否把现状、规划、历史区分清楚
- 是否误把 archive 内容当成现行事实
