# 基础设施能力地图

本地图用于从“系统问题”找到当前 owner、composition root、事实源和降级语义。不要从某个 Redis/NSQ/GORM package 反推它拥有全部业务规则。

## 1. 能力到责任边界

| --- | --- | --- | --- | --- |

## 2. 三个 composition root

### apiserver

- `internal/apiserver/process`：连接、Redis runtime、MQ/Event、transport、scheduler 与 shutdown。
- `internal/apiserver/container`：业务模块安装、跨模块 binding、REST/gRPC export。
- `internal/apiserver/container/modules/*`：模块内 Wire/Assemble/Install。

### collection-server

- `internal/collection-server/process`：Redis runtime、IAM、apiserver gRPC、HTTP 和 lifecycle。
- `internal/collection-server/container`：BFF application、cache/report runtime、gate/resilience。
- `internal/collection-server/transport/rest`：准入顺序、public/protected route。

### worker

- `internal/worker/process`：Redis/Mongo、event catalog、gRPC、metrics、MQ/dead-letter/hold。
- `internal/worker/container`：handler dependencies 和 application runtime。
- `internal/worker/integration/eventing|messaging`：dispatch、settlement、retry hold/replay。

## 4. 使用规则

1. 从用户可见现象或系统问题进入本表。
2. 找 composition root，确认能力是否真的装配，而不是只存在 package。
3. 找最终事实源和 scope：local process、Redis shared、DB global。
4. 追失败语义、状态码、settlement、retry/disposition。
5. 用日志/指标定位，用 durable state 下最终结论。
6. 目标态、注释和配置样例必须回到当前代码验证。

例如排查“答卷重复”时，不应从 Redis lock package 得出结论；完整链路必须追到 collection SubmitGuard、gRPC、apiserver transaction 和 Mongo 唯一索引。
