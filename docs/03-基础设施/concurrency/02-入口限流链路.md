# 入口限流链路

## 1. 解决什么问题

入口限流防止突发流量直接打穿 collection-server、qs-apiserver 和下游依赖。

## 2. 所在位置

入口限流位于 Nginx / 网关和应用层 handler 附近。Nginx 保护连接和入口洪峰，应用层按用户、接口和业务语义限流。

## 3. 设计目标

按用户、IP、接口、全局维度控制流量；超限快速失败；不同接口阈值不同；限流结果可观测。

## 4. 整体流程

请求到达后计算限流 key；未超限进入业务处理；超限直接返回错误和重试建议，不进入后端排队。

## 5. 核心数据结构

limit key、window、capacity、refill rate、route、user_id、ip、decision、retry_after。

## 6. 正常流程

正常用户请求在阈值内通过。查询接口阈值可高于提交接口，提交接口因为触发后续写入和事件链路，阈值更保守。

## 7. 异常流程

突发脚本或单用户高频刷接口时被拒绝；Redis 限流依赖异常时按配置选择本地降级或保守失败。

## 8. 幂等 / 降级 / 背压

限流不解决业务重复提交，只解决请求数量。被限流的请求不应进入 SubmitQueue。

## 9. 可选方案

只在 Nginx 做限流无法使用用户、租户、assessment task 等业务维度；只在应用做限流无法保护连接洪峰。

## 10. 当前方案取舍

采用入口和应用双层保护。入口挡流量洪峰，应用按业务语义保护热路径。

## 11. 观测指标

allowed count、rejected count、limit key topN、retry_after distribution、route-level QPS、限流后下游 QPS。
