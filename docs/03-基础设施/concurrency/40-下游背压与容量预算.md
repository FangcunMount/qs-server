# 下游背压与容量预算

## 1. 结论

Backpressure 的任务不是提高吞吐，而是在依赖已经接近饱和时阻止更多并发操作继续放大延迟和资源占用。

当前 apiserver 为 MySQL、Mongo 和 IAM 分别建立一个进程级共享信号量。同一依赖的多个 repository、业务请求和部分后台任务共享计数，而不是每个 repository 各有一套独立容量。

容量配置只能是压测起点。`max_inflight=120` 表示“单个 apiserver 进程最多允许 120 个 Mongo 操作同时持有槽位”，不表示 120 QPS，也不直接表示数据库能安全处理 120 个连接。

## 2. 当前装配

`internal/apiserver/resilience/subsystem` 创建三个 limiter：

| 依赖 | 生产 `max_inflight` | 槽位等待上限 | 作用域 |
| --- | ---: | ---: | --- |
| MySQL | 150 | 5000ms | 单个 apiserver 进程 |
| Mongo | 120 | 5000ms | 单个 apiserver 进程 |
| IAM | 100 | 4000ms | 单个 apiserver 进程 |

共享 limiter 通过 composition root 注入：

- MySQL `BaseRepository` 及相关 read store；
- Mongo `BaseRepository`、AnswerSheet/模型/解释等模块；
- MySQL/Mongo Outbox store；
- IAM client/module。

测试会用两个 repository 争用同一个 limiter，验证容量确实跨 repository 共享。

## 3. Acquire 语义

```text
Acquire(requestCtx)
  -> 有空槽：立即取得，返回原 requestCtx + release
  -> 无空槽：等待
  -> requestCtx 先取消：返回 requestCtx 错误
  -> limiter timeout 先到：返回 context deadline exceeded
```

等待 timeout 只限制“等槽位”阶段。成功取得槽位后返回的仍是原始 request context，不会给数据库操作凭空增加一个新的 5 秒执行预算。

因此有效等待上限是：

```text
min(请求剩余 deadline, backpressure timeout)
```

例如 collection reliable accept 的外层 deadline 是 2 秒，那么它下游某个 5 秒槽位等待配置不可能让这一请求多等 5 秒。

## 4. Backpressure 与连接池

两者都限制并发，但位置不同：

| 机制 | 位置 | 保护对象 | 满载表现 |
| --- | --- | --- | --- |
| Backpressure | repository/依赖调用前 | 应用 goroutine、下游请求量、跨模块公平性 | 应用内有界等待或超时 |
| DB 连接池 | 驱动层 | 实际连接 | 驱动内部等待、连接超时 |

生产 MySQL `max-open-connections=150` 与 MySQL `max_inflight=150` 对齐。这只是一个显式基线：

- 一次 repository 操作不一定始终只占一个连接；
- transaction 会延长连接占用；
- 连接池还可能被未经过统一 limiter 的路径使用；
- 相等的上限不保证零等待，也不保证数据库 CPU/IO 安全。

Mongo 的 limiter 是应用级共享预算；文档不能把它等同于 Mongo driver pool size，除非同时取得 driver 配置与运行指标证据。

## 5. QPS、并发与延迟

Little’s Law 提供第一步估算：

```text
L ≈ λ × W
```

- `L`：平均在途数量；
- `λ`：到达率，例如 operations/s；
- `W`：平均停留时间，例如 seconds。

例：若 Mongo 操作到达率为 200 ops/s，平均占槽 200ms，则平均在途约为 `200 × 0.2 = 40`。这不意味着配置 40 就够：

- 平均值看不到 P95/P99 尾延迟；
- burst 会在短时间内提高 λ；
- 慢查询、事务和网络抖动会提高 W；
- 不同操作成本不等；
- 还需留恢复与后台任务余量。

所以 Little’s Law 用于解释量纲和建立假设，最终上限必须通过分阶段压测与依赖指标校准。

## 6. 为什么 HTTP QPS 不等于 DB QPS

可以用近似放大模型：

```text
dependency_ops/s
  ≈ accepted_http_qps
  × 每请求平均依赖操作数
  + 后台任务操作数
  + 消费者与 Outbox 操作数
  + 重试操作数
```

HTTP 请求可能：

- 被 auth、preflight、RateLimit 或 Gate 提前拒绝，产生零次 DB 写；
- 在一次提交中读取 questionnaire/profile/testee，再执行 durable write；
- 触发多个 repository 查询；
- 命中 cache，避免回源；
- 超时后被客户端重试；
- 与 Outbox relay、worker、统计任务共享数据库。

因此不能用 “3 个 API 实例 × 每实例 80 HTTP QPS” 直接推导 DB 是 240 ops/s，更不能据此证明安全。

## 7. 多实例预算

当前 Backpressure 是 apiserver 进程内的。若有 `N` 个同配置实例，理论最大聚合在途上限接近：

```text
N × per_instance_max_inflight
```

但这只是上界，不是安全容量，因为：

- 负载可能偏斜；
- 所有实例共享同一 DB；
- DB 容量不会随 API 实例线性增加；
- 实例故障后存活实例的在途数和延迟会变化；
- 客户端重试可能抬高到达率；
- 后台消费者未必按同一实例比例缩放。

如果 DB 安全总在途预算为 `C_db`，一种保守起点是：

```text
per_instance_budget
  <= floor(C_db × 安全系数 / 最大计划实例数)
```

这会在实例减少时损失理论吞吐，但不会因为扩容让聚合上限越过 DB 预算。是否需要根据存活实例数动态重分配，必须引入可信成员信息、故障切换状态机和新的抖动风险。

## 8. 容量推导步骤

### 8.1 明确单位

分别记录：

- HTTP requests/s；
- gRPC calls/s；
- MySQL/Mongo/IAM operations/s；
- in-flight operations；
- 连接池 active/wait；
- P50/P95/P99 latency；
- timeout/reject/error rate。

### 8.2 找到依赖拐点

在固定实例拓扑下逐步升压，直到出现任一持续信号：

- 依赖 P99 明显上翘；
- 连接池等待增长；
- Backpressure utilization 持续高于 0.8；
- timeout 比例增长；
- DB CPU/IO/lock/slow query 超过安全线；
- 吞吐不再增加但在途和延迟继续增加。

拐点之前的稳定区间才是容量候选，不是“请求还能成功”的最高瞬时点。

### 8.3 留出安全余量

余量用于：

- 单实例故障；
- burst；
- GC 与网络抖动；
- schema/index 变化；
- Outbox backlog 恢复；
- 生产流量分布与压测模型差异。

### 8.4 反向校验入口预算

依据每类 HTTP 请求的依赖放大系数，反推 collection/apiserver RateLimit 与 Gate，而不是先拍一个入口 QPS 再让数据库兜底。

## 9. 超时与重试螺旋

一个危险链条是：

```text
依赖变慢
  -> 在途增加
  -> 客户端超时
  -> 立即重试
  -> 到达率增加
  -> 依赖更慢
```

保护方式需要协同：

- Gate/Backpressure 限制在途；
- 超时必须有界且分层一致；
- 429/503 带 `Retry-After`；
- 客户端指数退避 + 抖动 + 最大尝试次数；
- 幂等 key 保证写请求重试不制造新事实；
- 观测要区分服务端拒绝、客户端取消和依赖 timeout。

只把 limiter 从 80 提到 150，可能减少短期 503，也可能只是把等待转移到连接池并放大尾延迟。

## 10. 当前限制与规划

| 状态 | 内容 |
| --- | --- |
| `已实现` | MySQL/Mongo/IAM 使用固定上限信号量和有界槽位等待。 |
| `已实现` | resilience snapshot 暴露 `max_inflight`、`in_flight`、timeout 等当前值。 |
| `待补证据` | 生产实例数、依赖总容量、放大系数、连接池等待和 P99 拐点。 |
| `规划改造` | 自适应并发、依赖 Circuit Breaker、按健康实例动态预算。 |

自适应机制会引入反馈延迟和振荡，不能仅凭“自动”二字认为优于固定保守预算。

## 11. 学习问题

已知 Mongo Backpressure 单实例上限 120，3 个 apiserver 实例：

1. 为什么不能直接说 Mongo 安全并发是 360？
2. 如果 Mongo 平均延迟从 100ms 升到 800ms，即使 HTTP QPS 不变，在途数会怎样变化？
3. 如果只剩两个实例，固定每实例 120 是保护 DB，还是可能让两个实例把 DB 压得更重？还缺哪些证据才能回答？
