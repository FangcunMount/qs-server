# 压力模型与责任边界

## 1. 结论

高并发治理不是“加一个限流器”，而是让不同保护机制各自约束一个可度量的量，并明确失败时由谁承担损失。

qs-server 当前使用的基本分工是：

```text
RateLimit     控制到达速率 λ
Gate          控制某进程的在途请求数
Backpressure  控制某下游依赖的在途操作数
连接池        控制真实连接等稀缺资源
Lease         协调一段时间内谁有资格执行
唯一键/事务   决定什么业务事实可以被承认
```

把这些机制混在一起，会产生两类典型错误：用性能优化承担正确性，或者用正确性约束误判容量。

## 2. 六类问题

| 问题 | 关键问题 | 对应机制 |
| --- | --- | --- |
| 到达过快 | 一秒钟允许多少新请求进入？ | RateLimit、burst |
| 在途过多 | 同时有多少请求占用 goroutine、内存或下游等待？ | HTTP Gate、gRPC inflight Gate |
| 依赖饱和 | 同时允许多少 Mongo/MySQL/IAM 操作？ | Backpressure、连接池 |
| 重复意图 | 重试或攻击是否会制造两份业务事实？ | `idempotency_key`、指纹、唯一键、事务 |
| 执行权竞争 | 多实例中谁执行 leader/task/consumer 工作？ | LockLease |
| 运行时调整 | 谁能修改预算、让 leader 让权并留下证据？ | Resilience control、审计、状态回收 |

同一个请求可能依次经过多种保护。例如答卷提交既受 Gate 和 RateLimit 保护，又使用 SubmitGuard 降噪，最终仍由 Mongo 持久化约束裁决。

## 3. 正确性、可用性与效率

### 3.1 正确性机制

正确性机制回答“系统最后承认了什么事实”。可靠提交的当前真相包括：

- `(writer_id, idempotency_key)` 唯一约束；
- 同一 key 的内容指纹比较；
- AnswerSheet 与 Outbox 在同一 Mongo 事务内落盘；
- 只有查询到 durable AnswerSheet，collection 才能返回 202。

这些约束应尽量靠近持久化事实，跨过进程重启、Redis 故障和网络重试仍然有效。

### 3.2 可用性与效率机制

Gate、RateLimit、Backpressure、singleflight，以及 SubmitGuard 的设计意图，主要是减少资源放大：

- 提前拒绝超出预算的请求；
- 避免过多请求同时等待下游；
- 减少同一 key 的重复回源或重复提交；
- 在局部依赖故障时保住仍可工作的路径。

它们可以降级或被绕过，但绕过后仍不能制造错误业务事实。

当前 SubmitGuard 的争用者会直接进入 durable accept，实际并未消除这次重复 DB 调用；这是需要用指标评估的现行实现，而不是可以默认成立的“Redis 已经削峰”。

### 3.3 一个判断法

问自己：“删掉这个机制，系统会变慢/变贵，还是会写出错误事实？”

- 删除 SubmitGuard：可能增加 Mongo 竞争和重复调用，但唯一键仍收敛，主要是效率损失；
- 删除 Mongo 唯一键：两个实例可能各自写入一份答卷，是正确性缺口；
- 删除 Gate：突发流量可能压垮进程或下游，是容量与可用性缺口；
- 删除 Outbox 原子落盘：可能出现答卷已成功但事件永久丢失，是可靠性与正确性缺口。

## 4. 进程与作用域

| 进程 | 现行保护所有权 | 作用域提醒 |
| --- | --- | --- |
| collection-server | HTTP Gate、gRPC inflight Gate、可分布式 RateLimit、SubmitGuard | Gate 是单进程；Redis RateLimit/Lease 才能跨实例 |
| apiserver | 本地 RateLimit、MySQL/Mongo/IAM Backpressure、leader/task lease | RateLimit 与 Backpressure 都是单进程；多个实例容量会聚合 |
| worker | AnswerSheet 事件重复抑制 lease、实例心跳 | Redis lease 故障时 duplicate suppression degraded-open |

因此，“配置值相同”不等于“集群总量相同”。必须先说明机制是本地还是分布式、实例数如何变化、流量是否均匀、每个请求放大成多少依赖操作。

## 5. 失败语义不是一个状态码

| 场景 | 当前语义 | 原因 |
| --- | --- | --- |
| RateLimit 超额 | 429 + `Retry-After` | 客户端到达速率超过预算 |
| Submit Gate 50ms 未取得槽位 | 429 + `Retry-After: 1` | 当前实例的提交在途数已满 |
| 普通 query/catalog miss Gate 超时 | 503 + `Retry-After` | 服务当前没有处理容量 |
| report-status Try Gate 满 | 503 | 短查询快速失败，避免排队 |
| wait-report Gate 满且启用降级 | 200 pending + `Retry-After` | 没有伪造报告完成，只放弃本次服务端等待 |
| 同 key、同指纹 | 202，返回同一 AnswerSheet | 同一业务意图的重复投递 |
| 同 key、不同指纹 | 409 | 一个 key 被用于两个不同业务意图 |
| 依赖或 durable accept 不可用 | 503 | 服务无法确认 durable 结果 |

429、409、503 表达不同事实。不能因为“都是第二个请求”就统一返回 429，也不能用后续请求取消第一个已经成立的业务事实。

## 6. 与 Cache、Event 的边界

### 6.1 Cache miss 保护

`internal/pkg/loadguard` 提供 singleflight、回源超时和进程内 stale 降级。它只保护 cache miss 回源，不是通用 HTTP 准入器，也不能替代跨实例 RateLimit。

详细缓存边界见 [Cache](../cache/README.md)。

### 6.2 MQ 重投与 HTTP 重试

MQ redelivery 是同一事件的消费重试；HTTP 重试是客户端重复投递业务意图。它们可以都使用幂等或 duplicate suppression，但重试 owner、Ack/Nack 语义和持久化事实不同。

详细事件语义见 [Event](../event/README.md)。

## 7. 当前限制

| 状态 | 内容 |
| --- | --- |
| `已实现` | 数据面在控制面不可用时仍可使用配置基线运行；collection 的首次 control sync 是 readiness 条件。 |
| `已实现` | Redis 分布式限流运行期错误 degraded-open；SubmitGuard 与 worker duplicate suppression 在特定基础设施错误下也会 fail-open。 |
| `待补证据` | 各生产实例数、负载均衡偏斜、请求放大系数和真实依赖容量需要运行环境与压测证据。 |
| `规划改造` | 本地应急限流、Circuit Breaker、自适应并发与渐进恢复尚未实现。 |

## 8. 面试与自检

如果被问“限流能不能保证幂等”，完整回答应是：

> 不能。限流控制到达速率，只能减少重复请求造成的资源消耗。幂等必须用稳定业务 key、请求指纹和持久化唯一约束收敛；需要可靠异步处理时，还要把主事实和 Outbox 放在同一事务。

进一步追问：

1. Gate 满时为什么有的路由返回 429，有的返回 503，有的返回 200 pending？
2. 单实例 Gate 为什么不能约束集群总并发？
3. 为什么 fail-open 不是“什么都不管”，而是把正确性责任交给更后端的约束？
