# 可观测性、压测与验收

## 1. 结论

配置值说明“系统准备怎么限制”，运行快照说明“当前实例装配了什么”，Prometheus 时间窗口才说明“过去一段时间实际发生了什么”。

Concurrency / Resilience 验收必须同时回答：

1. 谁在拒绝：RateLimit、Gate、Backpressure 还是依赖？
2. 谁在等待：HTTP Gate、gRPC Gate、Backpressure、连接池还是数据库锁？
3. 是否丢失或重复了业务事实？
4. Redis 故障时请求落到了哪一层保护？
5. 实例变化后，集群聚合预算是否仍安全？

单个 `200/202` 比例或平均延迟无法回答这些问题。

## 2. 三类证据

| 证据 | 回答什么 | 局限 |
| --- | --- | --- |
| 配置 | baseline QPS、burst、inflight、timeout、action flag | 不证明已装配，也不证明安全 |
| Runtime snapshot | 当前实例 policy source/version、inflight、lock health、degraded reason | 是瞬时值，可能看不到刚才发生过的峰值 |
| Prometheus 时间窗口 | reject/timeout 增量、wait histogram、阶段延迟、故障频率 | 依赖 scrape、label 和查询窗口正确 |

system governance projector 会把 component snapshot 与 Prometheus evidence 合并。Backpressure utilization 当前按 `in_flight / max_inflight` 计算，`>=0.8` warning，`>=0.9` critical。

## 3. 核心指标目录

### 3.1 统一决策

```text
qs_resilience_decision_total{
  component,
  kind,
  scope,
  resource,
  strategy,
  outcome
}
```

它用于区分：

- rate allowed/limited/degraded-open；
- backpressure acquired/released/timeout；
- lock acquired/contention/error；
- duplicate skipped；
- queue full 等保留 contract 结果。

查询时必须带足够 label，避免把不同 component、scope 和 strategy 混成一个数字。

### 3.2 Gate 与提交

| 指标 | 含义 |
| --- | --- |
| `collection_http_gate_wait_seconds` | collection HTTP Gate 等待时长 |
| `collection_grpc_inflight_wait_seconds` | collection 到 apiserver gRPC 槽位等待 |
| `qs_collection_submit_gate_reject_total` | submit 50ms 未取得 Gate 的拒绝数 |
| `qs_collection_answersheet_submit_total{outcome}` | collection durable accept 结果 |
| `qs_collection_answersheet_submit_stage_duration_seconds{stage,outcome}` | preflight、identity、profile-link、gRPC、total 等阶段耗时 |
| `qs_apiserver_answersheet_durable_submit_total{outcome}` | apiserver created、idempotency hit/conflict、failed |
| `qs_apiserver_answersheet_durable_stage_duration_seconds{stage,outcome}` | Mongo transaction、Outbox stage 等 durable 阶段耗时 |

当前 `collection_http_gate_wait_seconds` 没有 route label，无法仅靠该 histogram 区分 catalog/query/submit。submit reject 有独立 counter，但其他 Gate 的路由级诊断仍需结合 HTTP route metrics 和日志。状态：`观测缺口`。

### 3.3 Backpressure

| 指标 | 含义 |
| --- | --- |
| `qs_resilience_backpressure_inflight` | 当前依赖在途 |
| `qs_resilience_backpressure_wait_duration_seconds{outcome}` | 取得槽位或 timeout 前的等待 |
| `qs_resilience_decision_total{kind="backpressure",...}` | acquired/released/timeout 次数 |

只看 in-flight 接近上限仍不够：

- 高利用率、低等待、零 timeout 可能是健康高效；
- 中等利用率、P99 wait 很高可能有流量偏斜；
- in-flight 低但 DB latency 高，可能是连接池、锁或下游容量问题。

### 3.4 LockLease

```text
qs_locklease_operation_total{
  component,
  name,
  operation,
  result
}
```

配合统一 decision 指标和 runtime lock snapshot，重点观察：

- acquire error；
- contention；
- renew lost/failed；
- release error；
- active run；
- Redis family degraded；
- body cancellation 后未退出告警。

### 3.5 控制面与审计恢复

| 指标 | 含义 |
| --- | --- |
| `resilience_control_operation_total{component,operation,outcome}` | control plane 操作结果 |
| `system_governance_audit_fallback_pending` | 等待回填 MySQL 的 terminal audit 数 |
| `system_governance_audit_fallback_total{operation,outcome}` | fallback load/put/list/delete 结果 |

fallback pending 持续大于 0 表示 action terminal 事实没有回到 MySQL 主审计存储，需要同时检查 MySQL 恢复和 Redis fallback 可用性。

## 4. 症状到责任层

| 症状 | 首先检查 | 可能原因 |
| --- | --- | --- |
| submit 429 增长，RateLimit limited 不增长 | Submit Gate reject、stage latency、gRPC wait | 在途变慢，不是到达 token 用完 |
| RateLimit limited 增长，Gate wait 很低 | global/user scope、热点 key、客户端重试 | 到达速率超预算 |
| `degraded_open` 增长 | Redis rate backend/family、下游 Gate/BP | Redis 故障后请求继续进入 |
| gRPC wait 上升 | collection gRPC inflight、apiserver latency | apiserver 或网络变慢 |
| Mongo BP timeout 增长 | Mongo operation latency、driver pool、Outbox backlog | 多类 Mongo 流量争用共享预算 |
| durable transaction failed 增长 | Mongo error、unique conflict、unknown commit recovery | 持久化链异常 |
| idempotency hit 增长 | 客户端 timeout/retry、重复 key | 合法重试或重试螺旋 |
| idempotency conflict 增长 | 同 key 指纹变化 | 客户端 bug、攻击或 key 生命周期错误 |
| lock contention 增长但业务吞吐正常 | workload/key 分布 | 合法单 owner 或热点 |
| lock renew failure/lost | Redis、网络、TTL、body 时长 | 可能出现双执行窗口 |

诊断顺序应从“哪个保护层做了决定”开始，再进入依赖内部，避免把所有 503 都归因于数据库。

## 5. 压测原则

### 5.1 固定事实

每次记录：

- commit SHA、镜像和配置；
- collection/apiserver/worker 实例数与资源规格；
- DB/Redis/MQ 拓扑；
- token 与测试用户规模；
- QPS profile、持续时间、VUs；
- 是否预热 cache；
- 客户端 timeout、retry 与退避策略；
- 压测前后 snapshot 和 Prometheus 窗口。

没有这些上下文，“系统支持 300 QPS”不可复现。

### 5.2 分层升压

推荐顺序：

1. smoke：验证 auth、数据与请求链；
2. 单场景：submit、query、report 分别找到拐点；
3. 稳态：长时间验证吞吐、P99、error 和资源泄漏；
4. burst：验证 burst、Gate、Retry-After 和恢复；
5. 混合场景：加入真实读写比例、Outbox/worker；
6. 单实例故障：验证剩余实例与重试行为；
7. 依赖故障：Redis、Mongo、MySQL、IAM 分别注入；
8. 恢复：验证 backlog、policy 和 error rate 是否回落。

每次只改变一个主要变量，才能解释拐点来自哪里。

## 6. 当前压测入口

可靠提交专项：

```bash
make perf-reliable-submit24
make perf-reliable-submit48-burst
make perf-reliable-submit96-boundary
```

跨实例相同 key 重复风暴：

```bash
COLLECTION_TOKEN='<same-writer-token>' \
SUBMIT_PAYLOAD_JSON='<valid-submit-json>' \
IDEMPOTENCY_KEY='coalescing-contract-<fresh-run-id>' \
PERF_ISOLATED_ENV=true \
COALESCING_SCENARIO=healthy \
make perf-submit-coalescing100
```

`scripts/perf/run-submit-coalescing.sh` 在 serverA 上按 Compose project/service
label 找到两个运行中的 collection 容器，读取它们在 `qs-network` 上的独立 IP，
逐一检查 `/readyz` 和 `/metrics`，再把请求以 round-robin 方式直接发到两个副本。
这样既不恢复已移除的宿主机端口，也不依赖 Nginx 是否恰好把一次小批次分到两个
副本。`EXPECTED_COLLECTION_REPLICAS` 默认是 2；发现数量不符或地址重复时直接失败。

指标使用压测前后的全局 counter delta，因此严格验收必须在隔离灰度环境运行，并
显式设置 `PERF_ISOLATED_ENV=true`。生产正常业务流量并存时不能把
`transaction committed == 1` 解释为单个 idempotency key 的精确证据。
`VERIFY_METRICS=false` 只允许用于脚本连通性调试，不能作为验收。

其余场景：

```bash
# 同 key、不同 fingerprint；两个 payload 必须仅表达两个不同提交意图
COLLECTION_TOKEN='<same-writer-token>' \
SUBMIT_PAYLOAD_JSON='<valid-submit-json-A>' \
CONFLICT_PAYLOAD_JSON='<valid-submit-json-B>' \
IDEMPOTENCY_KEY='coalescing-conflict-<fresh-run-id>' \
PERF_ISOLATED_ENV=true \
COALESCING_SCENARIO=conflict \
make perf-submit-coalescing-conflict

# 以下场景先按故障注入方案隔离对应 Redis family，再执行验收
PERF_ISOLATED_ENV=true COALESCING_SCENARIO=redis_lock_failure make perf-submit-coalescing-redis-lock-failure
PERF_ISOLATED_ENV=true COALESCING_SCENARIO=redis_signal_failure make perf-submit-coalescing-redis-signal-failure
PERF_ISOLATED_ENV=true COALESCING_SCENARIO=redis_unavailable make perf-submit-coalescing-redis-unavailable
```

Redis 故障 target 只负责生成受控流量并验证对应观测，不负责修改或停止 Redis。
执行前仍必须提供 `COLLECTION_TOKEN`、`SUBMIT_PAYLOAD_JSON` 和新的
`IDEMPOTENCY_KEY`，并完成经过评审的故障注入；不要对共享生产 Redis 直接执行。

脚本会在压测前后自动抓取两个 collection 实例与 apiserver 的 `/metrics`，
检查增量并输出 Redis RTT、contender wait 的 P95/P99、owner、contention、
signal、timeout、degraded-open 和逐实例 activity。
预期：

- 两个 collection 实例的 coalescer activity 增量都大于 0；
- 同步批次中 `qs_collection_answersheet_submit_coalescer_total{outcome="owner"}` 至少为 1，lease acquire 同时出现 acquired 和 contention；
- collection `readback_hit` 和 apiserver `explicit_readback{outcome="hit"}` 至少为 99；
- `qs_apiserver_answersheet_durable_operation_total{operation="transaction",outcome="committed"}` 增量为 1；
- questionnaire preflight、ProfileLink 和 durable save gRPC 的成功 `_count` 增量各为 1；
- 100 个响应均为 202 且 `answersheet_id` 相同；
- 健康场景 `degraded_open=0`、`signal_error=0`，Redis 和 wait histogram 必须有样本；
- Redis 故障场景仍只有一份 AnswerSheet，不把 contention 或 Redis error 映射为 429。

混合升档与脚本检查：

```bash
make perf-pretest120
make perf-mixed240-models
make perf-mixed280-models
make perf-mixed300
make perf-verify
```

脚本和 profile 位于 `scripts/perf`。完整环境准备与结果解释见 [300QPS 混合场景压测 SOP](../../04-接口与运维/11-300QPS混合场景压测SOP.md)。

这些 target 的名字不是性能结论。例如 `96-boundary` 表示测试输入档位，不表示生产已经证明 submit 96/s 可持续。

## 7. 故障验收矩阵

| 场景 | 施压/故障 | 必验不变量 | 预期证据 |
| --- | --- | --- | --- |
| Submit Gate 饱和 | 慢化 durable accept 并超过 96 在途 | 已受理请求不因后续请求取消 | 50ms 后 429、reject counter、无额外 AnswerSheet |
| RateLimit 饱和 | 提高到达率或热点 user | 不误报 409/503 | 429、Retry-After、limited label |
| Redis rate 故障 | backend timeout/error | 小流量仍可受理；正确性不变 | degraded-open、下游保护接管 |
| SubmitCoalescer 重复风暴 | 两实例并发 100 个同 key 同内容请求 | 一个 transaction；所有响应同一 AnswerSheet | owner≈1、explicit readback hit≈99、完整 preflight≈1、transaction committed=1 |
| SubmitCoalescer 故障 | lock/signal family 不可用 | 同 key 同内容仍只一份 AnswerSheet | degraded-open、Mongo idempotency hit |
| 同 key 不同内容 | 两实例并发提交 | 只承认一个指纹 | 一方 202，另一方 409 |
| Mongo 饱和 | 降低容量或增加延迟 | 不返回无 durable 结果的 202 | BP timeout/503、transaction metric |
| worker lock 故障 | Redis lease 不可用 | handler 可重入，不制造重复最终事实 | degraded-open、重复消费指标 |
| lease loss | renew 返回 false | body 收到 cancellation | lease-lost metric/error、无旧 owner 后续副作用 |
| control Pub/Sub 丢失 | 丢弃 signal | 最终仍应用 state | 周期 reconcile 后版本收敛 |
| action MySQL complete 故障 | action 执行后阻断 MySQL | 同 request ID 不重复执行 | Redis fallback、recovery 回填 |

## 8. 容量验收标准模板

一个 Gate/RateLimit/Backpressure 配置至少应有以下证据：

| 指标 | 验收问题 |
| --- | --- |
| Throughput | 稳态是否达到目标，而非只看发送量？ |
| P95/P99 | 尾延迟是否处于 SLO，升压后是否出现拐点？ |
| Reject/timeout | 哪一层拒绝，比例是否符合预期？ |
| Dependency utilization | DB/Redis/IAM 是否仍有恢复余量？ |
| In-flight/wait | 是否持续贴顶，是否有等待泄漏？ |
| Correctness | durable 数、唯一键、Outbox 数是否一致？ |
| Recovery time | 停止 burst/故障后多久回到基线？ |
| Retry amplification | 服务端 429/503 是否导致客户端发送量继续上升？ |

若缺失 correctness 与 recovery 两项，结果只能说明“短时间跑过”，不能说明系统可靠。

## 9. 文档与代码门禁

文档变更至少运行：

```bash
make docs-hygiene
make docs-facts
```

相关代码契约测试：

```bash
go test ./internal/pkg/resilience/...
go test ./internal/collection-server/concurrency/... \
  ./internal/collection-server/resilience/... \
  ./internal/collection-server/infra/redisops/... \
  ./internal/collection-server/application/answersheet/... \
  ./internal/collection-server/transport/rest/...
go test ./internal/apiserver/resilience/... \
  ./internal/apiserver/application/systemgovernance/... \
  ./internal/apiserver/application/survey/answersheet/...
```

Mongo integration test 需要真实 Mongo replica set 与权限；只通过 unit test 或编译不能声称 transaction/idempotency E2E 已验证。

## 10. 当前观测缺口

| 状态 | 缺口 |
| --- | --- |
| `观测缺口` | HTTP Gate wait histogram 缺少 route/gate 维度。 |
| `观测缺口` | 固定配置没有自动给出 HTTP 到 dependency 的放大系数。 |
| `待补证据` | Redis degraded-open 期间的安全小流量与过载拐点。 |
| `待补证据` | production renewal disabled 对长任务的实际风险。 |
| `待补证据` | 使用 `scripts/perf/k6-submit-coalescing.js` 在真实双实例与 Mongo replica set 上验证 transaction=1、尾延迟和 Redis 故障拐点。 |

## 11. 学习问题

看到 submit 429 从 0 升到 5%，你至少要再取哪些指标，才能判断应该：

- 提高 Submit Gate；
- 降低 RateLimit；
- 优化 Mongo；
- 缩短前置校验；
- 修复客户端重试？

请先写出证据链，再给调参结论。单看 429 比例不能选择上述任何一个方案。
