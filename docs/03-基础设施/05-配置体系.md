# 配置体系

本文介绍 `qs-server` 当前配置如何从 YAML 进入各个进程，以及哪些配置真正影响运行时行为。

## 30 秒了解系统

`qs-server` 当前的配置体系是“YAML 配置文件 + Options 结构 + 少量命令行参数”的组合。

三类进程各自有一份主配置文件：

- `configs/apiserver.{dev,prod}.yaml`
- `configs/collection-server.{dev,prod}.yaml`
- `configs/worker.{dev,prod}.yaml`

而真正被业务代码消费的，则是各服务自己的 `Options` 结构。换句话说：

- YAML 是运行配置来源
- `Options` 是进程内稳定配置模型

核心代码入口：

- [../../internal/apiserver/options/options.go](../../internal/apiserver/options/options.go)
- [../../internal/collection-server/options/options.go](../../internal/collection-server/options/options.go)
- [../../internal/worker/options/options.go](../../internal/worker/options/options.go)
- [../../internal/pkg/options](../../internal/pkg/options)
- [../../configs/apiserver.dev.yaml](../../configs/apiserver.dev.yaml)
- [../../configs/collection-server.dev.yaml](../../configs/collection-server.dev.yaml)
- [../../configs/worker.dev.yaml](../../configs/worker.dev.yaml)

## 核心架构

```mermaid
flowchart LR
    YAML[configs/*.yaml]
    Opts[Options 结构]
    Cfg[Config 结构]
    Runtime[运行时装配]

    YAML --> Opts
    Opts --> Cfg
    Cfg --> Runtime
```

## 核心设计原则

- 通用配置抽到 `internal/pkg/options`，避免三个进程重复维护数据库、Redis、IAM、gRPC 等基础配置结构。
- 每个进程再在自己目录下定义运行时特有选项，例如 `submit_queue`、`backpressure`、`worker` 并发参数。
- YAML 面向部署与运行；`Options` 面向代码消费；两者通过 `mapstructure` 字段绑定。
- 不是所有配置结构都已被当前运行时完整使用，读配置时必须区分“定义了”与“实际接线了”。

## 当前配置面分成哪些层

### 通用基础配置

这部分主要在 `internal/pkg/options`：

- `mysql`
- `redis`
- `mongodb`
- `iam`
- `grpc`
- `messaging`

这些配置会被多个进程复用。

### apiserver 专属配置

`apiserver` 在通用配置之外，还额外引入了：

- `cache`
- `backpressure`
- `rate_limit`
- `statistics_sync`
- `wechat`

这些配置决定了主业务服务的运行治理方式。

### collection-server 专属配置

`collection-server` 额外引入了：

- `grpc_client`
- `rate_limit`
- `submit_queue`
- `runtime`

这说明它的重点不在持久化，而在入口保护和对 `apiserver` 的调用治理。

### worker 专属配置

`worker` 额外引入了：

- `messaging`
- `worker`
- `grpc`
- `cache`

这说明它的重点是事件消费、回调 `apiserver` 和统计缓存开关。

## 哪些配置最值得优先理解

### apiserver

如果只关心最影响运行时行为的部分，优先看：

- `mysql / mongodb / redis`
- `messaging`
- `iam`
- `cache`
- `backpressure`

### collection-server

优先看：

- `grpc_client`
- `redis`
- `rate_limit`
- `submit_queue`
- `iam`

### worker

优先看：

- `messaging`
- `grpc`
- `redis`
- `worker.concurrency`
- `cache.disable_statistics_cache`

## 关键设计点

### 1. Options 才是代码真正消费的配置模型

三个服务最终都不是直接在业务代码里读 YAML，而是先构造成各自的 `Options`，再包装成运行时 `Config`：

- `apiserver` 的 `Config` 基本就是 `Options` 的薄包装
- `collection-server` 也是同样模式
- `worker` 会再把部分选项压成更适合运行时消费的 `Config`

因此，判断一个配置项是否真的生效，最终还是要看 `Options` 和运行时代码有没有把它接进去。

### 2. 通用配置被集中到了 internal/pkg/options

数据库、Redis、IAM、gRPC 这类基础选项都抽到了共享层。这使得：

- 三个进程对同一类依赖有一致的数据结构
- 配置命名和校验逻辑不会每个进程都写一遍
- 新增基础设施能力时，可以先扩共享配置，再按进程接线

### 3. 配置定义不等于运行时已经完全使用

当前最典型的例子是 `worker`：

- 配置结构里仍有 `MySQL / MongoDB`
- 但真正的 `DatabaseManager` 现在只初始化 `Redis`

所以读配置时必须把“可配置”和“当前运行时真的接了”区分开。

### 4. cache、submit_queue、backpressure 体现了不同进程的治理重点

- `apiserver.cache`
  - 关注业务读路径和统计读模型
- `collection-server.submit_queue`
  - 关注入口削峰和提交受理语义
- `apiserver.backpressure`
  - 关注保护下游依赖

这些配置不只是“参数集合”，也在反映每个进程在系统中的职责。

### 5. events.yaml 是配置体系中的特殊文件

大多数配置都只在单个进程里生效，但 `configs/events.yaml` 是一份被发布端和消费端共同依赖的基础设施配置：

- `apiserver` 用它决定事件怎么路由
- `worker` 用它决定订阅什么 Topic、把事件分发给谁

因此它更像“共享运行拓扑”，而不是某个服务自己的本地配置。

## 边界与注意事项

- YAML 文件里的字段很多，但文档更应该关注“真正改变运行时行为的配置”，而不是把全部字段抄成参数表。
- `dev` 和 `prod` 配置的存在，不代表两个环境的运行模式完全一致；真正的行为还取决于相应代码是否启用相关能力。
- `events.yaml` 与普通服务配置不同，它同时影响 `apiserver` 发布端和 `worker` 消费端。
- 配置体系这篇文档只讲“配置如何组织和进入系统”；具体的端口、部署、证书文件路径等运维细节，放到后续 `04-接口与运维` 里更合适。
