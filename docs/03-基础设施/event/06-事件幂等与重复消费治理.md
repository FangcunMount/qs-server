# 事件幂等与重复消费治理

## 1. 解决什么问题

MQ 消费天然可能重复投递。事件幂等治理避免同一个业务事实被重复推进成多份测评、多份报告或重复状态变更。

## 2. 所在位置

幂等治理位于 Outbox relay、worker handler、业务状态机和报告生成链路中。

## 3. 设计目标

依靠状态机、业务唯一性、claim、CAS 或锁防止重复推进；重复消费应返回成功或跳过。当前没有统一 processed-event 表或通用 business_key。

## 4. 整体流程

consumer 收到事件后先检查事件和业务状态；已处理则跳过；未处理则执行业务动作并记录处理结果。

## 5. 核心数据结构

aggregate_id、processing status、outcome_id、generation_id、report_id，以及各业务链路自己的唯一约束/lease/claim。event_id 会传给部分下游，但不是全局去重存储的统一键。

## 6. 正常流程

worker 处理事件后推进 assessment / report 状态。后续重复事件看到终态或处理中状态后不再重复执行重计算。

## 7. 异常流程

处理过程中失败时保留可重试状态；不确定是否成功时以业务事实源查询为准；payload 不合法进入失败告警。

## 8. 幂等 / 降级 / 背压

幂等检查必须在耗时计算前完成；重复消费不能占用大量 worker；重试使用退避，避免失败事件形成风暴。

## 9. 可选方案

只靠 MQ exactly-once 不现实；只靠数据库唯一索引能防部分重复写，但不能覆盖状态推进和外部副作用。

## 10. 当前方案取舍

使用业务状态机、唯一约束、claim、CAS 和锁的组合。技术幂等解决重复消息，业务幂等解决同一业务动作重复推进；每个事件的实际保护点和失败结算见 [09-事件契约矩阵.md](09-事件契约矩阵.md)。

## 11. 观测指标

duplicate event count、idempotency hit、state transition reject、retry after duplicate、handler skipped count。
