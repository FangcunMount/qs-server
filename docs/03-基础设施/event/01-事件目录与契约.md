# 事件目录与契约

**本文回答**：`configs/events.yaml` 当前定义了哪些事件目录；topic、event type、delivery、aggregate、domain、handler 分别代表什么契约；`eventcatalog` 如何把 YAML 变成发布端和消费端都可查询的 Catalog；代码常量、契约测试、worker registry 如何防止事件契约漂移。

---

## 30 秒结论

| 维度 | 结论 |
| ---- | ---- |
| 契约真值 | `configs/events.yaml` 是当前运行时事件目录真值，定义 topic、event、delivery、aggregate、domain、handler |
| 代码镜像 | `internal/pkg/eventcatalog/types.go` 中维护 19 个 event type 常量，并通过测试与 YAML 对齐 |
| Topic 数量 | 当前 4 个 topic：survey lifecycle、evaluation lifecycle、analytics behavior、plan task |
| Event 数量 | 当前 19 个事件 |
| Delivery class | 当前只有 `best_effort` 和 `durable_outbox` 两类 |
| Handler 绑定 | 每个 event 必须声明 worker handler；worker 启动时校验 handler 是否注册 |
| 发布约束 | delivery class 是出站可靠性合同，不是 RoutingPublisher 的内部策略 |
| 消费约束 | YAML handler 名称必须能被 worker explicit registry 创建 |
| 不包含 | YAML 不定义 worker 并发、重试次数、消费组细节、payload schema registry、下游服务清单 |
| 新增事件原则 | 先定义业务事实和 delivery，再补 event type 常量、YAML、publisher/outbox、handler、测试和文档 |

一句话概括：

> **事件目录是系统异步流程的契约表：发布端看它决定 topic 和 delivery，消费端看它决定订阅和 handler。**

## 1. 为什么事件目录要集中管理

qs-server 的事件驱动链路很长：

```text
answersheet.submitted
  -> assessment.submitted
  -> assessment.interpreted
  -> report.generated
  -> footprint.report_generated
```

如果 topic、handler、delivery 分散在各处代码里，会出现：

| 问题 | 后果 |
| ---- | ---- |
| 发布端和消费端 topic 不一致 | 事件发了但 worker 收不到 |
| handler 名称漂移 | worker 启动后无法分发 |
| delivery class 不明确 | durable 事件被 direct publish |
| 文档和代码不一致 | 排障时找不到事实源 |
| 新增事件漏测试 | 运行时才暴露错误 |

因此事件目录集中在：

```text
configs/events.yaml
```

并由 `eventcatalog` 统一解析、查询和校验。

## 2. YAML 结构

当前 `configs/events.yaml` 只有三层核心结构：

```yaml
version: "1.0"

topics:
  assessment-lifecycle:
    name: "qs.evaluation.lifecycle"
    description: "测评生命周期事件"

events:
  assessment.submitted:
    topic: assessment-lifecycle
    delivery: durable_outbox
    aggregate: Assessment
    domain: evaluation/assessment
    description: "测评已提交"
    handler: assessment_submitted_handler
```

### 2.1 顶层字段

| 字段 | 说明 |
| ---- | ---- |
| `version` | 事件目录版本 |
| `topics` | 逻辑 topic key 到物理 topic name 的映射 |
| `events` | event type 到 topic、delivery、handler 等契约的映射 |

### 2.2 TopicConfig

| 字段 | 说明 |
| ---- | ---- |
| `name` | 运行时 MQ topic name |
| `description` | topic 业务说明 |

### 2.3 EventConfig

| 字段 | 说明 |
| ---- | ---- |
| `topic` | 逻辑 topic key，必须存在于 `topics` |
| `delivery` | 出站可靠性合同：best_effort / durable_outbox |
| `aggregate` | 事件所属聚合 |
| `domain` | 事件所属业务域 |
| `description` | 事件语义说明 |
| `handler` | worker handler registry 中的 handler 名称 |

## 3. 当前 Topic 目录

| Topic Key | Topic Name | 描述 | 事件数量 |
| --------- | ---------- | ---- | -------- |
| `questionnaire-lifecycle` | `qs.survey.lifecycle` | 问卷和量表生命周期事件 | 2 |
| `assessment-lifecycle` | `qs.evaluation.lifecycle` | 测评生命周期事件 | 5 |
| `analytics-behavior` | `qs.analytics.behavior` | 行为足迹与测评服务过程投影事件 | 8 |
| `task-lifecycle` | `qs.plan.task` | 测评任务生命周期事件 | 4 |

### 3.1 Topic 设计原则

| 原则 | 说明 |
| ---- | ---- |
| 按领域流程分组 | survey/evaluation/analytics/plan |
| 不按数据库表分组 | 事件是业务事实，不是表变更日志 |
| 不按 handler 分组 | handler 是消费实现，不是事件归属 |
| topic 数量克制 | 避免过细 topic 增加订阅复杂度 |
| topic name 稳定 | 修改 topic name 会影响发布端、worker 和 MQ 运维 |

## 5. Delivery Class 契约

当前只有两种 delivery class：

```text
best_effort
durable_outbox
```

### 5.1 best_effort

语义：

```text
业务状态已经保存；
事件尽力发布；
发布失败一般不回滚主状态；
不承诺补发。
```

适合：

- 缓存刷新。
- 轻量通知。
- 可由下次查询/同步修复的投影。
- 不影响主业务闭环的副作用。

当前 best_effort：

```text
questionnaire.changed
scale.changed
task.opened
task.completed
task.expired
task.canceled
```

### 5.2 durable_outbox

语义：

```text
业务状态和 outbox record 在同一持久化边界写入；
relay 负责异步发布；
发布失败可重试；
下游仍需幂等。
```

适合：

- 驱动主业务链路。
- 事件丢失会导致流程卡住。
- 事件丢失会导致统计/标签/报告不可恢复。
- 需要可观测和补偿的关键事件。

当前 durable_outbox：

```text
answersheet.submitted
assessment.submitted
assessment.interpreted
assessment.failed
report.generated
footprint.*
```

### 5.3 delivery class 不是 publisher mode

`delivery` 描述**事件应该如何出站**。

`PublishMode` 描述**运行时 publisher 如何发送**：

```text
mq
logging
nop
```

它们不是同一个概念。

## 6. Catalog 模型

`internal/pkg/eventcatalog` 是 qs-server 对 component-base eventcatalog 的封装和别名层。

它提供：

| 能力 | 说明 |
| ---- | ---- |
| `Load(path)` | 从 YAML 加载配置 |
| `Parse(data)` | 从字节解析配置 |
| `NewCatalog(cfg)` | 构建 catalog |
| `GetTopicForEvent(eventType)` | 查询 event 对应 topic name |
| `GetDeliveryClass(eventType)` | 查询 delivery |
| `TopicSubscriptions()` | 生成 worker topic subscription |
| `Config()` | 获取原始配置 |

### 6.1 代码常量镜像

`internal/pkg/eventcatalog/types.go` 维护了全部 event type 常量。

例如：

```go
const (
    AnswerSheetSubmitted = "answersheet.submitted"
    AssessmentSubmitted = "assessment.submitted"
    ReportGenerated = "report.generated"
    FootprintReportGenerated = "footprint.report_generated"
)
```

`EventTypes()` 返回代码已知的全部 event type。

`ValidateEventTypes(cfg)` 会检查代码常量是否全部存在于 YAML。

这是一条重要防漂移机制：

```text
代码常量
  <-> configs/events.yaml
```

## 7. Catalog 如何服务发布端

发布端使用 `RoutingPublisher`：

```text
event type
  -> Catalog.GetTopicForEvent
  -> topic name
  -> BuildMessage
  -> MQ publish
```

如果 event type 不在 catalog 中：

- 记录 `PublishOutcomeUnknownEvent`。
- 打 error log。
- 返回错误。

这防止未登记事件悄悄发到错误 topic。

### 7.1 发布端不应该硬编码 topic

业务应用服务不应写：

```go
publisher.Publish("qs.evaluation.lifecycle", evt)
```

而应创建 domain event，让 RoutingPublisher 通过 catalog 选择 topic。

---

## 8. Catalog 如何服务 Outbox

Outbox 需要知道事件 delivery class。

`durable_outbox` 事件：

```text
业务事务
  -> BuildRecords
  -> outbox store
  -> relay
```

`best_effort` 事件不应进入 outbox。`outboxcore.BuildRecords` 会拒绝把 best_effort 事件写入 outbox。

### 8.1 为什么不能把 best_effort 写入 outbox

如果 best_effort 也写入 outbox，会造成：

- delivery 语义失效。
- outbox backlog 被轻量事件污染。
- 运维误以为所有事件都要补偿。
- 事件成本上升。

---

## 9. Catalog 如何服务 Worker

Worker 侧使用 catalog 生成订阅和 handler 绑定。

### 9.1 Dispatcher 校验

`Dispatcher.Initialize(catalog)` 会：

1. 检查 catalog 已加载。
2. 检查 handler registry 已配置。
3. 调用 `validateHandlerBindings`。
4. 确认每个 event config 中的 handler 都已注册。
5. 创建 Subscriber。
6. RegisterHandlers。

如果 YAML 中声明的 handler 不存在，worker 启动失败。

### 9.2 Subscriber 注册

`eventruntime.Subscriber.RegisterHandlers()` 会遍历 config events：

```text
eventType -> handlerName -> HandlerFactory -> handlerFunc
```

然后保存到内存 map：

```text
handlers[eventType] = handlerFunc
```

### 9.3 TopicSubscriptions

`TopicSubscriptions()` 会把 event type 按 topic 聚合，用于 worker messaging runtime 订阅 topic。

这使 worker 不需要硬编码订阅哪些 topic。

## 10. 事件命名规范

### 10.1 推荐格式

```text
<aggregate_or_domain>.<past_tense_fact>
```

示例：

```text
answersheet.submitted
assessment.interpreted
report.generated
task.opened
footprint.entry_opened
```

### 10.2 命名原则

| 原则 | 说明 |
| ---- | ---- |
| 使用过去式事实 | 表示事情已经发生 |
| 不用命令式 | 不写 `assessment.evaluate` |
| 不带技术实现 | 不写 `mysql.row_inserted` |
| 不带 handler 名 | handler 是消费实现 |
| 不带 transport | 不写 `nsq.assessment.submitted` |
| 保持稳定 | 改名会影响发布端、消费端和历史数据 |

---

## 11. aggregate / domain 字段语义

### 11.1 aggregate

`aggregate` 表示事件所属聚合或事实对象。

例如：

| Event | Aggregate |
| ----- | --------- |
| `answersheet.submitted` | AnswerSheet |
| `assessment.submitted` | Assessment |
| `report.generated` | Report |
| `task.opened` | AssessmentTask |
| `footprint.entry_opened` | BehaviorFootprint |

### 11.2 domain

`domain` 表示事件所属业务域或模块。

例如：

```text
survey/answersheet
evaluation/assessment
evaluation/report
statistics/behavior
plan
```

它用于文档、排障和治理，不应该作为代码路由的主要依据。路由以 event type 和 topic 为准。

---

## 12. handler 字段语义

`handler` 是 worker registry 的 handler 名。

例如：

```text
assessment_submitted_handler
behavior_projector_handler
task_opened_handler
```

### 12.1 handler 不是服务名

handler 表示 worker 内部处理函数，不表示：

- 哪个进程。
- 哪个部署实例。
- 具体 consumer channel。
- 并发数。
- 重试策略。

这些由 worker/messaging/runtime 配置和 MQ provider 负责。

### 12.2 多 event 复用同一个 handler

可以复用。例如所有 `footprint.*` 当前都使用：

```text
behavior_projector_handler
```

这表示同一 handler 内部按 event type 分发或解析不同 payload。

## 13. YAML 不表达什么

| 不表达 | 原因 |
| ------ | ---- |
| worker 并发 | 并发属于 worker runtime / MQ adapter 配置 |
| consumer group/channel 策略 | 当前 worker serviceName/channel 层处理 |
| retry 次数 | 重试属于 MQ provider / handler / outbox relay |
| payload schema registry | payload 由 domain event struct 和 handler decode tests 保护 |
| 下游服务清单 | handler 代码和依赖注入决定 |
| SLA / timeout | 属于 runtime/resilience 配置 |
| DLQ 策略 | 当前不在事件目录层表达 |

不要把 `events.yaml` 扩张成工作流编排 DSL。

## 14. 契约测试

### 14.1 代码常量与 YAML 对齐

`eventcatalog/types.go` 中 `EventTypes()` 是代码事件常量集合。测试会确保这些事件都存在于 YAML。

### 14.2 delivery class 校验

测试会确保：

- 每个 event 都声明 delivery。
- delivery 只能是合法值。
- 缺失/非法 delivery 会被拒绝。

### 14.3 direct publish 约束

`eventruntime` 架构测试会扫描 direct publish 调用点，避免 `durable_outbox` 事件绕过 outbox。

注意：这不是 RoutingPublisher 的职责。RoutingPublisher 仍可发布 durable event，因为 relay 最终也要通过它发布。

### 14.4 worker handler 绑定

worker dispatcher 和 handler registry 测试会确保：

- YAML 中 handler 可在 registry 找到。
- Subscriber 可以注册所有 event type。
- 缺 handler 会尽早失败。

## 15. 新增事件的最小变更面

新增事件至少要改：

```text
1. 领域事件 payload / constructor
2. eventcatalog/types.go 常量
3. configs/events.yaml
4. 发布路径：direct publish 或 outbox stage
5. worker handler 或复用已有 handler
6. handler registry
7. handler decode / idempotency / tests
8. 事件文档
```

如果是 durable_outbox，还要额外设计：

```text
outbox stage 边界
MySQL/Mongo store
事务一致性
relay 出站
补偿排障
```

## 16. 常见误区

### 16.1 “YAML 写了事件就能运行”

不够。还需要代码常量、事件构造、发布路径、worker handler、registry、测试。

### 16.2 “handler 字段就是消费者服务”

不是。handler 是 worker 内部处理函数名，不是部署拓扑。

### 16.3 “delivery=durable_outbox 代表不会重复消费”

错误。durable_outbox 只保证可靠出站；消费端仍要幂等。

### 16.4 “best_effort 就是不重要”

不准确。它可能业务上重要，但系统选择了轻量通知语义，不能依赖它推进主状态。

### 16.5 “topic 越细越好”

不一定。topic 太细会增加运维和订阅复杂度。当前按业务生命周期聚合 topic。

### 16.6 “events.yaml 应该配置所有重试和并发”

不建议。事件目录只做契约，不做 worker runtime 编排。

---

## 17. 修改指南

### 17.1 修改 topic name

风险很高，需要同步：

- publisher topic resolver。
- worker subscriptions。
- MQ topic 创建。
- dashboards / alerts。
- 文档。
- 部署环境。

不建议随意改 topic name。

### 17.2 修改 delivery class

从 best_effort -> durable_outbox：

1. 设计 stage 边界。
2. 选择 outbox store。
3. 保证同事务写入。
4. 增加 relay 验证。
5. 更新 architecture tests。
6. 更新排障文档。

从 durable_outbox -> best_effort：

- 通常不建议。
- 必须证明事件丢失不会卡住业务主链路。

### 17.3 修改 handler 名

需要同步：

- `configs/events.yaml`。
- worker registry。
- handler tests。
- dispatcher tests。
- 文档。
