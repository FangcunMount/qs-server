# MQ 发布与消费链路

## 1. 链路总览

```text
DomainEvent
   ↓ EffectiveRegistry.Lookup(event type)
RoutingPublisher
   ↓ eventcodec envelope + eventmessaging metadata
MQ topic
   ├─ worker 主 channel → handler registry → ACK/NACK
   └─ additional channel → projection handler → 独立 ACK/NACK
```

发布端只向 Registry 已知的 Topic 路由；消费端只向显式 handler registry 分发。未知事件不会凭命名约定动态加载 handler。

## 2. RoutingPublisher 模式

| 模式 | 行为 | MQ-backed | Durable Outbox 可标记 published |
| --- | --- | --- | --- |
| `mq` | 编码并发布到配置 Topic | MQ publisher 非 nil 时是 | 是 |
| `logging` | 记录事件，不发送 MQ | 否 | 否 |
| `nop` | 忽略直接发布 | 否 | 否 |

默认环境映射为 production 使用 MQ、development 使用 logging、test 使用 nop。环境默认只是兜底；关键安全门禁是 `IsMQBacked()`。

对 best-effort 事件，logging/nop 是允许的退化：它本来就没有可靠投递承诺。对 durable event，ImmediateDispatcher 和 relay 只有在 MQ-backed 时才启动，因此不会因为 logger 返回 nil 就把 Outbox 标成 published。

Registry 不认识的 event type 不会猜测 Topic，发布结果记录为 `unknown_event`。

## 3. MQ message 构造

`component-base/pkg/eventcodec` 把 DomainEvent 编码为统一 envelope，`component-base/pkg/eventmessaging` 构造 MQ message：

- message UUID 使用 event ID。
- body 是包含 `id`、`eventType`、`occurredAt`、`aggregateType`、`aggregateID`、`data` 的 JSON envelope。
- metadata 包含 `event_type`、`aggregate_type`、`aggregate_id`、`occurred_at`、`source`。

source 表示发布运行时来源，例如 apiserver。它用于观测和排障，不代替 event owner。

## 4. Worker 主消费者

worker 启动时：

1. 加载 `configs/events.yaml` 并构造 EffectiveRegistry。
2. 构造显式 handler registry。
3. 校验每个主 handler 都存在，且没有错误映射。
4. 按 Topic 建立订阅，channel 使用 worker service name。
5. 收到消息后提取 event type、解码 envelope、dispatch handler、执行统一结算。

handler factory/decorator 可以增加日志和指标，但不能改变业务 handler 的错误返回或运输层 ACK/NACK 语义。

## 5. 统一结算策略

dispatch 结果与 handler error 分开：

```go
type DispatchOutcome string

const (
    DispatchHandled DispatchOutcome = "handled"
    DispatchUnknown DispatchOutcome = "unknown"
)
```

运输层映射固定如下：

| 场景 | 判定 | MQ 动作 | Consume outcome | 原因 |
| --- | --- | --- | --- | --- |
| metadata 无 event type 且 envelope 无法解析 | decode-invalid | NACK | `decode_nacked` / `decode_nack_failed` | 保留统一的运输失败证据，并进入有界运输重试 |
| event type 未注册 | unknown | ACK | `unknown_acked` / `unknown_ack_failed` | 当前进程无法处理，持续重投无收益 |
| handler 返回 error | handler failure | NACK | `nacked` / `nack_failed` | 交给 MQ redelivery 重试 |
| handler 成功或识别为重复 | handled | ACK | `acked` / `ack_failed` | 本 channel 已完成结算 |
| automatic retry 开关暂停且 hold 成功 | held | ACK | `held` / `hold_failed` | `retry_event_hold` 已接管恢复责任，不继续消耗运输预算 |

ACK/NACK API 自身失败会记录独立 outcome，不能把“handler 已处理”和“broker 已成功结算”混为一个指标。

NACK 不是无限循环。subscriber 的默认和代码硬上限都是 8 次 delivery attempt，退避为 30 秒起步、5 分钟封顶、20% jitter；配置可以下调，关闭 bounded delivery 等价于只允许 1 次。预算耗尽后 provider 调用 failed-message handler：

1. worker 主消费者和 apiserver 附加消费者把原 message ID、event ID、组织、topic、channel、payload、attempt 和最后错误写入 MySQL `event_delivery_dead_letter`。
2. 相同 `(provider, topic, channel, message_id)` 重复终止时更新同一条证据。
3. 记录初始为 `manual_required`，不会自动重新投递。

`collection-server` 的 IAM authz version subscriber 目前只使用 terminal log/metric handler，没有写入这张业务 Event 死信表；它不属于本模块的领域事件 consumer。

handler 仍必须区分可重试基础设施错误与已经落为业务终态的失败。若 handler 已保存明确结论并返回 nil，运输层会 ACK。

## 6. 死信人工重放

```text
event_delivery_dead_letter(manual_required)
       │ operator selects org-scoped rows + expected attempts
       ↓
events.replay_delivery（high risk + confirmation）
       │ decode canonical event → RoutingPublisher.Publish
       ├─ publish success → terminal + replayed_at
       └─ decode/publish failure → manual_required + last_error
```

治理动作每次最多接受 100 个 target，并用组织、当前 disposition 和 `expected_delivery_attempts` 做授权与并发冲突检查。它不是自动重试器，也不会修改 payload：decode-invalid 消息即使形成死信，若 payload 不是合法 canonical event，重放仍会失败，需要先修复 producer/contract，再决定数据处置。

## 7. 主消费者与附加消费者

YAML 中的 handler 表示 worker 主消费者。一个事件需要第二个读模型投影时，不在 publisher 前后增加业务 hook，而是在 EventSpec 中登记 `AdditionalConsumers`：

```text
qs.evaluation.lifecycle
   ├─ worker channel → answersheet_submitted_handler → Evaluation 主链
   └─ qs-apiserver-modelcatalog-hot-rank-v1
          → modelcatalog.hot_rank_projection → Redis hot-rank
```

两条 channel 有独立消费进度和失败结算：

- Hot-rank Redis 错误只 NACK projection channel。
- worker 主链不会因 Hot-rank Redis 故障受阻。
- projection 使用 event ID 对应的 Redis processed key 防止重复计数。
- logging/nop 或 MQ 不可用时附加 consumer 显示为 disabled，不通过 publish hook 更新热度。

附加消费者由 apiserver EventSubsystem 在 Start 前冻结 binding，按 Consumer ID 排序启动和关闭。

## 8. 幂等不是统一 ledger

at-least-once 允许以下重复来源：

- MQ 成功但 Outbox mark published 失败。
- immediate 与 relay 竞争同一事件。
- consumer NACK 后 redelivery。
- ACK 成功状态对调用方不可见。
- 人工重放 terminal dead letter。

当前不引入全局 event-id ledger，而是按事件副作用选择业务幂等：

| 策略 token | 主要方法 |
| --- | --- |
| `answersheet-id-lease-and-ensure-assessment` | 以 AnswerSheet/Assessment 业务键租约和 ensure 保护创建流程 |
| `evaluation-run-state-claim` | 通过 Evaluation run state claim 避免重复执行 |
| `report-business-key-run-claim-cas` | 以报告业务键、run claim 和 CAS 保护生成 |
| `report-status-overwrite` | 状态投影允许按业务键覆盖 |
| `repeatable-attention-projection` | attention 投影设计为可重复执行 |
| `terminal-failure-fact` | 写入终态失败事实 |
| `notification-event-metadata` | 通知携带事件 metadata，发送失败不阻断 handler |
| `redis-processed-key-by-event-id` | 以 event ID processed key 去重 Hot-rank 计数 |

best-effort lifecycle 后置动作使用自身发布生命周期策略。具体 token 的权威映射见[领域事件设计](./02-领域事件设计.md)。

只有当某个外部副作用不可逆、现有业务键/CAS/状态机无法保护，并且重复成本明确时，才应为该事件立项 inbox/dedup record。不能为了抽象统一而先引入全局账本。

## 9. 错误边界

| 位置 | 错误处理 |
| --- | --- |
| envelope encode | 发布失败；durable Outbox 保留并重试 |
| MQ publish | best-effort 返回发布错误；durable 由 Store 记录失败后重试 |
| event type unknown at publisher | 不发布，记录 `unknown_event` |
| payload decode in handler | handler error，NACK |
| envelope/event type 无法解析 | NACK；运输预算耗尽后形成持久化死信 |
| 可重试依赖错误 | handler error，NACK |
| 已完成/重复事件 | handler nil，ACK |
| 非关键后置通知失败 | handler 内记录，按事件矩阵决定 ACK |
| dead-letter replay decode/publish 失败 | 回到 `manual_required`，保留最后错误 |

## 10. 可观测入口

发布、Outbox 和消费分别记录：

- `qs_event_publish_total`
- `qs_event_outbox_total`
- `qs_event_consume_total`
- `qs_event_consume_duration_seconds`

label 和 outcome 语义见[事件可观测性与故障处理](./05-事件可观测性与故障处理.md)。日志应带 event type、event ID、topic、aggregate、source/service 和 outcome，便于在 producer、Outbox 与 consumer 之间关联。
