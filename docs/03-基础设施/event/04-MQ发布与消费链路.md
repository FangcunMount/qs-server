# MQ 发布与消费链路

## 1. 解决什么问题

MQ 解决跨进程异步消费问题，让测评执行和报告生成从提交接口中剥离出来。

## 2. 所在位置

MQ 位于 Outbox relay 和 worker 之间。apiserver 发布业务事件，worker 消费后执行测评、报告生成或投影更新。

## 3. 设计目标

解耦提交与计算；支持 worker 独立扩容；支持 ACK / NACK、重试和消费并发控制；消费失败可观测。

## 4. 整体流程

relay 从 Outbox 发布事件到 topic；worker 按 channel 消费；处理成功 ACK，失败 NACK 或重试；消费结果写入业务状态或报告状态。

## 5. 核心数据结构

topic、channel、message_id、event_id、event_type、payload、attempt、ack deadline、consumer group。

## 6. 正常流程

`answersheet.submitted` 进入测评 worker；worker 读取答卷和模型，执行 evaluation；报告 worker 生成报告并推进 report status。

## 7. 异常流程

无法解析 envelope 的 poison 消息 ACK；可解析但没有已注册 handler 的 unknown 消息 warning 后 ACK；handler 返回 error 才 NACK 重试。重复投递由业务幂等处理；长时间积压需要降并发、扩 worker 或暂停上游。

## 8. 幂等 / 降级 / 背压

MQ 消费天然按至少一次处理设计。worker 必须依靠业务状态机、claim、CAS、业务唯一键或锁幂等；当前没有通用 `processed_event` 表、通用 business_key 或统一消费退避策略。

## 9. 可选方案

HTTP 同步调用 worker 会导致提交延迟不可控；本地 goroutine 不支持跨实例和重试；Redis Pub/Sub 不保存消息，不能承担可靠消费。

## 10. 当前方案取舍

MQ 只承担异步投递和消费缓冲，不承担业务事实存储。可靠事实由 DB 和 Outbox 保存；它也不驱动 `behavior_footprint` 的实时投影。

## 11. 观测指标

publish latency、MQ depth、consumer lag、handler duration、ACK / NACK count、retry count、worker concurrency、oldest message age。
