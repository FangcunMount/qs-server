# 事件模块方案取舍

## 1. 为什么需要 Outbox，而不是直接 publish MQ

直接 publish MQ 无法覆盖业务写入成功但消息发布失败的窗口。Outbox 把事件先落在本地事实边界内，再由 relay 重试发布。

## 2. 为什么需要 MQ，而不是 goroutine 异步

goroutine 只能在当前进程内运行，进程退出会丢任务，也无法独立扩容 worker。MQ 提供跨进程异步消费和消费端扩容能力。

## 3. 为什么需要一次性信令，而不是只靠轮询

只靠短轮询会把报告生成耗时线性放大成查询请求。一次性信令让长轮询在报告完成时立即返回，减少无效等待和请求量。

## 4. 为什么 Redis Pub/Sub 不能替代 MQ

Redis Pub/Sub 不保存消息，订阅方离线时会丢。它适合唤醒正在等待的请求，不适合承载 `answersheet.submitted`、`interpretation.report.generated` 这类不能丢的业务事件。

## 5. 为什么事件采用最终一致，而不是分布式事务

测评链路跨 MongoDB、MySQL、MQ、worker 和 Redis。分布式事务成本高、耦合强。最终一致配合 Outbox、幂等和补偿更符合提交短事务和异步执行目标。

## 6. 为什么要建立 Effective Event Registry

`events.yaml` 很适合作为 event type、Topic、delivery 和 handler 的 wire contract，但 owner、Outbox profile、immediate、priority、settlement 和 idempotency 不全是传输层字段。把所有信息继续塞入同一 YAML，会让跨进程契约与 apiserver 私有运行策略混在一起。

因此目标方案是保留 `events.yaml`，另建 `EventSpec`，并在启动时生成只读 Effective Registry。工程化的目标不是字面上只有一个文件，而是每项信息有唯一权威来源，并能产生单一有效视图。

## 7. 为什么 Outbox runtime 应归 EventSubsystem

Mongo domain-event Outbox 同时承载 AnswerSheet 和 Interpretation 事实，它的 relay、ready index 和状态已经不是 Survey 私有能力。由业务模块构造全局 relay 会导致运行时所有权与事件语义 owner 混淆。

目标设计由 EventSubsystem 持有 publisher、Outbox profile、relay、status 和 lifecycle；业务模块仍负责何时产生事件，并通过窄端口在本地事务中 staging。这不改变业务事务所有权。

## 8. 为什么暂不引入统一 event-id ledger

当前可靠主链已经使用业务唯一键、状态机、claim、CAS 和 lease 防止重复推进。统一 ledger 会增加写入、清理、事务和运维成本，但不能自动解决外部副作用的业务幂等。

只对“必须重试、重复不可接受、且业务状态无法去重”的外部副作用评估事件专用 inbox/dedup record。

## 9. 为什么 Signal 只统一治理视图

cache changed signal 和 report-status signal 都有自己的业务 owner 和降级查询逻辑。它们可以与 Domain Event 一起出现在契约治理页，但不应被 EventSubsystem 包装为另一种 Outbox 或 MQ。

选择是：统一契约视图，不统一运行机制和业务所有权。

## 10. 观测指标

Outbox backlog、MQ depth、worker lag、retry count、signal timeout、duplicate event count、report completion latency。
