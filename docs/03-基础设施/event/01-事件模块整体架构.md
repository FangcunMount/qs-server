# 事件模块整体架构

## 1. 解决什么问题

事件模块解决答卷提交后异步测评、报告生成、状态通知和可靠出站问题。提交接口不应该同步执行耗时计算，但业务事实也不能因为 MQ 短暂失败而丢失。

## 2. 所在位置

事件模块位于 Survey / Evaluation / Report 等领域服务和 worker 之间，同时和 report 查询链路的一次性信令协作。

## 3. 设计目标

写入链路短事务；业务事件可靠出站；worker 可独立扩容和重试；重复消费可幂等处理；积压可观测可补偿；临时唤醒不进入可靠事件职责。

## 4. 整体流程

领域服务产生 Domain Event；事件写入 Outbox；relay 扫描或索引待发布事件并发布 MQ；worker 消费事件并 ACK / NACK；worker 更新报告状态后发布一次性信令。

## 5. 核心数据结构

event_id、event_type、aggregate_type、aggregate_id、business_key、payload、version、outbox status、retry_count、next_attempt_at、published_at。

## 6. 正常流程

业务写入和 Outbox 写入处于同一一致性边界。relay 发布成功后更新 Outbox 状态。worker 消费后推进测评或报告状态。

## 7. 异常流程

发布失败后 Outbox 保持待重试；worker 失败后 NACK 或进入重试；重复消息由 event_id / business_key 和状态机处理；信令丢失时客户端继续查询状态。

## 8. 幂等 / 降级 / 背压

worker 必须按事件幂等；relay 和 worker 并发必须有上限；MQ 或 DB 慢时让上游降速；补偿任务必须避免二次风暴。

## 9. 可选方案

同步 HTTP 调 worker 会把耗时压回提交链路；goroutine 异步会丢可靠性；分布式事务成本高；只靠 Redis Pub/Sub 不能保存业务事实。

## 10. 当前方案取舍

采用最终一致事件链路：Outbox 保证可靠出站，MQ 保证异步解耦，worker 保证可重试推进，Signal 只做 best-effort 唤醒。

## 11. 观测指标

Outbox pending / failed、relay publish duration、MQ depth、worker consume duration、ACK / NACK count、retry count、dead letter count、report status transition duration。
