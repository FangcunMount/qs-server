# 事件模块整体架构

## 1. 设计结论

Event 的工程终局不是“业务模块调用 MQ”，而是把契约、可靠投递和运行时所有权分开：

```text
component-base event contract
              ↓
configs/events.yaml + EventSpec
              ↓
       EffectiveRegistry
              ↓
        EventSubsystem
     ┌────────┼─────────┐
     ↓        ↓         ↓
 Publisher  Profiles  Consumers
               ↓
     ready-index / immediate / relay
```

业务模块只表达“发生了什么”，并只依赖发布或事务 staging 的窄端口。EventSubsystem 负责组装、启动、关闭和观测运行时，不把 Store、relay、reconciler、MQ client 暴露给业务模块。

## 2. 分层与职责

| 层 | 位置 | 职责 | 不负责 |
| --- | --- | --- | --- |
| 基础事件契约 | `component-base/pkg/event` | `DomainEvent`、`BaseEvent` | qs-server 事件清单和业务策略 |
| wire codec | `component-base/pkg/eventcodec`、`eventmessaging` | envelope 编解码、MQ message 构造 | 路由、Outbox、handler |
| Contract Plane | `internal/pkg/eventing/catalog` | YAML + EventSpec 合并、校验、不可变查询 | 进程生命周期和存储 |
| Shared Runtime | `internal/pkg/eventing/runtime` | Publisher、subscriber、dispatch/settlement 抽象 | 业务 handler 装配 |
| Observe | `internal/pkg/eventing/observe` | Event/Outbox 指标和 outcome | 决定 ACK/NACK |
| Wire DTO | `internal/pkg/eventing/payload`、`outcome` | 共享 payload 与 outcome JSON 契约 | 业务流程编排 |
| Application ports | `internal/apiserver/application/eventing` | `ProfileBinding`、Stager、PostCommit 等窄端口 | 构造具体 Store |
| Process runtime | `internal/apiserver/eventing/subsystem` | 两个 Outbox profile、附加消费者、status、生命周期 | 关闭外部 DB/Redis/MQ 连接 |
| Infrastructure | Mongo/MySQL/Redis Outbox 包 | 持久化、claim、ready-index | 选择业务事件策略 |
| Worker runtime | `internal/worker/integration/eventing` | 主消费通道、显式 handler registry、结算 | apiserver profile 生命周期 |

依赖方向是单向的：业务模块可以依赖 component-base 契约和 application ports，但不能反向构造 EventSubsystem、relay 或 Store。

## 3. Contract Plane

Event 契约由两类事实组成：

- `configs/events.yaml` 描述稳定 wire 路由：事件类型、topic、delivery、主 handler。
- `EventSpec` 描述 qs-server 工程策略：owner、Outbox profile、immediate、priority、幂等、结算和附加消费者。

`EffectiveRegistry` 启动时合并二者，并提供 `Lookup`、`EventsByProfile`、`ImmediateTypes`、`PriorityTiers`、`Consumers` 和 `Snapshot` 等只读查询。Store、ready-index、immediate、relay、status 都消费同一份派生策略，避免各处维护独立事件切片。

Registry 会拒绝以下配置：

- YAML、代码常量和 EventSpec 事件集合不一致。
- durable event 缺少 profile、priority、幂等或 settlement。
- best-effort event 声明 profile 或 immediate。
- immediate 用于非 durable event。
- 主 handler 不在显式 handler registry 中。
- 附加消费者 ID/channel 不稳定或缺少运行时 binding。

完整契约见[领域事件设计](./02-领域事件设计.md)。

## 4. 唯一 Composition Root

`EventSubsystem` 只在 apiserver resource bootstrap 构造一次。resource stage 负责把以下依赖转成 Subsystem Options：

- MySQL、MongoDB、Ops Redis。
- EffectiveRegistry 与 RoutingPublisher。
- MQ publisher 和 subscriber factory。
- `outbox_relay.mongo.*`、`outbox_relay.assessment.*`。
- `eventing.consumers.*`。
- backpressure 和 observer。

构造完成的实例被注入 Container。Container 不再持有 MQ、Catalog、relay 参数等二次构造能力，而是从同一实例取得：

- `Publisher()`：供 best-effort 发布和其他直接发布入口使用。
- `Profile(profile)`：向业务模块注入 `ProfileBinding`。
- `RegisterConsumer(...)`：在 Start 前绑定附加消费者。
- `StatusService()` 与 `Outboxes()`：供内部治理接口读取。

缺少 EventSubsystem 时生产 Container fail-fast，status 也不再回查 Survey/Evaluation 的兼容出口。

## 5. 两个 Outbox Profile

| Profile | Store | 事件 |
| --- | --- | --- |
| `mongo_domain_events` | Mongo `domain_event_outbox` | `answersheet.submitted`、`interpretation.report.generated`、`interpretation.report.failed` |
| `assessment_mysql_events` | MySQL `domain_event_outbox` | `evaluation.requested`、`evaluation.outcome.committed`、`evaluation.failed` |

每个 profile 只有一个 Store、ready-index、immediate dispatcher、reconciler 和 relay。Survey、Evaluation、Interpretation 不拥有这些运行时，只持有对应的 `ProfileBinding`。

## 6. 生命周期

业务模块初始化期间注册附加 consumer；所有模块装配完成后，process 统一启动 Subsystem。

### Start

顺序固定且可测试：

1. 冻结并校验 consumer bindings，按 ID 排序启动 subscribers。
2. 按 `mongo_domain_events`、`assessment_mysql_events` 顺序启动全部 ready-index reconcilers。
3. 按同样顺序启动全部 relays。

任一阶段失败都会清理由本次 Start 已启动的资源，不进入后续阶段。logging/nop 或无 MQ-backed publisher 时不启动 subscriber 和 relay；durable Outbox 仍保持 pending。

### Close

关闭顺序与启动相反：

1. cancel Subsystem context。
2. 等待 relays 退出。
3. 逆序关闭 reconciler 和 immediate dispatcher。
4. 停止并逆序关闭 subscribers。

单个 subscriber Close 失败不会阻断其他资源清理，错误通过 `errors.Join` 汇总。重复或并发 Close 等待同一次清理并返回稳定结果。Subsystem 不关闭注入的数据库、Redis 或 MQ 连接。

## 7. 业务接入边界

### Durable producer

业务事务内调用 `ProfileBinding.Stager`，把事实和 Outbox 写入同一本地事务；事务成功后调用无返回错误的 `PostCommit.AfterCommit`。这个接口刻意不返回错误，因为提交后错误已经不能回滚业务事实。

### Best-effort producer

生产者直接使用 RoutingPublisher。logging/nop 可以记录或忽略事件，但不会获得持久化补偿，因此不得承载不可丢失的业务事实。

### Consumer

worker 主通道消费 YAML 中每个事件的 handler。一个事件需要独立失败域或独立消费进度时，通过 `AdditionalConsumers` 建立同 topic、独立 channel 的消费者，而不是在 publish hook 中执行业务投影。

当前唯一附加消费者是 `modelcatalog.hot_rank_projection`。它消费 `answersheet.submitted`，Redis 故障只会 NACK 自己的 channel，不影响 worker 的 Evaluation 主链。

## 8. Event 与 Signal 的边界

| 判断 | Event | Signal |
| --- | --- | --- |
| 表达持久业务事实 | 是 | 否 |
| 需要跨故障恢复 | durable event 是 | 否 |
| 需要 ACK/NACK | MQ event 是 | 否 |
| 主要用途 | 异步业务流程、投影 | 缓存失效、刷新唤醒 |
| 运行时所有者 | EventSubsystem / worker | CacheSubsystem 或专属 runtime |

Signal 始终独立于 EventSubsystem。详情见[一次性信令链路](./06-一次性信令链路.md)。

## 9. 设计保证与非目标

当前设计保证：

- durable 事实与 Outbox 共享本地事务。
- MQ 不可用时 durable 事件不被错误标记为 published。
- ready-index 失败不会丢失 Outbox，relay 可退回数据库扫描。
- immediate 失败后 relay 继续兜底。
- poison/unknown/handler failure 的结算行为稳定。
- 运行时、status 和工程契约来自同一个 Registry/Subsystem。

当前明确不提供：

- 跨 Mongo/MySQL 的分布式事务。
- exactly-once 或全局 event-id ledger。
- DLQ、人工 replay 平台和 schema negotiation。
- 用 Event 恢复已删除的 `behavior_footprint` 链路。

这些非目标不能通过文档措辞隐式升级为运行时保证；若业务需要，必须以具体副作用和失败模型单独立项。
