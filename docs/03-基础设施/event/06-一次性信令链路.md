# 一次性信令链路

## 1. Signal 不是 Event

Signal 是 Redis Pub/Sub 上的 best-effort 唤醒消息，用于提示另一个运行时“缓存或状态可能变化，请刷新”。它不表达必须永久保存的业务事实，也不进入 EventSubsystem 或 Outbox。

```text
业务提交/缓存变更
       ↓ best-effort Notify
Redis Pub/Sub channel
       ↓ Watch
缓存失效 / 状态刷新
       ↓
TTL、下一次 Signal 或主动查询继续兜底
```

选择标准：

| 问题 | 使用 |
| --- | --- |
| 消息丢失会导致必要业务流程永久缺失 | Durable Event |
| 只是提醒其他进程丢弃缓存或重新查询 | Signal |
| 轻量通知允许丢失且仍属于事件语义 | Best-effort Event |

## 2. 契约与 transport

- `configs/signals.yaml` 是 Signal 拓扑清单，不由运行时动态加载。
- `internal/pkg/signalcatalog` 保存稳定名称常量，并通过测试与清单同步。
- Cache Signal payload 位于 `internal/pkg/cache/signal`，保持 transport-agnostic。
- Report status Signal contract 位于 `internal/pkg/reportstatus`。
- Redis transport 使用 component-base signaling 包和 `redis.UniversalClient`。
- 顶层配置继续使用 `signaling.redis` 的 enabled、prefix、channel、buffer 等选项。

Signal 名称、SignalKey、JSON payload、prefix/channel 解析属于兼容契约。改变 Go 包所有权不能改变 Redis wire 行为。

## 3. 当前拓扑

| Signal | Delivery | Transport | Publishers | Subscribers | 作用 |
| --- | --- | --- | --- | --- | --- |
| `report_status_changed` | `ephemeral_signal` | `redis_pubsub` | apiserver、worker | collection-server | 唤醒报告状态刷新 |
| `questionnaire_cache_changed` | `ephemeral_signal` | `redis_pubsub` | apiserver | apiserver、collection-server | 清理问卷缓存 |
| `scale_cache_changed` | `ephemeral_signal` | `redis_pubsub` | apiserver | apiserver | 清理量表缓存 |
| `typology_model_cache_changed` | `ephemeral_signal` | `redis_pubsub` | apiserver | apiserver、collection-server | 清理类型模型缓存 |

清单中的 delivery policy 为 `ephemeral_signal`。publisher/subscribers 的顺序不表达优先级，拓扑测试比较前会排序。

## 4. 运行时所有权

### Apiserver

CacheSubsystem 持有：

- component-base Redis Signaler。
- 业务窄接口 `SignalNotifier`。
- Cache watchers。
- Start/Close 和重复启动保护。
- notify attempt/failure 指标。

Survey、ModelCatalog 等业务模块只依赖 notifier 窄接口，不获得具体 Redis client 或 Signaler。Signal disabled 时不创建订阅，也不发送通知。

### Collection-server

collection CacheSubsystem 使用自己的 Redis ops handle 构造 watchers，收到 Signal 后清理对应 L1 cache，并保留既有重连和 warmup 行为。

### Report status

Report status signaling 使用 component-base UniversalClient Signaler，自行管理专属状态刷新语义，不借用 CacheSubsystem 配置对象。

## 5. 失败语义

- Notify 失败只记录 attempt/failure，不返回业务错误。
- Redis Pub/Sub 不保存离线消息；subscriber 断连期间的 Signal 可以丢失。
- Watch decode 失败不能自动修复业务事实，应记录并等待下一次 Signal/TTL/查询。
- subscriber context cancel 后退出；重连属于各进程 runtime 行为。
- Signal 不使用 ACK/NACK、Outbox、relay、ready-index 或 event settlement。

这种 best-effort 是有意设计，不是尚未补齐的 Event 可靠性。缓存正确性必须仍由 TTL、版本、主动读取或下一次变更保证。

## 6. 禁止的耦合

- 不把 Signal 注册进 `configs/events.yaml` 或 EffectiveRegistry。
- 不让 `internal/pkg/cache/signal` 依赖 apiserver、collection-server、options、report status 或 Prometheus。
- 不让业务模块直接持有 Redis Signaler。
- 不使用 durable Outbox 驱动纯缓存失效。
- 不为了“可靠 Signal”复制 EventSubsystem；如果消息不可丢，应重新判断它是否是业务 Event。

## 7. 观测

Cache Signal 保留：

- `qs_cache_signal_notify_total{signal,service}`。
- `qs_cache_signal_notify_failed_total{signal,service}`。

指标反映通知尝试，不证明所有 subscriber 都已经收到。治理页面展示 Signal 拓扑时，也应明确 delivery 为 ephemeral，而非可靠事件。

## 8. 验证

```bash
go test ./internal/pkg/cache/... \
  ./internal/pkg/signalcatalog \
  ./internal/pkg/reportstatus \
  ./internal/apiserver/cache/... \
  ./internal/collection-server/cache/...
```

新增或修改 Signal 时，还必须运行拓扑同步测试，确保 `configs/signals.yaml`、代码常量和真实 publisher/subscriber 一致。
