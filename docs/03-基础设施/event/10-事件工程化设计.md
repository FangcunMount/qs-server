# 事件工程化设计

## 1. 结论

qs-server Event 已收口为“契约中心化、profile 隔离、统一生命周期”的 at-least-once 子系统：

```text
events.yaml + EventSpec
          ↓
EffectiveRegistry
          ↓
EventSubsystem
          ↓
Outbox Profile / Consumer Runtime / Status
          ↓
业务模块只依赖 Publisher、Stager、PostCommitDispatcher
```

现有 Topic、event envelope、Mongo/MySQL `domain_event_outbox` 和业务事务边界未改变。运输结算仍是 poison ACK、unknown ACK、handler failure NACK。`behavior_footprint` 不属于 EventSubsystem，只由 `behavior_journey_scan` 后台扫描重建。

本轮没有引入 exactly-once、统一 event-id ledger、DLQ、replay 或 schema negotiation。

## 2. 与 cache 工程化的关系

Event 借鉴 cache 的结构原则，而不是复制 cache family、L1/L2 或 warmup：

| cache | event |
| --- | --- |
| Capability Spec | EventSpec |
| Effective Registry | EffectiveRegistry |
| Cache Subsystem | EventSubsystem |
| typed cache adapter | Publisher / ProfileBinding / Consumer binding |
| governance status | event/profile/consumer status |

cache 是能力中心化的状态加速子系统；event 是契约中心化的异步投递子系统。

### 2.1 统一命名空间

qs-server 内部的共享 Event 能力统一从 `internal/pkg/eventing` 进入，不再以多个顶层 `internal/pkg/event*` 包平铺：

| 路径 | 责任 |
| --- | --- |
| `internal/pkg/eventing/catalog` | EventSpec、EffectiveRegistry、事件常量和契约校验 |
| `internal/pkg/eventing/runtime` | publisher、subscriber、dispatch result 和 ACK/NACK settlement |
| `internal/pkg/eventing/observe` | 稳定的指标名、label 和 outcome |
| `internal/pkg/eventing/payload` | 跨进程事件 payload wire DTO |
| `internal/pkg/eventing/outcome` | 测评结果 wire DTO |

`component-base/pkg/event`、`eventcodec` 和 `eventmessaging` 提供跨仓库基础契约；qs-server 不保留 `pkg/event` 或本地 codec forwarding package。`internal/pkg/eventing` 也不持有进程生命周期：apiserver EventSubsystem 和 worker consumer 才是各自的 runtime composition root。

## 3. Contract Plane

### 3.1 权威来源

- `configs/events.yaml`：wire contract，定义 event type、Topic、delivery、aggregate、domain 和主 worker handler。
- `internal/pkg/eventing/catalog.EventSpec`：工程契约，定义 owner、Outbox profile、immediate、priority、幂等和 settlement。
- `EffectiveRegistry`：两者合并后的进程内只读有效视图。
- [09-事件契约矩阵.md](09-事件契约矩阵.md)：面向维护者的可读镜像，由同步测试保护。

`signals.yaml` 与 `internal/pkg/signalcatalog` 继续独立治理；Signal 不进入 EventSubsystem 的 Outbox runtime。

### 3.2 启动校验

`EffectiveRegistry` 在启动阶段验证：

- YAML、代码事件常量和 EventSpec 集合严格一致。
- EventSpec owner 与 YAML domain 一致。
- durable event 必须声明 profile、priority、idempotency、settlement。
- best-effort 不得声明 profile、immediate 或 priority。
- immediate 只能用于 durable event。
- additional consumer 的 ID 全局唯一，并声明稳定 channel、runtime、幂等和 settlement。
- 生产 worker catalog 中的主 handler 必须存在于显式 handler registry。
- 契约矩阵必须覆盖所有有效事件，并与 delivery、store、immediate 和 handler 一致。

### 3.3 运行时查询

Registry 提供：

- `Lookup(eventType)`
- `EventsByProfile(profile)`
- `ImmediateTypes(profile)`
- `PriorityTiers(profile)`
- `Consumers(eventType)`
- `Snapshot()`

Store claim、ready-index bucket 和 relay bucket 顺序均由 Registry 派生；旧的 `outboxpriority` 可变全局变量和 `container/internal/outboxruntime` 已删除。

## 4. Runtime Plane

### 4.1 EventSubsystem 所有权

EventSubsystem 在 apiserver resource stage 构造并注入 Container。业务模块完成 consumer binding 后，process 统一调用 `Start`。

Subsystem 拥有：

- Event Catalog、EffectiveRegistry 与 RoutingPublisher。
- `mongo_domain_events` 和 `assessment_mysql_events` 两个 profile。
- 每个 profile 的 Store、ready index、immediate dispatcher、relay、reconciler 和 status reader。
- apiserver 本地 projection consumer 的订阅与生命周期。
- event/profile/consumer 状态装配。

`Start` 顺序为：冻结并校验 consumer binding、启动 projection subscribers、启动 reconciler、启动 relay。启动失败会中止 process runtime stage。`Start` 和 `Close` 均幂等；`Close` 只停止 Subsystem 创建的循环和 subscriber，不关闭 DB 或 Redis 连接。

### 4.2 Outbox Profile

| Profile | 本地事务 | Store | 事件 |
| --- | --- | --- | --- |
| `mongo_domain_events` | MongoDB transaction | Mongo `domain_event_outbox` | `answersheet.submitted`、Interpretation terminal events |
| `assessment_mysql_events` | MySQL transaction | MySQL `domain_event_outbox` | Evaluation requested/outcome/failed |

同一 profile 只构造一个 Store 和一个 relay。Survey 与 Interpretation 共享 Mongo profile；Evaluation 使用 MySQL profile。AnswerSheet Repository 只保存答卷及提交幂等记录，不再构造或代理 Outbox Store。

保留生产配置键：

- `outbox_relay.mongo.*`
- `outbox_relay.assessment.*`

它们只配置 profile 的 interval、batch size、publish workers 和 immediate concurrency，不定义事件类型或 priority。

### 4.3 ProfileBinding

业务模块只能获得窄端口：

```go
type ProfileBinding struct {
    Stager     EventStager
    PostCommit PostCommitDispatcher
}
```

`Stager` 在业务本地事务中写入 Outbox。`PostCommitDispatcher.AfterCommit` 只在事务成功后调用且无返回错误：提交已经完成，ready-index 或 immediate 失败不能伪装成可回滚错误。

所有提交后的事件先 best-effort 写 ready index；Registry 标记为 immediate 的事件再异步尝试 MQ。只有 publisher 明确 `IsMQBacked()==true` 才启用 durable immediate。logging、nop 或 MQ publisher 缺失时，Outbox 保持 pending，由未来 relay 排空。

当前 immediate 事件：

- `answersheet.submitted`
- `evaluation.requested`
- `evaluation.outcome.committed`

## 5. Consumer Plane

### 5.1 主 worker consumer

dispatch 结果与 handler error 已分开：

```go
type DispatchResult struct {
    Outcome DispatchOutcome // handled | unknown
}
```

运输映射保持：

| 输入 | 处理 | 结算 |
| --- | --- | --- |
| envelope 无法解析 | poison | ACK |
| 没有 handler | `DispatchUnknown` / `unknown_acked` | ACK |
| handler 返回 error | handler failure | NACK |
| handler 返回 nil | handled | ACK |

ACK/NACK 自身失败只记录专用 outcome，不能据此宣称消息已被可靠安置。

### 5.2 hot-rank 独立投影

`answersheet.submitted` 有一个附加消费者：

| 属性 | 值 |
| --- | --- |
| Consumer ID | `modelcatalog.hot_rank_projection` |
| Runtime | apiserver EventSubsystem |
| Topic | 与主 worker 相同，由 events.yaml 解析 |
| Channel | `qs-apiserver-modelcatalog-hot-rank-v1` |
| 幂等 | Redis processed key by event ID |
| 失败结算 | Redis/handler error 只 NACK 本 channel |

Survey 的 `BeforePublishHook` 已删除。hot-rank 不再位于 durable publish 前置路径，因此 Redis 故障不会阻断 Evaluation 主 worker channel。重复 event ID 由现有 Redis processed key 防止重复计数。

配置：

```yaml
eventing:
  consumers:
    modelcatalog-hot-rank:
      enabled: true
      channel: "qs-apiserver-modelcatalog-hot-rank-v1"
```

配置默认开启，但只有 MQ-backed publisher 可用时才实际订阅；logging/nop 模式的 status 明确显示 disabled。

## 6. Status 与治理

`GET /internal/v1/events/status` 保留原有 `catalog`、`outboxes` 字段，并增加：

- `events`：owner、delivery、profile、immediate、priority、handler、idempotency、settlement。
- `profiles`：running、relay、reconciler、immediate 状态。
- `consumers`：event、topic、channel、runtime、enabled、healthy、last error、settlement。

系统治理直接读取 EventSubsystem 的 Outbox readers，不再从 Survey/Evaluation module 回查 relay 或 status fallback。

## 7. 幂等决策

当前不建立统一 event-id ledger。每个事件继续使用契约矩阵中列出的业务保护：lease、业务唯一键、run claim、CAS、覆盖写或可重复投影。

只有同时满足以下条件时，才单独评估 inbox/dedup record：

1. 副作用位于外部系统或不可逆边界。
2. handler 失败后必须重试。
3. 重复执行会产生不可接受的业务后果。
4. 业务唯一键、状态机、claim、CAS 或覆盖写无法解决。

## 8. 架构保护

自动测试持续保护：

- YAML、事件常量、EventSpec、worker handler 和契约矩阵同步。
- logging/nop/缺失 MQ 不得将 durable Outbox 标记 published。
- profile-specific priority 不得跨 Mongo/MySQL 污染。
- 业务模块不得构造 Store、relay、reconciler、immediate dispatcher 或业务投影 publish hook。
- AnswerSheet Repository 不得重新代理 Outbox。
- transaction rollback 不调用 post-commit；成功提交每批只通知一次。
- poison ACK、unknown ACK、handler error NACK。
- hot-rank handler failure NACK 独立 channel；重复 event ID 不重复计数。
- status 旧字段保持兼容，新字段来自 EffectiveRegistry 和实际 runtime。

## 9. 明确非目标

- 不改变 Topic、envelope、Mongo collection、MySQL table 或本地事务边界。
- 不恢复 `behavior_footprint` event chain。
- 不引入跨 Mongo/MySQL/MQ 分布式事务或 exactly-once 承诺。
- 不默认增加统一 ledger、DLQ、replay 或 schema negotiation。
- 不将 Redis Pub/Sub Signal 包装成 Outbox event。
