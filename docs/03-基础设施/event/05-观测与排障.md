# 事件观测与排障

**本文回答**：事件系统现在有哪些可观测 outcome，publish/outbox/consume 三段分别如何定位问题，以及出现积压、丢事件疑似、poison message、Ack/Nack 异常时应按什么顺序排查。

## 30 秒结论

| 维度 | 当前事实 |
| ---- | -------- |
| 观测模型 | [`eventobservability.Observer`](../../../internal/pkg/eventobservability/outcomes.go) |
| 默认实现 | Prometheus observer，使用 bounded labels |
| publish outcome | `mq_published`、`mq_failed`、`fallback_logged`、`logged`、`nop`、`unknown_event`、`encode_failed` |
| outbox outcome | `claim_failed`、`published`、`publish_failed`、`mark_failed_failed`、`mark_published_failed` |
| consume outcome | `acked`、`ack_failed`、`nacked`、`nack_failed`、`poison_acked`、`poison_ack_failed` |
| backlog / lag | `qs_event_outbox_backlog`、`qs_event_outbox_oldest_age_seconds`，只看 `pending/failed/publishing` |
| handler 耗时 | `qs_event_consume_duration_seconds`，只记录已解析出 `event_type` 的 dispatch + settlement |
| operating 摘要 | [`GET /internal/v1/events/status`](../../04-接口与运维/10-operating%20事件只读状态接入.md)，只读，不提供 replay / repair |
| 高基数字段 | 不把 `event_id`、`aggregate_id`、error message 放进 metrics labels |

## Prometheus 指标

| 指标 | 类型 | labels | 用途 |
| ---- | ---- | ------ | ---- |
| `qs_event_publish_total` | counter | `source/mode/topic/event_type/outcome` | 发布路径成功、降级、失败计数 |
| `qs_event_outbox_total` | counter | `relay/topic/event_type/outcome` | relay claim/publish/mark 结果计数 |
| `qs_event_consume_total` | counter | `service/topic/event_type/outcome` | worker Ack/Nack/poison 结果计数 |
| `qs_event_consume_duration_seconds` | histogram | `service/topic/event_type/outcome` | worker handler 分发与 settlement 耗时 |
| `qs_event_outbox_backlog` | gauge | `store/status` | 当前未完成 outbox 条数 |
| `qs_event_outbox_oldest_age_seconds` | gauge | `store/status` | 当前最老未完成 outbox 年龄 |
| `qs_event_outbox_status_scrape_total` | counter | `store/outcome` | outbox 状态读取成功/失败 |

当前 outbox gauge 只覆盖 `pending`、`failed`、`publishing`。`published` 历史量通过 `qs_event_outbox_total{outcome="published"}` 观察，不做 DB 全表 count。

## Grafana 与只读状态入口

```mermaid
flowchart LR
    relay["OutboxRelay.DispatchDue"]
    store["MySQL / Mongo outbox"]
    observer["eventobservability.PrometheusObserver"]
    prom["Prometheus"]
    grafana["Grafana trends / alerts"]
    api["GET /internal/v1/events/status"]
    operating["operating read-only summary"]

    relay --> store
    relay --> observer
    observer --> prom
    prom --> grafana
    api --> store
    api --> operating
```

分工固定如下：

- Grafana 看趋势、告警和历史曲线，例如 backlog 是否持续上升、handler p95 是否升高。
- `GET /internal/v1/events/status` 给 operating 展示当前 catalog 摘要和 outbox snapshot。
- status API 是只读接口；不提供 replay、repair、手工 mark、dead-letter 操作。

PromQL 示例：

```promql
sum by (store, status) (qs_event_outbox_backlog)
max by (store, status) (qs_event_outbox_oldest_age_seconds)
histogram_quantile(0.95, sum by (le, topic, event_type) (rate(qs_event_consume_duration_seconds_bucket[5m])))
sum by (topic, event_type, outcome) (increase(qs_event_consume_total{outcome=~"nacked|nack_failed|ack_failed"}[10m]))
```

告警建议：

| 场景 | 建议条件 |
| ---- | -------- |
| outbox pending 积压 | `qs_event_outbox_backlog{status="pending"} > 0` 且 `oldest_age_seconds` 持续超过 relay 周期 |
| outbox failed 积压 | `qs_event_outbox_backlog{status="failed"}` 持续增长 |
| status scrape 失败 | `increase(qs_event_outbox_status_scrape_total{outcome="failure"}[5m]) > 0` |
| worker handler 变慢 | consume duration p95 持续高于业务 SLA |
| Ack/Nack 异常 | `ack_failed/nack_failed` 在短窗口内非零 |

## Publish 排障

| outcome | 含义 | 优先看哪里 |
| ------- | ---- | ---------- |
| `unknown_event` | catalog 找不到 event type | `configs/events.yaml`、`eventcatalog` 测试 |
| `encode_failed` | event payload JSON 编码失败 | 领域事件 data 结构 |
| `mq_failed` | MQ provider 发布失败 | MQ 连接、topic、provider 日志 |
| `fallback_logged` | `PublishModeMQ` 但 publisher nil | apiserver process resource bootstrap |
| `logged` / `nop` | 当前模式不实际发 MQ | 环境、`messaging.*` 配置、publish mode |
| `mq_published` | 已交给 provider | 下一步看 MQ 和 worker |

相关测试：[`publisher_test.go`](../../../internal/pkg/eventruntime/publisher_test.go)。

## Outbox 排障

```mermaid
sequenceDiagram
    participant R as Relay
    participant S as Store
    participant P as Publisher

    R->>S: ClaimDueEvents
    alt claim failed
        R-->>R: claim_failed
    else claim ok
        R->>P: Publish
        alt publish failed
            R->>S: MarkEventFailed
        else publish ok
            R->>S: MarkEventPublished
        end
    end
```

| outcome | 含义 | 处理方向 |
| ------- | ---- | -------- |
| `claim_failed` | store claim due events 失败 | DB 连接、索引、事务、Mongo query |
| `publish_failed` | relay 拿到事件但发布失败 | MQ provider 或 payload 编码 |
| `mark_failed_failed` | 发布失败后标记 failed 又失败 | DB 写路径优先级高 |
| `mark_published_failed` | 发布成功后标记 published 失败 | 可能重复发布，需查事件幂等 |
| `published` | relay 发布并标记完成 | 下一步看 worker |

相关测试：[`outbox_test.go`](../../../internal/apiserver/application/eventing/outbox_test.go)。

## Consume 排障

| outcome | 含义 | 处理方向 |
| ------- | ---- | -------- |
| `poison_acked` | payload 不能解析，已 Ack 避免堆积 | 查 publisher/envelope 兼容 |
| `poison_ack_failed` | 毒消息 Ack 失败 | 查 MQ adapter / connection |
| `acked` | handler 成功且 Ack 成功 | 链路完成 |
| `ack_failed` | handler 成功但 Ack 失败 | 可能重投，检查 handler 幂等 |
| `nacked` | handler 失败并 Nack | 查 handler 和 internal gRPC |
| `nack_failed` | handler 失败且 Nack 失败 | 查 MQ adapter / connection |

相关测试：[`integration/messaging/runtime_test.go`](../../../internal/worker/integration/messaging/runtime_test.go)。

## 常见问题速查

| 现象 | 第一检查点 | 第二检查点 |
| ---- | ---------- | ---------- |
| 新事件 worker 没处理 | `events.yaml` handler 名、`handlers.NewRegistry()` | `Dispatcher.Initialize` 日志和测试 |
| MQ 有积压 | worker `service-name` / concurrency | handler error 与 consume outcome |
| durable 事件没有补发 | outbox 表/集合状态 | relay 是否启动、outbox outcome |
| operating 只读状态为空 | container 是否注入 event catalog/outbox reader | `qs_event_outbox_status_scrape_total` |
| best-effort 事件疑似丢失 | publish outcome | 应用保存后是否调用 direct publish |
| poison message 增多 | producer payload envelope | NSQ envelope / legacy raw payload 兼容 |
| 重复处理 | MQ at-least-once、Ack 失败、mark published 失败 | handler 幂等和业务唯一约束 |
