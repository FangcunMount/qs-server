# 事件可观测性与故障处理

## 1. 观测目标

Event 可观测性必须回答三个不同问题：

1. 契约与运行时是否按预期装配：事件、profile、附加 consumer 是否启用。
2. 可靠出站是否健康：是否积压、最老事件多久、发布或状态迁移是否失败。
3. 消费是否健康：消息最终被 ACK、NACK、识别为 unknown，还是 poison。

这三类问题分别依赖静态契约、运行时 status 和时序指标。任何一个单独来源都不能代表端到端健康。

```text
Contract plane       Runtime snapshot       Time-series evidence
events/specs    +    profiles/consumers  +  publish/outbox/consume
      │                    │                         │
      └────────────────────┴───────────────┬─────────┘
                                           ↓
                              system governance / operator
```

## 2. 指标目录

指标由 `internal/pkg/eventing/observe` 统一定义，名称和 label 是稳定观测契约。

### 发布

| 指标 | Labels | 含义 |
| --- | --- | --- |
| `qs_event_publish_total` | `source,mode,topic,event_type,outcome` | RoutingPublisher 的发布结果 |

Publish outcomes：

- `mq_published`：MQ 发布成功。
- `mq_failed`：MQ 发布失败。
- `encode_failed`：envelope 编码失败。
- `unknown_event`：Registry 无法解析事件路由。
- `fallback_logged`：MQ publisher 缺失时进入日志兜底。
- `logged`：logging 模式记录。
- `nop`：nop 模式忽略。

`fallback_logged` 不是 durable publish 成功。durable immediate/relay 以 `IsMQBacked()` 为门禁，不能因为日志路径返回 nil 就标记 Outbox published。

### Outbox 投递

| 指标 | Labels | 含义 |
| --- | --- | --- |
| `qs_event_outbox_total` | `relay,topic,event_type,outcome` | relay/immediate 的发布与状态迁移结果 |

主要 outcomes：`published`、`publish_failed`、`claim_failed`、`mark_published_failed`、`mark_failed_failed`、`immediate_skipped`。

### 消费

| 指标 | Labels | 含义 |
| --- | --- | --- |
| `qs_event_consume_total` | `service,topic,event_type,outcome` | 消费结算数量 |
| `qs_event_consume_duration_seconds` | `service,topic,event_type,outcome` | 从 dispatch 到结算的处理耗时 |

Consume outcomes 与运输策略一一对应：

- `decode_nacked`、`decode_nack_failed`：envelope/event_type 无法解析，进入有界运输重试。
- `unknown_acked`、`unknown_ack_failed`。
- `acked`、`ack_failed`。
- `nacked`、`nack_failed`。
- `held`、`hold_failed`：automatic retry 暂停后的 durable hold 结算。
- `hold_replayed`、`hold_replay_failed`：hold 恢复发布结果。

重试预算必须分层阅读，不能把数字相加或互相替代：

| 层次 | 硬上限 | 退避 | 耗尽后的状态 |
| --- | ---: | --- | --- |
| Evaluation / Interpretation business attempt | 3 | 30 秒起步、5 分钟封顶 | `manual_required` 或 `terminal` |
| Retry hold / Outbox publish attempt | 30 | 10 秒起步、1 小时封顶、20% jitter | `manual_required` |
| MQ transport delivery attempt | 8 | 30 秒起步、5 分钟封顶、20% jitter | `event_delivery_dead_letter` |

配置只能下调这些预算。automatic retry 开关关闭时，业务重试事件写入 `retry_event_hold` 后 ACK，不消耗运输预算；开关恢复并重启 Worker 后，Hold Replayer 使用原 message UUID/event ID 自动重放。

### Backlog snapshot

| 指标 | Labels | 含义 |
| --- | --- | --- |
| `qs_event_outbox_backlog` | `store,status` | Store/status 维度的积压量 |
| `qs_event_outbox_oldest_age_seconds` | `store,status` | 最老积压年龄 |
| `qs_event_outbox_backlog_by_type` | `store,event_type,status` | 事件类型维度积压量 |
| `qs_event_outbox_oldest_age_by_type_seconds` | `store,event_type,status` | 事件类型维度最老年龄 |
| `qs_event_outbox_status_scrape_total` | `store,outcome` | status snapshot 读取成功/失败 |

Status reporter 对数据库快照做节流，当前不会为每个 HTTP 请求无限制扫描 Outbox。读取失败必须与“backlog 为 0”区分。

## 3. 内部状态接口

### 原始 Event status

```text
GET /internal/v1/events/status
```

该接口只读取 EventSubsystem，并保留历史 `catalog`、`outboxes` 字段，同时输出：

- `events`：owner、delivery、profile、immediate、priority、handler、幂等和 settlement。
- `profiles`：profile 运行状态以及 relay、reconciler、immediate 能力状态。
- `consumers`：附加 consumer 的 event、topic、channel、runtime、enabled、health、last error。

接口受内部组织管理员能力保护。它是当前进程的运行时快照，不是长期监控数据库。

### System governance

```text
GET /internal/v1/system-governance/events?window=5m
```

治理接口把以下证据聚合为面向运维的视图：

- EventSubsystem status snapshot。
- Outbox store/type backlog 与 oldest age。
- Signal 拓扑状态。
- 指定窗口内的 Prometheus 证据。
- 当前组织的最新业务失败、Outbox、retry hold 和运输死信；全局系统事件不会进入组织管理员可重放集合。

默认严重度规则包含：

- Store 状态为 failed 或读取退化：critical。
- 最老 pending 达到 5 分钟：warning。
- 最老 pending 达到 15 分钟：critical。

阈值用于治理展示，不改变 relay 重试或 Store 状态机。

qs-operating-system 的入口为 `/operations/system-governance?tab=events`；旧 `/operations/event-governance` 仅作为重定向入口。前端应展示证据及其时间窗口，不能把缺失数据渲染成健康。

## 4. Status 字段的语义边界

必须避免把“已配置”误读成“进程活着”：

- profile `running` 表示 EventSubsystem 生命周期已经进入运行态，不证明每一次 DB/MQ 操作成功。
- relay/reconciler/immediate enabled 表示对应 runtime 存在并满足启用条件；真实健康仍看失败指标和 backlog 趋势。
- consumer status 当前只覆盖 EventSubsystem 管理的附加消费者，例如 Hot-rank。
- worker 主 channel 不属于 apiserver EventSubsystem，其健康要结合 worker consume 指标、日志和 MQ lag。
- Outbox snapshot 读取失败不是零积压，必须显示 degraded/failed evidence。

因此“全绿”的最低证据是：契约装配正确、运行时启用、status 可读、backlog/age 正常、publish/consume outcome 在观察窗口内无异常趋势。

## 5. 运维诊断路径

| 现象 | 首先检查 | 继续定位 | 常见解释 |
| --- | --- | --- | --- |
| durable backlog 持续增长 | profile/relay enabled、oldest age | `publish_failed`、MQ 可用性 | MQ 故障、relay 未启动、发布持续失败 |
| backlog 只在某事件增长 | by-type backlog/age | 该事件 topic、payload encode、priority | 单一事件编码或路由问题 |
| ready-index 错误 | Redis 日志与 reconciler 状态 | DB polling 是否仍 claim | 加速层退化；不应直接判定事件丢失 |
| `mark_published_failed` 增长 | Store 错误 | consumer 是否收到重复 event ID | MQ 可能已成功，后续会重复发布 |
| `unknown_acked` 增长 | event type、部署版本、YAML/Spec | producer 与 consumer 版本 | 新旧版本契约错位，消息已 ACK |
| `decode_nacked` 增长 | message body/metadata | 非标准 producer 或 codec 变更 | 无法解析的消息正在消耗有界运输预算，耗尽后形成死信 |
| `held` 增长 | automatic retry 开关、`retry_event_hold` | Worker 重启后的 Hold Replayer | 暂停事件已有持久化恢复记录，不应在 MQ 中反复投递 |
| `hold_replay_failed` 增长 | MQ publisher、hold attempt/next_attempt_at | `events.replay_pending` 的 `retry_hold` store | 恢复发布失败；第 30 次后需人工一次授权 |
| `nacked` 持续增长 | event type + handler 日志 | 下游依赖、业务状态、payload | handler 返回可重试错误，MQ 正在重投 |
| `nack_failed` / `ack_failed` | MQ client/broker | channel/连接状态 | handler 结果与 broker 最终结算可能不一致 |
| Hot-rank unhealthy | consumer last error、Redis | 独立 channel lag/redelivery | 只影响投影 channel，不应阻断 Evaluation |

## 6. 建议告警原则

- 以 oldest age 和增长趋势为主，单次 backlog 非零不等于故障。
- `status_scrape` 失败单独告警，避免监控盲区被当成零积压。
- `unknown_acked` 应低容忍；`decode_nacked` 应结合运输 attempt 和死信增长告警。
- `mark_published_failed` 同时提示重复消费风险，消费者幂等是后续保护。
- `nacked` 结合 event type、持续时间和 MQ redelivery/lag 判断，避免瞬时依赖抖动造成噪声。
- logging/nop 环境明确显示 MQ runtime disabled，不生成“relay 没运行”的虚假生产告警。

具体 PromQL 和告警阈值属于部署治理配置，应以 Prometheus scrape interval、业务吞吐和恢复目标校准，不在基础设施文档中硬编码。

## 7. 日志关联字段

生产、Outbox 和消费日志应尽量保留：

- event ID、event type。
- aggregate type、aggregate ID。
- topic、channel/service。
- source、publisher mode。
- store/profile、relay。
- outcome、错误和尝试上下文。

诊断单个事件时，先用 event ID 串联 producer → Outbox → MQ → consumer，再用 aggregate ID 回到业务事实。不要只靠错误字符串搜索。
