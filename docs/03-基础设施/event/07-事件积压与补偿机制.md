# 事件积压与补偿机制

## 1. 解决什么问题

事件积压治理解决 Outbox 堆积、MQ depth 增长、worker 处理慢和 relay 卡住时如何发现、降速、修复和补偿。

## 2. 所在位置

积压治理覆盖 Outbox store、relay、MQ、worker pool、报告生成和治理端观测接口。

## 3. 设计目标

积压可发现；原因可定位；补偿可重复执行；补偿任务不制造二次流量风暴。

## 4. 整体流程

监控 pending / failed / depth / lag；定位 publish、consume、DB 或外部依赖瓶颈；调整并发或暂停上游；对失败事件执行重放或人工补偿。

## 5. 核心数据结构

outbox status、retry_count、next_attempt_at、last_error、claimed_by、claimed_until、topic depth、worker lag。

## 6. 正常流程

短暂积压由 worker 消费追平；relay 按批 claim pending；失败事件按退避重试。

## 7. 异常流程

relay 进程退出后 claim 过期；MQ 长时间积压时扩容 worker 或降上游；毒丸消息进入 failed / dead letter 并人工处理。

## 8. 幂等 / 降级 / 背压

补偿任务必须使用同一幂等规则；重放速率受限；下游慢时先保护 DB、MQ、worker，而不是继续增加补偿并发。

## 9. 可选方案

纯人工修复不可规模化；无限自动重试会扩大故障；直接删除失败事件会丢业务事实。

## 10. 当前方案取舍

采用状态化 Outbox + 可观测积压 + 有界补偿。补偿只能推进未完成事实，不能绕过业务状态机。

## 11. 观测指标

pending count、oldest pending age、failed count、retry count、MQ depth、worker lag、dead letter count、compensation success / failed。
