# 领域事件设计与契约矩阵

## 1. 本文是可执行契约

领域事件表达已经发生的业务事实。事件名称、topic、delivery、handler 和 wire payload 是跨进程契约；owner、Outbox profile、immediate、priority、幂等和 settlement 是 qs-server 的工程契约。

本页的事件矩阵由测试与 `EffectiveRegistry` 逐字段比较。修改表格中的代码 token 时，必须同步修改实现；描述性的 Producer 和 Handler failure behavior 列用于解释业务上下文，不参与路由。

事实来源优先级：

1. `component-base/pkg/event`、`eventcodec`、`eventmessaging`：基础接口和 envelope。
2. `configs/events.yaml`：topic、事件类型、delivery、主 handler。
3. `internal/pkg/eventing/catalog`：EventSpec、附加消费者和 Registry 校验。
4. 本文矩阵：面向维护者的完整投影。

## 2. 契约如何合并

```text
configs/events.yaml                    DefaultSpecs()
event → topic/delivery/handler         owner/profile/immediate/priority
                           \           idempotency/settlement/consumers
                            \         /
                         EffectiveRegistry
                                  ↓
             publisher / stores / ready-index / relay
                consumers / status / contract tests
```

YAML 保留 wire schema，Signal 继续由独立的 `configs/signals.yaml` 治理。工程策略不塞入 YAML，避免部署配置决定事务和幂等语义；运行时也不维护脱离 Registry 的硬编码事件切片。

## 3. DomainEvent 与 envelope

领域对象产生的事件实现 component-base `event.DomainEvent`。基础字段包括：

- Event ID：一次事件事实的全局标识，也是 MQ message UUID 和消费幂等的重要输入。
- Event Type：稳定 wire 名称，例如 `evaluation.requested`。
- Occurred At：事实发生时间，不是 MQ 发布时间。
- Aggregate Type / ID：事件所属聚合及业务标识。
- Data：事件 payload。

MQ envelope 的 JSON 字段固定为：

```json
{
  "id": "event-id",
  "eventType": "evaluation.requested",
  "occurredAt": "2026-01-01T00:00:00Z",
  "aggregateType": "assessment",
  "aggregateID": "assessment-id",
  "data": {}
}
```

MQ message 同时携带 `event_type`、`aggregate_type`、`aggregate_id`、`occurred_at` 和 `source` metadata。消费者优先从 metadata 识别事件类型，缺失时才尝试从 envelope 提取；这不改变 envelope 本身的兼容要求。

事件类型、字段名和 payload JSON 都是 wire contract。Go 类型、文件路径和内部函数名不是 wire contract，但移动或重构时必须保留表征测试。

## 4. Delivery 语义

| Delivery | 生产方式 | 持久化 | 适用条件 |
| --- | --- | --- | --- |
| `durable_outbox` | 业务事实与 Outbox 同一本地事务 | Mongo 或 MySQL Outbox | 事件丢失会造成业务状态或必要投影不完整 |
| `best_effort` | 提交后或生命周期后置动作直接发布 | 无 | 允许因进程/MQ 故障丢失，且有其他事实来源或无需恢复 |

`immediate` 不是第三种 delivery。它只表示 durable event 在事务提交后可以异步尝试一次 MQ 快速投递；失败时 Outbox 仍由 relay 处理。

### Additional consumers

主 handler 由 worker 主 channel 消费。需要独立投递进度和独立失败域的投影，登记为附加消费者：

| Consumer ID | Event | Runtime | Topic | Channel | Idempotency policy | Settlement policy |
| --- | --- | --- | --- | --- | --- | --- |
| `modelcatalog.hot_rank_projection` | `answersheet.submitted` | `apiserver` | `qs.evaluation.lifecycle` | `qs-apiserver-modelcatalog-hot-rank-v1` | `redis-processed-key-by-event-id` | `handler_error_nack` |

Hot-rank 与 `answersheet_submitted_handler` 使用同一个 topic、不同 channel。Redis 错误只使 projection channel NACK；主 worker channel 的 Evaluation 链路不受影响。

## 6. Topic 拓扑

| Catalog topic ID | MQ topic | 事件 |
| --- | --- | --- |
| `questionnaire-lifecycle` | `qs.survey.lifecycle` | `questionnaire.changed`、`assessment_model.changed` |
| `assessment-lifecycle` | `qs.evaluation.lifecycle` | 答卷、Evaluation、Interpretation 六个 durable event |
| `task-lifecycle` | `qs.plan.task` | 四个 task best-effort event |

Topic 是 wire contract。事件 owner 或代码目录变化不能顺带改 Topic；任何 Topic 迁移都需要独立的生产者/消费者兼容方案。

## 7. Priority、Immediate 与 settlement

- `p0`：优先处理的入口事件，目前是 `answersheet.submitted`、`evaluation.requested`。
- `p1`：后续业务结果和终态事实。
- `p2`：Registry 支持的兜底层级，当前没有事件声明为 p2。

Priority 只影响 Outbox claim 顺序，不改变业务重要性、重试次数或 ACK/NACK。ready-index bucket 和 Store claim tier 都从 Registry 派生。

所有当前事件使用 `handler_error_nack`。poison 和 unknown 不是逐事件策略，而是全局运输策略：无法解析的 poison message ACK，未注册事件 ACK；已识别 handler 返回错误时 NACK。详见[MQ 发布与消费链路](./04-MQ发布与消费链路.md)。

## 8. Payload 演进规则

1. 优先新增可选字段，消费者必须容忍暂未使用的字段。
2. 不修改既有字段的 JSON 名称、类型或语义。
3. 不复用已删除的事件名表达新事实。
4. 需要破坏性变化时新增事件类型，并设计双写、双读和退场窗口。
5. payload DTO 与领域模型分离；内部模型重构不能自动改变 wire JSON。
6. 时间、ID、枚举和空值语义必须通过 JSON 表征测试固定。

共享 payload 位于 `internal/pkg/eventing/payload`，Interpretation terminal outcome 位于 `internal/pkg/eventing/outcome`。这些包不依赖进程级 runtime。

## 9. 已删除的历史链路

`behavior_footprint` 已从事件清单、EventSpec、handler、Outbox 和配置中删除。当前行为足迹由 `behavior_journey_scan` 后台扫描器读取 entry、answersheet、assessment、report 等事实数据构建。

因此不得：

- 新增 footprint Topic 或 event type。
- 把 footprint 写入 durable Outbox。
- 把后台扫描补偿误写成事件消费。

如果扫描器设计需要调整，应在行为足迹模块讨论，而不是扩展本事件矩阵。
