# 领域事件设计

## 1. 解决什么问题

领域事件用于表达已经发生的业务事实，避免把“希望做某事”的命令和“某事已经发生”的事实混在一起。

## 2. 所在位置

领域事件由 Survey、Evaluation、Interpretation、Plan 等领域或应用服务产生。只有 `durable_outbox` 事件进入 Outbox；`best_effort` 事件直接投递 MQ。行为轨迹不由领域事件投影，而由后台扫描器重建。

## 3. 设计目标

事件命名清晰；payload 可版本化；事件和数据库状态可关联；消费者可以幂等；事件目录可被治理端读取。

## 4. 整体流程

业务状态变更后创建 event；delivery 决定它进入 Outbox 或直接发布。worker 根据 event_type 调用 handler。完整事件矩阵见 [09-事件契约矩阵.md](09-事件契约矩阵.md)。

## 5. 核心数据结构

`id`、`eventType`、`occurredAt`、`aggregateType`、`aggregateID`、`data`。当前 envelope 没有通用 `event_version`、`business_key` 或 `trace_id`。

## 6. 正常流程

例如 `answersheet.submitted` 表示答卷已经提交；`evaluation.outcome.committed` 表示评分事实已可靠提交；`interpretation.report.generated` 表示报告成品已生成。

## 7. 异常流程

无法解析 envelope 的 poison 消息 ACK；可解析但无 handler 的 unknown 消息 warning 后 ACK；已找到 handler 的错误才 NACK。事件重放由业务状态机、claim、CAS 或业务键幂等，而非通用运行时去重表。

## 8. 幂等 / 降级 / 背压

事件处理以 event_id 和 business_key 幂等；投影可延迟，但不能修改原业务事实；消费积压时降低并发或启用补偿。

## 9. 可选方案

直接调用下游服务会耦合链路；把事件写成命令会导致消费者职责不清；只存数据库状态会失去异步推进和审计能力。

## 10. 当前方案取舍

事件表达事实，不表达命令。命令由调用方发起，事件由事实源产生，消费者只对事实做投影或后续推进。

## 11. 观测指标

event type count、invalid payload count、unknown event count、consumer lag、projection delay、event version distribution。
