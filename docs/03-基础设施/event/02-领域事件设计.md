# 领域事件设计

## 1. 解决什么问题

领域事件用于表达已经发生的业务事实，避免把“希望做某事”的命令和“某事已经发生”的事实混在一起。

## 2. 所在位置

领域事件由 Survey、Evaluation、Report、Plan、Statistics 等领域或应用服务产生，再进入 Outbox 和 MQ。

## 3. 设计目标

事件命名清晰；payload 可版本化；事件和数据库状态可关联；消费者可以幂等；事件目录可被治理端读取。

## 4. 整体流程

业务状态变更后创建 event，写入 Outbox，发布到对应 topic，worker 或投影消费者根据 event_type 处理。

## 5. 核心数据结构

event_id、event_type、event_version、occurred_at、aggregate_id、business_key、payload、trace_id。

## 6. 正常流程

例如 `answersheet.submitted` 表示答卷已经提交；`assessment.interpreted` 表示测评解释已完成；`report.generated` 表示报告已生成。

## 7. 异常流程

payload 版本不兼容时消费者拒绝并记录；未知事件类型不应被静默吞掉；事件重放时消费者依赖幂等键。

## 8. 幂等 / 降级 / 背压

事件处理以 event_id 和 business_key 幂等；投影可延迟，但不能修改原业务事实；消费积压时降低并发或启用补偿。

## 9. 可选方案

直接调用下游服务会耦合链路；把事件写成命令会导致消费者职责不清；只存数据库状态会失去异步推进和审计能力。

## 10. 当前方案取舍

事件表达事实，不表达命令。命令由调用方发起，事件由事实源产生，消费者只对事实做投影或后续推进。

## 11. 观测指标

event type count、invalid payload count、unknown event count、consumer lag、projection delay、event version distribution。
