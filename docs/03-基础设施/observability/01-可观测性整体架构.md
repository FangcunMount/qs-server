# 可观测性整体架构

## 1. 解决什么问题

可观测性整体架构让高并发保护、事件链路和缓存治理能够被验证，而不是只停留在设计说明。

## 2. 所在位置

HTTP handler、应用服务、cache store、Outbox relay、MQ worker、DB 访问和 report wait 都需要埋点。

## 3. 设计目标

请求可追踪；事件可追踪；状态可聚合；异常可告警；压测结果可复盘。

## 4. 整体流程

request_id / trace_id 进入接口后贯穿日志；业务 ID 进入事件和 report 状态；指标按 cache、queue、outbox、worker、DB、report 分类暴露。

## 5. 核心字段

request_id、trace_id、answer_sheet_id、assessment_id、report_id、event_id、outbox_id、tenant_id、user_id。

## 6. 异常流程

当报告生成慢时，先看 report wait timeout 和 worker lag，再看 Outbox pending 和 MQ depth，最后看 DB / Redis / 下游耗时。

## 7. 观测指标

QPS、P95/P99、error rate、cache hit rate、queue depth、Outbox pending、MQ depth、worker lag、report generation duration。
