# 基础设施设计原则

这些原则用于设计和 review，不表示当前代码已经全部满足。已知例外必须在对应专题写成“当前风险”，不能用原则文本覆盖实现事实。

## 1. 业务事实优先

只有权威存储提交成功，系统才能承认业务事实。Cache、Redis lock、Signal、metrics 和日志都不能伪造成功。

例：AnswerSheet `202 Accepted` 依赖 Mongo AnswerSheet + Outbox durable commit；SubmitGuard acquired 不构成 202 依据。

## 2. 正确性机制靠近事实源

跨实例竞态最终由唯一键、transaction、claim/CAS、状态前置条件和 fingerprint 收敛。入口 guard 用于降低重复工作，不承担最终正确性。

这意味着 Redis 完全不可用时，低流量下仍可让请求到 DB 判定；高流量是否继续放行则由独立容量保护决定，而不是通过取消第一次业务事实解决。

## 3. 投递语义必须显式

每种传播只能属于一种主要语义：

- `durable_outbox`：本地事实与 Outbox 同事务。
- `best_effort`：允许 producer 侧丢失。
- `signal`：可丢失唤醒/驱逐。

MQ 的 bounded at-least-once、consumer ACK/NACK 和 dead letter 是另一层，不能与 producer delivery 混写。

## 4. 事务只承诺本地边界

MySQL transaction、Mongo session transaction 各自原子；系统没有 MySQL+Mongo+Redis+MQ 的分布式事务。跨存储流程用 durable intermediate state、Outbox、幂等 consumer 和 checkpoint 收敛。

“先写 A，再写 B”不是原子性；“调用成功返回”也不等于响应一定到达客户端。

## 5. 容量机制各管一个量

- RateLimit：单位时间到达量与 burst。
- Gate：一个进程的在途请求。
- gRPC inflight：一个 client process 的下游 RPC 在途量。
- Backpressure：一个 process 对某依赖的在途操作。
- connection pool：driver 可复用连接与等待。
- DB capacity：整个数据库可承载的 workload。

不同量不能直接换算。单实例 80 QPS、三个实例也不能直接证明全局 240 QPS，因为请求分布、依赖服务时间、事务连接占用、重试和热点都可能不同。

## 6. 降级必须保留语义

降级可牺牲性能、新鲜度或部分非核心功能，不能：

- 把依赖不可用当作“无数据”。
- 把权限不可确认当作允许。
- 把 Outbox/MQ 失败当作业务已完成。
- 用缓存旧值扩大权限或跨组织访问。

每个 dependency 要声明 fail-open、fail-closed、fallback 和恢复条件。切换与恢复需要迟滞、半开探测和渐进放量时，应在状态机、指标和测试中实现，而不是只写在文档。

## 7. Scope 与所有权单一

每个预算、锁、consumer channel、checkpoint、cache capability 和治理 action 都要回答：

- owner 是哪个进程/模块？
- scope 是 request、user、org、instance 还是 global？
- key 如何构造，何时释放/过期？
- 多实例是否共享？
- 谁能恢复，如何防重复？

“每实例 local”与“Redis global”必须明确区分。

## 8. 安全是连续证明链

TLS/mTLS、JWT、principal、tenant domain、QS org、capability、resource ownership 缺一不可。新增接口先定义 actor 和 scope，再写 handler。

配置禁用认证不应成为匿名成功路径。当前部分 HTTP/gRPC 装配仍有 fail-open 风险，目标应是 production fail-fast/fail-closed。

## 9. Unknown 不能伪装为 Zero

Prometheus 不可用、远程 status timeout、read model 未同步都应保留 `unavailable/unknown`。指标为零只在 source 可用且查询成功时成立。

同理，静态 liveness 200 只能证明 handler 可响应，不能被解释为数据库和 MQ 健康。

## 10. 运行时资源必须可关闭

所有 client、listener、subscriber、publisher、scheduler 和 goroutine 都需要唯一 lifecycle owner：

```text
construct → initialize → start → stop accepting → drain → close
```

启动部分失败要清理已创建资源；关闭先停入口/新消息，再停后台任务，最后释放下游。deadline 与 fallback 必须明确。

当前 apiserver shutdown 顺序、gRPC 无界 GracefulStop、worker 双信号监听属于需要收敛的例外。

## 11. 可治理不等于可随意修改

治理 action 必须具备：

- 组织/capability 授权。
- `confirm=true` 与风险级别。
- `(org_id, request_id)` 幂等 claim。
- 目标状态/attempt 冲突检查。
- 脱敏审计、结果回放和失败恢复。

人工 replay 是有状态恢复协议，不是“再发一次消息”按钮。

## 12. 可替换通过 Port，不通过万能抽象

业务模块依赖 repository/read-model/transaction/cache capability/event publisher 等窄 port，不依赖 Redis/GORM/NSQ client。infra adapter 负责驱动语义和错误翻译。

但不要为了“可替换”制造一个跨所有存储的万能 repository 或 transaction；抽象必须保留必要的唯一键、claim、session 和查询能力。

## 13. 验证必须覆盖失败

基础设施验收至少包括：

- contract/unit：key、policy、route、状态机、错误映射。
- integration：真实 MySQL/Mongo/Redis/MQ transaction 与故障。
- multi-instance：并发、重复、lease、consumer channel。
- load：QPS、并发、pool、尾延迟、重试放大。
- operation：SIGTERM drain、dead letter/replay、migration/rollback。
- security：跨组织/越权、disabled/misconfigured startup、端点暴露。

编译成功、mock 通过、一次 happy path 或 health 200 都不是完整证明。
