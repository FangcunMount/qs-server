# 基础设施核心链路全景

本页把各专题放回用户可见链路。每条链都要区分入口保护、协调优化、最终事实与恢复证据。

## 1. HTTP 安全前置链

apiserver protected route：

```text
TLS / network
  → JWT verify
  → User principal
  → tenant_domain
  → active operator OrgScope
  → IAM authorization snapshot
  → route capability
  → application resource ownership
```

collection 报告/测评查询在 JWT/snapshot 后，还要通过 IAM User → Testee active ProfileLink 证明资源归属。认证成功不等于有能力，能力允许也不等于拥有某条 Assessment/Report。

当前 IAM disabled/verifier missing 时部分 route group 会不安装认证 middleware；这是装配风险，不是推荐链路。详见 [Security](./security/README.md)。

## 2. AnswerSheet 可靠提交链

```text
Client
  → collection Submit Gate（单实例在途）
  → collection global/user RateLimit（Redis 或本地）
  → application validation / actor
  → Redis SubmitCoalescer（建议性）
  → collection gRPC inflight Gate
  → Mongo Backpressure
  → Mongo transaction:
       AnswerSheet
       + (writer_id, idempotency_key, fingerprint)
       + answersheet.submitted Outbox
  → commit
  → 202 + durable answersheet_id + current request_id
```

状态语义：

| 情况 | 结果 |
| --- | --- |
| 同 key、相同规范化内容 | 回读并返回同一 AnswerSheet；`request_id` 仍标识当前这次 HTTP 请求 |
| 同 key、不同内容 | 409 conflict |
| Gate/RateLimit 饱和 | 429 或对应准入错误 |
| Mongo/transaction 不可确认 | 不返回 durable 202 |
| Redis SubmitCoalescer 争用 | contender 有界等待 completion signal，再由 Mongo durable lookup 裁决 |
| Redis SubmitCoalescer 故障 | degraded-open，继续 durable accept，由 Mongo 最终收敛 |

因此相同业务意图的第 n 次请求不能取消第 1 次事实，也没必要返回“你重复了”。完整推导见 [可靠提交：跨实例合并与幂等](./concurrency/30-可靠提交-跨实例合并与幂等.md)。

## 3. AnswerSheet 到 Assessment/Report 的异步链

```text
Mongo transaction committed
  → post-commit ready hint / immediate
  → Mongo Outbox relay
  → MQ answersheet.submitted
  → worker decode + dispatch
  → apiserver internal gRPC
  → Assessment intake idempotency / MySQL facts + Outbox
  → evaluation.requested
  → worker evaluation
  → Outcome commit + Outbox
  → interpretation worker
  → Report lifecycle
```

可靠性分层：

- 202 只承诺 AnswerSheet + Outbox durable，不承诺报告已经生成。
- ready-index/immediate 是加速，DB relay 能恢复发现。
- MQ 是 at-least-once，handler 必须幂等。
- invalid payload/handler error NACK，unknown event ACK。
- delivery attempt 有界；耗尽写 `event_delivery_dead_letter`。
- automatic business retry 可进入 retry hold；Outbox exhaustion 进入 `manual_required`。
- 人工 `events.replay_delivery`/`events.replay_pending` 必须带组织范围、request ID、reason 和状态冲突检查。

详见 [Event](./event/README.md)。

## 4. Cache-Aside 读链

```text
application query
  → capability Policy
  → L1（若该 owner 有）
  → Redis L2
  → negative sentinel / object decode
  → singleflight/loadguard
  → DB/read model
  → write cache with TTL+jitter
  → return domain/read DTO
```

写后链：

```text
business write commit
  → local invalidate
  → best-effort signal
  → remote L1 eviction / warmup hint
  → TTL final convergence
```

Cache 不参加业务事务。signal 丢失可能延迟新鲜度，但不能丢业务事实；权限和资源归属查询不能因 cache failure 扩大访问。详见 [Cache](./cache/README.md)。

## 5. Report status / wait 链

```text
Client GET/wait
  → collection Testee access proof
  → assessment ownership
  → Redis report_status
  → miss/error: apiserver/DB fact fallback
  → optional local waiter registration

worker status change
  → write operational report_status
  → Redis Pub/Sub report_status_changed
  → wake waiter
  → waiter re-read status fact
```

Signal 只减少等待延迟。丢 signal、断线或超时后必须回查；收到 signal 也不能直接把报告判为完成。

## 6. 下游容量保护链

```text
arrival QPS
  → RateLimit
  → process-local Gate
  → client inflight Gate
  → dependency Backpressure
  → driver pool
  → DB/IAM/MQ service time
```

吞吐近似受 `concurrency / service_time` 约束，但一次请求可能执行多次操作、事务持有连接、请求分布不均且重试会放大负载。实例数与单实例阈值不能直接相乘得出安全 QPS。

Redis rate backend 运行期故障时，collection submit 会使用每实例 30 QPS global、10 QPS user 的本地保守预算；其他 budget 仍 degraded-open。半开与渐进恢复仍是规划。详见 [Concurrency / Resilience](./concurrency/README.md)。

## 7. Scheduler / Lease / Claim 链

```text
periodic trigger
  → optional Redis leader/lease
  → DB claim / state condition
  → bounded work
  → checkpoint / outcome
  → release or expiry
```

Redis lease 决定“谁先做”，DB claim/fencing 决定“谁的结果可被接受”。续租失败只会取消协作式 body；如果写入端不校验 owner/token/state，旧持有者仍可能提交结果。

## 8. System Governance 动作链

```text
admin request + OrgScope + capability
  → action registry / enabled / confirm
  → MySQL claim(org_id, request_id)
  → target state and attempt conflict check
  → execute cache/resilience/event action
  → persist terminal result
       ├─ MySQL success
       └─ Redis terminal fallback → recovery runner → MySQL
  → same request ID replays same result
```

如果 terminal audit 也无法持久化，action 可能已经生效但响应为 500。此时必须固定原 request ID 查目标状态，不能更换 key 盲重试。

## 9. 进程启动与关闭链

```text
config/flags/env
  → raw schema / unmarshal / complete / validate
  → resources
  → container/modules
  → integrations
  → transports/runtime
  → shutdown callback
  → serve/consume
```

SIGTERM 正常路径与 listener/stage failure 路径不同。当前 apiserver 关闭时 transport 最后停止，worker 有双信号监听，gRPC GracefulStop 无 timeout；不能仅凭“注册了 shutdown hook”断言 drain 安全。详见 [Runtime](./runtime/README.md)。

## 10. 从现象到证据

| 现象 | 第一观察面 | 最终证据 |
| --- | --- | --- |
| submit 429 增长 | gate/rate metrics、instance load | 无业务事实产生；检查客户端退避 |
| 相同 key 返回冲突 | request logs/fingerprint category | Mongo idempotency record + AnswerSheet |
| 202 后长期无 Assessment | Outbox backlog、consume NACK | Mongo Outbox、dead letter/hold、Assessment intake state |
| cache hit 降低 | cache/Redis family metrics | DB/read model 仍是事实 |
| `/readyz` 200 但请求失败 | runtime snapshot | 对应 DB/IAM/MQ/业务状态；ready 覆盖范围有限 |
| replay 500 | action audit fallback metrics | MySQL action run、Redis fallback、目标状态 |
| SIGTERM 后请求错误 | shutdown logs/inflight | listener/drain/DB close 的真实顺序 |

Observability 用来缩小范围，Data Access/Event 持久状态用来下最终结论。完整方法见 [Observability](./observability/README.md)。
