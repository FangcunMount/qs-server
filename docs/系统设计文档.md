# qs-server 系统设计文档

## 1. 项目定位

`qs-server` 不是一个单纯的问卷系统，而是一个面向多类测评模型的**测评后端平台**。

它的核心目标是：把一次用户作答，稳定地转化为可计算、可追踪、可解释、可查询的测评结果与解释报告。

系统围绕以下能力建设：

```text
采集作答事实
管理测评模型资产
执行测评计算
沉淀测评执行结果
生成解释报告
支撑周期测评、筛查、统计和前台收集入口
```

从业务语义上看，主链路是：

```text
Actor / Plan
    ↓
Survey
    ↓
ModelCatalog
    ↓
Calculation
    ↓
Evaluation
    ↓
Interpretation
    ↓
Statistics
```

其中各层边界如下：

```text
Survey 保存“答了什么”；
ModelCatalog 管理“用什么模型、规则和版本”；
Calculation 执行“怎么算”；
Evaluation 管理“一次测评如何运行、运行到什么状态、结果是什么”；
Interpretation 负责“结果如何解释成报告”；
Statistics 消费事件形成读侧聚合；
Collection Server 承载前台入口与入口保护；
Worker 承载异步事件执行。
```

更底层地说：

```text
模型结构归 ModelCatalog；
计算执行归 Calculation；
执行状态归 Evaluation；
报告表达归 Interpretation；
调用编排归 Application。
```

这是后续重构时最重要的边界。

## 2. 总体设计原则

### 2.1 主状态收口

主业务写模型、领域规则、持久化和可靠事件发布统一收口在 `qs-apiserver`。

`collection-server` 和 `qs-worker` 都不维护第二套主业务模型。

```text
collection-server 负责入口治理；
qs-worker 负责异步推进；
qs-apiserver 负责业务事实。
```

### 2.2 作答、模型、计算、执行、解释分离

系统不能把“问卷”“量表”“报告”混成一个大模型。

应该拆成：

```text
作答事实：Survey
模型资产：ModelCatalog
计算内核：Calculation
执行状态：Evaluation
报告解释：Interpretation
```

这样才能同时支撑医学量表、人格测评、行为能力、认知任务、筛查任务等不同类型测评。

### 2.3 模型驱动，而不是测评类型驱动

长期不应该通过大量类似下面的分支组织主链路：

```text
if kind == scale
if kind == personality
if algorithm == brief2
if algorithm == mbti
```

更理想的选择逻辑是：

```text
PublishedModelSnapshot
    ↓
AlgorithmFamily / PayloadFormat / DecisionKind
    ↓
RuntimeDescriptorRegistry
    ↓
EvaluationPipeline
```

具体测评 code 只应该长期存在于：

```text
AlgorithmCode / ModelCode 等模型资产数据值
模型配置
seed 数据
payload
fixture
migration
characterization test
```

不应该成为长期代码组织单元。

### 2.4 可靠事件与临时信令分离

系统中存在三种不同语义：

```text
durable_outbox
    可靠业务事件，必须可追踪、可重试、可补偿。

best_effort
    尽力投递事件，允许丢失，不能承载关键业务事实。

ephemeral_signal
    临时唤醒信令，例如 Redis Pub/Sub，只提升及时性，不参与正确性证明。
```

Outbox、MQ、Signal 不是替代关系：

```text
Outbox 解决业务写入成功但消息发布失败的问题；
MQ 解决跨进程异步消费问题；
Signal 解决在线等待请求的临时唤醒问题。
```

## 3. 运行时架构

`qs-server` 保持三进程形态：

```text
cmd/
├── qs-apiserver
├── collection-server
└── qs-worker
```

### 3.1 qs-apiserver

`qs-apiserver` 是主业务与状态中心。

它拥有：

```text
领域模型
应用服务
仓储接口
持久化适配
REST / gRPC 接口
internal gRPC
事件发布
Outbox relay
模型资产治理
测评执行编排
报告生成
统计投影入口
```

它是唯一的主业务写模型中心。

典型职责：

```text
保存 AnswerSheet；
创建 Assessment；
执行 Evaluation；
生成 InterpretReport；
发布 durable_outbox 事件；
维护模型资产、计划任务、统计入口。
```

### 3.2 collection-server

`collection-server` 是前台 BFF 与入口保护层。

它负责：

```text
小程序 / 前台接口
JWT / 身份前置
监护关系校验
入口参数治理
SubmitQueue 削峰
request_id 状态查询
idempotency_key 入口重复抑制
DTO 转换
调用 qs-apiserver gRPC
report-status / wait-report 查询
```

它不负责：

```text
不拥有 AnswerSheet 主写模型；
不拥有 Assessment 主写模型；
不拥有 InterpretReport 主写模型；
不发布主业务 durable event；
不替代 apiserver 仓储。
```

collection-server 中的 `request_id` 与 `idempotency_key` 需要区分：

```text
request_id
    collection 本地提交状态 key；
    用于查询 queued / processing / done / failed；
    生命周期受 collection 进程内存和 TTL 影响；
    不等于业务 durable 幂等键。

idempotency_key
    客户端或业务侧提交幂等 key；
    用于跨实例重复提交抑制；
    当前可由 collection SubmitGuard 使用 Redis lock + done marker 保护；
    但不能替代 apiserver 侧最终业务唯一约束。

SubmitGuard
    collection 入口侧保护；
    解决重复提交、并发提交、跨实例抢占问题；
    是入口治理，不是最终业务事实。
```

### 3.3 qs-worker

`qs-worker` 是异步事件执行器。

它负责：

```text
订阅 MQ；
分发 event handler；
使用 Redis lock / lease 做执行保护；
处理失败重试；
调用 apiserver internal gRPC；
触发报告后处理、统计投影、通知等副作用。
```

它不是第二套业务服务，不维护第二套领域聚合。

worker 可以驱动流程，但业务规则和持久化权威仍然在 `qs-apiserver`。

## 4. 核心业务链路

一次前台答卷提交的主链路如下：

```text
Client
  ↓
collection-server REST
  ↓
SubmitQueue / SubmitGuard
  ↓
qs-apiserver gRPC
  ↓
Survey 保存 AnswerSheet
  ↓
Outbox stage answersheet.submitted
  ↓
Outbox Relay 发布 MQ
  ↓
qs-worker 消费 answersheet.submitted
  ↓
apiserver internal gRPC
  ↓
创建 Assessment / 执行 Evaluation
  ↓
生成 AssessmentOutcome
  ↓
生成 InterpretReport
  ↓
写 report status cache
  ↓
发布 report.generated / report_status_changed
  ↓
collection wait-report 被唤醒或客户端按 next_poll_after_ms 查询
```

### 4.1 提交入口

collection-server 接收前台提交请求后：

```text
1. 校验身份和参数；
2. 生成或读取 request_id；
3. 进入 SubmitQueue；
4. 使用 idempotency_key / SubmitGuard 做入口重复抑制；
5. 异步调用 apiserver gRPC；
6. 返回 202 accepted + request_id；
7. 客户端查询提交状态或报告状态。
```

SubmitQueue 是入口削峰机制，不是 MQ，也不是 worker。

### 4.2 作答保存

apiserver 的 Survey 模块负责：

```text
1. 加载问卷结构和版本；
2. 校验题目存在性、必填、选项合法性；
3. 构造 AnswerSheet；
4. 保存 AnswerSheet；
5. 写入 durable submit 相关保护；
6. stage answersheet.submitted outbox。
```

这里不应该同步完成完整测评和报告生成。

原因：

```text
报告生成可能涉及模型解析、计分、常模、解释、通知等多阶段逻辑；
同步执行会拉长前台响应；
慢依赖会放大入口压力；
失败重试和补偿无法清晰治理。
```

### 4.3 异步测评执行

worker 消费 `answersheet.submitted` 后，通过 internal gRPC 回调 apiserver，推进：

```text
CalculateAnswerSheetScore
CreateAssessmentFromAnswerSheet
EvaluateAssessment
GenerateReport
SyncAssessmentAttention
ProjectBehaviorEvent
```

worker 不直接操作完整业务仓储。

### 4.4 报告查询

报告状态查询应该优先走 Redis report status cache。

基本策略：

```text
report-status
    非阻塞读取当前状态；
    返回 status / stage / message / next_poll_after_ms。

wait-report
    先查 cache；
    未完成则等待 signal 或 timeout；
    signal 丢失时靠 timeout + next_poll_after_ms 兜底；
    active waiters 必须受控。
```

wait-report 的核心不是“实时推送”，而是减少无效短轮询。

正确性来自：

```text
DB 最终状态；
report status cache；
客户端重试；
timeout 兜底。
```

Redis Pub/Sub signal 只提升及时性，不参与正确性证明。

## 5. 领域模块划分

`internal/apiserver/domain` 下长期目标模块如下：

```text
internal/apiserver/domain/
├── survey
├── modelcatalog
├── calculation
├── evaluation
├── interpretation
├── actor
├── plan
└── statistics
```

当前代码中部分能力可能还没有完全迁移到目标包结构。文档中的终局结构是演进方向，不应被理解为所有目录已经完全落地。

## 6. Survey：作答事实域

### 6.1 定位

Survey 只负责“用户答了什么”。

它不负责：

```text
不负责解释分数；
不负责模型资产；
不负责测评状态机；
不负责报告生成；
不负责统计聚合。
```

### 6.2 核心对象

```text
Survey
├── Questionnaire
├── QuestionnaireVersion
├── Question
├── QuestionOption
├── SubmissionSpec
├── AnswerSheet
├── AnswerItem
└── AnswerValue
```

### 6.3 领域事件

```text
answersheet.submitted
```

### 6.4 边界

Survey 可以校验：

```text
题目是否存在；
答案格式是否合法；
选项是否合法；
是否满足提交规格；
是否允许重复提交。
```

Survey 不应校验：

```text
这个答案得多少分；
这个分数代表什么风险；
这个测评最终是什么等级；
这个报告应该如何解释。
```

## 7. ModelCatalog：模型资产域

### 7.1 定位

ModelCatalog 管理“测评模型资产”和“规则配置”。

它回答：

```text
有哪些模型？
模型是什么版本？
绑定哪个问卷？
有哪些因子？
因子如何组织？
有哪些规则配置？
发布态快照是什么？
执行算法族是什么？
报告解释应该走什么机制？
```

它不直接执行计算，不保存某一次测评执行状态，也不生成用户报告实例。

### 7.2 核心对象

```text
ModelCatalog
├── AssessmentModel
├── ModelDefinition
├── PublishedModelSnapshot
├── QuestionnaireBinding
├── DecisionSpec
├── PayloadFormat
├── ProductChannel
├── Kind
├── SubKind
├── AlgorithmFamily
├── Algorithm
└── SourceRef
```

### 7.3 模型身份

建议稳定为：

```text
ProductChannel
Kind / ModelFamily
AlgorithmFamily
Algorithm
Code
Version
```

含义如下：

| 字段 | 含义 |
| ----- | ----- |
| ProductChannel | 产品分类 / 业务分类 |
| Kind | 当前代码中的模型族兼容字段 |
| AlgorithmFamily | 执行算法族 |
| Algorithm | 具体算法 / 具体测评模型 |
| Code | 模型资产编码 |
| Version | 模型版本 |

### 7.4 ProductChannel

`ProductChannel` 是分类字段，不是模型族能力。

建议枚举：

```text
medical_scale
personality
behavior_ability
cognitive
custom
screening
followup
```

它用于：

```text
产品入口；
前端分类；
列表筛选；
运营统计；
频道聚合。
```

它不决定：

```text
payload 怎么解析；
runtime 走哪个 evaluator；
report builder 用哪个；
计算算法是什么。
```

长期应该稳定为：

```text
behavior_ability = product channel / taxonomy
behavioral_rating = executable model family
cognitive = executable model family
```

### 7.5 Kind / ModelFamily

当前代码中的 `Kind` 短中期继续保留：

```text
scale
personality
behavioral_rating
cognitive
custom
```

`behavior_ability` 应逐步降级为 legacy product-channel kind，不再作为新模型族使用。

### 7.6 AlgorithmFamily

`AlgorithmFamily` 描述底层执行语义。

| Go 包名 | AlgorithmFamily 枚举 | 含义 |
| ----- | ----- | ----- |
| scoring | factor_scoring | 因子区间计分 |
| typology | factor_classification | pole / trait / pattern 分型 |
| norming | factor_norm | 常模查表与投影 |
| task_performance | task_performance | 任务 / 能力表现 |

### 7.7 Algorithm

`Algorithm` 更准确地说是模型资产中的算法标识或模型标识，用于运行时选择与配置索引，不应被理解为领域包名或硬编码分支入口：

```text
scale_default
mbti
sbti
bigfive
brief2
spm
behavioral_rating_default
snap_iv
conners
```

MBTI / SBTI / BigFive / Brief-2 / SPM 等只应作为模型资产数据、seed、payload、fixture、migration 或兼容 ACL 出现，不应成为 `domain/` 包名，也不应成为主链路硬编码分支。

## 8. ModelCatalog / Factor：通用模型构件

### 8.1 定位

`factor` 是 ModelCatalog 内的通用模型构件。

它回答：

```text
模型有哪些维度？
每个维度绑定哪些题？
维度之间是什么层级关系？
维度角色是什么？
维度使用什么计分配置？
维度是否参与常模或分类？
```

它不属于 Calculation。

### 8.2 核心对象

```text
factor
├── FactorSnapshot
├── FactorRole
├── ScoringSpec / ScoringStrategy
├── ScoringParams
├── ScoreRangeRule
├── ClassificationSpec
├── NormRef
├── ChildrenPolicy
├── Hierarchy
└── PublishValidation
```

### 8.3 FactorRole

建议稳定为：

```text
dimension
total
index
validity
subtest
task_set
report_group
ability_domain
```

含义如下：

| Role | 含义 |
| ----- | ----- |
| dimension | 普通计分维度 |
| total | 总分 |
| index | 综合指数 |
| validity | 效度指标 |
| subtest | 子测验 |
| task_set | 任务题组 |
| report_group | 报告分组，不参与计分 |
| ability_domain | 能力域 |

### 8.4 多级因子

终局设计支持：

```text
扁平 Factor 列表
+ ParentCode
+ ChildrenPolicy
+ SortOrder
+ Level
```

而不是递归嵌套树。

示例：

```text
GEC
├── BRI
│   ├── Inhibit
│   └── Self-Monitor
├── ERI
└── CRI
```

领域表达：

```json
{
  "code": "bri",
  "role": "index",
  "parent_code": "gec",
  "children_policy": {
    "strategy": "sum",
    "children": ["inhibit", "self_monitor"]
  }
}
```

### 8.5 Factor 的边界

Factor 应该保留：

```text
模型结构；
规则配置；
层级关系；
role；
question binding；
children policy；
norm ref；
classification seam。
```

Factor 不应该包含：

```text
Brief-2 专属执行逻辑；
SPM 专属执行逻辑；
T 分查表执行；
人格类型组合执行；
具体计算过程。
```

## 9. Calculation：无状态计算内核

### 9.1 定位

Calculation 是无状态计算内核。

它回答：

```text
给定计算节点、已有分数、算法参数和上下文，如何得到计算结果？
```

Calculation 不拥有模型资产。

它不应该 import：

```text
modelcatalog
modelcatalog/factor
survey
evaluation/assessment
```

调用方负责把领域资产翻译为中立输入。

边界应稳定为：

```text
ModelCatalog owns model structure and rule configuration.
Calculation executes those rules and produces calculation.Result.
Evaluation orchestrates one assessment run and maps calculation results into AssessmentOutcome.
```

### 9.2 核心对象

```text
calculation
├── ScoreNode
├── AggregationStrategy
├── ScoreValue
├── ScoreKind
├── Result
├── DimensionResult
├── ResultLevel
└── projection
```

### 9.3 ScoreNode

`ScoreNode` 是中立计算节点，不是 Factor。

示例结构：

```go
type ScoreNode struct {
    Code        string
    Name        string
    Role        string
    Kind        DimensionKind
    ParentCode  string
    Level       int
    SortOrder   int

    Aggregation AggregationStrategy
    Children    []string
    Weights     map[string]float64
}
```

它由 Application 层从 `factor.FactorSnapshot` 翻译而来。

### 9.4 Calculation Result

```go
type Result struct {
    Primary      *ScoreValue
    Level        *ResultLevel
    PrimaryLabel string
    Dimensions   []DimensionResult
}
```

### 9.5 Projection 模式

Calculation 中稳定使用 Projection 模式：

```text
Calculation Result
    ↓
Projection.Apply
    ↓
Enhanced Calculation Result
```

可保留：

```text
calculation/projection
├── CompositeProjection
├── HierarchyProjection
├── ScoreRangeProjection
└── ClassificationProjection seam
```

其中：

```text
CompositeProjection
    根据 ScoreNode.Children + AggregationStrategy
    从子维度分数派生父级 index / total 分数。

HierarchyProjection
    给 CalculationResult.Dimensions 补充层级 metadata
    并按层级顺序整理结果。
```

这里不应该使用 `display` 这样的 UI 命名，应该使用更接近领域语义的命名，例如：

```text
sortDimensionsByHierarchy
```

### 9.6 Calculation 不应该做的事

Calculation 不应该：

```text
读取模型快照；
解析 payload；
知道 Brief-2 / SPM / MBTI 的领域资产结构；
管理 EvaluationRun；
持久化结果；
生成报告。
```

## 10. Evaluation：测评执行域

### 10.1 定位

Evaluation 管理“一次测评执行”。

它回答：

```text
这次测评是谁提交的？
用哪个答卷？
用哪个模型快照？
当前跑到什么状态？
执行成功还是失败？
结果是什么？
是否需要重试？
```

### 10.2 核心对象

```text
evaluation
├── Assessment
├── EvaluationRun
├── AssessmentOutcome
├── DimensionResult
├── OutcomeScoreValue
├── OutcomeResultLevel
├── EvaluationStatus
└── Domain Events
```

### 10.3 Assessment

```text
Assessment
├── ID
├── SubjectRef
├── SubmitterRef
├── AnswerSheetRef
├── ModelRef
├── ModelSnapshotRef
├── QuestionnaireRef
├── PlanTaskRef
├── Scene
├── Status
├── CurrentRunID
├── ResultRef
├── ReportRef
├── CreatedAt
├── SubmittedAt
├── CompletedAt
└── FailedReason
```

### 10.4 EvaluationRun

`EvaluationRun` 是一次执行尝试。

它用于记录：

```text
第几次尝试；
什么时候开始；
什么时候结束；
是否成功；
失败类型是什么；
是否可重试；
trace 信息是什么。
```

推荐结构：

```text
EvaluationRun
├── RunID
├── AssessmentID
├── AttemptNo
├── Status
├── InputSnapshotRef
├── StartedAt
├── FinishedAt
├── ErrorCode
├── ErrorMessage
├── FailureKind
├── Retryable
└── TraceID
```

当前实现状态应按代码事实维护：

```text
evaluation_run 表已经落库；
Evaluate Start / Success / Fail 会写入 run 行；
assessment.current_run_id 在 Start 时持久化；
失败响应携带 retryable / run_id / failure_kind；
worker 可依据 retryable 决定 MQ nack 或 ack；
evaluation_run 仍未与 Statistics ProjectionCheckpoint 物理统一；
R113 新增 application/evaluation/consistency 对账扫描与 RepairInterpretedFinalization 补偿，覆盖 reporting 末步 assessment Save 失败窗口。
```

### 10.5 AssessmentOutcome

Evaluation 的 outcome 是测评执行域的事实结果，不是 Calculation 的内部结果。

Calculation 结果需要通过 Application adapter 映射为 `AssessmentOutcome`。

```text
calculation.Result
    ↓
Application adapter
    ↓
evaluation.AssessmentOutcome
```

### 10.6 Evaluation 不应该做的事

Evaluation 不应该：

```text
知道 Factor 如何聚合；
知道 Brief-2 T 分怎么查；
知道 SPM 能力等级怎么换算；
知道报告 section 如何生成；
知道产品频道如何展示。
```

### 10.7 Evaluation 包划分原则

`domain/evaluation` 不应该长期按照测评类型或模型族拆包。

不建议长期存在：

```text

domain/evaluation/personality
domain/evaluation/behavioral_rating
domain/evaluation/cognitive
domain/evaluation/brief2
domain/evaluation/mbti
```

Evaluation 的领域核心不是“这是哪种测评”，而是：

```text
一次测评执行如何被创建、运行、完成、失败、重试和落结果。
```

推荐结构：

```text
domain/evaluation
├── assessment
│   ├── aggregate.go
│   ├── outcome.go
│   ├── status.go
│   └── event.go
├── run
│   ├── run.go
│   ├── attempt.go
│   └── failure.go
├── input
│   └── snapshot.go
├── policy
│   ├── retry.go
│   ├── idempotency.go
│   └── timeout.go
└── pipeline
    ├── pipeline.go
    └── descriptor.go
```

Evaluation 中的差异应该由抽象承载，而不是由测评类型目录承载。

推荐抽象：

```text
EvaluationPipeline
RuntimeDescriptor
CalculationPipeline
InputAssembler
CalculationAdapter
OutcomeAssembler
```

执行选择逻辑应是：

```text
PublishedModelSnapshot
    ↓
AlgorithmFamily / PayloadFormat / DecisionKind
    ↓
RuntimeDescriptorRegistry
    ↓
EvaluationPipeline
```

总结：

```text
modelcatalog 可以认识模型族；
evaluation 只应该认识执行管线。
```

Application 层面向调用者和用例，因此它负责组织跨领域编排。

典型职责：


### 11.1 Adapter 模式

当前已经出现两个关键 adapter：


建议最终沉淀为：


### 11.2 RuntimeDescriptor

```text

    ↓

    ↓
```

## 17. 关键设计模式

### 17.1 Published Snapshot 模式

ModelCatalog 发布后生成不可变快照：

```text
Draft Model
    ↓ publish
PublishedModelSnapshot
    ↓
Evaluation 只读取 published snapshot
```

Evaluation 不读 draft。

### 17.2 Adapter 模式

关键 adapter：

```text
modelcatalog.factor.FactorSnapshot
    -> calculation.ScoreNode

calculation.Result
    -> evaluation.AssessmentOutcome

typology runtime spec
    -> canonical factors

scale execution snapshot
    -> canonical factors
```

Adapter 放在 Application 或 algorithm extension 中，不放在 Calculation kernel 中。

### 17.3 Projection 模式

Calculation 中使用 Projection：

```text
Result
    -> CompositeProjection
    -> NormProjection
    -> HierarchyProjection
    -> Final Result
```

通用 projection 放 Calculation。

算法专属 projection 先放 Application / algorithm orchestration。

等第二个同类算法出现后，再上抽到 Calculation。

### 17.4 Registry 模式

适合以下场景：

```text
EvaluatorRegistry
ReportBuilderRegistry
RuntimeDescriptorRegistry
ScoreProjectorRegistry
```

短期可以使用 switch。

当模型族和算法超过一定数量后，迁移为 registry。

### 17.5 Pipeline 模式

Evaluation 应该使用 Pipeline 模式承载不同模型运行时的执行差异。

Pipeline 不应按测评 code 命名，而应按算法族、payload format、decision kind 或 runtime descriptor 选择。

推荐抽象：

```go
type EvaluationPipeline interface {
    Supports(snapshot modelcatalog.PublishedModelSnapshot) bool
    Execute(ctx context.Context, input EvaluationInput) (*assessment.AssessmentOutcome, error)
}
```

或：

```go
type RuntimeDescriptor struct {
    AlgorithmFamily  modelcatalog.AlgorithmFamily
    PayloadFormat    string
    DecisionKind      modelcatalog.DecisionKind
    InputAssembler    InputAssembler
    Calculator        Calculator
    OutcomeAssembler  OutcomeAssembler
}
```

最终选择逻辑：

```text
PublishedModelSnapshot
    ↓
AlgorithmFamily / PayloadFormat / DecisionKind
    ↓
RuntimeDescriptorRegistry
    ↓
EvaluationPipeline
```

Pipeline 模式用于替代：

```text
if kind == scale
if kind == personality
if algorithm == brief2
```

### 17.6 Anti-Corruption Layer

主要用于 legacy：

```text
legacy model kind
legacy ruleset payload
legacy product-channel adapter
legacy execution snapshot
```

这些不应污染主模型。

建议所有 legacy 逻辑收口在：

```text
modelcatalog/legacy
application/modelcatalog/legacy
infra/.../legacy mapper
```

## 18. 高并发与一致性设计

### 18.1 入口保护

前台提交通过 collection-server 入口保护：

```text
SubmitQueue
    进程内 memory channel + worker pool；
    用于削峰；
    队列满返回 429；
    状态仅在本进程内维护。

request_id
    查询本地提交状态；
    不是业务 durable 幂等键。

SubmitGuard
    Redis lock + done marker；
    用于跨实例重复提交抑制；
    是入口治理，不是最终业务唯一约束。
```

### 18.2 写入一致性

答卷保存与事件发布必须避免：

```text
AnswerSheet 已保存，但 answersheet.submitted 没有发布。
```

因此关键业务事件应使用 durable outbox：

```text
业务写入
    ↓
stage outbox
    ↓
outbox relay
    ↓
MQ
    ↓
worker consume
```

### 18.3 异步执行

worker 消费 MQ 后，只负责推进流程，不直接拥有业务主状态。

```text
MQ event
    ↓
worker handler
    ↓
internal gRPC
    ↓
apiserver application service
    ↓
domain + repository + outbox
```

### 18.4 报告查询治理

报告查询应采用：

```text
report_status cache
next_poll_after_ms
wait-report long polling
Redis Pub/Sub signal
active waiter limit
timeout fallback
```

原则：

```text
report status cache 提供低成本读；
next_poll_after_ms 对客户端反压；
long polling 减少无效短轮询；
signal 只负责临时唤醒；
timeout 是正常结果，不是失败；
DB fallback 是正确性兜底。
```


```text

```

例如：

```text
第二个 factor_norm 算法出现：Conners
    -> 抽 calculation/norm

第二个 classification 算法出现：
    -> 抽 calculation/classification

第二个 task_performance 算法出现：
    -> 抽 calculation/task_performance
```

```text
ScoreNode.Code required；
duplicate code；
children ref exists；
cycle detection；
aggregation strategy valid；
weighted_sum weights valid。
```

### 20.2 Application adapter 通用化

当前若存在：

```text
behavioral_rating/outcome_adapter.go
behavioral_rating/scorenode_adapter.go
```

终局建议沉淀为：

```text
application/evaluation/calculationadapter
├── outcome.go
├── score_node.go
└── score.go
```

避免 cognitive / scale 重复实现。

### 20.3 KindCapability 拆薄

当前 `KindCapability` 若混合了：

```text
domain capability
API kind
display name
QRCode
preview
options
legacy product channel
```

终局应拆为：

```text
domain/modelcatalog/ModelFamilyCapability
application/modelcatalog/ModelCatalogOption
```

Domain 层只关心：

```text
是否可创建；
是否可发布；
是否可绑定问卷；
是否可执行；
执行路径是什么。
```

Application 层关心：

```text
APIKind；
DisplayName；
QRCode；
Preview；
Options；
ProductChannel aggregation。
```

### 20.4 ProductChannel 彻底降级

> **当前事实（R101 后）**：`behavior_ability` 仅在 application `option` 注册表与列表 API 作产品通道聚合；`evaluation/pipeline` 与 `application/evaluation/{execute,runtime,registry}` 禁止引用 `behavior_ability` 做执行路由。

`ProductChannel` 应稳定为普通分类字段。

```text
behavior_ability = product channel / taxonomy
behavioral_rating = executable model family
cognitive = executable model family
```

### 20.5 Primary dimension 配置化

主维度由已发布 norming 扩展 `primary_dimension_code` 配置；`calculation/norm/projection` 仅按配置码选取主维度，不再硬编码 `gec`/`total`。发布路径经 `norming.RequirePrimaryDimensionCodeForPublish` 强制非空；旧快照 decode 在 `norming/snapshot` 保留 `gec`/`total` 读路径兜底。

### 20.6 Typology decision 完全配置化

`decision.kind` 来自 payload / 已发布快照；`DecisionKindForIdentity` 不为 personality 或 behavioral_rating 做 algorithm 推断。`AlgorithmFromTypologyPayload` 与 `legacy.FallbackPersonalityDecisionKind` 仅 legacy decode；behavioral_rating 的 `NormLookup` 由 `norming.HasDefinitionExtension` 驱动 publish 决策。

### 20.7 Evaluation / Interpretation 去测评类型包

> **当前事实（R78–R86 后）**：`domain/evaluation` 六包（assessment/run/input/policy/pipeline/event）；`domain/interpretation` 五包且无 `*_detail.go` 测评 code 文件名。application 层机制 registry + legacy adapter；具体测评 code 仅作数据值。`application/modelcatalog` 顶层 service 已机制化命名（typology/norming/task_performance gateway）。

长期目标：

```text
evaluation 不出现 personality / scale / behavioral_rating / cognitive / brief2 / mbti 这类测评类型包；
interpretation 不出现 personality / scale / behavioral_rating / cognitive / brief2 / mbti 这类测评类型包。
```

迁移路径：

```text
第一阶段：
    保留 application 层的模型族 adapter；
    但 domain/evaluation 和 domain/interpretation 不再新增测评类型包。

第二阶段：
    将 application/evaluation/behavioral_rating 等适配逻辑沉淀为
    calculationadapter、runtime descriptor、pipeline。

第三阶段：
    将 interpretation 中的测评类型 builder 收敛为机制级 builder 和模板配置。

第四阶段：
    具体测评 code 只存在于 AlgorithmCode / ModelCode 等模型资产数据值、模型 seed、payload、fixture、migration 中。
```

最终原则：

```text
modelcatalog 认识稳定模型机制与资产元数据，不认识具体测评 code 包；
evaluation 只认识执行管线；
interpretation 只认识报告模板和解释规则。
```

## 22. 终局一句话

`qs-server` 的最终设计应该是：

```text
以 Survey 沉淀作答事实；
以 ModelCatalog 管理模型资产、Factor 结构和规则配置；
以 Calculation 作为无状态计算内核执行规则；
以 Evaluation 管理一次测评执行、状态机和结果事实；
以 Interpretation 生成报告解释；
以 Actor / Plan / Statistics 支撑参与者、任务计划和读侧聚合；
通过 collection-server 和 worker 分别承载前台入口与异步执行。
```

更精炼地说：

```text
模型结构归 ModelCatalog；
计算执行归 Calculation；
执行状态归 Evaluation；
报告表达归 Interpretation；
调用编排归 Application。
```

这是 qs-server 后续重构时最重要的边界。
