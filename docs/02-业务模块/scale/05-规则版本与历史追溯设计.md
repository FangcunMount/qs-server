# Scale 规则版本与历史追溯设计草案

**本文回答**：当 Scale 已发布的规则在下架后被再次编辑、再次发布时，历史 Evaluation 如何追溯到「提交那一刻」的规则；为什么当前只靠 `published` 冻结不够；`MedicalScaleVersion` / `ScaleRuleSnapshot` 应如何分两步演进。

> 本文档为设计草案，**不立即编码**。落地时机：当出现第一个真实「需要追溯历史规则」的业务场景（例如临床合规、报告版本比对、规则下架重发后误判排查）。

---

## 30 秒结论

| 维度 | 结论 |
| ---- | ---- |
| 当前事实 | `MedicalScale` 只有 `status`，没有规则版本；`published` 后规则冻结，但 `unpublish → 编辑 → 再 publish` 后无法区分新旧规则 |
| 历史风险 | 一份 Assessment 的 `MedicalScaleCode` 解析出的规则永远是「最新已发布」，可能与提交时不一致 |
| 阶段 1（小） | 给 `MedicalScale` 加 `ruleVersion`，每次进入 `published` 时递增；现有 `ScaleSnapshot` 透出版本号；Evaluation 记录 `MedicalScaleRuleVersion`，但仍按 code 解析（最新） |
| 阶段 2（大） | 引入 `ScaleRuleSnapshot` 仓储：版本化只读快照；Evaluation 按 `(code, ruleVersion)` 精确解析；Report 重生成能力 |
| 不做 | 不把规则深拷贝塞进 Evaluation 聚合根；不让 Survey/Actor 感知 ruleVersion |
| 风险 | 阶段 1 是「打标记」，治不了已有历史；阶段 2 才能精确还原。先做阶段 1，留观测，再决定阶段 2 |

一句话总结：

> **当前发布冻结只保证「发布后不可改」，不保证「同一个 code 永远是同一份规则」。要让 Evaluation 可精确追溯，必须给规则本身赋版本号。**

---

## 1. 为什么现状不够

### 1.1 现状

- `MedicalScale.status ∈ {draft, published, archived}`
- 上轮重构后：`published` 态下因子 / 计分 / 解读规则 / 问卷绑定均被 `ensureRuleEditable` 冻结
- 但 `Unpublish` 是合法状态转换：`published → draft`，下架后即可重新编辑
- 重新 `Publish` 后，量表代码（`scaleCode`）不变，状态又回到 `published`

### 1.2 后果

```text
T1: Scale=S1 published, rules=R1   →   Assessment A1 提交, 用 R1 解读
T2: Scale=S1 unpublish              →   draft
T3: Scale=S1 编辑解读规则 R2
T4: Scale=S1 publish                →   published, rules=R2

现在重新加载 A1 的报告：
  evaluationinput.Resolve(MedicalScaleCode=S1) → 拿到 R2
  但 A1 提交时实际用的是 R1
```

代码锚点：

- 解析路径：[../../../internal/apiserver/infra/evaluationinput/repository_resolver.go](../../../internal/apiserver/infra/evaluationinput/repository_resolver.go) 的 `GetScale(ctx, code)`
- 解析时入参：[../../../internal/apiserver/port/evaluationinput/input.go](../../../internal/apiserver/port/evaluationinput/input.go) 的 `InputRef.MedicalScaleCode`

`InputRef` 已经持有 `QuestionnaireCode + QuestionnaireVersion`，但 `MedicalScale` 这一侧只有 code、**没有版本**。结构上不对称。

### 1.3 风险评估

| 场景 | 当前是否可观测 | 风险等级 |
| ---- | ---- | ---- |
| 临床合规审计「这份报告依据哪一版规则」 | ❌ 无法回答 | 高（涉及合规） |
| 规则上线后发现解读规则误差，回滚后历史报告复盘 | ❌ 看到的是回滚后的规则，对不上 | 中 |
| 业务侧手动报表「比对新旧规则差异」 | ❌ 无法对比 | 中 |
| 普通用户重新打开历史报告 | ⚠️ 内容可能与生成时不一致 | 低（视产品策略） |
| 量表从未 unpublish 过 | ✅ 不受影响 | — |

结论：**「从未下架」是当前默认假设**。但只要平台允许 unpublish，这个假设迟早被打破。

## 2. 阶段 1：给规则赋版本号

### 2.1 目标

让「这次评估用的是 S1 的第几版规则」**可观测**。不强行还原历史，但为未来还原打基础。

### 2.2 模型变化

#### 2.2.1 MedicalScale 加 `ruleVersion`

```go
// 仅示意：domain/scale/medical_scale.go
type MedicalScale struct {
    // ... 现有字段
    ruleVersion uint32 // 当前已发布的规则版本；draft 阶段保持上次发布的版本号
}

func (m *MedicalScale) RuleVersion() uint32 { return m.ruleVersion }

// publish 时递增
func (m *MedicalScale) publish() error {
    if err := m.updateStatus(StatusPublished); err != nil {
        return err
    }
    m.ruleVersion++ // 每进入 published 一次，版本 +1
    m.addChangedEvent(ChangeActionPublished)
    return nil
}
```

要点：

- 只在「进入 published」时 +1；`Unpublish/Archive` 不变
- draft 阶段允许编辑，但 `ruleVersion` 保持「上次发布时的版本号」
- 首次 publish 从 `0 → 1`

#### 2.2.2 ScaleSnapshot 透出版本

```go
// port/evaluationinput/input.go
type ScaleSnapshot struct {
    // ... 现有字段
    RuleVersion uint32
}
```

`infra/evaluationinput/snapshot_mappers.go` 中 `scaleToSnapshot` 写入。

#### 2.2.3 Evaluation 持久化版本号

`Assessment` 写入 / 更新时，把 `ScaleSnapshot.RuleVersion` 记入持久化字段 `medical_scale_rule_version`。不改变现有解析流程（仍按 code 解析最新），但**历史 Assessment 留下版本指纹**。

### 2.3 事件

`scale.changed.published` 事件 data 增加 `rule_version` 字段：

```yaml
# configs/events.yaml （示意）
scale.changed:
  data:
    scale_id: uint64
    code: string
    version: string         # 现有：保留兼容
    rule_version: uint32    # 新增
    name: string
    action: published|unpublished|updated|archived
    changed_at: time
```

下游订阅方（缓存、统计、外部审计）可凭 `rule_version` 决定是否做差异处理。

### 2.4 落地清单（阶段 1）

| 改动点 | 文件 |
| ---- | ---- |
| `MedicalScale.ruleVersion` 字段 + publish 递增 | `domain/scale/medical_scale.go` |
| ScalePO + mapper 持久化 ruleVersion | `infra/mongo/scale/po.go`, `mapper.go` |
| Mongo 迁移：已存量量表初始化 `rule_version = published ? 1 : 0` | 一次性脚本 |
| ScaleChangedData 加字段 | `domain/scale/events.go` |
| ScaleSnapshot 加字段 + mapper 写入 | `port/evaluationinput/input.go`, `infra/evaluationinput/snapshot_mappers.go` |
| Assessment 持久化 ruleVersion | `domain/evaluation/assessment/*`, `infra/mysql/evaluation/po.go` |
| 测试：unpublish→publish 后 ruleVersion +1 | `domain/scale/medical_scale_factor_test.go` |

### 2.5 不做的事

- 不做精确还原：解析仍按 code 取最新已发布规则
- 不为旧 Assessment 补 `ruleVersion`（NULL 即可，代表「未知版本」）
- 不影响 REST / gRPC 现有契约（只新增可选字段）

## 3. 阶段 2：ScaleRuleSnapshot 与精确解析

### 3.1 触发条件

阶段 1 落地后，观察到以下任一信号再启动阶段 2：

- 合规要求强制「报告必须能还原签发时的规则」
- 出现一次实际生产事故，需要按历史规则复算
- 业务允许频繁 unpublish→republish，规则漂移成为常态

### 3.2 模型

#### 3.2.1 新增独立聚合：ScaleRuleSnapshot

```text
domain/scale/snapshot/
  snapshot.go           // ScaleRuleSnapshot 聚合根
  repository.go         // SnapshotRepository
```

```go
type ScaleRuleSnapshot struct {
    scaleCode    meta.Code
    ruleVersion  uint32
    publishedAt  time.Time

    // 不可变冻结快照
    questionnaireCode    meta.Code
    questionnaireVersion string
    factors              []FactorSnapshot   // 已经是只读结构
    // 注意：不持有展示信息（title/desc/tags 等）
}
```

关键点：

- 聚合**只读**：构造后任何字段都不变；没有 mutator
- 与 `MedicalScale` 分聚合：`MedicalScale` 保留可编辑状态机；`ScaleRuleSnapshot` 是事实事件流
- 主键：`(scale_code, rule_version)`

#### 3.2.2 写入时机

```text
MedicalScale.publish()
  → ruleVersion++
  → SnapshotRepository.Save(buildSnapshot(m))   // 同事务
  → addChangedEvent(ChangeActionPublished)
```

封装为 application 层的 lifecycle workflow，确保「发布」与「快照落库」要么都成、要么都不成（outbox or 同库事务）。

#### 3.2.3 解析流程升级

```go
// port/evaluationinput/input.go
type InputRef struct {
    AssessmentID            uint64
    MedicalScaleCode        string
    MedicalScaleRuleVersion uint32 // 新增：阶段 1 已写入
    AnswerSheetID           uint64
    QuestionnaireCode       string
    QuestionnaireVersion    string
}
```

`evaluationinput.Resolve` 根据 `(MedicalScaleCode, MedicalScaleRuleVersion)` 命中 snapshot；命中失败时（旧数据没有版本）退回到「按 code 取最新」并打告警日志。

### 3.3 不向 Evaluation 聚合深拷贝规则

诱惑：让 `Assessment` 直接持有完整规则副本。**不要**这么做：

- Evaluation 聚合膨胀，跨域职责混淆
- 规则副本字段众多，序列化/反序列化成本高
- 真实快照应该是 Scale 域的事实，由 Scale 域负责生命周期

正确做法：Evaluation 只持有指针字段 `(scaleCode, ruleVersion)`，规则事实由 `ScaleRuleSnapshot` 仓储提供。

### 3.4 与 `scale.changed` 的关系

- `scale.changed` 仍只是「规则变更通知」，不是「历史重算命令」
- 历史报告重算应作为独立能力：用户/运营触发，按 `(scaleCode, oldRuleVersion → newRuleVersion)` 标记重算任务
- 阶段 2 的 `ScaleRuleSnapshot` 让重算任务真正可执行（能拿到旧规则）

## 4. 反例：不要做什么

| 反模式 | 为什么不做 |
| ---- | ---- |
| 让 `published` 量表的规则也能改 | 直接破坏「发布即冻结」语义，规则漂移失控 |
| 用 `updated_at` 作为版本号 | 时间戳不能表达「同一秒内多次发布」「时区差异」「人工回拨」 |
| 在 Evaluation 聚合内深拷贝完整 Scale | 跨域职责，膨胀，重复 |
| 按 questionnaire_version 反推 rule version | 问卷版本与规则版本是不同事实，不能复用 |
| 把 ScaleVersion 当成 ScaleListCache 的一部分 | 缓存不是事实源，版本号必须落库 |
| MBTI / 其他非医学量表也共用同一个 `ruleVersion` | 那是另一个规则域的问题，先抽 `EvaluationModelRef`，再讨论版本 |

## 5. 取舍与门槛

| 决策 | 阶段 1 | 阶段 2 |
| ---- | ---- | ---- |
| 改动范围 | Scale + Evaluation 两侧加字段 | 新增 Snapshot 聚合、仓储、迁移、查询 |
| 数据迁移 | 一次性脚本，给存量量表初始化 | 大：需要把已发布量表回填快照 |
| 风险 | 低，纯增量字段 | 中，影响解析主路径 |
| 收益 | 可观测、可审计「这次评估用的版本」 | 可精确重放、可比对、可重算 |
| 启动门槛 | 任意时刻都可做 | 阶段 1 跑稳定 ≥ 1 个月，且出现真实诉求 |

**建议节奏**：

```text
现在     ── 阶段 0 完成：发布冻结（已落地）
下一轮   ── 阶段 1：ruleVersion 打标记
观察期   ── 收集真实追溯/重算诉求
触发后   ── 阶段 2：ScaleRuleSnapshot 与精确解析
```
