# Survey 领域服务与策略

## 1. 本文回答

本文区分 Survey 的聚合行为、领域服务、领域策略、应用服务和基础设施适配器，避免把所有 `Service` 都误写成领域服务。

## 2. 责任分层

```text
聚合行为
  维护单个 Questionnaire / AnswerSheet 自身不变量

领域服务与规格
  处理发布、版本、跨题校验和提交规格等领域规则

应用服务
  面向用例装配仓储、事务、规则引擎、事件和跨模块 port

基础设施
  实现 Mongo、缓存、规则执行、Outbox 和传输协议
```

文件名包含 `service` 并不自动意味着领域服务；判断标准是它是否表达业务规则、是否独立于传输和持久化技术。

## 3. Questionnaire 领域服务

### 3.1 Lifecycle

`questionnaire.Lifecycle` 负责 Publish、Unpublish、Archive：

- 校验前置状态；
- 发布时调用 Validator；
- 发布时调用 Versioning 递增大版本；
- 调用聚合包内状态方法；
- 让聚合产生 `questionnaire.changed`。

它不保存 Repository、不发布事件、不发送缓存信令，也不调用 ModelCatalog；这些是应用层责任。

### 3.2 Versioning

`Versioning` 封装历史兼容版本规则：初始化、小版本递增、大版本递增。版本变更作用于 Questionnaire 聚合，但算法不依赖数据库或接口。

### 3.3 Validator

`Validator` 是发布前完整性策略：

- `ValidateBasicInfo`；
- `ValidateQuestion / ValidateQuestions`；
- `ValidateForPublish`；
- 将多个 `ValidationError` 聚合为领域错误。

它检查问卷是否“足以发布”，不检查一次具体答案是否满足该发布版本。

### 3.4 SubmissionSpec

`SubmissionSpec` 是从已发布 Questionnaire 派生的提交规格，负责：

- 拒绝未知 question code；
- 以服务端题型校验客户端 question type；
- 校验 Radio / Checkbox 选项属于当前题目；
- 根据 ShowController 计算题目可见性；
- 要求可见的 required 题存在且非空；
- 产生 `PreparedSubmissionAnswer`，供应用层继续校验和构造 AnswerValue。

这是领域规格，不是持久化模型，也不应由 transport 自行重写。

## 4. AnswerSheet 领域行为与适配

### 4.1 Submit

`answersheet.Submit` 是聚合工厂行为：验证 ID、QuestionnaireRef、SubmissionContext 和答案唯一性，创建答卷并产生提交事件。

### 4.2 AnswerValue adapters

`NewAnswerValueAdapter` 和 `NewScorableValue` 将 AnswerValue 暴露为规则引擎需要的校验/计分视图。它们只做类型适配，不拥有规则选择和用例编排。

### 4.3 UpdateScores

`AnswerSheet.UpdateScores` 把 `ScoredAnswerSheet` 投影回聚合，按 question code 更新单题分数和总分。评分算法本身由 `ruleengine.AnswerScorer` 实现。

## 5. 应用服务

| 应用服务 | 行为者 / 用例 | 主要协作者 |
| --- | --- | --- |
| `QuestionnaireLifecycleService` | 管理员创建、保存草稿、发布、下架、归档、删除 | Repository、Lifecycle、Validator、EventPublisher、BindingSyncer、CacheSignalNotifier |
| `QuestionnaireContentService` | 管理员编辑题目和顺序 | Repository、Question factories |
| `QuestionnaireQueryService` | 管理和公开查询 | Repository、ReadModel、IAM identity、hotset recorder |
| `AnswerSheetSubmissionService` | 答题者提交和查询自己的答卷 | Questionnaire Repository、SubmissionSpec、AnswerValidator、DurableStore |
| `AnswerSheetManagementService` | 管理员查询和删除答卷 | Repository、ReadModel |
| `AnswerSheetScoringService` | 跨模块 intake 前计算基础题分 | AnswerSheet Repository、Questionnaire Repository、AnswerScorer |

这些服务可以编排多个依赖、做日志和错误码映射，但不应复制聚合不变量。

## 6. 关键策略与 Port

| Port / Strategy | 作用 | 失败语义 |
| --- | --- | --- |
| `QuestionnaireBindingVersionSyncer` | 医学量表问卷发布后同步 ModelCatalog binding version | 同步返回错误会使发布 API 返回失败，但此前持久化步骤可能已完成 |
| `CacheSignalNotifier` | 发布后发送 best-effort 缓存失效信令 | 空实现或发送失败不能成为主事实失败依据 |
| `AnswerValidator` | 批量执行 required、长度、数值等答案校验 | 未配置或校验失败则拒绝提交 |
| `AnswerScorer` | 根据题目选项/计算规则生成基础分数 | intake 将计分失败视为硬失败，不继续创建 Assessment |
| `SubmissionDurableStore` | 幂等地保存 AnswerSheet 并 stage 领域事件 | AnswerSheet 与 Outbox 必须处于同一事务 |
| `surveyreadmodel.*Reader` | 查询列表和详情 | 不承载写模型不变量 |

## 7. 装配关系

`container/modules/survey.New` 是当前模块装配根：

```text
Questionnaire submodule
  LifecycleService + ContentService + QueryService

AnswerSheet submodule
  SubmissionService + ManagementService + ScoringService
```

Mongo Repository、缓存 decorator、规则引擎、Mongo transaction runner 和 EventSubsystem profile 都由容器注入。Survey 不应在 domain 或 application 中自行创建这些基础设施对象。

## 8. 设计边界

- 生命周期状态规则放 domain；发布快照持久化顺序放 application/infra。
- 提交规格和可见 required 判断放 domain；DTO、日志、错误码映射放 application/transport。
- 规则引擎执行是 port；Survey 决定何时校验和何时计分。
- 跨模块 Assessment intake 不放入 Survey application，由 `application/journey/assessmentintake` 编排。
