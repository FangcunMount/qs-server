# Survey 提交规格与答案校验

## 1. 本文回答

本文说明已发布 Questionnaire 如何派生 `SubmissionSpec`，以及服务端如何依次校验问卷版本、题目引用、题型、选项、显示条件、required 规则和答案值规则。

## 2. 30 秒结论

`SubmissionSpec` 是 Questionnaire 聚合向 AnswerSheet 提交用例暴露的只读领域规格。它把一份已发布问卷转换为“服务端接受什么答案”的契约，避免 transport 或客户端自行解释题型。

```text
Published Questionnaire
  -> BuildSubmissionSpec
  -> PrepareAnswers(raw answers)
  -> PreparedSubmissionAnswer
  -> AnswerValue + validation tasks
  -> Answer
  -> AnswerSheet.Submit
```

客户端传入的 `question_type` 不是事实源。它必须与服务端发布快照中的题型严格一致；最终用于构造 Answer 的题型来自 `PreparedSubmissionAnswer`。

## 3. 为什么需要 SubmissionSpec

直接把 Questionnaire 交给 transport 或在 application service 中遍历题目，会导致以下问题：

- REST、gRPC 和管理入口各自实现一套题型规则；
- 客户端可以通过伪造 `question_type` 改变值解码；
- 校验使用 active head，而不是 AnswerSheet 指定的发布版本；
- 选项、显示条件和 required 规则在多个层重复且逐渐漂移。

`SubmissionSpec` 只保留提交所需的最小视图：question code/type、validation rules、option code 集和 ShowController。它不是 Mongo PO，也不是对外 DTO。

## 4. 输入与产出模型

### 4.1 RawSubmissionAnswer

| 字段 | 来源 | 信任级别 |
| --- | --- | --- |
| `QuestionCode` | 客户端 | 不可信，必须存在于发布快照 |
| `QuestionType` | 客户端 | 不可信，只用于与服务端题型对比 |
| `Value` | 客户端/transport 解码 | 不可信，必须经过题型归一化与规则校验 |

### 4.2 PreparedSubmissionAnswer

`PrepareAnswers` 成功后产出：

- 从服务端规格取得的 question code；
- 从服务端规格取得的 question type；
- 已通过题型/选项预检查的 raw value；
- 从发布快照复制的 validation rules。

它仍不是 `Answer`；application service 还需要将 value 转成 `AnswerValue`，执行规则引擎，再构造 Answer。

## 5. 四层校验模型

### 5.1 问卷契约存在性

`EnsureSubmittable` 要求：

- Questionnaire 非 nil；
- `status=published`；
- questionnaire code 非空；
- version 非空。

应用层在此之前已按请求的 code/version 解析发布快照。这一层防止 draft、archived 或无版本对象成为提交契约。

### 5.2 题目与题型一致性

`PrepareAnswers` 逐项检查：

1. question code 非空；
2. question code 存在于该问卷版本；
3. 客户端 question type 非空；
4. 客户端 question type 与服务端 `QuestionType.Value()` 完全相等。

当前比较是大小写敏感的字符串比较。因此 `Radio` 和 `radio`、以及 `Text` 和 `text` 不是同一份提交契约。外围系统的兼容转换不得被误写为 domain 接受多种命名。

### 5.3 题型特有结构校验

Radio 和 Checkbox 的选项 code 必须属于发布题目：

- Radio 必须可归一化为单个 option code；
- Checkbox 必须可归一化为 option code 列表；
- 列表中的每个 code 都必须在当前题目选项集合内。

这一层验证的是“值是否属于题目定义”，不是 min/max selections 等通用校验规则。

### 5.4 可见性、required 与通用规则

`ensureVisibleRequiredQuestionsAnswered` 先使用全部 raw answers 计算 ShowController：

- 没有 controller 的题默认可见；
- `rule=or` 时任一条件命中即可见；
- 其它值按 and 处理，所有条件都命中才可见；
- Section 不作为可作答题；
- 只有“可见 + 可作答 + required”的题必须提交非空值。

随后 application service 将每个 PreparedSubmissionAnswer 转为 `AnswerValidationTask`，由 `ruleengine.AnswerValidator` 执行 required、长度、数值范围、选择数量等通用规则。

## 6. 题型值转换

`CreateAnswerValueFromRaw` 在规格检查后执行：

| 题型 | 构造结果 | 主要拒绝条件 |
| --- | --- | --- |
| `Radio` | `OptionValue` | 不能归一化为单选值 |
| `Checkbox` | `OptionsValue` | 不是 `[]string / []interface{}{string...}` |
| `Text / Textarea` | `StringValue` | 不是 string |
| `Number` | `NumberValue` | 不是 `float64 / int / int64` |
| `Section` | 当前会构造 `StringValue` | 不是 string |

对 Section 的当前处理需要特别说明：SubmissionSpec 不会对 Section 执行 required 检查，但 `PrepareAnswers` 尚未显式拒绝客户端为 Section 传值。因此“Section 无需作答”是领域意图，“Section 值一定被拒绝”不是当前已实现事实。

## 7. 规则不应落在哪里

| 层 | 应负责 | 不应负责 |
| --- | --- | --- |
| transport | JSON/proto 解码、必要的格式错误 | 判断选项是否属于问卷、重写 required/ShowController |
| application | 解析版本、组装任务、调用规格与规则引擎 | 重复题型或聚合不变式 |
| domain/questionnaire | 发布问卷的题型、选项、可见性和提交规格 | Mongo、gRPC、错误码映射 |
| domain/answersheet | AnswerValue、Answer 与 AnswerSheet 不变式 | 重新查询 Questionnaire 或决定版本 |
| ruleengine port/infra | 执行通用 validation/scoring rule | 决定应当使用哪份问卷快照 |

## 8. 失败语义

| 阶段 | 错误意义 | 是否已产生 AnswerSheet |
| --- | --- | --- |
| 版本解析 / EnsureSubmittable | 没有可用的发布契约 | 否 |
| PrepareAnswers | 题目、题型、选项或可见 required 不符合问卷 | 否 |
| CreateAnswerValueFromRaw | 值无法按服务端题型归一化 | 否 |
| AnswerValidator | 通用 validation rule 失败 | 否 |
| AnswerSheet.Submit | 上下文、答案或唯一性不变式失败 | 否 |
| DurableStore | 持久化/幂等/Outbox 提交失败 | 取决于事务结果与幂等恢复 |

校验成功只表示内存中可以构造答卷，不代表 Mongo 已提交；Mongo 提交成功也不代表后续 Assessment 已执行。
