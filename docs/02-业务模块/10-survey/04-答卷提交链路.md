# 答卷提交链路

## 1. 业务目标

把用户一次作答沉淀为可靠的 `AnswerSheet` 事实。面向“发起测评”的组合 REST 用例在 AnswerSheet 落库后同步创建 Assessment；Evaluation 计分和报告仍异步推进。

---

## 2. 参与对象

| 对象 | 角色 |
| ---- | ---- |
| `ActorContext` | 提供受试者、提交者和组织上下文 |
| `Questionnaire` | 提供题目结构和提交规格 |
| `AnswerSheet` | 答卷事实聚合 |
| `AnswerItem` | 单题答案 |

---

## 3. 前置条件

- 问卷存在且处于可提交版本。
- 提交者和受试者上下文已解析。
- 答案结构满足题型和必填校验。

---

## 4. 流程图

下图表达已确认、待实现的组合提交目标。当前代码仍是 AnswerSheet 落库后由 `answersheet.submitted` worker 创建 Assessment；迁移时保留该消费者作为幂等补偿路径。

```mermaid
sequenceDiagram
    participant U as User
    participant A as actor
    participant S as survey
    participant O as submission orchestrator
    participant E as evaluation

    U->>A: 解析业务 ActorContext
    U->>S: 获取 Questionnaire
    U->>S: 提交 AnswerItem 列表
    S->>S: 校验问卷版本 / 题型 / 必填 / 唯一性
    S->>S: 创建 AnswerSheet
    S-->>O: AnswerSheetID
    O->>E: 幂等创建 Assessment
    E-->>O: AssessmentID
    O-->>U: AnswerSheetID + AssessmentID
    S-->>E: answersheet.submitted（补偿路径）
```

---

## 5. 关键规则

- `AnswerSheet` 创建即代表提交事实，不把前端草稿建成后端领域状态。
- 答卷引用的是问卷和版本，不复制完整问卷模型。
- Survey 自身的提交成功只代表作答事实落库。
- 面向小程序的组合 REST 用例成功边界是 AnswerSheet 与 Assessment 均已持久化，但不等待 Evaluation 或 Report。
- 组合编排不进入 `domain/survey`；跨 Mongo/MySQL 的部分成功通过幂等、outbox 和补偿恢复。

---

## 6. 幂等与异常处理

| 场景 | 处理 |
| ---- | ---- |
| 答案缺失或重复 | 拒绝提交 |
| 问卷版本不可提交 | 拒绝提交 |
| 后续测评暂时失败 | 不回滚 `AnswerSheet`，由 `evaluation` 处理执行失败 |
| Assessment 同步创建失败 | 保留 AnswerSheet，由幂等命令或 `answersheet.submitted` 补偿创建 |
| 重复提交 | 由提交上下文、任务上下文或业务幂等键约束 |

---

## 7. 产出结果

- `AnswerSheet` 主事实。
- `answersheet.submitted` 可靠事件。
- 组合用例返回真实可查询的 `assessment_id`；事件消费者作为补偿路径可返回已有 Assessment。
- 面向统计的答卷提交行为投影。
