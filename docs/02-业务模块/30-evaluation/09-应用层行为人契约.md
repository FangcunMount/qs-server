# Evaluation 应用层行为人契约

## 1. 设计原则

Evaluation 领域层对 `Assessment`、`EvaluationRun`、`Execution` 和 `Outcome` 的规则负责；应用层对发起用例的行为人负责。

一个公开应用服务只能服务一个行为人及一组一致的权限、输入和演化原因。同一行为人可以拥有查询、命令等多个服务。`Committer`、runtime registry、projection、codec 和算法 mechanism 是应用实现机制，不作为独立行为人服务暴露。

## 2. 行为人契约

| 行为人 | 应用能力 | 入口 | 权限与信任边界 | Evaluation 责任 |
| ---- | ---- | ---- | ---- | ---- |
| 答卷编排系统 | `AnswerSheetAssessmentIntakeService` | internal gRPC；终局由 Assessment Intake Journey 调用 | 受信内部服务身份、组织范围、答卷幂等键 | 从已提交答卷确保 Assessment，并推进到 `submitted` |
| 受试者 | `TesteeAssessmentQueryService`；终局补齐受试者评分查询 | collection BFF → Evaluation gRPC | 请求必须携带受试者主体；应用服务校验 Assessment 所有权 | 查询“我的测评”和属于本人的评分事实 |
| 后台操作者 | protected query、recovery、batch execution | apiserver REST | transport 校验 capability；应用服务校验组织和受试者可见范围 | 查看 Assessment/Outcome/Run，重试和批量执行 |
| 评分 Worker | `WorkerExecutionService`、独立结果读取端口 | `evaluation.requested` → internal gRPC | 受信 Worker 身份；Run claim/lease 保证幂等和并发安全 | 完成一次评分尝试并返回 `evaluated` 或失败重试语义 |
| 调度器 | consistency audit service | apiserver scheduler | 进程内调度身份；只读审计 | 检测历史事实漂移，不自动改写事实 |
| 报告查看者 | Interpretation/Journey 查询与等待服务 | REST/gRPC BFF | 对应报告查询的主体和访问范围 | 不属于 Evaluation |

“数据分析系统”尚未形成独立入口和权限契约，不得以未来需求为由复用受试者或后台操作者应用服务。需要时建立单独的 analytics 用例。

## 3. Transport 约束

### REST

REST 路径可以继续按资源组织并保持兼容，但 handler 只负责认证上下文、参数和响应映射：

- Evaluation 查询和命令调用后台操作者应用服务；
- 报告查询和等待调用 Interpretation/Journey；
- 资源所有权、组织范围和受试者可见性不得只存在于 handler。

### gRPC

gRPC 终局按调用者和信任边界拆分：

- Assessment Intake：答卷编排系统；
- Testee Evaluation Query：collection BFF 代表受试者；
- Evaluation Worker：评分 Worker；
- Interpretation Query：报告查看者。

不得使用可选 `testee_id` 在同一个 RPC 中切换“受试者查询”和“内部无权限查询”。新的受试者评分查询必须携带受试者主体；Worker 使用独立内部入口。

## 4. H0 护栏

Batch H0 建立以下可执行约束：

1. 行为人应用端口必须能追踪到真实 REST、gRPC 或 scheduler 入口及权限边界。
2. Worker 结果读取不得复用后台操作者查询服务。
3. 受试者评分和高风险查询必须提供 `testee_id` 并执行所有权校验。
4. Worker 只把 `evaluated` 视为 Evaluation 成功终态；`interpreted` 属于旧报告生命周期语义。
5. gRPC transport 当前的直接 read-model 访问和跨模块答卷编排被记录为只减不增的兼容债。
6. EvaluationService 当前两个报告查询 RPC 被记录为只减不增的兼容债；不得新增报告生成、等待或重试能力。
7. 当前答卷计分、查重、创建、自动提交的调用顺序由表征测试保护，后续迁入 Journey 时不得静默改变。

H0 不完成 gRPC 服务物理拆分，也不把答卷编排迁入 Journey；这些属于后续独立迁移批次。
