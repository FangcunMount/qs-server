# Evaluation 设计问题与重构清单

> 状态：**规划改造**。本文是 Evaluation 的活动重构台账，汇总受理、执行输入、运行时路由、EvaluationRun、Outcome、查询投影和运维恢复中已由当前源码确认的问题。本文中的目标设计不等于已实现能力。

## 1. 本文的用法

本文不再重述 Evaluation 架构，而是回答：

1. 完成本轮领域模型与关键链路分析后，实际发现了哪些问题；
2. 哪些问题会影响业务正确性或可靠恢复，哪些只是扩展性债务；
3. 跨 Survey、ModelCatalog、Actor、Interpretation 的问题由谁保护哪一段语义；
4. 后续开启单独重构时，应从哪个稳定编号开始，如何判断已经关闭。

使用约定：

- 后续分析报告、Issue、提交和 PR 优先引用 `EV-Rxxx`；
- 实施前必须重新阅读当时源码、配置、migration 和真实数据；
- 一次重构只处理已确认的有界范围，不因为都属于 Evaluation 就一次重写整条链路；
- 涉及已受理 Assessment、已提交 Outcome 或历史 ReportInput 时，默认保护历史语义，不就地改写；
- 问题修复后更新状态、实施证据、验收结果和剩余风险，不直接删除条目。

## 2. 30 秒结论

Evaluation 的核心主干已经成立：

```text
AnswerSheet durable fact
  -> Assessment intake
  -> evaluation.requested
  -> EvaluationRun claim / lease
  -> exact InputSnapshot
  -> RuntimeDescriptor
  -> Calculation
  -> Outcome reliable commit
  -> evaluation.outcome.committed
```

当前问题不是“需要推倒重写 Evaluation”，而是五类具体缺口：

1. **受理语义不够显式**：系统仍在 Worker 阶段用 active binding 判断是独立问卷还是测评，`bound=false` 又同时表示多种不同业务结果；
2. **错误分类可能影响恢复**：输入解析失败统一被最终化为不可重试 validation，基础设施瞬时故障也可能进入 terminal；
3. **精确执行仍缺少部分强证据**：NormSubject 未统一物化，InputSnapshotRef 只是 `model:code@version`，运行时还保留 identity/route fallback；
4. **长任务与一致性治理不完整**：EvaluationRun 有 Lease 和 fencing，但没有通用续租；一致性 scheduler 只检测一种 drift；
5. **扩展接口和读模型仍有历史形状**：InputAssembler 使用 context 传递真实输入，新 family 需多处注册，`assessment_score` 仍以量表因子趋势为中心。

治理原则是：

> 先关闭会让系统创建错误业务事实、选错执行输入或错误终止重试的问题；再强化快照、恢复和可观测性；最后才调整 pipeline 接口和投影结构。

## 3. 优先级与状态

### 3.1 优先级

| 级别 | 含义 | 判定标准 |
| --- | --- | --- |
| P0 | 业务事实、结果正确性或可靠恢复风险 | 可能创建无意义 Assessment、使用非预期发布版本、选错常模或将瞬时故障变成 terminal |
| P1 | 执行可追溯性、状态契约和自动恢复能力 | 不一定立即算错，但会降低故障恢复和历史审计可信度 |
| P2 | 扩展性、可维护性和运行时噪声 | 新机制接入需多点修改，接口不表达真实依赖，或运维难以分类故障 |
| P3 | 需要真实业务或合规要求才启动 | 内容 hash、更强 append-only 保护等成本应由审计需求驱动 |

### 3.2 状态

| 状态 | 含义 |
| --- | --- |
| 规划改造 | 当前源码已能确认问题，尚未实施 |
| 待补证据 | 需要生产数据、延迟分布或部署证据确认真实影响 |
| 待业务决策 | 技术方案可实现，但仍缺必要的产品或合规语义 |
| 已实现待验收 | 代码已修改，但还缺数据、故障注入或生产观测证据 |
| 已关闭 | 代码、数据、测试、消费方和文档都已完成，无剩余必做项 |

## 5. P0：先保护测评事实与可靠恢复

### EV-R001：在可靠受理事实中冻结测评意图与 release

**当前事实**

`AnswerSheetSubmittedData` 保存 Questionnaire code/version、testee、filler、TaskID 和提交时间，不保存：

- 这次提交是 independent questionnaire 还是 assessment；
- 应使用哪个 Assessment release；
- 提交当时的 model code/version。

Worker 收到 `answersheet.submitted` 后，Journey 才根据 Questionnaire code/version 查询当时 active binding。因此在答卷被可靠受理到 Worker 解析之间，运营若刚好发布新 release，系统没有一份事实说明“本次提交原本准入哪个 release”。

**与 Plan 使用最新版本不矛盾**

Plan 仅保存 code，每次新 Task 使用当时最新发布版本，这是已确认的业务语义。但当一次 Task/扫码会话已经展示精确 Questionnaire 并提交 AnswerSheet 后，这一次作答应冻结自己的执行语义，而不应继续追踪后续发布变化。

**目标设计**

不把 Assessment 创建重新塞回 HTTP 请求，仍保持：

```text
202 Accepted
  = AnswerSheet + answersheet.submitted Outbox durable
```

但 AnswerSheet 或其可靠受理元数据应冻结一份明确 Admission：

```text
purpose = independent_questionnaire

or

purpose = assessment
assessment_release_ref = kind/subKind/algorithm/code/version
questionnaire_ref       = code/version
```

**验收条件**

- 202 仍以 AnswerSheet + Outbox 为边界，不等待评分或 Assessment 创建；
- independent 与 assessment 不再由 Worker 猜测；
- 运营在事件延迟期发布新版本，已受理答卷仍使用原 release；
- 新 Plan Task 仍在创建时使用最新发布版本；
- admission 与 Questionnaire/Model 不匹配时 fail closed，并形成可治理失败事实。

**证据**

- [`AnswerSheetSubmittedData`](../../../internal/pkg/eventing/payload/answersheet.go)
- [`assessmentintake.Service`](../../../internal/apiserver/application/journey/assessmentintake/service.go)
- [Survey 从作答事实到测评执行](../10-survey/32-关键链路-从作答事实到测评执行.md)

### EV-R002：独立 Questionnaire 不应创建 Assessment

**当前事实**

Journey 的 `Ensure` 即使得到 `bound=false`，仍会调用 `CreateForAnswerSheet`，形成没有 ModelRef 的 pending Assessment。新建后还可能：

- best-effort 完成 Plan Task；
- 写入 queued report-status；
- 让 readiness 返回 ready。

当前 `TestEnsureUnboundAnswerSheetCreatesWithoutAutoSubmit` 明确锁定了这一现状。

**根因**

`bound=false` 同时压缩了：

```text
合法的 independent questionnaire
本应有模型但 binding/release 缺失
binding resolver 没有装配
```

三者需要正常结束、可重试失败或系统配置错误三种不同处理，不能共用一个 boolean。

**目标设计**

```text
independent_questionnaire
  -> base question score derived
  -> no Assessment
  -> no Plan assessment completion
  -> no queued report status
  -> journey result = no_assessment_required

assessment + exact release
  -> create/reuse Assessment
  -> submit
  -> complete Plan Task best-effort
  -> queued report status

assessment but release unavailable
  -> explicit governed failure
  -> retry/manual compensation
```

**验收条件**

- independent 答卷只保留 AnswerSheet 与基础题分；
- 不创建 Assessment，不发 `evaluation.requested`；
- `answersheet.submitted` 重放仍幂等；
- Plan 与 report-status 只在真正 Assessment 建立后更新；
- 现有 unbound pending Assessment 有先审计、后迁移/归档的处置方案。

**跨模块关系**

本项与 ModelCatalog `MC-R001` 指向同一业务问题。ModelCatalog 保护“问卷是否存在可执行 release”；Evaluation 保护“只有明确测评意图才创建 Assessment”。

### EV-R003：把 ModelRef 准入收回 Assessment 聚合

**当前事实**

`Assessment.ApplyScoringProjectionAt()` 已强制 `HasEvaluationModel()`，但 `Assessment.Submit()` 只检查 pending，没有检查 ModelRef。当前主链依赖 Journey 中的 `bound && pending` 条件阻止 unbound submit。

**风险**

领域不变式只存在于某个 application service 的 `if`：

- 新调用方可能绕过 Journey；
- 管理脚本或补偿工具可能直接调用 Intake；
- 聚合仍能发出没有 model identity 的 `evaluation.requested`。

**目标设计**

`Submit()` 在领域层同时强制：

```text
status == pending
ModelRef is complete
QuestionnaireRef is complete
AnswerSheetRef is complete
```

历史 questionnaire-only Assessment 应经过显式兼容/迁移路径读取，不得为了它们继续放宽新写入不变式。

**验收条件**

- unbound Assessment 在聚合层无法 Submit；
- 事件 payload 的 Model identity 必定完整；
- Intake、补偿、管理端和重放测试均覆盖该不变式；
- legacy 数据的处理不会修改已有事实。

**证据**

- [`Assessment.Submit`](../../../internal/apiserver/domain/evaluation/assessment/assessment.go)
- [`Evaluation Intake`](../../../internal/apiserver/application/evaluation/intake/service.go)

### EV-R004：区分语义错误与瞬时依赖故障

**当前事实**

Input Resolver 已能区分 model not found、unsupported model、AnswerSheet not found、Questionnaire not found 和 version mismatch，但它们只表达业务类别，没有统一表达 retryability 和 dependency category。

Engine 对任意 Input Resolver error 都使用：

```text
FailureKind = validation
Retryable   = false
```

因此“发布模型确实不存在”和“MongoDB/MySQL 短暂超时”可能形成相同业务处置：Assessment failed + terminal。

**目标契约**

```text
ResolveFailure
  semantic_kind       model_not_found / version_mismatch / ...
  dependency_category modelcatalog / survey / actor / database / transport
  retryable           true / false
  safe_reason         operator-readable
  cause               internal error chain
```

建议映射：

| 失败 | 业务分类 | 默认处置 |
| --- | --- | --- |
| exact model/questionnaire 明确不存在 | validation | terminal |
| Assessment identity 与快照不一致 | validation | terminal + 人工治理 |
| DB timeout/connection unavailable | dependency | automatic retry |
| context canceled by shutdown | interrupted | 由 Lease recovery 接管 |
| unsupported AlgorithmFamily | configuration | terminal + 发布门禁缺口告警 |

**验收条件**

- repository 错误不再被包装成“不存在”；
- 瞬时依赖故障不会直接进入 terminal；
- 稳定配置错误不会无限自动重试；
- RetryDecision 保留当前持久化授权、预算和审计语义；
- 每种错误均有故障注入或稳定分类测试。

**证据**

- [`evaluationinput.ResolveError`](../../../internal/apiserver/port/evaluationinput/input.go)
- [`evaluationInputWorkflow`](../../../internal/apiserver/application/evaluation/execute/evaluation_workflows.go)
- [`execute.Service`](../../../internal/apiserver/application/evaluation/execute/service.go)

### EV-R005：补齐 NormSubject 物化链

**当前事实**

`InputSnapshot` 已定义 `NormSubjectSnapshot{AgeMonths, Gender}`，Norming 机制也会尝试从该对象构造常模 Subject；但 `RepositoryResolver` 的生产组装只注入 Model、AnswerSheet、Questionnaire 和 Norm repositories，没有 Actor/Testee 人口学快照读取器。

**风险**

需要年龄、性别分层的 Norm 可能：

- 获得空 Subject；
- 命中非预期 generic/band；
- 无法在 Outcome 中证明测评当时使用的人口学事实。

**目标设计**

1. 从 Actor 权威事实读取出生日期与性别；
2. 按 Assessment/AnswerSheet 的发生时间计算 AgeMonths，不按查询当天计算；
3. 将人口学快照纳入 InputSnapshot 与可验证快照引用；
4. 缺失必需人口学数据时，按发布策略明确拒绝或使用 generic fallback，不隐式匹配。

**验收条件**

- 年龄上下界、性别、出生日期缺失、generic fallback 均有 golden test；
- 评分结果能回答实际使用哪个 Norm version/band；
- 历史重试不会因受试者当前年龄增长而改变分层。

**跨模块关系**

ModelCatalog `MC-R002` 负责 Norm 匹配语义与发布约束；Actor 负责人口学事实；Evaluation 负责在正确时点物化并传入 Calculation。

## 6. P1：执行可追溯性、契约与恢复

### EV-R006：只有明确 NotFound 才能进入 Create

**当前事实**

Journey 仅在 `findErr == nil && existing != nil` 时命中幂等记录；其它结果都继续 `CreateForAnswerSheet`。这意味着 Assessment NotFound 与 database timeout/corrupted result 没有严格分支。

**目标设计**

```text
FindByAnswerSheetID
  -> found: reuse
  -> explicit not found: create
  -> dependency/system error: return error, let event settlement recover
```

duplicate insert 后的再查也应对查询错误做同样分类，不能将“再查失败”变成原始 duplicate error。

**验收条件**

- repository NotFound 有稳定 sentinel/code；
- timeout 时 Create 不被调用；
- 并发 duplicate-then-read 仍解析为同一 Assessment；
- 日志和指标可分辨 not_found、duplicate_hit 和 dependency_error。

### EV-R007：重新定义 assessment-readiness

**当前事实**

collection-server 的 readiness 通过 internal gRPC 按 AnswerSheet ID 查找 Assessment。只要能返回 Assessment ID，就返回 `status=ready`，没有返回 Assessment status，也没有 independent questionnaire 结果。

因此当前 ready 实际只表示：

> 按 AnswerSheet ID 存在一条 Assessment。

它不能保证 Assessment 已 submitted，更不保证执行请求已进入 Outbox。

**目标契约**

```text
pending                  Assessment 尚未建立/提交
no_assessment_required   独立问卷正常结束
ready                    Assessment >= submitted，可进入后续状态查询
failed                   准入已形成可治理失败
```

是否要将 failed 直接暴露给小程序，需由产品文案与安全契约决定；但应用层至少不能把 pending/unbound 伪装成 ready。

**验收条件**

- gRPC response 携带稳定 readiness phase，不只有 ID；
- collection-server 对每种 phase 有明确 polling/terminal 策略；
- 安全校验仍保护 writer/testee 关系；
- 兼容窗口内旧客户端不会因新状态无限轮询。

**证据**

- [`ResolveAssessmentByAnswerSheetID`](../../../internal/apiserver/transport/grpc/service/assessment_intake.go)
- [`GetAssessmentReadiness`](../../../internal/collection-server/application/answersheet/submission_service.go)

### EV-R008：严格消费发布时冻结的 RuntimeIdentity

**当前事实**

`RuntimeResolver` 优先从 InputSnapshot 构造 ModelRoute，无法构造时回退 Assessment ModelRef；Provider Registry 对 typology/behavioral_rating 也保留 default identity fallback。

这些兼容路径对历史 snapshot 有价值，但新发布模型若继续依赖它们，Evaluation 仍在执行时做本应在发布时完成的路由决策。

**目标设计**

- 新 AssessmentSnapshot 冻结 AlgorithmFamily、Algorithm、PayloadFormat 和 DecisionKind；
- InputSnapshot 完整传递该 RuntimeIdentity；
- Evaluation 仅验证并查找 exact DescriptorKey；
- 历史数据经过独立 compatibility resolver，命中次数可观测；
- 不支持的组合在发布阶段拒绝，不等到 Worker terminal。

**跨模块关系**

ModelCatalog `MC-R006` 负责冻结 RuntimeIdentity，`MC-R014` 负责 capability 同源；Evaluation 负责严格消费并退出正常路径 fallback。

### EV-R009：让 InputSnapshotRef 能证明实际执行输入

**当前事实**

InputSnapshotRef 当前是：

```text
model:<code>@<version>
```

无模型时兼容退化为 `answersheet:<id>`。该引用会写入 EvaluationRun 和 Outcome，但它没有包含 Questionnaire、AnswerSheet、Norm、Model payload 内容 hash 或 release digest。

**目标不是拷贝整份输入**

应先定义可验证的 composite identity：

```text
model snapshot id/version/digest
questionnaire code/version/digest
answersheet id/content digest
norm table/version/band
subject snapshot digest when applicable
```

具体可以是结构化 `InputSnapshotIdentity` 加 composite hash，而不是把所有 JSON 再存一遍。

**验收条件**

- 任一成分变化都会形成不同引用；
- Run 和 Outcome 保留同一 snapshot identity；
- 重试能验证重新物化内容与原引用一致；
- 算法不依赖 JSON map 的非稳定序列化顺序；
- 在引入 hash 前明确哪些字段属于语义内容。

### EV-R010：为长执行建立 Lease 策略

**当前事实**

EvaluationRun 默认 Lease 为 2 分钟。Engine 在 Claim 时一次设置 `lease_until`，随后解析 InputSnapshot、计算并提交 Outcome，期间没有通用 Renew/Heartbeat。

fencing token 会阻止旧 Worker 在 Lease 过期被接管后提交，因此不会产生两份 canonical Outcome；但可能产生重复计算、旧 Worker 最后 `claim lost` 和额外压力。

**选择顺序**

1. 先量化不同 AlgorithmFamily 的 p95/p99/max 执行时间；
2. 若所有正常任务远小于 2 分钟，保留简单 Lease 并设定时长预算；
3. 若认知任务确实较长，再增加 token-safe heartbeat 或按 family 配置 Lease；
4. 不为了“看起来完整”在短任务上过早引入续租线程。

**验收条件**

- 执行时长按 AlgorithmFamily/descriptor 可观测；
- 续租必须匹配 run ID + attempt + claim token；
- 旧 Worker 不能续租或提交；
- Worker 崩溃后 heartbeat 停止，Scheduler 仍能接管同一 attempt；
- 引入续租后不把进程卡死伪装成永不过期的健康任务。

### EV-R011：建立完整一致性矩阵，但不盲目自动修复

**当前事实**

`evaluation_consistency_reconcile` 当前只扫描 submitted Assessment，并检测：

```text
Outcome exists + Assessment is still submitted
```

它记录指标和告警，不自动改状态。这个保守边界是正确的，因为 Outcome 存在不能单独证明 Run、Projection 和 committed Outbox 都完整。

**缺失的检测矩阵**

| Assessment | Run | Outcome | Projection | committed Outbox | 应分类 |
| --- | --- | --- | --- | --- | --- |
| submitted | running Lease expired | none | none | none | lease recovery candidate |
| submitted | succeeded | exists | maybe | maybe | success projection drift |
| evaluated | succeeded | missing | any | any | canonical outcome missing |
| evaluated | failed/running | exists | any | any | run/status mismatch |
| failed | succeeded | exists | any | any | terminal conflict |
| evaluated | succeeded | exists | missing | exists | rebuildable projection missing |
| evaluated | succeeded | exists | exists | missing | downstream trigger gap |

**目标设计**

- 完整扫描与自动恢复分离；
- 每类 drift 有 severity、evidence 和 recommended action；
- 只对可从 canonical Outcome 确定性重建的 projection 提供自动修复；
- 需要修改 Assessment/Run/Outbox 的情况进入审计迁移或人工确认；
- 任何修复都有 org scope、reason、operator/request ID 与执行结果。

### EV-R012：用数据证据退出 Outcome 历史兼容

**当前事实**

新 Outcome 只写 schema v2，但 `evaluationfact/codec` 仍支持 schema 0、1 和 2。部分历史 scale Outcome 缺少 ReportInput，`ScoreFactReader` 会记录告警，并从当前 Scale catalog 补 factor name/max score。

这一降级保住了可读性，但也意味着历史查询可能混入当前 catalog 展示元数据，不能将它当成精确历史重放。

**正确步骤**

1. 审计生产 Outcome 的 schema_version 分布；
2. 统计各 model kind 缺失/解码失败 ReportInput 的数量；
3. 确认哪些数据能从 exact historical release 回填；
4. 以 dry-run、可重跑、有审计的 migration 补齐；
5. 证明某个 decoder/fallback 命中量归零后再删除。

**不允许的捷径**

- 不用当前模型重算历史 Outcome；
- 不用新文案覆盖历史 code/结果；
- 不在没有生产数据分布证据时删除 decoder；
- 不把 current-catalog fallback 扩展到新 model kind。

## 7. P2：执行扩展性、运行时契约与读模型

### EV-R013：让 InputAssembler 真正组装 CalculationInput

**当前事实**

`InputAssembler.Assemble` 只接收 ModelRoute，并返回主要包含 Route 的 `CalculationInput`。真正的 Assessment 与 InputSnapshot 由 DescriptorExecutor 放入 context，Calculator 再通过 `ExecutionInputFromContext` 取出。

**问题**

- 从 Calculator 签名看不出它依赖完整业务输入；
- InputAssembler 的名字与实际职责不一致；
- context 成为隐式参数通道；
- 纯 Calculation 边界更难独立测试与复用。

**目标设计**

```text
InputAssembler.Assemble(ExecutionInput)
  -> explicit mechanism-neutral CalculationInput

Calculator.Calculate(ctx, CalculationInput)
  -> raw result
```

context 仅携带 cancellation、deadline、trace 和 request metadata，不携带业务输入。

**实施边界**

- 先用现有 descriptor golden tests 保护行为；
- 逐个 family 迁移，不同时改算法；
- 保留 OutcomeAssembler 对机制结果的边界；
- 不把 ModelCatalog DTO 直接暴露给纯计算内核。

### EV-R014：收敛新 AlgorithmFamily 的注册面

**当前事实**

增加一个全新 family 通常需同步：

- ModelCatalog capability/ExecutionPath；
- runtime materialization spec；
- InputProvider 物化；
- RuntimeDescriptor Registry；
- Pipeline components attach；
- Outcome detail/codec；
- container wiring 与架构测试。

这些注册点各自有真实职责，不应粗暴合并成一个巨型 switch；但当前缺少一份能验证“某 family 已全链路注册”的聚合描述。

**目标设计**

- 使用一份 capability/descriptor manifest 描述 identity、route、provider factory 和 pipeline factory；
- 各模块仍拥有自己的实现，manifest 只做组装与完整性验证；
- 启动或发布测试能在漏注册时立即失败；
- 同类新 model code 仍只需配置/发布，不需新增 manifest。

**启动条件**

当前四类 family 已稳定，本项可在第五个全新机制出现前做小步收敛，不必为了减少文件数量提前建设动态插件系统。

### EV-R015：事件 payload 不应在 canonical Assessment 之前决定是否执行

**当前事实**

Worker handler 先调用 `EvaluationRequestedData.NeedsEvaluation()`。若 payload 没有 `model_code` 或 legacy `scale_code`，handler 直接返回 nil，消息被 ACK，不会回读 Assessment。

这为历史 questionnaire-only 事件保留了宽松兼容，但会让一条 payload 损坏或不完整的有效测评请求静默结束。

**目标设计**

- EV-R002/003 关闭后，新 `evaluation.requested` 必须携带完整 identity；
- payload 缺失应进入 invalid/poison/legacy 分类，不默认当作正常 questionnaire-only；
- 是否执行的最终事实来自 canonical Assessment；
- 兼容分支命中量可观测，有退出条件。

### EV-R016：减少 active claim 的重投噪声

**当前事实**

当另一个 Worker 持有有效 Run Lease 时：

```text
Engine Claim -> Claimed=false
Worker Service receipt -> Assessment submitted + Run running
handler -> no durable terminal disposition
MQ -> NACK
```

这是“宁可重投，不误 ACK 不确定状态”的保守设计，正确性优先。只有当指标证明重复消息和长计算会造成明显 NACK 压力时，才值得调整。

**候选改进**

- 在持久化回执中显式返回 `processing + lease_until`；
- transport 支持受控 delayed requeue，避免紧密重投；
- 或依赖原消息的后续 redelivery，但根据 Lease 剩余时间设定 backoff。

**不能做的事**

不能因为“已有 Worker 在处理”就无条件 ACK 唯一恢复触发。必须先证明原执行者崩溃后还有 scheduler/relay 能稳定恢复。

### EV-R017：建立 internal gRPC 错误分类

**当前事实**

`AssessmentIntakeService.EnsureAssessment` 和 `EvaluationWorkerService.ExecuteEvaluation` 除基本参数校验外，将 application service 的大部分错误统一映射为 gRPC `Internal`。

Worker 仍会对错误 NACK，因此不会直接丢消息；但运维和 transport 层无法仅通过 code 区分：

- invalid argument；
- not found；
- failed precondition；
- unavailable/deadline；
- claim lost/aborted；
- canonical consistency failure。

**目标设计**

建立统一 `toEvaluationGRPCError`，保留安全外部消息，同时在 structured details/metadata 中提供内部错误分类。不应把数据库 SQL、隐私数据或完整内部 error chain 直接暴露给调用方。

### EV-R018：按查询需求建立模型无关或专用投影

**当前事实**

`assessment_score` 存储：

```text
assessment_id
testee_id
factor_code / factor_name
raw_score
risk_level
is_total_score
```

它非常适合医学量表的因子趋势，也已与 Outcome ID 关联，是可重建查询投影；但不适合自然表达人格 profile、pole preference、行为常模派生分或认知任务 validity。

**目标原则**

- Outcome 继续是唯一不可变事实；
- `assessment_score` 可继续服务 scale factor trend，不必为了“统一”强行替换；
- 只在出现稳定查询需求时，为 typology/behavioral/cognitive 建立专用 read model；
- 投影必须能从 Outcome 幂等重建；
- 趋势的模型版本、因子 code 连续性和跨版本可比性必须明确。

**启动条件**

当前业务已确认需要患者级 factor trend；如果还没有人格/认知趋势或组合检索需求，不应预先建设一个“万能结果表”。

### EV-R019：在真实演进时为 ReportInput 增加 schema identity

**当前事实**

Outcome 有 `SchemaVersion=2`，但该版本主要描述 Outcome fact payload。ReportInput 只根据 Model kind 解码成 Scale/Typology/Behavioral/Cognitive payload，没有独立 `report_input_format` 和 `report_input_schema_version`。

**当前结论**

这是已知演进点，但不是必须立即实施的缺陷。在只有一种结构时，为版本号而版本化会增加无效复杂度。

**启动条件**

任一条成立时启动：

- 同一 Model kind 出现第二种不兼容 ReportInput；
- 模型发布需要在不改 Outcome schema 的情况下演进解释资产；
- Interpretation 需并行支持两种解码器。

目标字段可以是：

```text
report_input_format
report_input_schema_version
report_input_digest
```

## 8. P3：由合规要求驱动的 Outcome 防篡改

### EV-R020：在需要时增强 Outcome 内容完整性证明

**当前事实**

Outcome Repository 只提供创建和查询，Assessment ID 与 Run ID 唯一约束阻止多份 canonical Outcome。但 MySQL longtext payload 的不可变主要依赖：

- application repository 不提供 Update/Delete；
- 数据库账号权限；
- 组织管理约定。

对当前辅助测评业务，这一强度可以接受。不应在没有合规目标时盲目引入区块链或复杂签名系统。

**启动条件**

- 医疗/心理合规要求证明结果未被篡改；
- 存在多系统复制或长期归档后的完整性校验需求；
- 运维需要定期检测 payload/report input 损坏。

**候选能力**

- canonical serialization + payload/report input digest；
- Outcome 创建审计与数据库权限隔离；
- 定期完整性扫描；
- 必要时使用组织级密钥签名，并建立密钥轮换与验证策略。

## 12. 不应被重构破坏的已确认决策

1. 202 Accepted 表示 AnswerSheet 与 Outbox 已可靠提交，不表示 Evaluation 已完成；
2. Questionnaire 可以独立使用，只有绑定已发布 AssessmentModel 后才创建可执行 Assessment；
3. Plan 保存 code，每个新 Task 使用当时最新发布版本；已受理测评的精确版本不再漂移；
4. Worker 只拥有异步消费、调用和 settlement，不拥有 Evaluation/Interpretation 核心业务规则；
5. EvaluationRun 分离执行 attempt，Assessment 只表达业务结果状态；
6. MQ 重投、Outbox 发布尝试和 business retry attempt 是三个不同预算；
7. 失败已持久化 automatic/manual_required/terminal RetryDecision 后，当前消息 ACK，后续由治理事件接管；
8. Lease recovery 接管同一 attempt，不消耗新的 business retry budget；
9. Calculator 保持无状态，Evaluation 编排输入、路由、计算与提交；
10. 只有 Outcome、Assessment evaluated、Run succeeded、适用的查询投影和 committed Outbox 在可靠事务中成立，评分才算成功；
11. Outcome 是不可变评分/判定事实，`assessment_score` 和 Assessment summary 是可重建/兼容投影；
12. `Assessment=evaluated` 不表示 Report 已生成，Interpretation 失败不得回写 Evaluation 成 failed；
13. 系统产生医生判断、治疗观察与随访的辅助信息，不产生医学诊断。
