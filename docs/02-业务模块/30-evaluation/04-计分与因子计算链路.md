# 计分与因子计算链路

## 1. 业务目标

把答卷答案和**发布态模型快照**中的 Factor 规则结合，计算维度原始分、总分、等级和结构化结果。

Evaluation **读取** ModelCatalog 快照里的 Factor，**写出** `AssessmentOutcome.Dimensions`（`DimensionResult`）。不在此层重新定义维度结构。

`assessment_score` 是该结构化结果的持久化评分事实之一，归属 Evaluation。它不是 Report 投影；同一次评估可以基于同一组评分事实生成不同受众、模板或版本的报告。

---

## 2. 参与对象

| 对象 | 层 | 角色 |
| ---- | -- | ---- |
| `AnswerItem` | survey | 单题答案输入 |
| `FactorSnapshot` | model-catalog | 快照中的维度定义（题目归属、计分策略、解释/常模引用） |
| `AlgorithmFamily` | model-catalog（派生） | 决定 Factor 如何被解释（记分/分类/常模/任务表现） |
| `AssessmentOutcome` | evaluation | 结构化执行结果 |
| `DimensionResult` | evaluation | 单维度得分、等级、派生分（T 分、百分位等） |
| `assessment_score` | evaluation / MySQL | 因子或维度评分事实的查询投影 |

命名对照见 [../04-术语表.md](../04-术语表.md)：catalog 侧 **Factor**，输出侧 **Dimension**。

## 3. 流程图

```mermaid
flowchart LR
    Snapshot["PublishedModelSnapshot"]
    Factors["FactorSnapshot[]"]
    Answers["AnswerItem[]"]
    Family["AlgorithmFamily"]
    Executor["Family Executor"]
    Raw["原始 DimensionResult"]
    Enrich["算法增强 可选"]
    Outcome["AssessmentOutcome"]

    Snapshot --> Factors
    Answers --> Executor
    Factors --> Executor
    Family --> Executor
    Executor --> Raw --> Enrich --> Outcome
```

典型路径（**包名** / **`AlgorithmFamily` 枚举**）：

- **scoring** / `factor_scoring`：题目分 → Factor 聚合 → 分数区间解释 → `DimensionResult`
- **norming** / `factor_norm`（Brief-2）：先走 scale-like 原始分，再由 `EnrichBrief2Outcome` 按 `dim.Code` 查常模补 T 分/百分位
- **typology** / `factor_classification`：维度分 → 极性/类型组合 → `ProfileResult` + `DimensionResult`
- **task_performance** / `task_performance`（SPM）：任务/题组判定 → 能力等级（可选常模）

---

## 4. 关键规则

- 题目分计算依赖**发布态快照**中的 Factor，不依赖可变问卷配置或草稿模型。
- **Factor 分是输入规则的结构化结果**，不是报告文案；报告由 `interpretation` 生成。
- 等级、T 分、百分位等应保存在 `DimensionResult` / `DerivedScores`，便于追溯。
- `risk_level / level_code / severity` 若由发布态模型规则确定，属于 Evaluation 事实；对应的解释文案属于 Interpretation。
- `factor_name` 可以作为执行时模型元数据快照随评分事实保存，但权威模型定义仍在 ModelCatalog。
- `conclusion / suggestion`、报告章节、图表和受众化表达不属于 `assessment_score` 的目标事实模型。
- `behavioral_rating` / `cognitive` 由已发布 `DefinitionV2` 构造各自执行 DTO，并复用共享因子计分机制；payload 只保留 wire/runtime DTO 职责。

### 4.1 评估事实与报告投影

| 维度 | 评估事实 | 报告投影 |
| ---- | -------- | -------- |
| 产生阶段 | Evaluation | Interpretation |
| 输入 | AnswerSheet + PublishedModelSnapshot | 持久化 EvaluationOutcome + 模板/受众 |
| 典型内容 | 原始分、标准分、维度分、分类、等级代码 | 结论文案、建议、章节、图表、受众表达 |
| 稳定性 | 同一输入和模型版本应保持一致 | 可随报告模板、受众和版本变化 |
| 重建 | 不随报告重建而覆盖 | 可从评估事实独立重建 |

当前 `AssessmentScorePO` 同时保存 `RawScore / RiskLevel` 与 `Conclusion / Suggestion`，属于历史混合模型。目标态以前者为权威评分事实，后两者迁入 Report 或停止作为权威字段写入；迁移期保留兼容列不等于继续扩大其职责。

### 4.2 EvaluationOutcome 高于 assessment_score

`assessment_score` 主要服务分数型和维度型查询，不是所有机制的统一结果：

- scoring 产出维度分数；
- typology 产出分类与 profile；
- norming 产出常模位置与等级；
- task_performance 产出任务表现和能力结果。

统一事实边界是 canonical `EvaluationOutcome`；`assessment_score` 只是其中一种持久化明细投影。

## 6. 异常处理

| 场景 | 处理 |
| ---- | ---- |
| 答案无法映射规则 | 执行失败或记录不可计分项 |
| 快照中 Factor 定义缺失 | 执行失败，不生成半成品报告 |
| 规则版本不匹配 | 使用执行记录引用的模型快照排查 |
| 常模表与 Factor 代码不匹配 | Brief-2 增强跳过该维度或记录无效 |
| 报告生成失败 | 保留 EvaluationOutcome 与 assessment_score，Report 独立失败和重试 |
