# 核心设计：评估事实与数据存储

## 1. 本文回答

本文说明 Evaluation 成功后究竟保存哪些事实，为什么 `evaluation_outcome` 高于 `assessment` 摘要和 `assessment_score` 投影，schema v2 如何分离评分事实与报告素材，以及 Assessment、Run、Outcome 和查询投影如何落在 MySQL。

## 2. 30 秒结论

| 存储 | 性质 | 能否作为评分事实源 |
| --- | --- | --- |
| `assessment` | 业务生命周期 + 结果摘要 | 否，摘要不足以重建完整结果 |
| `runtime_checkpoint(scope=evaluation_run)` | 执行尝试、claim、lease 和失败审计 | 否，它证明执行过程 |
| `evaluation_outcome` | 不可变 canonical 评估事实 | **是** |
| `assessment_score` | 量表维度/趋势查询投影 | 否，不覆盖 typology、norm、task performance 的全部结果 |
| `domain_event_outbox` | 可靠事件交付记录 | 否，只传递事实引用和进度 |

```text
Execution (in memory)
  -> schema v2 pure-fact payload
  -> EvaluationOutcome Record
       + ModelIdentity
       + RuntimeIdentity
       + InputSnapshotRef
       + frozen ReportInput
  -> read-only evaluationfact port
  -> Interpretation / Worker receipt / Evaluation queries
```

## 3. Execution 不等于 Record

Evaluator 返回的 `Execution` 可含计算期的标签和临时 detail，它只是成功提交的输入。Committer 会从中生成新 Outcome ID，固化模型/运行时身份和输入引用，再把适合长期持久化的内容编码为 schema v2。

这个分离保护三个不变式：

- 计算成功不等于事实已提交；
- 只有与 Assessment、Run、score projection 和 Outbox 同事务提交的 Record 才可对外读取；
- Preview 可以使用 transient Execution，但不得伪造 Outcome Record 或触发生产状态迁移。

## 4. Outcome Record 结构

| 字段组 | 主要字段 | 用途 |
| --- | --- | --- |
| 来源 | outcome ID、org ID、assessment ID、testee ID、run ID | 业务归属和追溯 |
| 模型身份 | kind、subKind、algorithm、code、version、title | 固化执行时使用的模型身份 |
| 运行时身份 | AlgorithmFamily、DecisionKind、PayloadFormat | 说明当时如何路由和判定 |
| 输入审计 | InputSnapshotRef | 可读的模型/答卷快照引用 |
| 评分事实 | Payload + SchemaVersion | 版本化 canonical Execution 事实 |
| 报告素材 | ReportInput | 执行时 ModelPayload 的冻结 JSON |
| 时间 | EvaluatedAt | Assessment、Run、Outcome 共用的成功时间 |

Record 不保存 AnswerSheet 和 Questionnaire 的完整副本。`InputSnapshotRef` 用于审计，`ReportInput` 当前只冻结 `InputSnapshot.ModelPayload`，为 Interpretation 提供当时的模型规则与展示素材。

## 5. schema v2：纯事实 payload

新写入固定使用 `SchemaVersion=2`。`MarshalRecordV2` 序列化：

- ModelRef（清除 title）；
- Detail 中的纯计分/分类事实；
- Primary score、Level code、Profile code/traits；
- Dimensions 中的分数、派生分、等级、偏好和强度；
- Validity 的 code 与 passed 事实。

新 payload 不持久化：

- Summary / Tags；
- 分数、等级、维度和效度的展示 label/name/message；
- typology 的人格文案、建议、优势、弱项、图片和稀有度。

typology 分类的 v2 detail 只保留 TypeCode、Pattern、MatchPercent/Similarity 和 special trigger 等评分事实。展示文案从同一 Record 中冻结的 ReportInput 按选中 code 解析，不从当前发布模型偷换历史语义。

## 6. 版本兼容边界

`evaluationfact/codec` 当前接受 schema `0 / 1 / 2`：

| Schema | 用途 | 政策 |
| --- | --- | --- |
| 0 / 1 | 历史 Outcome 读取 | 只读兼容，包含 legacy factor/typology detail 解码 |
| 2 | 当前生产新写 | 纯评分事实 + 冻结 ReportInput |
| 其他 | 未知 | 拒绝读取，不猜测 payload |

schema v1 decoder 是历史数据兼容门禁，不是新生产路径。删除它需要先完成生产数据 schema 审计和必要的数据修复。

`DecodeReportInput` 按 Record.Model.Kind 恢复 scale、typology、behavioral_rating 或 cognitive ModelPayload。新 v2 报告路径应使用该冻结值；只有历史缺失 ReportInput 的量表评分查询会记录告警并回退当前 catalog 补充 factor name/max score。

## 7. MySQL 存储模型

### 7.1 `assessment`

| 字段组 | 主要内容 |
| --- | --- |
| 业务归属 | org_id、testee_id、origin_type/origin_id |
| 输入引用 | questionnaire code/version、answer_sheet_id |
| 模型引用 | model kind/subKind/algorithm/code/version/title |
| 状态 | status、submitted_at、evaluated_at、failed_at、failure_reason |
| 摘要投影 | primary score、level、severity；保留 total_score/risk_level 兼容字段 |
| 通用审计 | ID、version、created/updated/deleted 字段 |

`answer_sheet_id` 有唯一索引，是 Assessment intake 的最终业务幂等保障。Assessment 摘要供列表、风险队列和兼容 API 使用，不允许从摘要反向构造 Outcome。

### 7.2 `runtime_checkpoint`

EvaluationRun 不再使用独立 `evaluation_run` 表，而是写入共享 `runtime_checkpoint`，并固定 `scope=evaluation_run`：

| 字段组 | 主要内容 |
| --- | --- |
| 身份 | scope、resource_id(run ID)、attempt_no、assessment_id |
| 状态 | status、started_at、finished_at |
| 失败 | error_code(FailureKind)、error_message、retryable |
| 审计 | trace_id、input_snapshot_ref |
| 并发控制 | claim_token、lease_expires_at |

物理唯一键为 `(scope, resource_id, attempt_no)`。更新 claimed Run 还需匹配 claim token 与 running status。

### 7.3 `evaluation_outcome`

Outcome Repository 只有 `Save / FindByID / FindByAssessmentID`，没有 Update 和 Delete。表上的关键物理约束：

- primary key: outcome ID；
- unique `assessment_id`；
- unique `evaluation_run_id`；
- model identity 索引用于按模型排查；
- `payload_json` 和 `report_input_json` 为 longtext JSON，解析由版本化 codec 负责。

“不可变”由两层共同保护：Repository 不暴露变更操作，assessment/run 唯一索引拒绝重复成功提交。

### 7.4 `assessment_score`

该表每行保存一个 factor 分数：assessment ID、outcome ID、testee ID、factor code/name、raw score、risk level 和 total-score 标记。新写入必须携带 `evaluation_outcome_id`，证明投影来源。

它服务趋势和因子查询，不保存 conclusion/suggestion，也不表达 typology profile 或 task performance 的全部事实。

### 7.5 `domain_event_outbox`

Evaluation 复用 MySQL 通用 Outbox。`Stage` 强制要求活动事务，事件通过 unique event_id 幂等落库，并记录 status、attempt_count、next_attempt_at、last_error 和 published_at。它保护的是事件交付，不是 Outcome payload 备份。

## 8. 读模型与下游契约

| 读取者 | 首选事实 | 说明 |
| --- | --- | --- |
| Evaluation Worker 回执 | Outcome Record + latest Run | Assessment=evaluated 但 Outcome 缺失是一致性错误 |
| Testee / Operator 单次评分 | Outcome 解码 | 不从 `assessment_score` 反向伪造 Outcome |
| 趋势查询 | `assessment_score` | 读取多次 Assessment 的 factor 投影 |
| Interpretation | `evaluationfact.Repository` | 只读 Record，通过 codec 解码 Execution 和 ReportInput |
| Scheduler | Assessment + Outcome | 只读检测漂移，不修改事实 |

`port/evaluationfact.Repository` 只暴露 `FindByID` 和 `FindByAssessmentID`，这是 Evaluation 与 Interpretation 之间的稳定边界。下游不得直接导入 Evaluation application/domain 的可写对象。

## 9. 数据排查顺序

某个 Assessment 长时间未完成时，按事实因果顺序查询：

```sql
SELECT id, status, answer_sheet_id, evaluation_model_code,
       submitted_at, evaluated_at, failed_at, failure_reason
FROM assessment
WHERE id = ? AND deleted_at IS NULL;

SELECT resource_id, attempt_no, status, retryable, error_code,
       input_snapshot_ref, claim_token, lease_expires_at, started_at, finished_at
FROM runtime_checkpoint
WHERE scope = 'evaluation_run' AND assessment_id = ? AND deleted_at IS NULL
ORDER BY attempt_no DESC, id DESC;

SELECT id, evaluation_run_id, model_kind, model_algorithm, model_code,
       model_version, algorithm_family, decision_kind, schema_version, evaluated_at
FROM evaluation_outcome
WHERE assessment_id = ?;

SELECT evaluation_outcome_id, factor_code, raw_score, risk_level, is_total_score
FROM assessment_score
WHERE assessment_id = ? AND deleted_at IS NULL;

SELECT event_type, status, attempt_count, last_error, created_at, published_at
FROM domain_event_outbox
WHERE aggregate_type = 'Evaluation' AND aggregate_id = ?
ORDER BY id;
```

判定原则：

- Assessment=submitted 且有活动 lease：仍在执行或等待 lease 过期；
- Assessment=failed：看 latest Run 的 FailureKind/Retryable，不只看 failure_reason 文本；
- Assessment=evaluated 但 Outcome 缺失：一致性故障，不能返回空成功结果；
- Outcome 存在但 Assessment=submitted：历史漂移，由 audit 报警并审计修复；
- Outcome 存在但报告缺失/失败：属于 Interpretation 链路，不否定 Evaluation 事实。
