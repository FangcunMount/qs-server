# Evaluation 设计终局

## 1. 文档目的

本文是 Evaluation 后续重构的判定基线。实现细节可以变化，但任何改动都不得突破本文定义的事实归属、状态边界和跨模块契约。

## 2. 终局职责

Evaluation 负责把答卷和已发布模型快照执行为可信、可重放的评分事实；它不生成报告成品，也不以报告结果改写评分事实。

```text
AnswerSheet + Published Model Snapshot
  → Assessment
  → EvaluationRun (attempt 1..n)
  → EvaluationOutcome (immutable fact)
  → score projections / evaluation.outcome.committed
```

## 3. 终局领域模型

| 对象 | 身份与生命周期 | 责任 |
| ---- | -------------- | ---- |
| `Assessment` | 一次测评业务实例；`pending → submitted → evaluated / failed` | 关联答卷、受试者、模型身份与评分摘要；`evaluated` 是成功终态 |
| `EvaluationRun` | 一次执行尝试；可有多个 | 记录 attempt、输入快照、trace、运行状态、失败分类和重试资格 |
| `EvaluationOutcome` | 每次成功 Run 的不可变事实 | 固化评分/分类事实、运行时身份、输入快照引用、冻结 ReportInput 与 schema 版本 |
| score projection | Outcome 的查询投影 | 供趋势、风险、工作台等查询；不保存报告结论或建议 |

`Assessment` 只表达“评分是否已经可靠成立”；`EvaluationRun` 只表达“某次执行是否成功”；`EvaluationOutcome` 才是 Interpretation 可重复消费的事实源。

## 4. 不变量

1. `Assessment.evaluated` 表示 Outcome、Run、必要 score projection 已可靠提交。
2. `EvaluationRun.succeeded` 不包含报告生成成功。
3. Outcome 一经成功提交不可被报告重试、报告失败或读侧投影修改。
4. 报告失败不得调用 `Assessment.MarkAsFailed()`，不得清除分数、等级或摘要。
5. 新的 Evaluation 重试必须创建新的 Run；不得覆盖旧 Run 的失败审计。
6. `assessment_score` 是评分投影，不承载 conclusion、suggestion 等报告内容。

## 5. 事件与事务边界

| 事实 | 可靠事件 | 同事务提交 |
| ---- | -------- | ---------- |
| 可执行 | `evaluation.requested` | Assessment `submitted` 与 outbox |
| 成功 | `evaluation.outcome.committed` | Outcome、score projection、Assessment `evaluated`、EvaluationRun `succeeded` 与 outbox |
| 失败 | `evaluation.failed` | Assessment `failed`、EvaluationRun `failed` 与 outbox |

`evaluation.outcome.committed` 是 Interpretation 的唯一生产触发输入。它不是“生成报告命令”，而是“评分事实已经成立”的通知。

## 6. 明确不属于 Evaluation

- `ReportGeneration`、`InterpretationRun`、`InterpretReport` 的状态或持久化。
- 报告模板、结论、建议、渲染内容。
- 报告失败重试与报告失败事件。
- 客户端 `completed / interpreted` 组合进度。

## 7. 完成判据

- Evaluation 容器不接收 ReportBuilder、ReportSaver 或 Interpretation 用例。
- Evaluation 生产路径不内联调用报告生成。
- `evaluation.outcome.committed` 之后的所有报告处理只读取 Outcome。
- Assessment 领域状态不含 `interpreted`。
- 任何报告失败测试都证明 Assessment、Run、Outcome 与 score 不变。

## 8. 与 Interpretation 的接口

Evaluation 输出稳定的 Outcome 事实；Interpretation 对该事实创建报告生成意图和尝试。两者通过事件和 Outcome ID 协作，不共享可写聚合。

参见：[Interpretation 设计终局](../40-interpretation/08-设计终局.md)。

## 9. Batch E 结构收敛状态

| 项目 | 终局约束 | 当前状态 |
| ---- | -------- | -------- |
| 评分事实契约 | Interpretation 只通过显式 port 消费 Outcome 事实，不引用 Evaluation application/domain 实现包 | 已建立 port 边界并由跨模块 import 守卫保护 |

### 事实契约与运行时注册收敛

- `EvaluationCommitter` 只接受 `CommitRequest`：Assessment、InputSnapshot、Execution、DescriptorKey、EvaluationRun 与统一完成时间；应用层不再维护介于 Execution 和 Record 之间的可变 Outcome carrier。
- `port/evaluationfact` 只定义不可变、只读的 Record/Execution 契约；payload 与 report input 以防御性副本暴露。版本化反序列化位于独立 `evaluationfact/codec`，schema v1 legacy decoder 在生产数据审计完成前保留。
- Preview 使用独立 `PreviewOutcome` 组合 Evaluation Execution 与 Interpretation builder，不伪造持久化 Record，也不复用生产 Committer。
- `RuntimeDescriptorRegistry` 是运行能力唯一注册源。Input provider 直接按 Registry 导出的 `ExecutionPath` 物化，不再经过 `ModelDescriptor` 投影，也不注册 MBTI/SBTI/BigFive 算法别名 executor。
| Assessment/Run 关系 | Run 通过 `assessment_id` 归属 Assessment；Assessment 不持有 `current_run_id` | 已删除领域字段、PO 映射、双写与索引，并提供数据库迁移 |
| 报告进度 | Report status 属于 Interpretation/Worker/Journey，Evaluation 不保存也不对外导出 | 已迁出 Evaluation 容器及执行服务 |
| 执行入口 | Worker 单次执行与后台操作者批量用例分离 | `execute.Engine` 仅服务 Worker；同步批量编排已迁入 `application/evaluation/operator` |
| 一致性审计 | Scheduler 只读检测并告警历史漂移，不自动修改事实 | `consistency` 已并入 `scheduler.AuditOnce`，指标时间序列保持不变 |
| typology 兼容适配 | 新执行身份只使用 canonical model identity | 已删除 MBTI/SBTI/BigFive `ExecutionIdentity` 别名与 application legacy scorer；仅 `evaluationfact/codec` 保留 schema v1 decoder |

schema v1 decoder 是数据兼容门禁，不是新生产主路。删除它必须附带可复核的生产数据审计结果与必要的数据修复记录。

## 10. Batch H 应用层收敛状态

- 公开应用入口按 `intake / testee / operator / worker / scheduler` 五类行为人组织；`execute / outcome / runtime / registry / calculationadapter` 只作为内部机制。
- 跨模块答卷流程由 `journey/assessmentintake` 编排；AnswerSheet ID 是业务幂等键，Assessment 创建前不得完成 collection submit，Plan 回写与 ReportStatus 仍为 best-effort。
- gRPC 已删除 actor-neutral `EvaluationService`。Evaluation proto 暴露 `TesteeEvaluationService`、`AssessmentIntakeService`、`EvaluationWorkerService`；Interpretation proto 暴露 `ParticipantReportService` 与 `InterpretationAutomationService`。
- 通用 `InternalService` 不再声明 Evaluation/Interpretation 流程 RPC；Worker 侧通过对应 actor client 调用。
- REST URL 保持不变，后台请求进入携带 `Actor{OrgID, OperatorUserID}` 的 Operator 服务；报告查询仍进入 Interpretation/Journey。
- `submit-status=done` 的协议不变量是同时返回 `answersheet_id` 与 `assessment_id`，不再在状态查询时异步补齐。

## 11. Batch I 终局收敛状态

`application/evaluation` 的公开行为人包固定为 `intake/testee/operator/worker/scheduler`。旧 `assessment/runquery/consistency` 包、通用 ScoreQuery、WorkerResultReader 和兼容构造器均已删除。REST、gRPC、数据库、领域状态机、Outcome Record 与生产事件名称保持原契约。

## 12. Batch J 语义纯化

- Operator 的 Query、Recovery、Batch 共用包私有授权协作者；命令执行顺序固定为 Actor、Assessment、机构、受试者访问关系、状态。批量预检未全部通过前不得执行 Evaluator。
- `domain/evaluation/routing` 只保存 `ExecutionIdentity / ModelRoute / DescriptorKey` 及算法家族、DecisionKind 的纯推导规则。`RuntimeDescriptor`、Calculator、Assembler、执行上下文和 Registry 位于 `application/evaluation/runtime/descriptor`。
- `EvaluationRun` 的状态全部私有；仓储使用 `ReconstructInput`，执行方只能经 `Claim / AttachInputSnapshot / Succeed / Fail` 修改。
- Outcome 新写 schema 固定为 v2。v2 payload 只含评分、分类、层级、维度、特殊规则与运行时事实，不含 summary、suggestion、strengths、weaknesses、commentary、图片、稀有度或完整模型配置。
- schema 0/1 继续由 versioned codec 只读兼容；未知 schema 拒绝。v2 报告文案必须用 Outcome 中的选中代码从同一 Record 冻结的 ReportInput 解析，缺失代码属于不可恢复的输入/配置错误，禁止回退到当前发布模型。
- Scheduler 使用 submitted Assessment ID keyset cursor 分批只读遍历；到达末尾后下一轮从起点开始。Assessment、Outcome、candidate reader 任一缺失都必须报告模块配置错误。
- 兼容不变量保持：REST/gRPC、数据库表与列、领域状态机、生产事件名、scheduler 配置/锁名及 `qs_evaluation_consistency_*` 时间序列不变。

## 13. Batch K 终局清债

- Worker 的 `evaluated` 回执必须读取 canonical Outcome；Assessment 只提供生命周期状态，Run 提供执行审计元数据。Outcome 缺失属于一致性错误，不允许返回空 Outcome ID。
- schema v2 writer 会清除 ModelRef、Profile、Dimension、Score、Level 与 Validity 中的展示文本，只持久化评分、分类、层级和运行时事实。历史 schema 0/1 与已有 schema v2 继续宽容读取。
- Scale factor 的 MaxScore 由冻结执行输入进入 Outcome；Score 查询优先读取 Outcome 与冻结 ReportInput，仅历史缺失记录允许回退到当前 ScaleCatalog。
- `DecisionKindForFamily` 是算法家族默认判定类型的唯一领域规则。生产执行先解析一次 RuntimeDescriptor，再使用同一个 `ResolvedExecution` 完成计算、日志和提交。
- Outcome Repository 的查询契约以 `(nil, nil)` 表示未找到；Scheduler 不感知 GORM 或其他基础设施错误类型。
- Workbench 最新风险队列拥有独立 `workbenchreadmodel` port；Evaluation Module 不再公开 Worker Engine、Assessment Reader 或 LatestRiskReader。
- Preview 当前仍使用富 transient typology detail。它不进入 schema v2 持久化，但在 Preview 完成纯事实输入切换前不得删除。
