# Evaluation 模块定位与边界

## 1. 模块定位

Evaluation 是测评执行层。它接收作答事实和模型快照，完成计分、因子计算、等级判定和结构化结果生成。

---

## 2. 负责范围

| 范围 | 说明 |
| ---- | ---- |
| 测评实例 | `Assessment` 的创建和生命周期 |
| 执行尝试 | `EvaluationRun`、错误、重试 |
| 模型执行 | 加载快照，选择执行器，运行规则 |
| 结果生成 | 总分、因子分、等级、原始结果 |
| 执行事件 | 提交、完成、失败等事件 |

`Assessment` 归属本模块。目标边界中，它在结构化评估结果可靠落库后进入 `evaluated`，生命周期到此结束；报告是否已生成由 Interpretation 的 Report 状态表达。

---

## 3. 不负责范围

| 不负责 | 归属模块 |
| ------ | -------- |
| 问卷定义 | `survey` |
| 模型资产发布 | `model-catalog` |
| 报告模板和解释文案 | `interpretation` |
| Report 生命周期与生成失败 | `interpretation` |
| 面向客户端的全流程完成态 | Journey / ReadModel 投影 |
| 周期任务调度 | `plan` |
| 统计读模型 | `statistics` |

---

## 4. 边界反例

- `Evaluation` 不是 `AnswerSheet`。
- `EvaluationResult` 不是 `InterpretReport`。
- `AssessmentModelSnapshot` 不是 Evaluation 的内部可变配置。
- `Assessment=evaluated` 不代表报告已经生成。
- 报告生成失败不能把 Assessment 从 `evaluated` 改成 `failed`。
- legacy `interpreted` 可以是查询投影，但不应继续作为 Assessment 的领域终态。

---

## 5. 当前实现状态

Batch 4 已删除 `Assessment.ApplyOutcome` 和持有 Assessment Repository 的旧 Interpretation Writer。`MarkAsFailed` 仅允许从 `submitted` 进入 Evaluation 失败；`evaluated` 后的 Report 失败不能改写 Assessment。对旧 API 暴露的 `interpreted` 由查询层根据 `Assessment=evaluated + Report=generated` 派生。

Batch 5 已从 Evaluation 的 `WiringDeps`、container facade 和 runtime materialize 中移除 `ReportBuilder` 装配；旧的 Interpretation `ScoreProjector` materialize 入口也已删除，生产分数投影只走 EvaluationCommitter。Evaluation runtime registry 现在只物化 Evaluator；typology 的报告 builder、adapter registry 和模板选择实现已迁入 Interpretation。

Batch 6 删除 Redis scoring snapshot 与 `EVALUATION_ASYNC_INTERPRETATION` 开关。`EvaluationOutcome` 是唯一的可恢复评分事实；旧 consistency reconciler 由它恢复 `Assessment=evaluated` 投影。Assessment 存储以 `evaluated_at` 取代 `interpreted_at`，并在迁移时将历史 `interpreted` 行归一为 `evaluated`。`assessment_score` 不再保存 conclusion/suggestion，解释文本只由 Report 持有。
