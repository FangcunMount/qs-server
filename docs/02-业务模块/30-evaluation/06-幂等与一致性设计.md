# 幂等与一致性设计

## 1. 设计目标

Evaluation 通常由事件驱动，必须处理重复消费、部分失败、下游异步和历史追溯。

---

## 2. 幂等边界

| 边界 | 幂等依据 |
| ---- | -------- |
| 从答卷创建测评 | `AnswerSheetID`、模型绑定、任务上下文 |
| 执行模型快照 | `AssessmentID`、`SnapshotID`、执行尝试 |
| 写入 EvaluationOutcome / assessment_score | `AssessmentID`、`EvaluationRunID`、模型快照引用 |
| 发布事件 | outbox 可靠出站语义 |

---

## 3. 一致性原则

| 场景 | 原则 |
| ---- | ---- |
| 答卷已提交但执行失败 | 不回滚 `AnswerSheet` |
| 执行完成但报告未生成 | 保留 `EvaluationResult`，由报告层补偿 |
| 执行完成但统计未更新 | 主链路成功，统计最终一致 |
| 模型配置变更 | 历史执行仍引用旧快照 |
| Report 失败或重建 | 不覆盖 EvaluationOutcome，不新增 EvaluationRun |

EvaluationRun `succeeded`、Assessment `evaluated`、EvaluationOutcome 和 `assessment.evaluated` outbox 应尽量位于同一可靠提交边界。若受跨存储限制无法原子完成，必须由 consistency reconciler 明确定义检测与修复规则，不能让长期的 `Assessment=evaluated + Run=running` 成为正常状态。

Batch 4 后，Evaluation consistency reconciler 只检测和修复“评估产物已落库但 Assessment 仍为 submitted”的 Evaluation 终态窗口。它不再检测 `report_without_interpreted_status`，也不再通过 `ApplyOutcome` 修复 Assessment。Report 的 pending/generating/generated/failed 一致性归 Interpretation 状态机。

---

## 4. 重试边界

Evaluation 可以重试执行失败，但不应重试 Survey 答卷提交，也不应直接重试报告渲染。跨模块补偿通过事件、任务或运维入口协调。

Report 重试属于 Interpretation。它读取已持久化 EvaluationOutcome，不调用 Evaluation runtime resolver，不改变 Assessment，也不创建新的 EvaluationRun。
