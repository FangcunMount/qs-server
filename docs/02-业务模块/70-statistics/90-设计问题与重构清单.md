# Statistics 设计问题与后续优化清单

> 状态：**V2-only 重构已经进入最终验收阶段**。V1 退役、内部正名和调用方切换属于已经完成的结构性工作，不再列为后续重构项。本文只记录当前唯一 Statistics 架构中仍需用运行证据关闭的问题，以及未来达到明确触发条件后才值得投入的优化。

## 1. 使用原则

这份清单不是愿望池。一个条目只有同时满足以下条件才进入实施：

1. 问题可以由代码、数据、执行计划或故障演练证明；
2. 对业务正确性、可恢复性、容量或维护成本有明确影响；
3. 已经确定验收方式；
4. 不以重新引入 V1 双轨、实时 Projector 或持久 Checkpoint 为解决方案。

当前优先级：

| 优先级 | 含义 |
| --- | --- |
| P0 | 上线或宣布重构完成前必须关闭 |
| P1 | 当前架构的重要可靠性与容量保护 |
| P2 | 有明确触发条件的后续优化 |

## 3. ST-001：Migration 完整演练

### 问题

Migration 56 删除 V1 表并完成快照正名，Migration 57 补齐 canonical Collector 来源索引。即使业务历史数据已经清空，也必须证明它们不误删三类 Fact、五类 Result、SyncRun 或任何业务表。

### 必须验证的三条路径

1. 从 migration 55 前向执行 56、57；
2. 从空库执行全部 migration；
3. 依次执行 57、56 down，恢复上一版镜像所需的空 V1 Schema。

前两条路径最终得到的 canonical Statistics Schema 必须一致。Down 只验证结构回退能力，不声明恢复旧统计数据。

### 验收

- 最终只存在三张 Fact、五张 Result 和 SyncRun；
- `statistics_org_snapshot` 的列、索引和约束一致；
- 三类事件扫描索引存在；
- 业务表和 canonical Statistics 数据不被删除；
- migration contract test 与真实 MySQL 结果一致。

## 4. ST-002：冷启动端到端验收

一次性环境至少包含：

- MySQL；
- 单节点 MongoDB Replica Set；
- Redis；
- qs-apiserver 与 qs-worker 所需最小运行组件。

建立最小 Actor、Entry、AnswerSheet、Assessment、Outcome、InterpretReport、PlanEnrollment 和 Task 后，执行一次 publish，逐层核对：

```text
权威数据
  → Access / Assessment / Plan Fact
  → 五类 Result
  → Read Service
  → /api/v2/statistics
```

然后重跑相同窗口，Fact 和 Result 行数不得增长，聚合值不得漂移。

## 5. ST-003：首次生产发布

历史数据清空不等于不需要初始化。没有成功的 SyncRun 时，Read Service 必须返回 `statistics_not_ready`；因此发布后需要为每个启用机构执行一次 publish。

如果清空发生在当天、最近完整日没有业务数据，仍执行空 publish，用于建立：

- `statistics_org_snapshot`；
- `as_of_date/snapshot_at`；
- 首个机构级 Cache Generation；
- 可审计的成功 SyncRun。

验收时记录：

- 机构；
- Run ID；
- 窗口；
- Collector 计数；
- Projection 结果计数；
- Generation；
- API freshness。

## 6. ST-004：Collector 数据库级验证

每个生命周期来源已经拆成单独扫描，后续验收必须证明：

- 使用 `(event_time,id)` 跨页时无遗漏、无重复；
- 恰好落在窗口边界的数据只属于一个窗口；
- 迟到数据能被 repair 重新采集；
- 相同 FactKey、相同核心字段是幂等成功；
- 相同 FactKey、不同核心字段使整个 Run 失败；
- 来源查询命中 `(org_id, deleted_at, event_time, id)` 索引；
- 单页 500 时不会退化为多时间列 OR 全表扫描。

新来源接入时必须复用这组合同测试。

## 7. ST-005：Organization Snapshot 不变量

Organization Snapshot 表达“截至最近发布水位，机构当前资源状态与累计业务量”，不是历史资源趋势。

必须守卫：

- 当前 clinician、entry、testee 等资源数来自当前 MySQL 读模型；
- 累计测评量来自 Fact；
- 结果事务内不扫描 MongoDB；
- 删除或转移资源不会反写历史 Fact 归属；
- Snapshot 不伪装成任意历史日期的资源快照；
- 若未来需要资源历史趋势，应新增明确的维度历史或 Resource Daily，而不是回填当前状态。

## 8. ST-006：Plan Fulfillment 容量边界

第一版按机构全量重建 Plan Fulfillment，这是符合当前规模的奥卡姆选择。只有出现以下任一证据才启动增量化设计：

- 单机构重建耗时逼近夜间批次 SLA；
- 锁持有时间影响其他机构运行；
- Enrollment/Task 数据量使全量查询稳定超过容量预算；
- 故障恢复窗口无法接受。

届时优先按受影响 Enrollment 重建，不引入通用 Metric DSL 或分布式计算平台。

## 9. ST-007：缓存故障闭环

需要通过故障注入验证四个停点：

| 故障点 | 期望结果 |
| --- | --- |
| Collector 失败 | Run failed，已提交 Fact 可幂等重采 |
| Projection 失败 | Result 事务整体回滚 |
| 结果提交后进程退出 | Run 保持 `data_committed` |
| Generation/预热失败 | 不重复计算，使用 `resume-cache` 闭环 |

查询侧还需验证：

- Redis 可用时命中当前 Generation；
- Redis 不可用时优先返回带 stale 语义的 L1；
- 没有 stale 时由 LoadGuard 限制 MySQL 回源；
- 超限返回 503，不让故障流量直接打垮数据库。

## 10. ST-008：保留与归档

当前不提前删除 Fact 或 SyncRun。连续运行三个月后，根据以下证据制定策略：

- 每日 Fact 增长量；
- SyncRun 记录增长量；
- repair 最大回看窗口；
- 审计和问题定位实际使用周期；
- 索引体积和查询延迟。

归档策略不得破坏指标重建所需的事实窗口，也不得把 Statistics 变成业务事实的唯一保存者。

## 11. ST-009：Plan 规则修订

当前 PlanEnrollment 固定一轮参与，Task 保存本轮实际计划、截止和完成状态，足以支持履约统计。

只有当业务允许“同一 Enrollment 运行中修改周期策略并要求按旧规则解释历史任务”时，才需要引入 PlanScheduleRevision。触发前不增加该概念。

## 12. 已关闭且不得重新打开的迁移项

以下内容已经从活动设计中移除：

- V1/V2 影子对账；
- Statistics API 版本开关；
- Behavior Projector、Journey Scanner、Pending Reconcile；
- V1 三段夜间同步；
- `ProjectBehaviorEvent`；
- V1 Statistics REST 和缓存治理兼容入口；
- 历史 Statistics 回填；
- `statisticsv2` 内部包名；
- 旧 Redis Statistics Key 主动扫描删除。

若未来出现新需求，应基于 canonical Collector、Fact、Projection 和 Run 扩展，不能通过恢复这些机制解决。

## 13. 宣布完成的门槛

P0 项全部关闭，并同时满足：

- 全量 Go 测试与静态 ratchet 通过；
- qs-operating-system 测试和生产构建通过；
- OpenAPI 与 gRPC 生成物一致；
- Migration 真实环境演练通过；
- 冷启动 E2E 通过；
- 首次生产 publish 成功；
- 所有机构 API 具有明确 freshness；
- Statistics 故障不影响答卷受理、测评执行和报告生成。

P1/P2 不阻塞结构性重构完成，但必须保留负责人、触发条件和验收证据，避免再次演化成无法解释的统计旁路。
