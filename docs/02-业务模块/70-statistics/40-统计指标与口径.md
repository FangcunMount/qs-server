# Statistics 统计指标与口径

> 状态：**当前指标词典**。本文定义 Statistics 指标的名称、事实来源、时间归属、维度、分母和查询语义。任何实现和调用方文案都必须以本文与当前 API 契约为准。

## 1. 本文回答

1. 每个统计字段究竟数什么；
2. 它是存量、动作流量还是 cohort 履约；
3. 按哪个时间归日、从哪类 Fact 计算；
4. Questionnaire、AssessmentModel、Outcome、Report 怎样区分；
5. 完成率、按时率和零分母怎样解释。

## 2. 30 秒结论

一个可用指标必须同时定义：

```text
名称
业务问题
统计对象
事实来源
所属 Projection
时间归属
维度
去重方式
分子/分母
新鲜度
不包含什么
```

Statistics 最重要的三条口径边界是：

1. AnswerSheet、Assessment、Outcome、Report 是四个阶段，不能互相近似；
2. Plan Activity 按事件发生日，Plan Fulfillment 按 planned/due cohort；
3. Organization Snapshot 是 `snapshot_at` 的当前资源观察，Daily 完整性由 `as_of_date` 表达。

每个指标必须归属且只归属于一个 Projection。Data Collector 只构建 Fact，不计算查询指标；Projection Engine 只调度，不通过分支拥有指标口径。

## 3. 通用规则

### 3.1 统计类型

| 类型 | 例子 | 时间语义 |
| --- | --- | --- |
| 当前存量 | active clinician、active entry | snapshot_at 时当前状态 |
| 累计存量 | total assessment、total report | 截至 snapshot_at 的累计对象数 |
| 动作流量 | answersheet submitted、task completed | occurred_at 所属自然日 |
| Cohort | due task、on-time completion | planned_at/due_at 所属自然日 |
| 去重主体数 | enrolled testee、participating testee | 指定窗口内 DISTINCT testee |

不能用同一个“count”混合这些时间语义。

### 3.2 业务日

- 时区：`Asia/Shanghai`；
- 日窗口：`[00:00, next 00:00)`；
- Daily 最新完整日：`as_of_date`；
- Statistics 不承诺实时 `today`；
- 7d/30d 包含 `as_of_date`。

### 3.3 动作次数与人数

除明确带有 `testee`、`clinician` 或 `distinct` 的指标外，所有 Fact count 都是事实发生次数。例如：

- 同一人打开 Entry 三次，`entry_opened_count=3`；
- 同一患者提交两份答卷，`answersheet_submitted_count=2`；
- 同一 Task 只能产生一次 `task_completed`，由 FactKey 保护。

### 3.4 空归属

医生、Entry、Plan 等不适用或无法恢复时进入未归属桶。事实层使用 `NULL`；结果层可以使用明确技术维度键表示 `unattributed`，但 API 不得把它解释成 ID 为 0 的业务对象。

## 4. Organization Snapshot

| 指标 | 定义 | 来源 | 时间 |
| --- | --- | --- | --- |
| `testee_count` | 机构内未删除 Testee 数 | Actor | snapshot_at |
| `clinician_count` | 机构内未删除 Clinician 数 | Actor | snapshot_at |
| `active_clinician_count` | 当前 active Clinician 数 | Actor | snapshot_at |
| `entry_count` | 机构内未删除 Entry 数 | Actor | snapshot_at |
| `active_entry_count` | 当前有效且未过期 Entry 数 | Actor | snapshot_at |
| `active_enrollment_count` | 状态为 active 的 PlanEnrollment 数 | Plan | snapshot_at |
| `answersheet_submission_count` | 已可靠受理 AnswerSheet 累计数 | AssessmentFact | snapshot_at |
| `assessment_count` | 已创建 Assessment 累计数 | AssessmentFact | snapshot_at |
| `report_count` | 已生成 InterpretReport 累计数 | AssessmentFact | snapshot_at |
| `content_count` | 已发生统计事实的内容身份去重数 | AssessmentFact | snapshot_at |

### 4.1 Content 身份

```text
content_identity = content_kind + content_code
```

建议类型：

```text
questionnaire
scale
typology
behavioral_rating
cognitive
```

同一个 code 如果分属不同 kind，视为不同内容。独立 Questionnaire 和绑定模型后的测评也不能因为 code 相同被无条件合并。

### 4.2 Snapshot 不是历史序列

Snapshot 只保存最近一次资源观察，不回答“一个月前有多少 active clinician”。如果未来需要资源历史趋势，应增加明确的 Resource Daily/Dimension History，而不是把当前状态回填成历史。

OrganizationSnapshotProjection 只直接读取 Testee、Clinician、Entry、active Enrollment 等当前 MySQL 资源状态；AnswerSheet、Assessment、Report 累计量从 Fact 读取。结果事务内不扫描 MongoDB。

## 5. Access Funnel

| 指标 | FactType | 统计对象 | 时间归属 |
| --- | --- | --- | --- |
| `entry_opened_count` | `entry_opened` | Entry 成功解析次数 | resolved_at |
| `intake_confirmed_count` | `intake_confirmed` | Intake 确认次数 | intake_at |
| `testee_created_count` | `testee_created` | Intake 中新建 Testee 的次数 | intake_at |
| `care_relationship_established_count` | `care_relationship_established` | 新建服务关系次数 | occurred_at |
| `care_relationship_transferred_count` | `care_relationship_transferred` | 服务关系转移次数 | occurred_at |

### 5.1 为什么节点不保证单调

这些指标按各自事实发生时间独立统计，不是同一用户 cohort：

- 同一人可多次打开 Entry；
- Intake 可能使用已有 Testee；
- Intake 不一定新建关系；
- 关系可能在其他流程建立。

因此某日：

```text
entry_opened >= intake_confirmed >= testee_created
```

不是必须成立的不变量。

### 5.2 转化率

第一版不提供严格 cohort 转化率。若展示简单比值，必须命名为窗口比值，例如：

```text
intake_to_open_ratio = intake_confirmed_count / entry_opened_count
```

并明确它不是“打开入口的人最终有多少完成 Intake”的用户级漏斗。

## 6. Assessment Service

| 指标 | FactType | 含义 | 时间归属 |
| --- | --- | --- | --- |
| `answersheet_submitted_count` | `answersheet_submitted` | 最终 AnswerSheet 可靠受理数 | submitted_at |
| `assessment_created_count` | `assessment_created` | Assessment 创建数 | created_at |
| `outcome_committed_count` | `outcome_committed` | EvaluationOutcome 可靠提交数 | committed_at |
| `assessment_failed_count` | `assessment_failed` | Assessment 进入失败事实数 | failed_at |
| `report_generated_count` | `report_generated` | InterpretReport 成品生成数 | generated_at |
| `report_failed_count` | `report_failed` | 报告生成失败事实数 | failed_at |

### 6.1 独立 Questionnaire

没有模型绑定的 Questionnaire：

```text
answersheet_submitted_count += 1
assessment_created_count     不增加
outcome_committed_count      不增加
report_generated_count       不增加
```

这不是失败，而是信息收集业务的正常终点。

### 6.2 节点不保证同日单调

同一 AnswerSheet 的 Assessment、Outcome、Report 可能发生在不同日期。窗口内节点数不保证：

```text
submitted >= created >= outcome >= report
```

严格漏斗需要按 `answersheet_id/assessment_id` 做 cohort 分析，第一版 Daily 提供的是阶段流量。

### 6.3 Assessment completion

内容统计中的“测评完成”统一指：

```text
Outcome committed
```

不是：

- Assessment created；
- Report generated；
- Assessment 状态字段的模糊近似。

## 7. Assessment 维度

AssessmentDaily 支持：

```text
org
clinician
entry
origin_type
questionnaire_code
model_kind
model_code
```

### 7.1 归属维度

- 新数据使用受理时 frozen attribution；
- 历史数据可能是 derived_legacy/unknown；
- 归属质量应可单独统计；
- 查询历史医生/Entry 时不能再按当前关系重新分组。

### 7.2 版本维度

Questionnaire/Model version 保存在 Fact 中，第一版 Daily 不按版本拆行。原因是当前主要查询按内容 code，版本进入 Daily 会显著增加稀疏组合。

如果未来需要版本级质量对比，应增加专用查询或版本 Daily，不能静默修改现有 code 级指标。

## 8. Clinician Statistics

### 8.1 当前资源

| 指标 | 定义 | 来源 |
| --- | --- | --- |
| primary/attending/collaborator testees | 当前各关系类型下去重患者数 | Actor Relation |
| total accessible testees | 当前授权可访问患者去重数 | Actor/Authz |
| active entry count | 当前有效 Entry 数 | AssessmentEntry |

### 8.2 窗口指标

| 指标 | 来源 |
| --- | --- |
| intake count | AccessDaily by clinician |
| assigned count | care relationship established by clinician |
| answersheet/assessment/outcome/report | AssessmentDaily by frozen clinician |

当前资源和窗口行为来自不同时间语义，API 必须分别放在 Snapshot/Window 结构，不能合并成一个不带时间说明的数字。

## 9. Entry Statistics

| 指标 | 定义 |
| --- | --- |
| resolve/open count | 指定 Entry 的 `entry_opened` 次数 |
| intake count | 指定 Entry 的 `intake_confirmed` 次数 |
| assigned count | 指定 Entry 的关系建立次数 |
| answersheet submitted | 归属该 Entry 的 AnswerSheet 数 |
| outcome committed | 归属该 Entry 的 Outcome 数 |
| last resolved at | AccessFact 中最大 resolved occurred_at |
| last intake at | AccessFact 中最大 intake occurred_at |

Entry 当前是否 active、目标内容和过期时间直接读取 AssessmentEntry 业务对象。

## 10. Content Batch

### 10.1 Questionnaire

```text
total_submissions = answersheet_submitted_count
total_completions = 对独立问卷不定义测评完成
```

如果现有 API 必须返回 completion，独立问卷应返回“不适用”能力标志，而不是强行将提交同时算作完成。

### 10.2 AssessmentModel

```text
total_submissions = assessment_created_count
total_completions = outcome_committed_count
completion_rate   = total_completions / total_submissions
```

模型类型由 `model_kind` 区分，不再只支持 Questionnaire/Scale 二分。

### 10.3 Report

报告生成率可以单独定义：

```text
report_generation_rate
  = report_generated_count / outcome_committed_count
```

它不是模型计算完成率。

## 11. Plan Activity

### 11.1 Enrollment Activity

| 指标 | FactType | 归属时间 |
| --- | --- | --- |
| `enrollment_joined_count` | enrollment_joined | joined_at |
| `enrollment_closed_count` | enrollment_closed | closed_at |
| `enrollment_terminated_count` | enrollment_terminated | terminated_at |
| `enrolled_testee_count` | DISTINCT testee with joined | joined_at window |

### 11.2 Task Activity

| 指标 | FactType | 归属时间 |
| --- | --- | --- |
| `task_created_count` | task_created | created_at |
| `task_opened_count` | task_opened | open_at |
| `task_completed_count` | task_completed | completed_at |
| `task_expired_count` | task_expired | expired_at |
| `task_canceled_count` | task_canceled | canceled_at |

`participating_testee_count` 定义为窗口内至少发生一次 Enrollment/Task Activity 的去重 Testee，不等于当前 active Enrollment 人数。

当前 active Enrollment 数属于 OrgSnapshot。

## 12. Plan Fulfillment

### 12.1 Planned Cohort

```text
planned_task_count
  = planned_at 落在窗口内且未取消的 Task 数
```

### 12.2 Due Cohort

```text
due_task_count
  = due_at 落在窗口内且未取消的 Task 数
```

Due Cohort 内：

```text
completed_task_count
  = 最终状态已完成

on_time_completed_count
  = completed_at <= due_at

overdue_task_count
  = completed_at > due_at
    OR 到固定 cutoff 仍未完成且 due_at < cutoff
```

`canceled_task_count` 单独记录，用于解释为什么原计划任务没有进入履约分母。

### 12.3 比率

```text
completion_rate
  = completed_task_count / due_task_count

on_time_completion_rate
  = on_time_completed_count / due_task_count
```

分母为零时：

```text
rate = 0
has_sample = false
```

### 12.4 集合关系

正常情况下：

```text
on_time_completed_count <= completed_task_count <= due_task_count
overdue_task_count <= due_task_count
```

逾期完成同时属于 completed 和 overdue，所以：

```text
completed + overdue
```

不能直接与 due 比较是否相等。

## 13. Testee Periodic

患者周期明细直接来自 PlanEnrollment 和 Task：

| 字段 | 口径 |
| --- | --- |
| enrollment round | 同一患者参与同一 Plan 的轮次 |
| start date | Enrollment.start_date |
| task seq | Enrollment 内任务序号 |
| planned at | 计划时间 |
| due at | 截止时间 |
| completed at | 实际完成时间 |
| status | Task 当前业务状态 |
| assessment id | 完成该 Task 的 Assessment |

项目完成率按该 Enrollment 内应履约 Task 计算，不能跨多个 Enrollment 轮次混合。

## 14. 新鲜度字段

每个统计响应至少应能够解释：

| 字段 | 含义 |
| --- | --- |
| `as_of_date` | Daily 已完整到哪一天 |
| `snapshot_at` | 当前资源何时扫描 |
| `is_stale` | 是否来自旧 Generation/降级 |

如果返回 stale 数据，必须保留原 `as_of_date`，不能把响应时间写成数据时间。

## 15. 对账规则

### 15.1 Access

```text
ResolveLog count
  == entry_opened Fact count

IntakeLog count
  == intake_confirmed Fact count
```

派生的 created/relationship 数按布尔条件对账。

### 15.2 Assessment

分别对账：

```text
AnswerSheet -> submitted Fact
Assessment -> created/failed Fact
Outcome -> committed Fact
Report -> generated Fact
```

不能只比较总数，也不能要求各阶段相等。

### 15.3 Plan

```text
Enrollment lifecycle -> PlanFact
Task lifecycle times  -> PlanFact
ActivityDaily         -> 按 occurred_at 汇总
FulfillmentDaily      -> 按 planned/due cohort 汇总
```

### 15.4 API

API 对账必须跳过缓存或指定相同 Generation，然后从响应逐层追溯到 Daily、Fact 和业务来源。

## 16. 当前口径相对旧实现的变化

以下内容用于解释为什么当前指标不沿用旧近似语义，不表示运行时仍有两套口径：

| 已退役口径 | 当前口径 |
| --- | --- |
| Assessment submitted 近似 AnswerSheet | 直接统计 AnswerSheet |
| evaluated 近似 Report generated | Outcome 与 Report 分开 |
| Questionnaire/Scale 二分 | Questionnaire 与四类 Model kind 正交 |
| `active_testees` 语义模糊 | participating testee 与 active enrollment 分开 |
| 履约查询时使用 `time.Now()` | 使用固定 as_of cutoff |
| today 混合实时/日聚合 | latest complete day |
| 当前关系回推历史归属 | frozen attribution |

这些变化已经固化到 DTO、OpenAPI 和调用方文案。后续不得只修改 SQL 而静默恢复旧语义。

## 17. 指标变更规则

修改任何指标前必须回答：

1. 业务问题变了吗，还是旧实现有误？
2. 事实来源是否变化？
3. 时间归属是否变化？
4. 分母、去重或空值是否变化？
5. 是否需要新版本字段/API？
6. 历史结果是否重算？
7. 调用方文案是否同步？
8. 如何对账和回滚？
9. 指标归属哪个 Projection，是否导致一个指标出现两条计算路径？

禁止：

- 不改字段名却静默改变业务语义；
- 用当前关系重算历史医生归属；
- 用 Report 数代替 Outcome 数；
- 用状态更新时间代替业务发生时间；
- 直接修改 Daily 数字修复差异。
- 在 Data Collector 或 Projection Engine 中复制具体 Projection 的指标计算逻辑。

## 18. 验收场景

- 独立问卷提交不产生测评失败；
- 同一 Entry 多次打开按次数计数；
- 新建 Testee 与建立关系分别统计；
- Outcome 与 Report 跨日时进入不同日期；
- 历史医生关系变化不改变 frozen 新数据；
- 同一患者两轮 Enrollment 分开统计；
- 逾期完成同时计入 completed 和 overdue；
- canceled Task 不进入 due 分母；
- 分母为零时 rate 与 has_sample 语义正确；
- stale 响应保留旧 as_of_date。
