# 核心设计：业务数据、事实与统计分层

> 状态：**当前唯一设计**。九张 canonical Statistics 表、三类 Fact 和五类结果已经由代码与 migration 实现；真实 MySQL migration 演练和冷启动端到端验收仍需完成。

## 1. 本文回答

1. 为什么不能继续让统计接口直接拼装多套业务表和投影；
2. Business Data、Fact、Result 在数据所有权上怎样分开；
3. 三类 Fact 与五类结果表分别保存什么；
4. 哪些字段必须结构化，哪些细节可以进入 JSON；
5. 哪些当前数据可以迁移，哪些必须先修改业务模块。

## 2. 30 秒结论

Statistics 的分层规则是：

> 业务数据层回答“业务真相是什么”；事实层回答“哪些可统计事件已经发生”；结果层回答“按稳定口径观察得到什么”。

```text
业务数据层：模块拥有、按业务事务写入
事实层：Statistics 拥有、按稳定来源身份构建
结果层：Statistics 拥有、按窗口删除并确定性重建
缓存层：可丢弃副本，不属于真值
```

权威业务数据仍然按所属模块保存在 MongoDB 或 MySQL；Statistics 自己拥有的 Fact、Result 和 SyncRun 第一阶段统一使用现有 MySQL。MongoDB AnswerSheet 由 Data Collector 批量读取并构建为 MySQL Fact。当前不建设独立统计数据库实例，也不复制完整业务维表。

Data Collector 和 Projection Engine 是两类应用行为，不是第四、第五个数据层：

```text
权威业务数据
  - Data Collector -> 标准事实层
  - Projection Engine -> 统计结果层
```

## 3. 所有权矩阵

| 数据 | 写入者 | 权威性 | Statistics 能否修改 |
| --- | --- | --- | --- |
| Testee / Clinician / Entry | Actor | 业务真值 | 否 |
| AnswerSheet / Admission | Survey | 业务真值 | 否 |
| Assessment / Outcome | Evaluation | 业务真值 | 否 |
| InterpretReport | Interpretation | 业务真值 | 否 |
| PlanEnrollment / Task | Plan | 业务真值 | 否 |
| Access / Assessment / Plan Fact | Statistics Data Collectors | 标准化统计事实 | 仅受控重建 |
| Daily / OrgSnapshot | Typed Projection Engine 中的显式 Projection | 派生结果 | 可以重建 |
| SyncRun | Statistics Runtime | 运行账本 | 按状态机更新 |
| Redis Statistics Cache | Cache | 可丢弃副本 | 可以删除、切换、预热 |

## 4. 第一层：业务数据

### 4.1 Access 来源

当前可直接复用：

- `assessment_entry_resolve_log`：一次 Entry 解析；
- `assessment_entry_intake_log`：一次 Intake，并带有是否创建 Testee/Assignment 的结果；
- Actor 关系变更的持久化数据或事件：关系建立、转移。

Access 中“打开过一次入口”是瞬时事实，不能只从当前 Entry 状态恢复。因此 Resolve/Intake 日志仍需保留，它们不是多余的统计表。

### 4.2 Assessment 来源

来源跨越 MongoDB 与 MySQL：

| 阶段 | 权威对象 | 关键时间 |
| --- | --- | --- |
| AnswerSheet submitted | MongoDB AnswerSheet | `submitted_at` |
| Assessment created | MySQL Assessment | `created_at` |
| Outcome committed | MySQL EvaluationOutcome | `committed_at` |
| Assessment failed | Assessment / Run | `failed_at` |
| Report generated | InterpretReport | `generated_at` |
| Report failed | Interpretation Run/Generation | `failed_at` |

Statistics 不从 Assessment 一个对象推断全部阶段，也不把 `evaluated_at` 解释为报告生成时间。

### 4.3 Plan 来源

当前来源：

- `plan_enrollment`：加入、自然关闭和提前终止；
- `assessment_task`：计划、开放、完成、过期、取消；
- 必要的 Task 生命周期时间字段。

PlanEnrollment 已经持久化，并与初始 Task、终止和 Task 终态迁移共同维护周期事实。详细边界见 [Plan 文档](../60-plan/README.md)。

## 5. 第二层：统一 Fact 头

三类 Fact 使用统一技术结构：

| 字段 | 含义 | 约束 |
| --- | --- | --- |
| `id` | Statistics 内部主键 | 无业务语义 |
| `org_id` | 机构作用域 | 必填 |
| `fact_key` | 稳定幂等身份 | 全表唯一 |
| `fact_type` | 事实词汇 | 受枚举/注册表约束 |
| `occurred_at` | 上海时间下实际发生时刻 | 必填，`DATETIME(3)` |
| `stat_date` | 上海自然日 | 从 occurred_at 派生 |
| `source_type` | 来源类型 | 必填 |
| `source_ref` | 来源稳定身份 | 必填 |
| `schema_version` | Fact 载荷版本 | 从 1 开始 |
| `payload_json` | 非核心扩展信息 | 可空，不承担常用过滤 |
| `created_at` | Fact 写入时间 | 不等于发生时间 |

推荐索引：

```text
UNIQUE(fact_key)
INDEX(org_id, stat_date, fact_type)
INDEX(source_type, source_ref)
```

### 5.1 为什么不是一个通用 Fact 大表

Access、Assessment 和 Plan 的身份、查询维度和生命周期差异很大。如果使用一个只含 JSON 的通用事件表：

- 每次聚合都要解析 JSON；
- 无法建立针对性索引；
- 字段兼容只能靠运行时约定；
- 一个领域新增字段会污染所有查询；
- 容易退化为没有治理的埋点平台。

因此三类 Fact 共享头部、分开承载领域字段。

## 6. `statistics_access_fact`

### 6.1 结构化字段

```text
clinician_id          NULL
source_clinician_id   NULL
entry_id              NULL
testee_id             NULL
target_type           NULL
target_code           NULL
```

无法归属时使用 `NULL`，不能用 `0` 伪装业务身份。

### 6.2 来源到事实映射

| 来源 | 条件 | 产生 Fact |
| --- | --- | --- |
| Resolve Log | 每一行 | `entry_opened` |
| Intake Log | 每一行 | `intake_confirmed` |
| Intake Log | `testee_created=true` | `testee_created` |
| Intake Log | `assignment_created=true` | `care_relationship_established` |
| Relation transfer | 每次确认的转移 | `care_relationship_transferred` |

同一 Intake Log 可以派生多行 Fact：

```text
source_ref = intake_log_id
fact_key   = intake:{id}:{fact_type}
```

不要求为此额外引入 `operation_id`。

## 7. `statistics_assessment_fact`

### 7.1 结构化字段

```text
testee_id / filler_id
answersheet_id / assessment_id / outcome_id / report_id

questionnaire_code / questionnaire_version
model_kind / model_code / model_version

clinician_id / entry_id
origin_type / origin_id
plan_id / enrollment_id / task_id
attribution_mode
```

跨存储业务 ID 在采集前必须验证。进程内 Run ID、Generation ID 等非数字身份如果需要追踪，使用明确的字符串列或 `payload_json`，不能强行转换为数值主键。

### 7.2 新数据归属

Survey 在 `202 Accepted` 可靠受理边界内保存：

```text
AnswerSheet
+ Admission
+ AttributionSnapshot
+ answersheet.submitted Outbox
```

Collector 读取冻结快照，不查询当前医生、Entry 或服务关系。

### 7.3 归属不变量

新 AnswerSheet 必须在可靠受理边界内冻结 `AttributionSnapshot`。Collector 只读取这份冻结归属；关系后来发生变化时，已经形成的 Assessment Fact 不重新归属。无法确认的身份保持 `NULL`，禁止使用当前关系猜测或伪造。

## 8. `statistics_plan_fact`

### 8.1 结构化字段

```text
plan_id
enrollment_id
testee_id
task_id
task_seq
scale_code
planned_at
due_at
completed_at
task_status
```

### 8.2 Enrollment 事实

| FactType | occurred_at | 说明 |
| --- | --- | --- |
| `enrollment_joined` | `joined_at` | 一轮参与正式成立 |
| `enrollment_closed` | `closed_at` | 周期自然结束 |
| `enrollment_terminated` | `terminated_at` | 提前终止 |

### 8.3 Task 事实

| FactType | occurred_at | 说明 |
| --- | --- | --- |
| `task_created` | `created_at` | Task 进入计划序列 |
| `task_opened` | `open_at` | 任务开放并获得入口 |
| `task_completed` | `completed_at` | 由一个 Assessment 履约 |
| `task_expired` | `expired_at` | 到期后进入 expired |
| `task_canceled` | `canceled_at` | 明确取消 |

为了让这些事实可重建，Plan 重构还需要补齐当前缺少的 `expired_at`、`canceled_at` 等不可替代时间，不能用最后一次 `updated_at` 代替。

当前不统计重排次数。新 Enrollment 创建新 Task；如果业务仍允许同一 Enrollment 内修改 `planned_at/due_at`，Plan 必须保存明确 Schedule Revision，否则 Statistics 不接纳该指标。

## 9. 第三层：四类 Daily

Daily 的组合唯一键不能直接依赖可空列，因为 MySQL 唯一索引允许多行 `NULL`。第一版采用明确的存储归一规则：

```text
Fact 中不适用/未知的业务 ID：NULL
Daily 维度键中的空 ID：0（仅表示 unattributed 技术桶）
Daily 维度键中的空字符串：'' 或明确的 none 枚举
```

`0` 只存在于派生结果的维度桶，API 不得把它序列化为真实 Clinician、Entry 或 Plan ID。这样可以使用普通组合唯一键保证同一维度每天只有一行，不需要额外 Hash 身份。

### 9.1 `statistics_access_daily`

粒度：

```text
org_id + stat_date + clinician_id + entry_id
```

指标：

```text
entry_opened_count
intake_confirmed_count
testee_created_count
care_relationship_established_count
care_relationship_transferred_count
```

### 9.2 `statistics_assessment_daily`

粒度：

```text
org_id + stat_date
+ clinician_id + entry_id
+ origin_type
+ questionnaire_code
+ model_kind + model_code
```

指标：

```text
answersheet_submitted_count
assessment_created_count
outcome_committed_count
assessment_failed_count
report_generated_count
report_failed_count
```

Fact 保留版本，Daily 首版按 code 聚合。需要版本级报表时新增显式查询模型，不提前放大所有 Daily 行数。

### 9.3 `statistics_plan_activity_daily`

粒度：

```text
org_id + stat_date + plan_id
```

指标分为 Enrollment Activity、Task Activity 和去重参与人数。全部按 `occurred_at` 归日。

### 9.4 `statistics_plan_fulfillment_daily`

粒度：

```text
org_id + cohort_date + plan_id
```

同一行分别保存该日的 planned cohort 和 due cohort 指标。完成率不落库，由查询时使用 due 分母计算。

## 10. `statistics_org_snapshot`

Snapshot 一机构一行：

```text
org_id
as_of_date
snapshot_at

testee_count
clinician_count / active_clinician_count
entry_count / active_entry_count
active_enrollment_count

answersheet_submission_count
assessment_count
report_count
content_count
```

资源数量从当前 MySQL 业务表读取；累计提交、测评和报告数量从已完成 Fact 汇总。这是固定口径，避免 Snapshot 在业务表和 Fact 之间随意选择。Projection 结果事务中不扫描 MongoDB。

Snapshot 不保存“today submissions”。查询使用 `latest_complete_day` 从 AssessmentDaily 读取最后一个完整日。

## 11. `statistics_sync_run`

SyncRun 一机构一批次一行：

```text
id / org_id
batch_key / attempt / trigger_type
window_start / window_end / as_of_date
status / stage
source_counts_json / fact_counts_json / result_counts_json
operator_id / reason
started_at / data_committed_at / finished_at
error_code / error_message
```

推荐唯一键：

```text
UNIQUE(batch_key, attempt)
INDEX(org_id, as_of_date, status)
```

`batch_key` 识别一个统计意图，`attempt` 允许失败后创建新尝试。Run 不是扫描游标。它记录“做过什么以及结果怎样”，而不是驱动每条来源从某个 offset 继续消费。

## 12. 为什么不建设资源维度状态层

机构资源快照依赖 Testee、Clinician、Entry 和 Enrollment 当前状态。我们已经选择更轻的方案：

- OrganizationSnapshotProjection 在结果事务中读取当前 MySQL 资源状态；
- 每日 T+1 刷新 OrgSnapshot；
- 不复制完整 Actor/Resource 维度历史；
- 不支持任意历史时点的资源状态还原。

这是当前 qs-server 体量下的有意取舍。未来只有在历史组织结构分析或查询压力得到生产证据后，才考虑独立维度状态层。

## 13. 可重建性矩阵

| 目标 | 来源 | 是否可自动重建 | 备注 |
| --- | --- | --- | --- |
| AccessFact | Resolve/Intake/关系事实 | 是 | 前提是原始日志仍保留 |
| AssessmentFact 新数据 | AnswerSheet、Assessment、Outcome、Report | 是 | AttributionSnapshot 必须已冻结 |
| AssessmentFact 归属 | AttributionSnapshot | 是 | 新数据在受理时冻结 |
| PlanFact | Enrollment、Task | 是 | Enrollment 已持久化 |
| 四类 Daily | 三类 Fact | 是 | 确定性重建 |
| OrgSnapshot | 当前业务表 + 累计事实 | 是 | 表达 snapshot_at |
| Redis Cache | MySQL 结果 | 是 | 可删除后预热 |

## 14. 保留与删除边界

第一版不自动删除 Fact：

- 先获得真实增长率和查询量；
- Daily 长期保留；
- SyncRun 保留策略由运行量和审计需要另行确定；
- 已退役 V1 表由前向 migration 删除，不再承担回滚或对账；
- 任何未来物理归档都必须有增长率、查询和恢复需求证据。

## 15. 数据模型评审清单

- 字段属于业务真值、统计事实还是查询结果？
- 身份是否来自稳定业务来源，而不是扫描批次？
- 时间是否明确是 occurred、planned、due、completed、snapshot 之一？
- 空值表示“不适用/未知”，还是被错误填成了 `0`？
- 新数据是否在业务边界冻结了归属？
- 指标是否可以只从一个结果表回答？
- 该表是 Data Collector 的输出、Projection 的输出，还是不必要的中间层？
- 新数据源是扩展现有 Collector，还是真的出现了新 Fact 家族？
- 是否为了未来想象中的分析提前增加高基数维度？
- 删除结果后能否从 Fact 确定性重建？
- 当前代码未实现的字段是否明确标记为规划？
