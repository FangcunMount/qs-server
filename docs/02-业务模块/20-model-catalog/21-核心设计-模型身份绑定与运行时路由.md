# 核心设计：模型身份、绑定与运行时路由

## 1. 本文回答

本文说明 ProductChannel、Kind/SubKind/Algorithm、DecisionKind、AlgorithmFamily 和 ExecutionPath 分别解决什么问题，QuestionnaireBinding 与多种 version 如何组合成稳定运行时引用，以及为什么这些维度不能合并成单个 `type` 字段。

## 2. 30 秒结论

| 维度 | 回答的问题 | 是否事实输入 |
| --- | --- | --- |
| ProductChannel | 产品上放在哪个目录？ | 目录分类，不参与 evaluator 路由 |
| Identity | 这是什么模型、采用什么算法？ | `Kind + SubKind + Algorithm` |
| DecisionKind | 计算值如何形成结论？ | 发布时从 Definition 推导 |
| AlgorithmFamily | 属于哪类执行机制？ | 从 identity/decision 派生 |
| ExecutionPath | 组合根怎样物化 input provider？ | capability 描述符 |
| QuestionnaireBinding | 该模型解释哪一份问卷？ | `questionnaire code + version` |

```text
ProductChannel ──> catalog filtering only

Identity + DefinitionV2
  -> DecisionKind
  -> AlgorithmFamily / ExecutionPath
  -> Evaluation runtime descriptor
```

## 3. Canonical Identity

当前只接受四种 canonical Kind：

| Kind | 常见 Algorithm | 默认 ProductChannel | ExecutionPath |
| --- | --- | --- | --- |
| `scale` | `scale_default` | `medical_scale` | `scale_descriptor` |
| `typology` | `mbti / sbti / bigfive` | `typology` | `typology_descriptor` |
| `behavioral_rating` | `brief2 / spm_sensory` | `behavior_ability` | `behavioral_rating_descriptor` |
| `cognitive` | `spm` | `behavior_ability` | `cognitive_descriptor` |

`behavior_ability` 是聚合 behavioral_rating 与 cognitive 的产品通道，不是领域 Kind。`personality`、`medical_scale`、`mbti` 也不能作为新模型的 Kind；它们分别属于历史称呼、产品分类或 Algorithm。

SubKind 当前主要用于区分 typology；Algorithm 在 family 内选择具体算法。运行时引用必须保留完整 identity，不能只保存 code 后根据命名猜算法。

## 4. ProductChannel 与模型身份分离

ProductChannel 当前 canonical 值为：

```text
medical_scale | typology | behavior_ability
```

它服务于创建表单默认值、目录筛选和产品展示。`DefaultProductChannelFor` 只提供默认映射，`ValidateNewProductChannel` 约束新建 behavioral/cognitive 进入 behavior_ability。历史行可以保留较宽分类，因此运行时禁止用 ProductChannel 选择 evaluator。

反例：

- `behavior_ability` 无法区分 BRIEF-2 与 Raven SPM。
- 同为 typology 产品目录，MBTI、SBTI、Big Five 的算法仍不同。
- scale 的 Category、Stages、ApplicableAges、Reporters 都是目录维度，不是执行注册键。

## 5. DecisionKind、AlgorithmFamily 与 ExecutionPath

### 5.1 DecisionKind

DecisionKind 描述测量值到结果的判定方式：

| DecisionKind | 典型含义 | AlgorithmFamily |
| --- | --- | --- |
| `score_range` | 分数区间判定 | `factor_scoring` |
| `pole_composition / trait_profile / nearest_pattern` | 类型或特征分类 | `factor_classification` |
| `norm_lookup` | 常模查表 | `factor_norm` |
| `ability_level` | 任务表现到能力等级 | `task_performance` |

发布时 `AssessmentModel.DecisionKindForDefinition` 结合 identity 和 DefinitionV2 推导最终值。typology 必须从 TypeConclusion 明确获得 decision；behavioral_rating 有 NormRef/NormConclusion 时为 norm lookup，无常模时可以是 score range。

### 5.2 AlgorithmFamily

AlgorithmFamily 是对执行机制的归类，用于 API 展示和架构判断，不是独立持久化的 authoring identity。draft summary 通常按 identity 派生；published snapshot 另保存实际 DecisionKind。两者信息相关，但不能互相覆盖。

### 5.3 ExecutionPath

`ModelFamilyCapability` 为每个 Kind 声明 runtime executable 与 ExecutionPath。Evaluation composition root 从 runtime descriptor registry 取得已注册路径，再物化对应 input provider。ExecutionPath 是装配选择，不承载模型内容。

## 6. QuestionnaireBinding

QuestionnaireBinding 冻结 `questionnaire_code + questionnaire_version`。不同 kind 的 application policy 不同：

| Kind | 当前绑定规则 |
| --- | --- |
| scale | 问卷必须存在且类型为 MedicalScale；同一问卷不能绑定另一个 scale；发布模型前可能先发布/同步问卷版本 |
| typology | 必须解析到已发布且非空的问卷；未指定 version 时使用当前发布版本并写回精确值 |
| behavioral_rating / cognitive | 聚合要求发布前 code/version 完整；当前没有与 scale/typology 等价的外部问卷 policy |

Survey 发布问卷后会通过可信服务 actor 调用 `SynchronizeQuestionnaireVersion`。该用例只更新仍处于 draft 的已绑定模型，不改 published 或 archived 模型，也不会替换现有 AssessmentSnapshot。

## 7. 四种版本不能混用

| 版本 | 形态 | 用途 |
| --- | --- | --- |
| AssessmentModel revision | Mongo `version` int64 | draft 配置修订和乐观锁 |
| AssessmentSnapshot version | 通常 `v{revision}` | published model ref |
| Questionnaire version | 业务版本字符串 | 绑定问卷及答卷解释锚点 |
| Norm table version | 字符串 | 定位独立常模资料 |

Definition schema version 是 BSON/JSON 结构版本，也不等于上述任何业务版本。

`AssessmentSnapshot.Version` 当前来自 model revision，不是 SemVer。Questionnaire version 才回答模型绑定哪份题目定义；Norm version 才回答使用哪份校准资料。

## 8. Runtime Ref

`port/modelcatalog.Ref` 保存：

```text
Kind + SubKind + Algorithm + Code + Version + Title
```

精确执行读取要求 code/version；`RefMatchesPublished` 还比较 identity。按 Questionnaire 解析时应同时传 questionnaire code/version，避免问卷重新发布后选到错误模型。

当前 active snapshot 在重新发布时会被替换，ModelCatalog 不是 append-only 版本档案。因此创建 Assessment 时冻结 model ref 很重要，但如果业务要求长期按旧 version 重放，还需要确保存储策略保留可读历史版本；不能只依赖 ref 字段本身。

## 9. Ports 与依赖方向

| Port | 读写边界 |
| --- | --- |
| `ModelRepository` | 可编辑 AssessmentModel |
| `PublishedModelRepository` | 发布用快照写入、下架与管理查询 |
| `PublishedModelReader/Lister` | published-only runtime/C 端/Plan 读取 |
| `QuestionnaireResolver` | code/version 到 model Ref |
| `NormRepository` | versioned Norm 读写 |

Evaluation、Plan 和 collection 依赖 published ports，不依赖 ModelRepository。任何新运行时消费者如果注入 draft repository，都会绕开发布校验和 snapshot 稳定边界。

## 10. 禁止的捷径

- 用 ProductChannel 或 Category 选择 evaluator。
- 把 `behavior_ability` 当成领域 Kind。
- 只保存 model code，不保存 version 和 identity。
- 用 AssessmentModel revision 代替 questionnaire version。
- 用 active draft/management query 为 Evaluation 补 Definition。
- 从 payload bytes 猜测 DecisionKind、identity 或缺失 Definition。
- 新增 flat `mbti/sbti` kind，而不是 typology + algorithm。
