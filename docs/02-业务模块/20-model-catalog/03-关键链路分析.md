# ModelCatalog 关键链路分析

**本文回答**：ModelCatalog 当前关键链路如何运行，包括草稿创建、绑定问卷、更新定义、发布快照、运行读取和 legacy 医学量表路径。

---

## 1. 草稿模型创建链路

典型入口：

- `application/modelcatalog/typology.Service.Create`
- `application/modelcatalog/norming.Service.Create`
- `application/modelcatalog/taskperformance.Service.Create`

链路：

```text
CreateInput
  -> domain.NewAssessmentModel
  -> optional BindQuestionnaire
  -> port.ModelRepository.Create
  -> mongo assessment_models
```

当前特点：

- 不同产品 / family 仍有不同 application service。
- 统一草稿聚合是 `AssessmentModel`。
- 产品分类写入 `ProductChannel`，模型身份写入 `Kind/SubKind/Algorithm`。
- 医学量表仍有 legacy `scoring/definition.MedicalScale` authoring 路径。

---

## 2. 问卷绑定链路

链路：

```text
BindQuestionnaire
  -> load AssessmentModel
  -> validate questionnaire code/version
  -> model.BindQuestionnaire
  -> ModelRepository.Update
```

设计语义：

- 问卷版本是测评快照的业务版本锚点。
- 目标规则是 `questionnaire_code + questionnaire_version -> AssessmentSnapshot`。
- 当前代码仍保留 `model_version` 兼容字段，后续应降级为技术兼容而非业务版本。

---

## 3. 定义更新链路

链路：

```text
UpdateDefinition
  -> load AssessmentModel
  -> validate payload format / JSON
  -> model.UpdateDefinition(DefinitionPayload)
  -> ModelRepository.Update
```

当前事实：

- 定义仍以 `DefinitionPayload{Format, Data}` 保存。
- typology 会做 payload normalization 和 runtime spec 相关校验。
- behavioral_rating / cognitive 当前主要保存 JSON payload，发布时再按 family 解析。

目标方向：

```text
DefinitionPayload -> Definition
Definition -> MeasureSpec + Calibration + Conclusion + ReportMap
```

---

## 4. 发布链路

典型链路：

```text
Publish(modelCode)
  -> load AssessmentModel
  -> validate publish preconditions
  -> model.MarkPublished
  -> publishedmodel.Build(model)
  -> PublishedModelRepository.Save
  -> mongo published_assessment_models
```

发布 assembler：

```text
application/modelcatalog/publishedmodel.Build
```

当前按 `Kind` 分支：

| Kind | 发布构建 |
| ---- | -------- |
| `scale` | `buildScoring` |
| `typology` | `buildTypology` |
| `behavioral_rating` | `buildNorming` |
| `cognitive` | `buildTaskPerformance` |

当前发布产物 port 名称是 `PublishedModel`；目标设计语义是 `AssessmentSnapshot`。

---

## 5. 已发布查询链路

后台和 C 端列表读取：

```text
PublishedModelLister
  -> infra/modelcatalog.PublishedStore
  -> infra/mongo/modelcatalog.Repository
  -> published_assessment_models
```

运行按问卷解析：

```text
RuntimePublishedCatalog.ResolveByQuestionnaire
  -> FindPublishedModelByQuestionnaire(questionnaireCode, questionnaireVersion)
  -> RefFromPublished
```

精确引用读取：

```text
GetPublishedModelByRef(ref)
```

当前要求 `ref.Version` 必填。目标语义上该字段应逐步从“测评业务版本”降级为兼容字段，业务锚点转向 `questionnaire_version`。

---

## 6. Evaluation input decode 链路

运行侧读取发布模型后，会按 family decode 成 evaluation input：

| Family / Kind | Decode 入口 | 产物 |
| ------------- | ----------- | ---- |
| scale | `DecodeScaleFromPublished` | `ScaleSnapshot` |
| typology | `PublishedTypologyCatalog` | `typology.Payload` |
| behavioral_rating | `DecodeBehavioralRatingFromPublished` | `norming/snapshot.Snapshot` |
| cognitive | `PublishedCognitiveCatalog` | `taskperformance/snapshot.Snapshot` |

当前 behavioral_rating 和 cognitive 会投影为 scale-like execution shape，以复用已有 factor scoring 组件。这是运行兼容策略，不应反向决定目标领域模型。

---

## 7. Legacy 医学量表链路

legacy 医学量表仍有独立路径：

```text
MedicalScale
  -> infra/mongo/scale.Repository
  -> scales collection
  -> scale publisher / backfill
  -> published_assessment_models
```

该链路当前仍服务历史医学量表 authoring、查询和迁移。

目标定位：

```text
legacy scale authoring store
```

它不再作为目标领域模型来源。
