# ModelCatalog 领域模型设计

## 聚合与发布态

`AssessmentModel` 是唯一后台可编辑聚合。它承载 identity、产品分类、问卷绑定、生命周期、修订号和 `DefinitionV2`。

发布生成不可变 `AssessmentSnapshot`，在 port 中兼容命名为 `PublishedModel`。它保存 `DefinitionV2`、identity、问卷版本、发布 payload bytes 和路由所需的 `DecisionKind`；不是第二个可编辑聚合。

```text
AssessmentModel
  Identity = Kind + SubKind + Algorithm
  Binding = QuestionnaireCode + QuestionnaireVersion
  DefinitionV2
  Lifecycle = draft / published / archived

AssessmentSnapshot
  immutable published DefinitionV2
  wire payload artifact
```

`questionnaire_code + questionnaire_version` 是模型绑定的业务锚点。draft `Revision` 和 snapshot `Version` 是技术兼容版本字段，不能替代问卷版本语义。

## DefinitionV2 四层

```text
Definition
  MeasureSpec
    Factor[] + FactorGraph + Scoring[]
  Calibration
    NormRef[]
  Conclusions
    RiskConclusion | NormConclusion | AbilityConclusion | TypeConclusion
  Outcomes[] + ReportMap
```

| 层 | 核心对象 | 规则 |
| --- | --- | --- |
| 测量 | `Factor`、`FactorGraph`、`Scoring` | `Factor` 只表示 `Code/Title/Role` 身份；题目来源、因子聚合、层级和排序分别属于计分与图 |
| 校准 | `Calibration`、`NormRef`、`Norm` | 常模资料独立存储在 `assessment_norms`；模型仅引用版本和 factor |
| 结论 | 四类 `Conclusion`、`Outcome` | 风险、T 分、能力范围、人格决策/outcome profile 不属于 Factor |
| 报告 | `ReportMap` | 显式声明 section、adapter、template 和分类标签 |

`TypeConclusion` 保存 decision、poles、special rules、outcome mapping 和 profile；typology 的运行 ID、payload envelope、报告 adapter DTO 留在 payload/interpretation 边界。

## Definition JSON

`DefinitionV2` 是 REST definition 编辑和 generic gRPC `definition_json` 的 canonical JSON。由于 `Conclusions` 是多态接口，每项必须带 `Kind`：`risk`、`norm`、`ability` 或 `type`。缺少 `Kind` 的 JSON 被拒绝，避免把具有重叠字段的常模和能力结论猜错。

Mongo 使用显式 `DefinitionPO`/`ConclusionPO` 映射保存相同语义，保留 nil/empty 和 map/slice clone 行为。

## 领域边界

- `definition.Validate` 负责跨层引用、图、计分、常模引用、结论和报告映射的领域校验。
- family-specific payload 约束由 `definition.Registry` 对应 strategy 在发布前补充校验。
- `FactorSnapshot`、`ScaleSnapshot`、`typology.Payload` 只存在于 `port/modelcatalog/payload/*`，不是领域对象。
