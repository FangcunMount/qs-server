# ModelCatalog 目标设计草案

**本文回答**：在现有模块事实基础上，ModelCatalog 目标领域模型应该长什么样。

---

## 1. 目标模型

```text
AssessmentModel
└── Definition
    ├── MeasureSpec
    │   ├── Factor[]
    │   ├── FactorGraph
    │   └── Scoring
    ├── Calibration
    │   └── NormRef[]
    ├── Conclusion
    │   ├── RiskConclusion
    │   ├── TypeConclusion
    │   ├── NormConclusion
    │   └── AbilityConclusion
    ├── Outcome[]
    └── ReportMap

AssessmentModel --publish--> AssessmentSnapshot
AssessmentSnapshot --uses--> Definition
```

---

## 2. 产品到配置的映射

| 产品 | 测量层 | 校准层 | 结论层 |
| ---- | ------ | ------ | ------ |
| 医学量表 | `Factor` / `Scoring` | 可选 `NormRef` | `RiskConclusion` |
| 人格测评 | `Factor` / `FactorGraph` | 通常无 | `TypeConclusion` |
| 行为能力 | `Factor` / `Scoring` | `NormRef` | `NormConclusion` |
| 认知任务 | `Factor`，role 可为 `task_set` / `ability_domain` | 可选 `NormRef` | `AbilityConclusion` |

---

## 3. 版本语义

测评的前置是问卷。问卷已经有多版本设计，因此测评模型不再引入独立业务版本号。

目标规则：

```text
questionnaire_code + questionnaire_version -> AssessmentSnapshot
```

不支持常规业务形态下的：

```text
同一个 questionnaire_version -> 多个 assessment_model_version
```

如果已发布测评规则需要修正，优先方式是发布新问卷版本并生成新快照。只有历史数据修复场景才允许 correction 流程；correction 必须显式标记，不能伪装成常规“同问卷版本多测评版本”能力。

## 5. 设计不变量

1. `AssessmentModel` 是唯一后台测评模型配置聚合。
2. `AssessmentSnapshot` 是不可变发布态读模型 / 值对象，不是第二个 domain aggregate。
3. `QuestionnaireVersion` 是测评快照的业务版本锚点。
4. 同一个 `model_code + questionnaire_code + questionnaire_version` 只能有一个常规有效 Snapshot。
5. `Definition` 分为测量、校准、结论和报告映射四层。
6. `Factor` 是跨产品、跨算法配置复用的测量节点。
7. 认知任务也先用 `Factor` 表达，不单独建 TaskMeasurement。
8. `Norm` 是常模资料，不等于 `norming` adapter。
9. `TypeConclusion` 取代 `FactorClassificationModel` 作为人格分类配置。
10. ModelCatalog 不消费用户答卷，不产出用户测评结果。
