# ModelCatalog 设计问题与重构清单

> 状态：**规划改造**。本文是 ModelCatalog 的活动重构台账，汇总领域模型、发布、运行时、常模、存储和四种模型类型专题中已确认的问题。每一项都区分“当前事实”与“目标设计”，不得把本文中的计划当成已实现能力。

## 1. 本文的用法

本文不是另一篇 ModelCatalog 架构说明，而是后续改进的统一入口。它解决三个问题：

1. 我们在深度分析中究竟发现了哪些问题；
2. 这些问题哪些会影响结果正确性，哪些是发布与扩展性债务；
3. 后续单独开启重构时，应从哪个编号开始，完成到什么程度才能关闭。

使用约定：

- 后续的分析报告、Issue、提交或 PR 优先引用稳定编号，例如 `MC-R002`；
- 实施前必须重新核对当时源码、数据和运行环境；
- 一次重构只承诺经过确认的有界范围，不因为“都在 ModelCatalog”就一次清理所有兼容代码；
- 涉及历史 snapshot、Outcome、Norm 或已受理 Assessment 时，必须保护历史可重放性；
- 问题关闭后更新状态、证据、验收结果和剩余风险，不直接删除条目。

## 2. 30 秒结论

ModelCatalog 的主干设计已经成立：

```text
AssessmentModel head
  -> DefinitionV2
  -> Questionnaire + Model 联合发布
  -> 不可变 AssessmentSnapshot
  -> active admission / exact-version execution
  -> Calculation
  -> Evaluation Outcome
  -> Interpretation
```

当前问题不是“整个架构没实现”，而是五类已经可以具体改造的缺口：

1. **结果正确性**：NormSubject/Band 与 behavioral Norm 发布门禁已改；Decision/Outcome 区间与 OutcomeCode 发布契约已部分落地（见 MC-R002～R004），DecisionResult 结构与 cognitive 主 Outcome 仍待后续；
2. **业务契约**：独立 Questionnaire 空壳 Assessment 分流与 behavioral Norm 发布门禁已改（见 MC-R001/MC-R003）；Decision/Outcome 契约仍待统一；
3. **发布与运行一致性**：AlgorithmFamily 没有冻结到 snapshot/InputSnapshot，运行时仍会按 identity/path 重新推导；
4. **扩展性**：Definition handler 按 Kind 聚合了语义校验、算法参数、外部资产、兼容 payload 与预览，新 Algorithm 会继续放大分支；
5. **存储与兼容债务**：标准 migration 与 unified schema 尚未完整对齐，DefinitionV2 与 compatibility payload 双存储、历史 Algorithm alias 和 default fallback 仍需有计划退出。

治理原则：

> 先修正可能改变结果或违反业务契约的问题，再将错误前移到发布阶段，最后才做 handler 拆分、payload 收敛和兼容代码退出。

## 3. 优先级与状态语义

### 3.1 优先级

| 级别 | 含义 | 典型标准 |
| --- | --- | --- |
| P0 | 结果正确性、业务契约或持久化约束风险 | 可能选错常模、生成无意义 Assessment、临界分结果不确定、新环境无法重建正确 schema |
| P1 | 发布门禁、版本一致性和执行可追溯性 | 错误未在发布时拒绝，Worker 需运行时猜测 |
| P2 | 模块边界、扩展性与可维护性 | handler 职责膨胀、DTO 表达力不一致、多份兼容投影 |
| P3 | 由新业务需求驱动的能力扩展 | 多常模、Norm 独立生命周期、新认知任务 |

P0 不表示必须在一个 PR 中一次完成，只表示在开启低优先级架构清理之前，应优先关闭或建立可验证的风险防线。

### 3.2 状态

| 状态 | 含义 |
| --- | --- |
| 规划改造 | 问题已由当前源码或存储契约确认，尚未实施 |
| 待补证据 | 静态源码不足以确认生产现状，需要数据或部署证据 |
| 待业务决策 | 技术可以实现，但缺少必要的产品/领域语义 |
| 已实现待验收 | 代码已改，但尚缺少对应风险层级的验收证据 |
| 已关闭 | 代码、数据迁移、测试和文档均完成，且无剩余必做项 |

## 5. P0：正确性、业务契约与存储防线

### MC-R001：独立 Questionnaire 不应创建空壳 Assessment

**状态**：已实现待验收

**当前事实（改造后）**

`assessmentintake.Service.Ensure` 在保存 AnswerSheet 基础分并解析绑定后：

- `bound=false` 且无已有 Assessment：直接成功返回（`AssessmentID=0`），不调用 `CreateForAnswerSheet`，不写 queued report-status，不匹配/完成 Plan；
- `bound=false` 但已有历史空壳 Assessment：幂等复用该记录，不自动 Submit；
- `bound=true`：创建（或复用）并自动提交 Assessment，行为不变。

**已确认业务边界**

> 独立 Questionnaire 可以作为信息收集器；未绑定 AssessmentModel 时，提交成功只产生 AnswerSheet 及基础题分，不应产生 Assessment。

**目标设计**

```text
AnswerSheet 已持久化
  -> 计算并保存基础题分
  -> 查询 active AssessmentBinding
      -> 未绑定：链路成功结束
      -> 已绑定：创建并提交 Assessment
```

**验收条件**

- 独立问卷提交保留 AnswerSheet 和基础题分；
- 不创建 Assessment，不设置 queued report status，不发出 `evaluation.requested`；
- `answersheet.submitted` 重放仍幂等；
- 已绑定问卷的现有自动提交链路不变。

**实施证据**

- `Ensure` 在 `!bound` 时提前返回 `no_assessment_required`；
- 单元测试：`TestEnsureUnboundAnswerSheetEndsWithoutCreatingAssessment`、`TestEnsureUnboundReplayReusesLegacyAssessmentWithoutSubmit`、`TestEnsureBoundAnswerSheetCreatesAndAutoSubmits`。

**剩余风险**

- `bound=false` 仍同时覆盖「合法独立问卷」与「本应有模型但 binding 缺失 / resolver 未装配」；细粒度分流见 Evaluation `EV-R002`；
- 存量 unbound pending Assessment 需另行审计与归档，本项不迁移历史数据；
- gRPC `EnsureAssessmentResponse.AssessmentId=0` 表示无需 Assessment，调用方若误判为失败需跟进（Worker 已按成功 ACK）。

**证据**

- [assessment intake](../../../internal/apiserver/application/journey/assessmentintake/service.go)
- [unbound intake test](../../../internal/apiserver/application/journey/assessmentintake/service_test.go)
- [已发布模型准入与执行输入](./31-关键链路-已发布模型准入与执行输入.md)

### MC-R002：补齐 NormSubject 并修正人口学匹配语义

**状态**：已实现待验收（与 Evaluation `EV-R005` 同批落地）

**当前事实（改造后）**

- `bandMatchesSubject` 与 direct lookup 共用严格规则：常模行声明了年龄/性别时，输入缺失不得命中；
- 参数化路径优先命中专用 Band，再回退显式通用 Band（无年龄/性别约束）；
- 生产 `BehavioralRating` / `Cognitive` input provider 通过 `NormSubjectReader` 从 Actor Testee 装配 `NormSubjectSnapshot`；
- `InputRef` 携带 `TesteeID` 与 `AsOf`（Assessment.SubmittedAt），AgeMonths 按测评时点计算，不按查询当天。

**目标设计**

1. 从 Actor/Testee 权威事实装配性别和出生日期；
2. 按测评发生时间计算 AgeMonths，不按当前时间；
3. Lookup 与 Band 采用同一严格规则：缺少必需人口学资料时不得命中专用层；
4. 区分 generic fallback、missing subject、no matching norm 和 invalid norm；
5. 把实际命中的 NormReference 保存到 Outcome 证据。

**验收条件**

- 覆盖年龄下界、上界、性别、信息缺失和 generic fallback 的 golden tests；
- Import → Publish → Input materialization → Calculation → Outcome 的端到端测试；
- 结果能回答“使用了哪个 Norm 版本和哪个人群层”。

**实施证据**

- `domain/calculation/norm`：`bandMatchesSubject` 严格化 + parametric generic fallback；
- 单元测试：`TestLookupNormScoreParametricRejectsMissingDemographics`、`TestLookupNormScoreParametricFallsBackToGenericBand`；
- `infra/evaluationinput`：`NormSubjectReader` 装配 + `AgeMonthsAt`；
- `container/modules/evaluation`：`NewNormSubjectReader(TesteeQueryService)` 接线。

**剩余风险**

- 完整错误分类（missing subject / no matching norm / invalid）尚未形成稳定 FailureKind，未命中仍可能只是缺少派生分；
- Import → Publish → Outcome 端到端 golden 尚未补齐；
- `AsOf` 在 Assessment 尚未 Submit 时回退 `time.Now()`（正常评估路径应已 Submit）；
- Outcome 已有 NormReference 字段，但人口学快照本身尚未写入独立可验证 InputSnapshotRef。

**证据**

- [Norm lookup](../../../internal/apiserver/domain/calculation/norm/tables.go)
- [Evaluation input contract](../../../internal/apiserver/port/evaluationinput/input.go)

### MC-R003：在发布时强制 behavioral_rating 的常模语义

**状态**：已实现待验收

**当前事实（改造后）**

- `behavioralDecisionKind` 仅返回 `norm_lookup`；无 NormRef / 无 NormConclusion / 非恰好一个 Primary 均发布失败；
- 新发布拒绝 `behavioral_rating_default` 与空 Algorithm，必须使用 `brief2` 或 `spm_sensory`；
- `ValidateForPublish` 额外校验：NormRef factor 必须存在于常模表；NormConclusion factor 必须有对应 NormRef；
- `BuildSnapshotPayload` 不再静默回退 default algorithm。

**已确认领域边界**

> `behavioral_rating` 必须使用 Norm 校准；如果业务只需要原始因子分和区间结果，应建模为 `scale`。

**目标设计**

发布门禁同时要求：

- 至少一个 NormRef；
- 至少一个 NormConclusion；
- 有且仅有明确 Primary 结果；
- NormRef Factor、Norm table Factor 和 Conclusion Factor 一致；
- DecisionKind 只能为 `norm_lookup`；
- 新发布不允许 `behavioral_rating_default` 代替真实 Algorithm。

**验收条件**

- 无 Norm、无 Primary、多 Primary、Factor 不一致均在发布时拒绝；
- BRIEF-2 与 SPM Sensory 正常发布/执行回归通过；
- 对历史 snapshot 的兼容读取不伪装成新发布规则。

**实施证据**

- [`decision.go`](../../../internal/apiserver/domain/modelcatalog/assessmentmodel/decision.go)
- [`behavioral_rating_handler.go`](../../../internal/apiserver/application/modelcatalog/definition/behavioral_rating_handler.go)
- 测试：`TestDecisionKindForDefinitionRejectsBehavioralWithoutNormSemantics`、`TestBehavioralValidateForPublish*`

**剩余风险**

- 历史已发布的 `behavioral_rating_default` / `score_range` snapshot 仍可被 exact-version 执行读取；需运行命中审计后再定下线；
- BRIEF-2 / SPM Sensory 全链路回归依赖现有 seed/characterization，未在本项新增端到端 Import→Publish golden。

**证据**

### MC-R004：统一 DecisionResult、ScoreRange 和 OutcomeCode

**状态**：已实现待验收

**当前事实（改造后）**

- 发布校验：Risk/Norm/Ability 的 ScoreRange 必须有 `OutcomeCode`，且引用顶层 Outcomes registry；
- 发布校验：默认半开 `[min,max)`；**仅末段**可设 `max_inclusive` 或 `unbounded_max`；拒绝重叠/空洞/非末段端点标志；
- 运行时匹配同源：`conclusion.MatchBounds` / `MatchScoreRangeOutcomes`（无显式端点标志时，旧 snapshot 末段 max-inclusive 兼容）；
- scale 运行时不再用硬编码绝对分阈值兜底风险等级（无规则命中 → `none`）；
- interpretation 文案路径已去掉「未命中取末段规则」fallback，与 calculation 判定对齐；
- `outcome.DecisionResult` + `DecisionResultFromExecution` 落地；cognitive `AbilityConclusion.Primary`（或 `Role=total`）提升为 `Execution.Level`；
- 临界值 golden：validate endpoint、ability boundary/unbounded、interpretation no-fallback、scale/norm Matches。

**仍待后续（非本批阻塞）**

- ScoreRangeOutcome 上 Title/Summary/Description 从判定事实彻底剥离到 InterpretationAssets；
- Import → Publish → Outcome 端到端各 family 全链路 golden（当前以单元/机制级 golden 为主）；
- 旧 snapshot 兼容窗口的可观测指标与下线计划。

**目标设计**

```text
Score fact
  -> DecisionSpec
  -> DecisionResult
       OutcomeCode
       LevelCode?
       PrimaryScore?
       Dimension facts
  -> InterpretationAssets
       title / summary / description / suggestions
```

新发布契约：

- 区间端点语义统一（半开 + 显式 `max_inclusive` / `unbounded_max`）；
- 区间不重叠；
- 对完整合法输入域不存在意外空洞；
- 所有可选结果引用顶层 OutcomeRegistry；
- 正式 scale 发布不使用默认风险阈值；
- cognitive 明确 primary ability dimension 并投影主 Outcome。

**验收条件**

- 每个 family 均有 minimum/boundary/maximum/gap/overlap golden cases；
- 持久化 code 与 label/prose 分离；
- 历史 OutcomeCode 不被新版本或文案修改改写；
- 兼容 fallback 只对旧 snapshot 生效且可观测。

- [`definition/validate.go`](../../../internal/apiserver/domain/modelcatalog/definition/validate.go)：`endpoint.required` / `endpoint.non_last` / overlap / gap

### MC-R005：让标准 migration 成为 unified schema 的真正部署入口

**状态**：部分实现待验收（标准 migration + IndexManager reconcile/verify + 启动校验 + contract/integration 测试已落地；生产 dirty@13 需运维恢复 + 部署核验）

**当前事实（改造后）**

- `000013_unified_modelcatalog_schema`：**仅** `createIndexes`（role-based partial unique + `assessment_norms.table_version` unique）；**不再**在 JSON 里 `dropIndexes`（已 cutover 环境 IndexNotFound → dirty）；
- 启动时：migration 成功后执行 `ReconcileUnifiedModelCatalogIndexes`（幂等 drop legacy + ensure）再 `VerifyUnifiedModelCatalogIndexes`；
- 契约/集成测试覆盖 create-only migration + Go reconcile。

**生产 dirty@13 恢复（当前阻塞）**

1. mongosh `qs`：`db.schema_migrations.find()` → 确认 `version:13, dirty:true`；
2. 对照 `RequiredUnifiedIndexNames` / `ForbiddenLegacyIndexNames` 检查三集合 `getIndexes()`；
3. **required 齐全且无 legacy** → 清 dirty 后重启：
   `db.schema_migrations.updateOne({}, {$set:{version:NumberLong(13), dirty:false}})`；
4. **缺 required**（drop 失败导致 create 未跑）→ 手工补齐或清 dirty 到 `version:12, dirty:false` 后部署含本修复的包再启（会重跑 13 + reconcile）；
5. **切勿**在未核验索引前盲目 `force 13`。

**仍待部署核验 / 后续**

- 生产/测试环境实际索引清单对照；
- Mongo read/write concern 配置固化；
- 非生产 Replica Set 上联合发布 commit + model/questionnaire 三类 rollback 证据。

**目标设计**

1. 增加不回改历史文件的后续 migration；
2. 清理冲突旧索引，建立 role-based partial unique indexes；
3. 为 Norm table version 建立唯一约束；
4. 增加 fresh database migration test 和 index contract test；
5. 启动时验证关键索引，缺失时拒绝写流量或产生高优先级告警；
6. 使用非生产、可重置的 Replica Set 数据库验证联合发布。

**关闭前必须补的部署证据**

- 生产/测试环境实际索引清单；

- `calcscoring.Evaluator`：先算 leaf（sign/weight/option_override），再按依赖序算 composite（sum/avg/weighted_sum）；
- 历史无 Measure 扁平载荷行为不变。

**目标选择**

| 方案 | 状态 |
| --- | --- |
| A. ScaleSnapshot 挂 Measure（过渡） | **已落地** |
| B. 直接中性 Calculation DTO | 长期 → EV-R013 / MC-R019 |

**验收条件**

- DefinitionV2 中发布允许的 Measure 语义均能被 factor_scoring 消费；
- 不再存在“可保存/可发布，但投影后丢失”的字段（canonical Measure 路径）；
- 历史 scale payload 仍可读；
- 代表性扁平量表回归不变。

**实施证据**

- [`payload/scale/payload.go`](../../../internal/apiserver/port/modelcatalog/payload/scale/payload.go)
- [`payload/scale/definition.go`](../../../internal/apiserver/port/modelcatalog/payload/scale/definition.go)
- [`payload/scale/projection_loss_test.go`](../../../internal/apiserver/port/modelcatalog/payload/scale/projection_loss_test.go)
- [`mechanisms/scoring/converters.go`](../../../internal/apiserver/application/evaluation/registry/mechanisms/scoring/converters.go)
- [`calculation/scoring/collect.go`](../../../internal/apiserver/domain/calculation/scoring/collect.go) / [`pipeline.go`](../../../internal/apiserver/domain/calculation/scoring/pipeline.go)
- 测试：`measure_semantics_test.go`、`converters_measure_test.go`

### MC-R016：逻辑与物理分离 DecisionSpec 和 InterpretationAssets

**当前事实**

DefinitionV2 中的 Conclusion、Outcome 和 TypeOutcomeProfile 同时包含：

- Decision 规则；
- 稳定结果 code；
- 标题、描述、建议、图片等解释资产；
- 部分 ReportMap/adapter/template 信息。

当前将它们随一次 AssessmentSnapshot 统一冻结，有利于历史报告不漂移；问题不是“数据必须马上拆库”，而是代码与 schema 的责任语义混合。

**目标设计**

```text
DefinitionV2
├── Measure
├── Calibration
├── DecisionSpec
├── OutcomeRegistry
└── InterpretationAssets
    ├── OutcomePresentation
    ├── TypeProfiles
    └── ReportSpec
```

第一阶段仍在一个模型 release 中联合发布，但使用独立类型、validator 和 package。只有解释资产未来真正出现独立团队、独立权限、跨模型复用或独立发布时，才评估独立聚合/存储。

**验收条件**

- Calculation/Evaluation 不消费长文案来做判定；
- Interpretation 不重新计分或反推 OutcomeCode；
- 修改解释文案需要新 release，但不改变旧 Outcome 事实；
- 发布时完整校验 ReportMap source/adapter/template 与 DecisionKind 兼容性。

### MC-R017：治理 DefinitionV2、compatibility payload 与 ReportInput 多份投影

**当前事实**

- DefinitionV2 是 canonical authoring 与发布语义；
- AssessmentSnapshot 仍同时保存 DefinitionV2 和 family payload；
- typology payload 仍包含 RuntimeSpec 与多个历史字段投影；
- ReportInput 当前冻结完整 ModelPayload，而不是裁剪后的 InterpretationAssetsSnapshot。

**风险**

- Definition 与 payload 可能漂移；
- compatibility converter 进入历史重放和报告链路；
- 一个 schema 改动同时影响 authoring、runtime 和 Interpretation；
- 存储与消息载荷被放大。

**目标设计**

1. 发布时对 canonical DefinitionV2 计算内容哈希；
2. compatibility payload 只能由 DefinitionV2 确定性投影；
3. 保存 projection hash/schema version，增加周期 audit；
4. 新 runtime 逐步直接消费 canonical 或中性 Calculation DTO；
5. ReportInput 改为最小、版本化、可重放的 InterpretationAssetsSnapshot；
6. 只在读取老 snapshot 时保留旧 payload decoder。

不能在没有历史数据审计和 backfill 方案时直接删除 payload。

### MC-R018：有计划地退出历史身份与运行时 fallback

**当前兼容面**

- typology 新建选项倾向 `personality_typology`，但历史数据可保存 `mbti/sbti/bigfive`；
- `behavioral_rating_default` 与空 Algorithm 仍可映射到 factor_norm；
- cognitive 空 Algorithm 仍可默认为 SPM；
- payload format 仍接受旧 family-specific 标识；
- registry/provider 仍使用 typology alias、legacy kind mapping 和默认 identity。

兼容代码不是“无用代码”，直接删除会破坏历史 Assessment 和 snapshot。真正的问题是：新发布与老数据共享同一个宽松入口，且缺少可观测下线条件。

**目标设计**

- new-write path：只接受 canonical identity/payload format；
- retained-read path：保留明确版本化 compatibility adapter；
- audit：统计每种 alias/fallback 的存量与运行命中量；
- backfill：只对可证明语义等价的数据进行受控迁移；
- retirement：当存量清零、命中清零且回归通过后再删除兼容分支。

**验收条件**

- 新发布 snapshot 不再保存空/default 身份；
- 历史读取兼容不污染新写入契约；
- 每个兼容分支都有使用证据和退出条件；
- 删除前完成数据审计、回放测试和回滚方案。

### MC-R019：将纯计算从 Evaluation application 收敛到 Calculation

**当前事实**

scale scoring、typology classification 和常模数学已经大量进入 `domain/calculation`，但 Raven SPM 的正确答案比较、题组分和总分计算仍位于 Evaluation application 的 `task_performance/spm_calculator.go`。部分 family 也仍在 application adapter 中组合较多领域计算细节。

**目标边界**

```text
ModelCatalog
  定义模型、算法绑定与参数

Calculation
  纯函数计分、校准与分类

Evaluation
  物化输入、选择算法、编排执行、保存事实
```

**实施原则**

- 先建 characterization/golden tests；
- 仅移动无 I/O、无应用状态的计算；
- 保留 InputAdapter 和 OutcomeAssembler 在 Evaluation；
- 不在这一步同时改变 Algorithm 标识或结果 schema。

## 8. P3：需要真实业务驱动的扩展

### MC-R020：Norm 资产治理与更广认知任务

以下项目已经可以说明设计方向，但当前不应在没有产品需求时提前实现：

| 候选能力 | 何时才应开始 | 目前不实施的原因 |
| --- | --- | --- |
| Norm draft/publish/archive/retired 生命周期 | 运营需要审核、回滚、禁用常模且 Import 即 available 无法满足 | 会引入新状态机与权限模型 |
| Norm usage 反向查询 | 需要在退役/删除前查看引用，或运营需要影响分析 | 可先用只读审计工具满足低频需求 |
| 多 NormTableVersion / CalibrationProfile | 一个 release 的不同 Factor 确实需要不同常模资产 | 当前 behavioral payload 使用单表假设，盲目放开会模糊选择规则 |
| scale + factor_norm | 出现业务语义仍是医学量表、但必须常模化的真实模型 | 需先完成 AlgorithmFamily 冻结和多对多路由 |
| 服务端强制认知任务时限 | 超时确实影响结果有效性或计分 | 需一并决定开始、暂停、断网、重进和超时语义 |
| 更多认知任务 Algorithm | 第二个真实任务需要反应时、遗漏/误报、自适应或新详情结构 | 当前只有 Raven SPM，不应预先设计“万能认知引擎” |

这一组不作为近期技术清理，只保留决策门槛。

## 9. 建议的实施顺序

不建议按文件顺序或 package 顺序实施，而应按风险与依赖关系分批。

### 批次 0：建立现状证据

先进行只读审计：

- 有多少 `behavioral_rating` release 没有 NormRef/Primary NormConclusion；
- 现有 behavioral Outcome 有多少缺失 NormReference；
- 有多少 pending Assessment 没有 model ref；
- 新旧 Algorithm/payload format/legacy kind 各有多少存量；
- 生产实际 Mongo 索引与 migration 有什么差异；
- DefinitionV2 与 compatibility payload 是否可确定重投影一致。

本批次不修数据，只生成数量、样本和影响范围。

### 批次 1：关闭直接正确性缺口

```text
MC-R001 独立 Questionnaire 分流
MC-R002 NormSubject + strict demographic matching
MC-R003 behavioral_rating 发布强制 Norm
MC-R004 Decision/Outcome 边界正确性（已实现待验收）
```

MC-R004 已按“区间校验 → OutcomeCode → scale fallback → max_inclusive/无上界 → interpretation 对齐 → DecisionResult/cognitive primary → golden”拆批落地。

### 批次 2：补齐发布和存储门禁

```text
MC-R005 canonical migrations/indexes/integration tests
MC-R007 统一发布完整性
MC-R008 首次发布后身份不变式
MC-R009 Questionnaire Binding Policy
MC-R011 Norm compatibility
MC-R012 release/concurrency semantics
```

### 批次 3：让运行时严格消费发布事实

```text
MC-R006 freeze RuntimeIdentity
MC-R010 InputInvariantValidator
```

完成该批次后，新模型的 Worker 路径应已不需要默认 family/decision。

### 批次 4：在行为保护下调整代码边界

```text
MC-R013 handler responsibility split
MC-R014 capability/validation single source
MC-R015 scale canonical Calculation input
MC-R016 Decision/Interpretation logical split
MC-R019 pure calculation extraction
```

### 批次 5：收敛双投影与兼容面

```text
MC-R017 projection/hash/minimal interpretation snapshot
MC-R018 legacy identity and payload retirement
```

这一批必须在运行命中审计、历史数据 backfill 和回放测试之后进行。

## 10. 每个重构项的开始与关闭门槛

### 10.1 开始前

- [ ] 重新阅读当前源码，不仅依赖本文；
- [ ] 确认工作树中用户未提交修改；
- [ ] 界定输入、输出、持久化和兼容契约；
- [ ] 完成影响数据审计；
- [ ] 补齐 characterization/golden tests；
- [ ] 明确是纯代码重构、行为改造还是数据迁移；
- [ ] 对需要业务决策的语义先得到确认。

### 10.2 关闭前

- [ ] 目标验收条件均有测试或部署证据；
- [ ] 历史 exact-version 执行与重试不漂移；
- [ ] 新发布契约已收紧，老数据兼容位于独立边界；
- [ ] 数据迁移可重跑、可审计、可回滚；
- [ ] 对应 ModelCatalog、Evaluation、Interpretation 文档已同步；
- [ ] `make docs-hygiene docs-facts` 通过；
- [ ] 相关领域、应用、基础设施和运行时测试通过；
- [ ] 对外 API/gRPC 契约变化已完成消费方迁移；
- [ ] 监控能证明旧 fallback 使用量下降或清零。

## 11. 不应被重构破坏的已确认决策

下列内容不是“待优化项”，而是重构时必须保护的领域决策：

1. Questionnaire 可以独立发布并产生 AnswerSheet；
2. Questionnaire 与 AssessmentModel 绑定后才构成可执行测评；
3. `AssessmentModel` 是可变工作头，`AssessmentSnapshot` 是不可变发布版本；
4. 新测评使用 active release，已受理测评和重试使用 exact retained release；
5. Algorithm 语义发生不兼容变化时新增 Algorithm identifier，当前不引入独立 AlgorithmVersion；
6. Factor 必需，Decision 必需；不完整模型不得发布；
7. `behavioral_rating` 必须常模化，无常模的原始分区间模型使用 `scale`；
8. `cognitive` 当前只宣称 Raven SPM 已实现，不宣称已建成通用认知任务平台；
9. ProductChannel 只用于产品分类，不用于 Evaluation 路由；
10. ModelCatalog 配置模型，Calculation 实现纯算法，Evaluation 编排执行，Interpretation 解释结果；
11. compatibility payload 是 DefinitionV2 的投影，不是第二个事实源；
12. 已被历史 Assessment 引用的 Model、Questionnaire 和 Norm 版本不得物理删除或就地改写；
13. 该系统产生辅助测评信息，不产生医学诊断。

## 12. 维护规则

每次更新某个 `MC-Rxxx` 时，至少记录：

| 字段 | 要求 |
| --- | --- |
| 状态 | 规划改造 / 待补证据 / 待业务决策 / 已实现待验收 / 已关闭 |
| 当前证据 | 实际代码、数据、部署或测试，不只引用旧文档 |
| 决策 | 接受的方案、未采用方案及原因 |
| 实施范围 | 本次修改什么，明确不修改什么 |
| 数据迁移 | 是否需要 audit/backfill/rollback |
| 验收 | 测试、集成环境、生产观测或人工核验 |
| 剩余风险 | 关闭后仍保留的兼容面或未覆盖场景 |

当源码已修复某个问题，但本文尚未更新时，以源码与真实运行证据为准；修改者应随后将条目改为“已实现待验收”或“已关闭”。
