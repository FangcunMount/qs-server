# 核心设计：人格题目贡献与因子计分

## 1. 本文回答

本文说明人格类型测评中，问卷选项分值、答卷 `Answer.Score`、题目贡献、因子聚合、结果决策和报告之间的职责边界，以及 `question_score`、`option_override` 和旧快照兼容分支的精确计算规则。

## 2. 30 秒结论

人格测评的标准计分链路是：

```text
Questionnaire.Option.Score
  -> AnswerSheet.Answer.Score
  -> QuestionContribution
  -> Factor score
  -> DecisionKind
  -> OutcomeMapping / TypeProfile
  -> ReportMap / report
```

必须遵守三个事实源边界：

| 事实 | 唯一所有者 | 不能做什么 |
| --- | --- | --- |
| 题目、选项和选项基础分值 | Survey Questionnaire | 人格模型不能默认复制一份选项分值 |
| 题目如何贡献给因子 | ModelCatalog DefinitionV2 | 问卷不能决定题目属于 E、I 或其它人格因子 |
| 一次作答的原始选项和基础题分 | Survey AnswerSheet | 执行期不能重新猜测或静默补成 0 |

新配置默认使用 `question_score`：

```text
贡献分 = Answer.Score × sign × weight
叶子因子分 = constant + sum(该因子的所有贡献分)
```

只有问卷基础分值无法表达模型语义时，才显式选择高级模式 `option_override`。16 人格、九型人格等是“因子结构 + 结果决策 + outcome/profile”的不同配置，不应新增同名专属计分算法。

## 3. 为什么问卷有分值，模型仍需要贡献配置

问卷分值和人格模型配置解决的是两个不同问题：

```text
问卷：用户选择了 B，这个选择的基础分是多少？
模型：Q1 的基础分应贡献给哪个因子，以什么方向和权重贡献？
```

例如，问卷将 Q1 的 A/B/C/D 配置为 `1/2/3/4`。作答选择 D 后，Survey 产生：

```yaml
question_code: Q1
value: D
score: 4
```

人格模型只需要声明：

```yaml
question_code: Q1
factor_code: E
scoring_mode: question_score
sign: -1
weight: 0.5
```

此时贡献分为：

```text
4 × (-1) × 0.5 = -2
```

模型没有重新定义 D 的分值，只定义 Q1 如何进入 E 因子。因此，同一份问卷基础计分可以被不同模型修订复用，而题目分值调整也只需要在问卷版本中完成。

## 4. QuestionContribution 契约

领域语义统一为：

```go
type QuestionContribution struct {
    QuestionCode string
    FactorCode   string
    ScoringMode  QuestionScoringMode
    Sign         int
    Weight       float64
    OptionScores map[string]float64
}
```

DefinitionV2 为兼容现有 REST wire，仍把贡献保存为对应 factor 的 `Scoring.Sources`：

```yaml
FactorCode: E
Strategy: sum
Sources:
  - Kind: question
    Code: Q1
    ScoringMode: question_score
    Sign: 1
    Weight: 1
```

typology runtime 使用 snake_case：

```yaml
question_code: Q1
scoring_mode: question_score
sign: 1
weight: 1
```

字段语义：

| 字段 | 默认值 | 规则 |
| --- | --- | --- |
| `question_code` | 无 | 必须引用绑定问卷中的题目 |
| `factor_code` | 无 | 在 DefinitionV2 中由所属 Scoring/Factor 确定 |
| `scoring_mode` | `question_score` | 新配置必须显式保存；空值只用于旧快照兼容 |
| `sign` | `1` | 只允许 `1` 或 `-1` |
| `weight` | `1` | 必须是大于 0 的有限数字 |
| `option_scores` | 无 | 只允许在 `option_override` 中出现 |

同一 `question_code + factor_code` 只能配置一次；同一道题可以分别贡献给多个不同因子。题目贡献的 `weight` 与复合因子的子因子权重是两个独立概念，不能共用字段或互相替代。

## 5. 精确计算规则

### 5.1 `question_score`：默认模式

```text
baseScore = Answer.Score
contributionScore = baseScore × sign × weight
```

`Answer.Score` 允许任意有限数字，包括 `0`、负数和小数，不限制为 Likert 1–5。

示例：

| Answer.Score | sign | weight | 贡献分 |
| ---: | ---: | ---: | ---: |
| 0 | 1 | 1 | 0 |
| -2.5 | -1 | 0.5 | 1.25 |
| 4 | -1 | 1 | -4 |
| 3.5 | 1 | 2 | 7 |

### 5.2 `option_override`：高级覆盖模式

```text
baseScore = OptionScores[Answer.Value]
contributionScore = baseScore × sign × weight
```

示例：

```yaml
question_code: Q1
scoring_mode: option_override
sign: -1
weight: 0.5
option_scores:
  A: 2
  B: 1
  C: 0
```

当 `Answer.Value=B`、问卷生成的 `Answer.Score=100` 时：

```text
baseScore = option_scores[B] = 1
贡献分 = 1 × (-1) × 0.5 = -0.5
```

这里明确忽略 `Answer.Score=100`，因为模型已主动声明覆盖。覆盖分值本身允许 `0`、负数和小数，但必须是有限数字。

`option_override` 当前只支持有选项的 `Radio` 题，并要求：

- 覆盖绑定问卷中全部有效选项；
- 不包含未知选项；
- 执行时保留原始 `Answer.Value`；
- 未知或缺失选项返回结构化执行错误，不能静默记零或回退到 `Answer.Score`。

### 5.3 `legacy_implicit`：只读兼容分支

缺少 `scoring_mode` 的已发布快照进入旧语义：

- 有 `OptionScores` 时，继续按历史 option scoring 的 strict/compat 行为执行；
- 无 `OptionScores` 时，继续按历史 Likert 1–5 与 sign 行为执行；
- 历史 override 未应用 sign，新实现仍保持该结果；
- 发布校验产生 `question_contribution.legacy_implicit` warning，但 warning 不阻断旧快照读取和执行。

`legacy_implicit` 不是新建配置的第三种选择。已发布旧快照不回填、不改写；只有旧草稿被用户保存时，前端 Mapper 才将其迁移为显式新格式。

## 6. 从贡献分到因子分

typology runtime 先按拓扑顺序计算叶子因子，再计算复合因子。

### 6.1 叶子因子

当前显式人格 runtime 的叶子因子公式为：

```text
leafFactorScore = constant + sum(CalculateQuestionContribution(...))
```

题目贡献权重只作用于单道题对当前叶子因子的贡献。如果同一道题同时贡献给 E 和 I，两个贡献分别计算，互不覆盖。

### 6.2 复合因子

复合因子只消费已计算的子因子分，不允许携带 `scoring_mode`、`sign`、题目 `weight` 或 `option_scores`。当前人格 runtime 支持：

```text
sum          = sum(child scores)
avg          = sum(child scores) / child count
weighted_avg = sum(child score × child weight) / sum(child weight)
```

复合因子的 child weight 与题目贡献 weight 必须分别配置：

```text
Answer.Score
  × contribution.sign
  × contribution.weight
  -> leaf factor score
  × composite child weight
  -> composite factor score
```

## 7. 因子分之后为什么还需要结果决策机制

计分只负责得到因子向量；结果决策负责把因子向量映射成类型、模式或画像。两者不能合并，否则每增加一种人格测评名称都会复制一套计分引擎。

人格类型当前使用有限的 `DecisionKind`：

| DecisionKind | 用途 | 典型配置 |
| --- | --- | --- |
| `pole_composition` | 根据多个因子的左右极点组合类型码 | MBTI、16 人格的 E/I、S/N、T/F、J/P |
| `nearest_pattern` | 将因子等级向量匹配到最近模式 | 预定义人格模式集合 |
| `trait_profile` | 直接输出连续特质画像 | Big Five 等连续人格特质 |
| `dominant_factor` | 选取最高的一个或 Top-K 因子 | 九型人格主型、主型加候选类型 |

因此：

- 增加“16 人格”通常复用 `pole_composition`，差异放在 factors、poles、outcomes 和 profiles；
- 增加“九型人格”通常复用 `dominant_factor`，通过 `top_k` 决定输出一个或多个候选因子；
- 只有现有四种机制无法表达新的“因子向量到结果”的数学规则时，才评估新增 DecisionKind；
- 产品名称、理论流派或报告模板名称都不是新增算法枚举的理由。

## 8. OutcomeMapping、TypeProfile 与报告边界

完整后半段链路是：

```text
factor scores
  -> DecisionKind
  -> outcome code
  -> OutcomeMapping / TypeProfile
  -> ReportMap
  -> Interpretation report
```

| 配置 | 责任 |
| --- | --- |
| Poles | 定义因子阈值两侧对应的类型极点 |
| Patterns | 定义可被 nearest-pattern 匹配的目标模式 |
| Outcome | 保存稳定结果 code 和基础内容 |
| OutcomeMapping | 声明结果详情类别和 adapter |
| TypeProfile | 将 outcome 与类型说明、维度特征等报告输入关联 |
| ReportMap | 声明 section、adapter 和 template 的展示映射 |

这些配置消费计分和决策结果，不参与 `Answer.Score` 或题目贡献的计算。报告类型也不能反向决定计分算法。

## 9. 预览与生产执行必须一致

预览不是另一套简化计分器。预览先依据绑定问卷构造与生产答卷相同的 `InputSnapshot`，然后进入同一个 typology runtime：

```text
preview answers
  -> validate against bound questionnaire
  -> derive Answer.Value and Answer.Score
  -> InputSnapshot
  -> configured typology runtime
  -> factor scores / outcome / report
```

输入规则：

- 有选项题必须提交 `value`；服务端从绑定问卷选项生成 `Answer.Score`；
- 同时提交 `value` 和 `score` 时，两者必须一致，否则返回 `answer.score.mismatch`；
- 无选项数值题必须直接提交有限 `score`；
- `option_override` 仍保留 `value`，供覆盖表查询；
- 缺少 score 时不允许默认写入 0。

同一组答案在预览和模拟生产执行中必须得到相同的因子分、outcome 和报告。

## 10. 校验责任与稳定错误码

`/validate` 和 `/publish` 共享 typology 的完整校验入口，依次检查 DefinitionV2、绑定问卷、runtime 投影和报告配置。服务端校验是最终权威，前端本地校验只用于更快反馈。

题目贡献的主要稳定错误码：

| 错误码 | 含义 |
| --- | --- |
| `scoring_mode.invalid` | 模式未知，或 override 用于不支持的题型 |
| `sign.invalid` | sign 不是 1/-1 |
| `weight.invalid` | weight 非有限数字或不大于 0 |
| `option_scores.required` | override 未配置覆盖分值 |
| `option_scores.forbidden` | question_score 携带覆盖分值 |
| `option_scores.missing_option` | 未覆盖问卷中的有效选项 |
| `option_scores.unknown_option` | 覆盖表包含问卷不存在的选项 |
| `question_contribution.duplicate` | 同一题目重复贡献给同一因子 |

只有 warning 时校验仍为通过；error 才阻断保存后的发布或预览执行。

## 11. 前端配置原则

后台编辑状态以各叶子 factor 的 `contributions` 为唯一事实来源。“题目贡献”页面只是将这些 contributions 展平展示，不维护第二份 `question_mappings` 副本。

默认表单只展示：

- 题目；
- 目标因子；
- 计分来源；
- 正向/反向；
- 权重；
- 操作。

交互规则：

- 新增贡献默认为 `question_score/sign=1/weight=1`；
- `question_score` 只读展示问卷选项及实际分值，并引导用户去问卷编辑；
- 用户主动选择“自定义选项计分”后才展开 `option_scores`；
- 切回 `question_score` 或更换题目时清除不再适用的覆盖分值；
- 保存时 Mapper 将 QuestionContribution 投影到 DefinitionV2 source，并无损保留表单未编辑的其它 Definition 字段；
- 旧草稿有 OptionScores 时迁移为 override，没有时迁移为 question_score；历史 override 的 sign=-1 规范化为 1，以保持旧实现的实际结果。

## 12. 发布与兼容边界

发布数据流：

```text
DefinitionV2
  -> factor domain
  -> typology payload
  -> configured graph
  -> AssessmentSnapshot
  -> classification execution
```

新发布快照必须显式保存 `scoring_mode/sign/weight`。任一转换阶段丢失这些字段都属于契约错误。旧快照仍走 legacy 分支，本轮不批量迁移，也不删除旧 MBTI/SBTI 解码或 `OptionScoring strict/compat`。

删除 legacy 分支的前置条件是：线上 warning 已盘点清零、旧发布快照完成独立迁移方案、跨仓 legacy fixture 可以被正式下线。不能仅因为新前端已经发布就删除兼容代码。
