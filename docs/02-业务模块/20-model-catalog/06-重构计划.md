# ModelCatalog 迁移与运行手册

## 上线前置条件

1. 所有 `assessment_models` 和 `published_assessment_models` 都有可校验的 `definition_v2`。
2. 所有 `NormRef` 都能从 `assessment_norms` 按版本解析。
3. 所有 identity 和 product channel 已 canonical 化；ModelCatalog 不接受 `personality`、`medical_scale` 或 `custom` kind alias。
4. 历史 `scales` collection 已通过一次性迁移写入目标 collection。生产代码不保留 dual-read。

迁移使用 `scripts/oneoff/migrate_modelcatalog_definition_v2`。默认 dry-run；`--apply` 使用 CAS 写入并保留已有 payload bytes。重复执行应为 skip，norm version 内容冲突必须阻塞处理。

## 发布顺序

```text
dry-run -> resolve conflicts -> apply -> audit definitions/norms
-> deploy V2-only runtime -> observe resolver/evaluation -> retire old code
```

不能通过在运行时恢复 payload 或旧 collection fallback 来绕过审计失败。应修复数据、重新执行脚本并复核。

## 故障定位

| 现象 | 优先检查 |
| --- | --- |
| 发布被拒绝 | `definition.Validate`、family strategy 校验、问卷 binding、NormRef 是否存在 |
| resolver 失败 | published row 是否存在 `definition_v2`，service actor capability 是否正确 |
| 常模运行失败 | `assessment_norms` 的 version、form variant、factor lookup 是否匹配 |
| collection 人格目录异常 | generic catalog gRPC 的 `definition_json`，随后检查 typology Definition 的 TypeConclusion/Measure graph |
| payload bytes 差异 | 对比 strategy 的 `BuildSnapshotPayload` contract test；不要以 payload 反向修正领域模型 |

## 删除守卫

生产代码必须保持以下约束：

- 不重新建立 family command、option router、旧 scale read service 或 legacy metadata binding。
- 不新增旧 scale REST 路由或专用 typology gRPC。
- runtime 不读取 draft，也不调用 payload parser 来补齐 Definition 语义。
- behavioral 发布从 `DefinitionV2 + NormRepository` 重建 wire payload；不得从 draft payload 复制 BRIEF-2 常模表。
- plan 的模型校验和标题投影只读 published scale，不能注入 `ModelRepository`。

`go test ./...`、OpenAPI/proto generation、`git diff --check` 和双范围 `rg` 是每次删除后的最低门禁。
