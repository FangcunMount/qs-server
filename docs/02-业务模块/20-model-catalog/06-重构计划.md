# ModelCatalog 重构计划

**本文回答**：如何从当前 modelcatalog 代码结构迁移到目标模型设计。

---

## 1. 当前包到目标包的映射

| 当前包 | 目标定位 |
| ------ | -------- |
| `assessmentmodel.AssessmentModel` | 后台测评模型配置聚合，已从 `publishing` 迁出 |
| `binding` | 迁入或改名为 `identity` |
| `factor` | 保留并增强为通用 Factor 模型 |
| `scoring/definition.MedicalScale` | 标记为 legacy medical scale authoring，逐步迁移 |
| `scoring/snapshot` | 运行 payload schema，不应定义核心 Factor |
| `norming` | 拆分为 `norm` 领域对象 + `NormConclusion` / compatibility adapter |
| `taskperformance` | 拆分为 `AbilityConclusion` / cognitive compatibility adapter |
| `typology` | 拆分为 `TypeConclusion` + 人格测评 profile / legacy adapter |

---

## 2. 目标包结构

```text
internal/apiserver/domain/modelcatalog
├── assessmentmodel
├── factor
├── norm
├── conclusion
├── reportmap
└── identity
```

迁移时不要求一步到位。优先守住行为兼容和发布快照读取契约。

---

## 3. 阶段计划

### 阶段一：文档和契约统一

- 以本目录新文档作为当前设计入口。
- 旧文档只保留在 `_archive`。
- 明确 `QuestionnaireVersion` 是业务版本锚点。
- 明确 `AssessmentSnapshot` 是查询、缓存、执行的稳定对象。

### 阶段二：命名收口

- `publishing` 兼容门面已删除；新代码直接使用 `assessmentmodel`、`identity`、`payloadformat`、`binding`。
- 将 `ProductChannel`、`Kind/SubKind/Algorithm`、`AlgorithmFamily` 的概念统一到 `identity`。
- 保留对外字段字符串，不在本阶段改 REST / proto / Mongo 字段形态。

### 阶段三：Definition 强类型化

- 将裸 payload 逐步收口为 `Definition`。
- 将 Factor 统一成通用测量节点。
- 将 `norming` / `taskperformance` / `typology` 中的 payload adapter 降级为 compatibility adapter。

### 阶段四：Snapshot 语义迁移

- 将 published 记录领域命名收口为 `AssessmentSnapshot`。
- 让查询、缓存、evaluation 都依赖 snapshot。
- `model_version` 保留兼容，目标语义改为由 `questionnaire_version` 派生或等同。

### 阶段五：legacy scale 退出核心模型

- 将 `scoring/definition.MedicalScale` 明确标记为 legacy authoring model。
- 医学量表目标表达为 `AssessmentModel + Definition + MeasureSpec + RiskConclusion`。
- 旧 `scales` collection 保持兼容读取，逐步迁移写路径。

---

## 4. 验证重点

- REST / proto / Mongo 字段形态不应在早期重构中改变。
- `published_assessment_models` payload bytes 和 payload format 保持兼容。
- evaluation runtime 继续从 published reader 读取，不回退到 draft model。
- collection-server 已发布列表和 typology 列表不被破坏。
- legacy `scales` 读取、backfill、publisher 保持可用，直到迁移完成。
