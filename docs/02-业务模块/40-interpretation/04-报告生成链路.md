# 报告生成链路

## 0. 当前实现 vs 目标

| 维度 | 当前实现 | 目标（Phase 2/3 重构后） |
| ---- | -------- | ------------------------ |
| 编排入口 | `evaluation/execute.Service` 直接调用 Interpretation service | `assessment.evaluated` 后由 Interpretation 独立生成报告 |
| 分数解读 | `domain/interpretation/scoring` 和 report builder 组装 `InterpretReport` | 按模型 identity 与 report map 选择 builder |
| 事件触发 | 同步或 `assessment.evaluated` 两种路径并存 | 统一消费 `assessment.evaluated` / 持久化 EvaluationOutcome |
| 状态所有权 | Writer 调用 `Assessment.ApplyOutcome` 推进 `interpreted` | Report 独立进入 `generated`；Assessment 保持 `evaluated` |
| 失败处理 | 报告失败可能把 Assessment 标记为 failed | Report 独立 failed，Assessment 不变 |

---

## 1. 业务目标

在 Evaluation 完成后，选择合适的报告 builder 和解释适配器，生成并持久化 `InterpretReport`。

---

## 2. 参与对象

| 对象 | 角色 |
| ---- | ---- |
| `EvaluationResult` | 结构化测评结果 |
| `ModelIdentity` | 报告选择依据 |
| `ReportBuilder` | 报告构建器 |
| `InterpretReport` | 报告实例 |

---

## 3. 前置条件

- Evaluation 已生成结构化结果。
- 能识别模型身份和报告构建器。
- 报告持久化能力可用。

---

## 4. 流程图

```mermaid
sequenceDiagram
    participant E as evaluation
    participant R as report/report
    participant M as model-catalog
    participant ST as statistics

    E-->>R: assessment.evaluated + EvaluationOutcomeRef
    R->>M: 读取模型身份 / 解释绑定
    R->>R: 选择 ReportBuilder
    R->>R: 匹配解释规则和模板
    R->>R: 保存 InterpretReport
    R-->>ST: report.generated
```

---

## 5. 关键规则

- 报告生成不重新执行测评。
- 生产目标态不保留 Evaluation 内联生成 Report 的同步模式；Interpretation 始终位于 `assessment.evaluated` 之后。
- 报告失败不应改写 EvaluationResult。
- 报告失败不应修改 Assessment 状态或清除其结果摘要。
- 报告事件代表报告实例生成，不代表统计投影完成。
- 报告内容必须能追溯到执行结果和模型身份。

---

## 6. 幂等与异常处理

| 场景 | 处理 |
| ---- | ---- |
| 重复生成同一报告 | 按 EvaluationID / ReportID 保持幂等 |
| Builder 缺失 | 报告生成失败，保留执行结果 |
| 模板错误 | 修复后可重建报告 |
| EvaluationOutcome 缺失 | 报告失败并暴露契约问题，不回退到重新计分 |
| 统计消费失败 | 不影响报告主事实 |

---

## 7. 产出结果

- `InterpretReport`。
- `report.generated` 事件。
- 查询端可读取的报告状态和内容。
- Journey / ReadModel 可据此把客户端进度投影为 `completed` 或兼容 `interpreted`。

Preview / 测试可以在进程内直接组合 Evaluation 计算结果与 ReportBuilder，但不持久化生产 Assessment 状态，也不改变上述生产边界。
