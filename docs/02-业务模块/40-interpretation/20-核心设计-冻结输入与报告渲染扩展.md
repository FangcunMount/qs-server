# 核心设计：冻结输入与报告渲染扩展

## 1. 本文回答

本文说明 Interpretation 如何不读取“当前模型”也能稳定重建报告，以及新计算机制、决策类型、报告模板或呈现 profile 应如何接入 Builder Registry。

## 2. 30 秒结论

报告不是对当前 ModelCatalog 的二次查询，而是对持久化 EvaluationOutcome Record 的确定性投影：

```text
Outcome.Record
  |- schema-versioned execution facts
  |- frozen ReportInput / ModelPayload
  |- model + runtime identity
  |- org / assessment / testee association
  -> InterpretationInput
  -> rendering.Key
  -> exact TemplateVersion fallback candidates
  -> Builder
  -> Report Draft
```

扩展点不是“模型 code -> 报告处理器”，而是完整机制身份。TemplateVersion 始终参与解析，不会回退到当前新版模板。

## 3. 为什么必须冻结报告输入

Evaluation 新写 Outcome 使用 schema v2：

- `Payload` 只保存分数、分类、维度、等级和效度等评估事实；
- `ReportInput` 冻结本次解释所需的 ModelPayload，包括解释规则、类型文案、模板选择和归因资料；
- `ModelIdentity / RuntimeIdentity` 冻结模型和运行时身份。

这个拆分避免两种错误：

1. 把解释文案塞进评分 payload，使 Evaluation 与报告呈现耦合。
2. 重建历史报告时读取当前 Published Model，导致同一 Outcome 随资产发布而改变含义。

`FromOutcomeRecord` 只通过 `evaluationfact/codec` 解码 Record，不重建 Assessment，不调用 Evaluator，也不查 ModelCatalog Repository。

## 4. InterpretationInput 的五组事实

| 部分 | 主要内容 | 用途 |
| --- | --- | --- |
| `Association` | OrgID、AssessmentID、TesteeID | Report 追溯与查询关联，不授予写权 |
| `Model` | kind、subkind、algorithm、code/version/title、channel/family | 报告标题、追溯和高级路由 |
| `Runtime` | AlgorithmFamily、DecisionKind、PayloadFormat | 确定评估事实的机制语义 |
| `Result` | Primary Score、Result Level | 报告主分与等级摘要 |
| `ReportSpec` | ReportType、TemplateVersion、Algorithm、ProductChannel、Profile、AdapterKey、TemplateID | 确定渲染路由与呈现资产 |

机制专属事实通过可选部分表达：

| 事实组 | 来源 | 当前使用者 |
| --- | --- | --- |
| `FactorScoringFacts` | Execution dimensions + 冻结 scale/norm/task 模型 | FactorScoring、Norm、Task Builder |
| `PersonalityTypeFacts` | 分类结果 + 冻结 outcome mapping/profile/template | pole composition、nearest pattern、dominant factor |
| `TraitProfileFacts` | trait dimensions + source attribution | trait profile |

不同事实组不强行折叠成“万能分数列表”。Builder 只接受自己机制所需的结构。

## 5. schema v2 与历史解码

### 5.1 分数、常模和能力型

Adapter 从冻结 scale snapshot 中读取 Factor 名称、最大分和 InterpretRules，再与 Execution dimensions 按 factor code 组合。评分值来自 Outcome，解释规则来自冻结 ReportInput。

### 5.2 schema v2 类型学

Outcome payload 只保存 `TypeCode / MatchPercent / SpecialTrigger` 等分类事实。Adapter 必须在冻结 typology payload 的 outcome mapping 中找到 TypeCode，才能恢复 TypeName、OneLiner、Strengths、Suggestions、Rarity 和授权资料。

如果 code 不存在，报告以输入失败结束，不会去查当前 Published Model 补齐。

### 5.3 历史 Outcome

codec 仍支持从旧 payload 读取已带文案的 PersonalityTypeDetail / TraitProfileDetail。这是 versioned 读兼容，不是新写模型。

## 6. Builder 的完整机制键

`rendering.Key` 包含：

```text
AlgorithmFamily
+ DecisionKind
+ ReportType
+ TemplateVersion
+ Algorithm
+ ProductChannel
+ ReportProfile
```

| 维度 | 解决的问题 |
| --- | --- |
| AlgorithmFamily | 是计分、分类、常模还是任务表现 |
| DecisionKind | 分数区间、极性组合、trait profile、nearest pattern 等决策语义 |
| ReportType | 同一 Outcome 要生成哪类报告 |
| TemplateVersion | 哪一批不可变模板、Builder 行为和内容 schema |
| Algorithm / ProductChannel | 机制内的精确算法或产品通道特化 |
| ReportProfile | scale、norm、task、personality_type、trait_profile、pattern_profile 呈现形态 |

`RoutingContextFromInput` 为历史输入补充明确的兼容默认：

- ReportType 缺失 -> `standard`；
- TemplateVersion 缺失 -> `legacy-v1`；
- DecisionKind 缺失 -> 按 AlgorithmFamily 映射默认决策；
- ReportProfile 缺失 -> 按 DecisionKind 映射。

这些默认是显式的历史兼容政策，不是任意回退。

## 7. 同模板版本内的 fallback

Registry 从最具体的键开始，逐步去掉 ReportProfile、ProductChannel 和 Algorithm，最后回到 AlgorithmFamily + ReportType。所有 candidate 都会重新写入原始 TemplateVersion。

因此：

- 可以为特定 algorithm/channel/profile 注册专用 Builder；
- 没有专用 Builder 时可在同版本内复用宽泛 Builder；
- 绝不会从 `legacy-v1` 回退到另一个 TemplateVersion。

Registry 启动时拒绝 nil Builder、缺失键、缺失 builder identity/content schema、模板版本不一致和重复键。

## 8. 当前 Builder 矩阵

| AlgorithmFamily | DecisionKind | Builder | 主要输出 |
| --- | --- | --- | --- |
| `factor_scoring` | `score_range` | `factor-scoring` | 总分、因子维度、风险等级、区间解释与建议 |
| `factor_norm` | `norm_lookup` | `norm-profile` | 常模结果摘要与维度报告 |
| `task_performance` | `ability_level` | `task-performance` | 能力等级和任务表现维度 |
| `factor_classification` | `pole_composition / trait_profile / nearest_pattern / dominant_factor` | `typology` | 人格类型或 trait profile、维度、ModelExtra 和归因资料 |

FactorScoring / Norm / Task 当前复用 `DefaultReportBuilder` 的结论、维度与建议组装能力。Typology Builder 按 `AdapterKey / TemplateID` 在 MBTI、SBTI、Big Five 和通用 personality/trait 模板之间选择。

## 9. 建议和解释为什么不是运行时策略

`rule.CollectConfiguredSuggestions` 只整理冻结的总体建议和维度建议，并做结构化去重。它不查数据库、不调 AI、不读当前配置。

这条边界保证：

- 同一 Outcome + TemplateVersion 的构建可重放；
- 报告文案可以追溯到冻结输入；
- 重试不会因外部服务状态而改变语义。

## 10. 生产生成与 ModelCatalog 预览

| 项目 | 生产 Interpretation | ModelCatalog Preview |
| --- | --- | --- |
| 输入 | 持久化 Outcome Record | 合成 Assessment + draft model input + 进程内 Execution |
| Builder 解析 | Registry 按完整机制键 | 当前直接调用 `NewTypologyBuilder` |
| Generation / Run | 创建并持久化 | 不创建 |
| InterpretReport | 成功后持久化 | 只返回 transient Draft |
| 事件 | generated / failed | 不发事件 |

预览复用领域渲染能力，但不能被写成“生产报告的特殊生成方式”。

## 11. 扩展新报告机制的最小步骤

1. 先确认 EvaluationOutcome schema 已持久化稳定评估事实。
2. 确认所有解释文案、模板选择和归因资料都能进入冻结 ReportInput。
3. 在 `InterpretationInput` 中新增或复用明确的机制事实组，不引入 Assessment 或 Repository。
4. 实现 Builder，声明 ReportType、TemplateVersion、BuilderIdentity、ContentSchemaVersion 和 MechanismKey(s)。
5. 把 Builder 加入 `DefaultBuilders`，覆盖精确键、fallback、重复键和未知版本测试。
6. 验证生产生成、预览、事件 payload 和多 audience 查询投影。

禁止通过修改 Evaluation 主链、按 model code 写 switch、读当前 Published Model 或在 Builder 中重新计分来完成扩展。
