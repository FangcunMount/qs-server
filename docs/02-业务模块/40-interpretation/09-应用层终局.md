# Interpretation 应用层终局

## 1. 设计原则

领域层对 `ReportGeneration`、`InterpretationRun`、`InterpretReport`、报告渲染和可见性规则负责；应用层对行为人的命令、查询、授权、事务和输出契约负责。

应用层不再按 `generation / input / reporting / query` 等机制建立一级包，而按以下行为人组织：

| 行为人 | 应用职责 |
| --- | --- |
| 自动化报告生成系统 | 根据 Outcome 生成或重试报告，返回 generated/processing/failed |
| 受试者/参与者 | 查询自己的报告，只看到 participant 视图 |
| 临床人员 | 查询获授权受试者报告，应用 clinician 可见性规则 |
| 机构管理员/运营人员 | 查询机构范围报告和管理视图 |
| 运维/审计人员 | 查询 Generation、Run、失败原因、历史模板与重试状态 |
| 模型设计者 | 保留在 Model Catalog application，通过稳定领域能力预览 |

## 2. 目标结构

```text
application/interpretation/
├── automation/
├── participant/
├── clinician/
├── administration/
├── operations/
└── internal/
```

`Starter`、`Executor`、`InterpretationCommitter` 和 Outcome input adapter 是 automation 用例的内部协作者；报告 Builder、Registry、模板路由和 Audience 可见性属于领域能力。

## 3. 固定不变量

1. 自动化生成只接受持久化 Outcome，不读取或修改 Assessment。
2. 重试不重新执行 Evaluator。
3. 一个 Generation 生成一份完整、不可变的标准报告。
4. Audience 不进入 Generation 身份；participant、clinician、admin 差异在查询投影阶段应用。
5. 每个行为人应用服务在访问报告前完成自己的授权。
6. 应用服务返回 DTO，不向 Transport 暴露可写领域聚合。
7. Generation、Run、Report、outbox 的事务和事件契约保持不变。
8. 模型预览不创建 Generation、Run 或 InterpretReport。

## 4. 迁移里程碑

- A0：目标不变量与 characterization。已完成。
- A1：领域渲染能力归位。已完成。
- A2：自动化报告生成行为人服务。已完成。
- A3：标准报告与 Audience 读取投影。已完成。
- A4：参与者报告应用服务。已完成。
- A5：机构管理员报告应用服务。已完成。
- A6：临床人员报告应用服务。已完成。
- A7：运维审计应用服务。已完成。
- A8：Model Catalog 预览解耦。已完成。
- A9：装配切换与兼容债清理。已完成。

当前应用层根包仅作为命名空间，生产行为分别位于 `automation`、`participant`、`clinician`、`administration`、`operations`；原 `generation`、`input`、`reporting` 一级机制包和无行为人的 `ReportQueryService`、`LifecycleQueryService`、`OutcomeReportService` 已删除。
