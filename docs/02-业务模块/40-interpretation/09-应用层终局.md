# Interpretation 应用层终局

## 1. 设计原则

领域层对 `ReportGeneration`、`InterpretationRun`、`InterpretReport`、报告渲染和可见性规则负责；应用层对行为人的命令、查询、授权、事务和输出契约负责。

应用层不再按 `generation / input / reporting / query` 等机制建立一级包，而按以下行为人组织：

| 行为人 | 应用职责 |
| --- | --- |
| 自动化报告生成系统 | 根据 Outcome 生成或重试报告，返回 generated/processing/failed |
| 受试者/参与者 | 查询自己的报告，只看到 participant 视图 |
| 临床人员 | 查询获授权受试者报告，应用 clinician 可见性规则 |
| 机构管理员/运营人员 | 查询机构范围报告和管理视图 |
| 运维/审计人员 | 查询 Generation、Run、失败原因、历史模板与重试状态 |
| 模型设计者 | 保留在 Model Catalog application，通过稳定领域能力预览 |

## 2. 目标结构

```text
application/interpretation/
├── automation/
├── participant/
├── clinician/
├── administration/
├── operations/
└── internal/
```

`Starter`、`Executor`、`InterpretationCommitter` 和 Outcome input adapter 是 automation 用例的内部协作者；报告 Builder、Registry、模板路由和 Audience 可见性属于领域能力。

## 3. 固定不变量

1. 自动化生成只接受持久化 Outcome，不读取或修改 Assessment。
2. 重试不重新执行 Evaluator。
3. 一个 Generation 生成一份完整、不可变的标准报告。
4. Audience 不进入 Generation 身份；participant、clinician、admin 差异在查询投影阶段应用。
5. 每个行为人应用服务在访问报告前完成自己的授权。
6. 应用服务返回 DTO，不向 Transport 暴露可写领域聚合。
7. Generation、Run、Report、outbox 的事务和事件契约保持不变。
8. 模型预览不创建 Generation、Run 或 InterpretReport。

## 4. 迁移里程碑

- A0：目标不变量与 characterization。已完成。
- A1：领域渲染能力归位。已完成。
- A2：自动化报告生成行为人服务。已完成。
- A3：标准报告与 Audience 读取投影。已完成。
- A4：参与者报告应用服务。已完成。
- A5：机构管理员报告应用服务。已完成。
- A6：临床人员报告应用服务。已完成。
- A7：运维审计应用服务。已完成。
- A8：Model Catalog 预览解耦。已完成。
- A9：装配切换与兼容债清理。已完成。

当前应用层根包仅作为命名空间，生产行为分别位于 `automation`、`participant`、`clinician`、`administration`、`operations`；原 `generation`、`input`、`reporting` 一级机制包和无行为人的 `ReportQueryService`、`LifecycleQueryService`、`OutcomeReportService` 已删除。

## 5. 传输契约终局

`api/grpc/proto/interpretation/interpretation.proto` 统一承载 `ParticipantReportService`、`InterpretationAutomationService` 及 Report/Generation 消息。报告中的 `ModelIdentity`、`ScoreValue`、`ResultLevel` 直接引用 Evaluation proto 的评分事实类型，不复制事实契约。Evaluation proto 不再定义 Report，Internal proto 不再定义报告生成流程 RPC。

## 6. H0-H8 查询与授权收尾

- H0-H3：新增轻量 `report_query_catalog`，成功提交时与 Report/Run/Generation/outbox 同事务投影；列表先分页目录，再按页读取 `interpret_report_artifacts` 或 `archived_reports` 正文。
- H4：管理员省略 TesteeID 时按当前 Org 查询；临床人员仍必须处于授权 Testee 范围。
- H5：临床报告通过 `/api/v1/clinicians/me/testees/{testee_id}/reports` 暴露。
- H6：运维审计通过 `/internal/v1/interpretation/*` 暴露，要求 IAM `qs:interpretation_reports/audit` 并校验资源 Org。
- H7：collection 报告、状态和等待入口同时校验登录用户到 IAM Profile 的 active link；自动化未分类错误只返回安全消息。
- H8：查询和授权架构守卫禁止恢复 legacy 全量合并、自声明 permissions 与 Audience 身份漂移。

`report_query_catalog` 只是每个 Assessment 一行的查询索引，保存范围、排序、筛选字段和正文来源引用；它不是第四个领域对象，也不保存报告正文。`archived_reports` 继续作为不可重试、无 Outcome 的历史成品正文。

## 7. R0-R6 终局退役

- R0：active archive/artifact 与 catalog 完成总量、优先级、引用和机构归属对账。
- R1：Statistics 也通过 catalog 解析当前报告，不再自行合并 artifact/archive。
- R2：删除旧查询 helper 和未使用身份中间件，所有未分类报告错误只返回安全消息。
- R3：生产代码与 Mongo 索引初始化不再包含 `interpret_reports`。
- R4：旧集合离线压缩备份、恢复抽样后 drop；备份保留 30 天。
- R5：删除旧 lifecycle/dedupe 命令和通用脚本中的旧集合分支。
- R6：终局守卫与文档固定集合、边界和验收命令。

最终 Interpretation 持久化集合固定为：

- `report_generations`：幂等生成身份和整体状态；
- `interpretation_runs`：每次执行尝试；
- `interpret_report_artifacts`：新三对象不可变成品；
- `report_query_catalog`：每个 Assessment 一行的当前来源选择与分页索引；
- `archived_reports`：有效的、不可重试的历史正文。

`interpret_reports` 不属于终局集合。`wait-report` 长轮询本轮保持兼容，另行制定退役计划。
