# 核心设计：生成幂等与可靠提交

## 1. 本文回答

本文说明重复 `evaluation.outcome.committed`、并发 Worker、进程崩溃、Builder 失败和 Mongo 事务失败下，Interpretation 如何避免重复报告、半成功状态和无限重试。

## 2. 30 秒结论

Interpretation 的幂等不是“查到报告就 return”，而是五层物理与事务约束：

| 边界 | 保护机制 |
| --- | --- |
| 一个报告意图只有一个 Generation | `(outcome_id, report_type, template_version)` unique index |
| 一个 Generation 的 attempt 不重复 | `(generation_id, attempt)` unique index |
| 同一 Generation 只有一份成品 | `interpret_report_artifacts.generation_id` unique index |
| 并发状态迁移不互相覆盖 | Generation version CAS |
| Report、catalog、Run、Generation、Outbox 同成同败 | 单一 Mongo 事务 |

Generation 回答“整份报告的当前结果”，Run 回答“某次尝试发生了什么”。两者不能合并。

## 3. Starter 的幂等决策

`Starter.Start` 以 Generation Key 读取聚合，再按状态决定：

| Generation 状态 | 处理 | 返回状态 |
| --- | --- | --- |
| 不存在 | 同一事务创建 Generation + attempt 1 running Run | `started` |
| `pending` | 创建 attempt 1 并 Begin | `started` |
| `generating` + active lease | 返回正在执行的 Run，不创建新 attempt | `processing` |
| `generating` + expired lease | 把旧 Run/Generation 结束为 failed，同事务创建下一 Run | `started` |
| `failed` | 从 latest failed Run 创建 attempt + 1 | `started` |
| `generated` | 按 Generation.ReportID 读取已有成品 | `generated` |

注意：当前 Starter 对所有 `failed` Generation 都允许再次调用创建下一 Run；是否再次调用由 Worker 对持久化 `Retryable` 的 settlement 决定。因此运维或新入口不应绕过失败语义盲目重放。

## 4. 并发 claim 与 CAS

两个 Worker 可能同时看到 Generation 不存在，或同时看到 failed Generation。Starter 把下列错误当作 claim conflict：

- Generation unique key duplicate；
- Generation expected version CAS 失败。

发生 conflict 后会重读一次对方已提交的状态，返回 `generated / processing / started`，而不制造第二个并发尝试。连续两次 conflict 则返回错误，由消息层决定重试。

## 5. lease 与崩溃恢复

新 Run 通过 `StartWithLease` 从 pending 进入 running。当前组合根设置 lease 为 5 分钟。

```text
running + lease_expires_at > now
  -> processing

running + lease_expires_at <= now
  -> old Run failed(timeout/lease_expired/retryable=true)
  -> Generation failed
  -> new Run running
  -> Generation generating
```

过期 Run 的失败与新 Run 的创建在同一 Mongo 事务中持久化，避免出现“旧 Run 已失败，但新 Run 未建立”的中间状态。

当前没有独立 Interpretation lease sweeper。lease 过期只会在后续 `Generate` 调用读到该 Generation 时恢复。排查长期 `generating` 时，不能假设后台定时任务会自动清理。

## 6. 失败分类与 Worker 重试

| 失败场景 | 持久化 Failure | Retryable | Worker |
| --- | --- | --- | --- |
| 缺少 AlgorithmFamily / DecisionKind | `input/unsupported_mechanism` | false | ACK |
| Registry 无 Builder | `template/builder_not_found` | false | ACK |
| Builder 返回 error | `build/build_failed` | true | NACK，重投后新 Run |
| Builder 返回 nil Draft | `build/empty_draft` | true | NACK，重投后新 Run |
| Artifact 不满足领域不变量 | `build/invalid_artifact` | false | ACK |
| Run lease 过期 | `timeout/lease_expired` | true | 下一次调用时恢复 |
| 读 Outcome、启动事务或终态事务失败 | 不一定已形成持久化 Interpretation failure | 传输回执默认 true | NACK |

`FailedError` 只表示 Run 与 Generation 失败已可靠提交。gRPC 将其转为含 GenerationID、RunID、FailureKind/Code 和 Retryable 的失败回执，Worker 不从错误文本猜测终态。

## 7. 成功可靠提交

`InterpretationCommitter.CommitSuccess` 先在 Generation / Run 副本上准备终态，再在一个 Mongo 事务中提交：

1. 不可变 `InterpretReport`；
2. `report_query_catalog` 当前来源投影；
3. `InterpretationRun = succeeded`；
4. `ReportGeneration = generated` + ReportID；
5. `interpretation.report.generated` Outbox event。

Generation、Run 和 Report 的互相引用在事务前校验：

- Run.GenerationID = Generation.ID；
- Generation.LatestRunID = Run.ID；
- Report.GenerationID / InterpretationRunID 与两者一致；
- OutcomeID / ReportType / TemplateVersion 与 Generation Key 一致。

任一一步失败都回滚全部写入。事务成功后再把事件交给 post-commit dispatcher，但 durable 事实已在 outbox 中。

## 8. 失败可靠提交

`CommitFailure` 在一个 Mongo 事务中提交：

1. `InterpretationRun = failed` + safe Failure；
2. `ReportGeneration = failed`；
3. `interpretation.report.failed` Outbox event。

失败路径不创建 InterpretReport，不更新 report_query_catalog，也不触碰 Evaluation 数据。

Committer 对原 Generation / Run 做副本后再迁移。如果事务失败，调用方持有的对象仍是 `generating/running`，不会被内存中的假终态污染。

## 9. 事件语义

| 事件 | 运行时 AggregateType | delivery | 主要下游 |
| --- | --- | --- | --- |
| `interpretation.report.generated` | `ReportGeneration` | Mongo durable outbox | Worker 高风险提醒与 Assessment attention 投影；其他读侧需显式订阅 |
| `interpretation.report.failed` | `ReportGeneration` | Mongo durable outbox | 失败审计与观测 |

当前 `configs/events.yaml` 的分类元数据仍把 aggregate 写成 `Report`，但领域事件 envelope 由 `domain/interpretation.AggregateType` 实际写入 `ReportGeneration`。排查生产事件时以 envelope 与生产者代码为准；该配置命名差异不应被文档隐藏。

## 10. 回执与 ACK/NACK

| gRPC 回执 | Worker 行为 | 后续 |
| --- | --- | --- |
| `success=true, status=generated` | ACK | 等待 generated event 驱动后置投影 |
| `success=true, status=processing` | ACK | 已有 active lease 的 worker 继续处理 |
| `success=false, status=failed, retryable=false` | ACK | 终态失败，不无限重投 |
| `success=false, status=failed, retryable=true` | NACK | 重投后在同 Generation 创建新 Run |
| nil / 未知失败状态 / gRPC error | NACK | 不确定状态不误 ACK |

`already_generated` 仍被 Worker 当作兼容终态并 ACK，但当前 Automation Service 对已生成 Generation 正常返回 `success=true, status=generated`。

## 11. 运维排查顺序

```text
Generation key 是否唯一
  -> Generation status/version/latest_run_id
  -> latest Run status/attempt/lease/failure
  -> generated 时 ReportID 是否存在
  -> report_query_catalog 是否指向该 artifact
  -> Mongo outbox 是否含对应 terminal event
  -> Worker settlement 是 ACK 还是 NACK
```

如果 Generation 是 `generated` 但 Report 不存在，或 catalog 指向缺失正文，这是持久化一致性问题，不应通过再跑 Builder 盲目修复。
