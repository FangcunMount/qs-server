# Interpretation 设计终局

## 1. 文档目的

本文是 Interpretation 后续重构的判定基线。它定义报告生成意图、执行尝试和报告成品的边界，防止报告生命周期再次回流到 Evaluation。

## 2. 终局职责

Interpretation 只消费已成立的 `EvaluationOutcome`，生成可追溯的报告成品；它不重新计分，也不修改 Assessment、EvaluationRun、Outcome 或 score projection。

```text
evaluation.outcome.committed
  → ReportGeneration
  → InterpretationRun (attempt 1..n)
  → InterpretReport (immutable artifact)
```

## 3. 终局领域模型

| 对象 | 身份与生命周期 | 责任 |
| ---- | -------------- | ---- |
| `ReportGeneration` | 聚合根；业务唯一键 `OutcomeID + ReportType + TemplateVersion` | 记录报告生成意图、总体状态、最新 Run 与成功成品引用 |
| `InterpretationRun` | 一次生成尝试；可有多个 | 记录 attempt、开始/结束、trace、失败原因与结果 |
| `InterpretReport` | 仅成功后存在的不可变成品 | 保存解释内容，并关联 Generation、Outcome 与成功 Run |

ReportGeneration 的独立身份允许同一 Outcome 以后按不同报告类型、模板版本或受众生成多个报告。`ReportID` 是独立身份，不再隐式复用 AssessmentID。

## 4. 状态机与幂等

```text
ReportGeneration: pending → generating → generated
                                  └→ failed → generating (new InterpretationRun)

InterpretationRun: pending → running → succeeded / failed
```

- 同一 Generation 已 `generated`：直接返回既有 Report。
- 同一 Generation `generating`：返回处理中，不创建并发重复 Run。
- `pending / failed`：创建下一次 InterpretationRun。
- 重试只读取同一个 Outcome；不得重新执行 Evaluator。

## 5. 事件与事务边界

| 事实 | 可靠事件 | 同事务提交 |
| ---- | -------- | ---------- |
| 报告成功 | `interpretation.report.generated` | Run `succeeded`、Generation `generated`、不可变 Report 与 outbox |
| 报告失败 | `interpretation.report.failed` | Run `failed`、Generation `failed` 与 outbox |

只保留两个 durable 终态事件：`interpretation.report.generated` 与 `interpretation.report.failed`；`pending / generating` 与一次尝试开始均是查询状态，不发送事件。

- 两个事件的 `aggregate_type` 固定为 `ReportGeneration`，`aggregate_id` 固定为 `generation_id`。
- 两个事件均携带 `generation_id`、`run_id`、`outcome_id`、`report_type`、`template_version`、关联主体标识、`attempt` 与发生时间。
- 成功事件额外携带 `report_id`、`builder_identity`、`content_schema_version` 及报告结果摘要。
- 失败事件额外携带 `failure_kind`、`failure_code`、`retryable` 与 `safe_reason`；失败时没有成品 `report_id`。

Registry 必须按完整机制键（包括 `template_version`）精确解析 Builder。模板版本由持久化 Generation/Outcome 输入冻结；未知版本必须失败，不能回退到刚发布的新模板。

`InterpretationCommitter` 是唯一的终态提交边界：它在同一事务内提交 Artifact（成功时）、Run、Generation 与 outbox；`Executor` 只负责构建 Artifact 并委派提交。提交前在 Generation/Run 副本上准备终态，事务失败不得污染调用方持有的 `generating/running` 对象。

## 6. 不变量

1. Interpretation 只读 EvaluationOutcome，不持有 Assessment Repository。
2. 报告失败不改变 Evaluation 任何事实；失败和重试只影响 Generation / Run。
3. `InterpretReport` 不保存 `pending/generating/failed`、attempt 或失败原因。
4. 生成成功后，Report 内容不可由重试或后续投影覆盖。
5. `interpretation.report.failed` 不能被订阅为“再次生成报告”的触发源，避免事件环。
6. 客户端 `completed / interpreted` 是读模型组合状态，不是 Report 或 Assessment 的额外写模型状态。

## 7. 明确不属于 Interpretation

- 计分、因子计算、等级判定和 EvaluationRun 创建。
- 修改 Assessment 状态、分数、等级、Outcome 或 score projection。
- 将报告失败投射为 `evaluation.failed`。
- 以报告查询结果替代 Outcome 作为生成输入。

## 8. 完成判据

- 三对象有独立领域类型、仓储和直接测试保护。
- 存储层存在 Generation 幂等唯一约束与 Run 的 attempt 审计。
- 成功、失败均在状态事实与 outbox 间保持同事务提交。
- worker 仅从 `evaluation.outcome.committed` 调用 Interpretation 用例。
- `assessment.interpreted`、`report.generated` 等旧 wire 名不再出现在生产配置或生产者中。

## 9. 与 Evaluation 的接口

Interpretation 的稳定输入是 EvaluationOutcome，不是 Assessment 可写对象。Evaluation 负责评分事实，Interpretation 负责报告生命周期与报告成品；二者通过 Outcome ID 与可靠事件解耦。
