# 核心设计：报告数据存储与查询索引

## 1. 本文回答

本文说明 Interpretation 为什么同时需要 Generation、Run、Artifact、Catalog 和 Archive 五类 Mongo 数据，每个 collection 的主键、唯一约束、写入时机与读取用途分别是什么。

## 2. 30 秒结论

Interpretation 的当前运行时存储模型是“三对象写模型 + 一个当前查询索引 + 一个历史正文库”：

| Collection | 角色 | 是否当前主写 |
| --- | --- | --- |
| `report_generations` | 幂等生成意图和整体状态 | 是 |
| `interpretation_runs` | attempt、lease、trace 和失败审计 | 是 |
| `interpret_report_artifacts` | 不可变成功报告正文 | 是 |
| `report_query_catalog` | 每个 Assessment 当前正文来源与分页索引 | 成功事务投影 |
| `archived_reports` | 无 Outcome / 无重试能力的历史报告正文 | 否，只读兼容 |

`interpret_reports` 不再被当前 Interpretation Repository 读写，但仓库的早期 Mongo migration 和 `scripts/mongodb/create-indexes.js` 仍存在对它的历史引用。因此它是已退出运行时主链的 legacy collection，不应写成“仓库所有运维痕迹都已删除”。

Mongo 没有用 DBRef 或强外键建模。一致性依赖领域不变量、唯一索引、Generation CAS 和 Mongo transaction。

## 4. report_generations

### 4.1 字段分组

| 分组 | 字段 | 语义 |
| --- | --- | --- |
| 身份 | `domain_id` | ReportGeneration ID |
| 幂等键 | `outcome_id / report_type / template_version` | 一份完整报告意图 |
| 状态 | `status / latest_run_id / report_id` | 总体进度和当前引用 |
| 并发 | `version` | 乐观锁 expected version |
| 时间 | `created_at / updated_at` | 生成意图生命周期 |

### 4.2 索引

| 索引 | 类型 | 保护/查询 |
| --- | --- | --- |
| `uk_generation_domain_id(domain_id)` | unique | Generation 领域 ID |
| `uk_generation_key(outcome_id, report_type, template_version)` | unique | 业务幂等 |
| `idx_generation_outcome_status_updated(outcome_id, status, updated_at desc)` | normal | Outcome 生成历史与运维查询 |

`Save` 使用 `domain_id + expected version` 做 update filter。MatchedCount 不为 1 就返回 VersionConflict，不做 last-write-wins。

## 5. interpretation_runs

| 分组 | 字段 | 语义 |
| --- | --- | --- |
| 身份 | `domain_id / generation_id / attempt` | 某 Generation 的第 N 次尝试 |
| 状态 | `status` | pending / running / succeeded / failed |
| 执行 | `trace_id / started_at / lease_expires_at / finished_at` | 调用追踪与 lease |
| 失败 | `failure.kind/code/safe_message/retryable` | 已持久化、可分类的安全事实 |

| 索引 | 类型 | 保护/查询 |
| --- | --- | --- |
| `uk_interpretation_run_domain_id(domain_id)` | unique | Run ID |
| `uk_interpretation_run_generation_attempt(generation_id, attempt)` | unique | attempt 不重复 |
| `idx_interpretation_run_generation_attempt_desc(generation_id, attempt desc)` | normal | 读取 latest Run |

Run 是 append-attempt 历史，但单个 Run 从 running 到 terminal 会更新原文档。“append-only”指失败后不覆盖旧 attempt 去伪造新尝试。

## 6. interpret_report_artifacts

Artifact 只在成功事务中 insert，不提供 update 方法。主要字段：

- provenance：GenerationID、OutcomeID、InterpretationRunID、ReportType、TemplateVersion、GeneratedAt；
- query association：OrgID、AssessmentID、TesteeID；
- content：Model、PrimaryScore、Level、Conclusion、Dimensions、Suggestions、ModelExtra；
- legacy-compatible summary：ScaleName/Code、TotalScore、RiskLevel。

| 索引 | 类型 | 保护/查询 |
| --- | --- | --- |
| `uk_artifact_domain_id(domain_id)` | unique | Report ID |
| `uk_artifact_generation_id(generation_id)` | unique | 一个 Generation 只有一份成品 |
| `idx_artifact_outcome_type_version(outcome_id, report_type, template_version)` | normal | Outcome 报告版本追溯 |
| `idx_artifact_assessment_generated(assessment_id, generated_at desc)` | normal | Assessment 历史报告 |
| `idx_artifact_testee_generated(testee_id, generated_at desc)` | normal | Testee 查询 |

`assessment_id` 可以对应多份不同 TemplateVersion 或 ReportType 的 Artifact，因此不能在 Artifact Repository 中靠“找最新”自行定义当前报告。

## 7. report_query_catalog

Catalog 每个 Assessment 唯一一行，只保存筛选、排序和正文来源：

```text
assessment_id
org_id / testee_id
source_kind = artifact | archive
source_id
model_code / risk_level
sort_at / sort_report_id
```

### 7.1 当前来源选择

Artifact 成功提交时 `ProjectCurrent` 做 upsert：

- 当前来源是 archive：artifact 总是获胜；
- 当前来源是较旧 artifact：新 `generated_at` 获胜；
- generated_at 相同：较大 ReportID 获胜；
- 新 artifact 较旧：不覆盖；
- duplicate key 表示更新的 artifact 已获胜，投影器幂等返回。

### 7.2 索引

| 索引前缀 | 用途 |
| --- | --- |
| unique `assessment_id` | 当前来源唯一 |
| `org_id + sort_at desc + assessment_id desc` | 机构报告分页 |
| `testee_id + sort_at desc + assessment_id desc` | 受试者报告分页 |
| `org_id/testee_id + model_code + sort_at` | 模型筛选 |
| `org_id/testee_id + risk_level + sort_at` | 风险筛选 |

列表查询先在 catalog 中 Count + 稳定排序分页，然后只批量读取当前页的 artifact/archive 正文，避免把两个大 collection 全量合并后再分页。

## 8. archived_reports

Archive 是历史报告正文：

- 可能没有 OutcomeID、GenerationID 和 RunID；
- 不进入 Generation / Run 重试状态机；
- 它的 `domain_id` 按历史契约对应 AssessmentID；
- 读模型会从 ScaleName/Code、TotalScore、RiskLevel 兼容推导 ModelIdentity、PrimaryScore 和 Level；
- 只有 catalog 指向它时才进入当前查询结果。

Archive 是为了保留有效历史报告，不是用来伪造 Generation / Run 历史。

## 9. 写事务边界

| 阶段 | 同一 Mongo 事务中的写入 |
| --- | --- |
| 初次 claim | Generation(generating) + Run(attempt 1, running) |
| failed 重试 | Generation(generating) + 新 Run(running) |
| lease 过期恢复 | 旧 Run(failed) + Generation 中间失败后的最终 generating 状态 + 新 Run(running) |
| 生成成功 | Artifact + Catalog + Run(succeeded) + Generation(generated) + Outbox |
| 生成失败 | Run(failed) + Generation(failed) + Outbox |

EvaluationOutcome 存在 MySQL，Interpretation 存在 Mongo。跨存储不使用分布式事务：Evaluation 先提交 Outcome 与 `evaluation.outcome.committed`，Interpretation 再以 OutcomeID 读取已成立事实。

## 10. 查询一致性与故障信号

| 现象 | 含义 | 正确方向 |
| --- | --- | --- |
| Generation generated，ReportID 指向缺失 | 成功事务不变量破坏 | 查 Mongo 事务、人工数据操作或还原 |
| Catalog 指向缺失正文 | dangling source | 视为内部一致性错误，不静默跳过 |
| Artifact 存在但 catalog 缺失 | 投影漏失或历史回填未完成 | 使用 catalog 对账/回填工具 |
| Catalog 仍指向 archive，artifact 已存在 | 优先级异常 | 检查成功事务投影和回填对账 |
| Generation 长期 generating、lease 过期 | 无后续 Generate 调用触发恢复 | 核对消息 settlement 与手工重放权限 |

## 11. 诊断查询示例

```javascript
// 某 Outcome 的 Generation
db.report_generations.find({outcome_id: NumberLong("<outcome_id>")})
  .sort({updated_at: -1})

// 某 Generation 的全部尝试
db.interpretation_runs.find({generation_id: NumberLong("<generation_id>")})
  .sort({attempt: -1})

// 某 Assessment 的新报告成品历史
db.interpret_report_artifacts.find({assessment_id: NumberLong("<assessment_id>")})
  .sort({generated_at: -1, domain_id: -1})

// 当前查询来源
db.report_query_catalog.findOne({assessment_id: NumberLong("<assessment_id>")})
```

生产查询请结合实际 Mongo ID 编码和只读权限；这些语句用于诊断关系，不是修复脚本。
