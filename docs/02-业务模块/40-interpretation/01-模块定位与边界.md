# Interpretation Model 模块定位与边界

## 1. 模块定位

Interpretation Model 是解释模型与报告产出层。它把 Evaluation 生成的结构化结果转换成用户、从业者或运营端可理解的报告。

当前代码包仍叫 `interpretation`，文档业务语言使用 `interpretation`。

**当前实现**：报告 Builder 与持久化已由 Interpretation 承担，但 `evaluation/execute` 仍直接调用 Interpretation，reporting writer 仍修改并保存 Assessment，把它推进到 `interpreted`。

**已确认目标**：Assessment 归属 Evaluation 并在 `evaluated` 结束；Interpretation 只消费持久化 EvaluationOutcome，独立维护 Report 生命周期，不持有 Assessment 仓储、不重新计分、不反向修改 Assessment。

---

## 2. 负责范围

| 范围 | 说明 |
| ---- | ---- |
| 解释模型 | 分数区间、因子解释、风险提示、建议文案 |
| 报告模型 | 报告 section、模板、输出结构 |
| 报告构建 | Builder registry、score / personality adapter |
| 报告实例 | `InterpretReport` 聚合与持久化 |
| 报告事件 | `report.generated` |
| 报告生命周期 | `pending / generating / generated / failed`，支持独立重试 |

---

## 3. 不负责范围

| 不负责 | 归属模块 |
| ------ | -------- |
| 作答提交 | `survey` |
| 模型资产发布 | `model-catalog` |
| 测评执行状态机 | `evaluation` |
| Assessment 聚合持久化 | `evaluation` |
| 周期任务调度 | `plan` |
| 统计聚合 | `statistics` |

---

## 4. 边界反例

- `InterpretReport` 不是 `Evaluation`。
- `InterpretationModel` 不是 `AssessmentModel`。
- 报告生成失败不应改写答卷事实。
- 报告生成失败也不应把已 `evaluated` 的 Assessment 改成 `failed`。
- 面向客户端的 `interpreted / completed` 是 Journey 投影，不是 Report 对 Assessment 的写操作。
- 报告查询不是统计查询。
