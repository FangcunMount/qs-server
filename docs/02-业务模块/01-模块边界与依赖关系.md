# 模块边界与依赖关系

**本文回答**：七个业务模块各自负责什么、不负责什么、上下游是谁，以及哪些反向依赖禁止出现。

---

## 1. 边界矩阵

| 模块 | 核心职责 | 不负责 | 上游 | 下游 |
| ---- | -------- | ------ | ---- | ---- |
| `survey` | 问卷定义、题目结构、答卷事实 | 计分、解释报告、周期计划 | `actor`、`plan` | `evaluation`、`statistics` |
| `assessment-model` | 测评模型、规则资产、模型快照 | 答卷事实、执行状态、报告实例 | 管理后台、模型配置 | `evaluation`、`interpretation-model` |
| `evaluation` | 一次测评执行、计分、结果生成 | 问卷定义、报告文案维护 | `survey`、`assessment-model` | `interpretation-model`、`statistics` |
| `interpretation-model` | 解释模型、报告构建、报告实例 | 原始答卷采集、测评任务调度 | `evaluation`、`assessment-model` | `statistics`、查询端 |
| `actor` | 参与者、受试者、访问上下文 | 登录认证、IAM 主身份管理 | IAM | `survey`、`plan`、`statistics` |
| `plan` | 测评计划、周期任务、任务调度 | 计分、报告生成 | `actor`、运营配置 | `survey`、`evaluation`、`statistics` |
| `statistics` | 统计指标、读模型、行为投影 | 业务事实源维护 | 所有业务事件 | 运营后台、数据看板 |

---

## 2. 禁止反向依赖

| 禁止依赖 | 原因 |
| -------- | ---- |
| `survey -> evaluation` | Survey 只沉淀作答事实，不能知道测评执行细节 |
| `survey -> interpretation-model` | Survey 不关心报告结构和解释文案 |
| `assessment-model -> evaluation` | 模型资产可被执行，但不依赖某一次执行实例 |
| `evaluation -> statistics` | Evaluation 发布事实，Statistics 消费事实；统计不能成为执行前置 |
| `interpretation-model -> statistics` | 报告生成可以发布事件，不应读取统计投影来决定报告事实 |
| `statistics -> 核心写模型` | Statistics 是读侧投影，不能反向驱动 `survey/evaluation/report/plan` 主状态 |

## 3. 业务名称与代码包

| 业务名称 | 当前代码位置 | 说明 |
| -------- | ------------ | ---- |
| `survey` | `internal/apiserver/domain/survey`、`container/modules/survey` | 文档名与代码名一致 |
| `assessment-model` | `internal/apiserver/domain/assessmentmodel`、`container/modules/assessmentmodel` | 文档用业务短横线，代码包仍是 `assessmentmodel` |
| `evaluation` | `internal/apiserver/domain/evaluation`、`container/modules/evaluation` | 文档名与代码名一致 |
| `interpretation-model` | `internal/apiserver/domain/report`、`container/modules/report` | 文档升级为解释模型语言，代码包当前仍是 `report` |
| `actor` | `internal/apiserver/domain/actor`、`container/modules/actor` | 文档名与代码名一致 |
| `plan` | `internal/apiserver/domain/plan`、`container/modules/plan` | 文档名与代码名一致 |
| `statistics` | `internal/apiserver/domain/statistics`、`container/modules/statistics` | 文档目录保持复数，匹配代码包 |

## 4. 迁移后的旧概念位置

| 旧概念 | 现行位置 | 解释 |
| ------ | -------- | ---- |
| `scale` 独立核心模块 | `20-assessment-model` | Scale 是医学量表模型资产，不再是业务主轴 |
| `personalitymodel` 独立核心模块 | `20-assessment-model` | Personality Model 是模型资产类型或兼容注册名 |
| `report` 文档目录 | `40-interpretation-model` | 当前代码包仍是 `report`，文档语言改为解释模型 / 报告产出层 |
| `statistic` 单数目录 | 不采用 | 当前代码包和文档目录统一为 `statistics` |

---

## 5. 边界自检

改业务模块时先问：

1. 这个事实是作答事实、模型资产事实、执行事实、报告事实，还是统计投影。
2. 这个对象是否有自己的生命周期，还是只是另一个聚合的快照或引用。
3. 是否把执行结果、报告解释、统计指标混进了同一个模型。
4. 是否引入了上游模块对下游模块的反向依赖。
5. 是否需要同步 [03-领域事件总览.md](./03-领域事件总览.md) 和 `configs/events.yaml`。
