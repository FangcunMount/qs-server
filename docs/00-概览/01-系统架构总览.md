## 文档概述

本文档基于 **领域驱动设计（DDD）** 理念，系统阐述问卷&量表限界上下文（Bounded Context）在 V2 架构中的领域模型设计。作为领域层设计的权威参考，本文档：

- ✅ 明确定义 BC 内的子域划分与职责边界
- ✅ 详细描述各聚合根的核心职责与关键属性
- ✅ 规范子域间的依赖关系与数据流向
- ✅ 说明问卷模式与量表模式的差异与适用场景
- ✅ 明确 Assessment 作为桥接聚合的中心地位
- ✅ 阐明内容存储层（Mongo）与业务编排层（MySQL）的职责分离

### 🔥 V2 核心改进

相比 V1 设计，V2 版本在以下方面做了重大优化：

1. **内容与业务分离**：AnswerSheet（内容）与 Assessment（业务）彻底解耦，所有业务来源字段（OriginType/PlanID/ScreeningID）只存在于 Assessment
2. **Assessment 中心化**：明确 Assessment 作为统计分析锚点，所有报表/图表/趋势分析基于 MySQL 的 Assessment 表
3. **FillerRef 替代 Writer**：废弃 Writer/Viewer 概念，使用 FillerRef 值对象记录填写人信息
4. **幂等创建策略**：EnsureTestee 支持多上游 BC 安全调用，避免重复创建
5. **事件驱动优化**：AssessmentSubmittedEvent/AssessmentInterpretedEvent 驱动评估工作流，实现完全异步解耦

### 💡 核心设计原则

| 原则 | 说明 | 体现 |
| ----- | ------ | ----- |
| **职责分离** | 问卷（survey）专注"数据收集"，量表（scale）专注"专业解读" | survey 不做评分解释，scale 不管问卷展示 |
| **聚合自治** | 每个聚合独立封装业务不变性，通过领域事件松耦合协作 | Assessment 状态变更发布领域事件 |
| **依赖单向** | 严格控制子域依赖方向，确保领域层无循环依赖 | survey → scale → assessment |
| **模式灵活** | 同一套模型支持纯问卷收集与专业量表测评两种场景 | 通过 MedicalScaleID 是否绑定区分 |

## 1. 限界上下文与子域划分

### 1.1 BC 定位与范围

**问卷&量表 BC** 是负责 **"问卷数据收集"** 和 **"专业量表测评"** 的核心业务限界上下文。设计目标：

1. **通用问卷能力**：支持任意类型的问卷调查（客户满意度、用户反馈、市场调研等）
2. **专业量表能力**：支持符合医学/心理学标准的量表测评（SDS、SAS、Conners 等）
3. **灵活模式切换**：同一套领域模型支持"纯问卷"和"量表测评"两种模式
4. **可追溯性**：完整记录测评生命周期，支持趋势分析与统计报表

**BC 边界：**

- ✅ 在边界内：问卷结构定义、答卷收集与校验、量表规则配置、评分算法封装、测评案例管理、计划/筛查项目配置
- ❌ 在边界外：用户身份认证（IAM BC）、支付（Payment BC）、通知推送（Notification BC）、组织架构（IAM BC）

**上游 BC 与集成关系：**

本 BC 为多个上游业务 BC 提供测评能力，典型上游包括：

| 上游 BC | 典型场景 | 集成方式 | 数据流向 |
| --------- | --------- | --------- | ---------- |
| **互联网医院 BC** | 患者出院后随访测评 | gRPC 调用创建 AssessmentPlan | 互联网医院 → 问卷&量表 BC |
| **行为训练中心 BC** | 学员入校后周期评估 | gRPC 调用创建 AssessmentPlan | 训练中心 → 问卷&量表 BC |
| **健康管理 BC** | 慢病患者定期测评 | gRPC 调用创建 AssessmentPlan | 健康管理 → 问卷&量表 BC |
| **学校管理 BC** | 新生入学心理筛查 | gRPC 调用创建 ScreeningProject | 学校管理 → 问卷&量表 BC |

**下游订阅者：**

| 订阅方 | 订阅事件 | 用途 |
| ------- | --------- | ------ |
| 通知模块 | AssessmentInterpretedEvent | 推送测评完成通知给用户 |
| 上游 BC | AssessmentInterpretedEvent | 回写测评结果到业务系统 |
| 数据分析服务 | 所有 Assessment 事件 | 实时统计与 BI 报表 |

### 1.2 子域划分总览

问卷&量表 BC 按 **职责内聚、依赖单向** 原则，拆分为 6 个子域：

| 子域编号 | 中文名 | 英文名 | 核心关注点 | 核心聚合 | 依赖关系 |
| --------- | ------- | -------- | ----------- | --------- | --------- |
| 1 | 问卷子域 | survey | 数据收集与输入校验 | Questionnaire, AnswerSheet | 无依赖 |
| 2 | 量表子域 | scale | 专业解读与评分规则 | MedicalScale | 依赖 survey |
| 3 | 测评案例子域 | assessment | 测评生命周期管理 | Assessment, AssessmentScore, InterpretReport | 依赖 survey、scale、user |
| 4 | 用户子域 | user | BC内用户领域视图 | Testee, Staff | 无依赖 |
| 5 | 测评计划子域 | plan | 周期性测评管理 | AssessmentPlan, AssessmentTask | 无依赖 |
| 6 | 入校筛查子域 | screening | 批量筛查项目管理 | ScreeningProject | 无依赖 |

### 1.3 各子域详细说明

#### 📋 1.3.1 survey 子域（问卷收集）

**核心定位：** 通用问卷收集能力层，专注数据采集而非专业解读

**职责清单：**

- ✅ **问卷结构定义**：题目、选项、页面布局、逻辑跳转
- ✅ **答卷数据收集**：结构化存储用户填写内容
- ✅ **输入层校验**：必填、格式、范围、题间依赖
- ✅ **计分元数据配置**：为 scale 子域提供原始分计算基础（ScoringConfig）
- ❌ **不做专业分数解读**：不涉及分数的临床/心理学意义
- ❌ **不管理测评生命周期**：不关心测评的执行者、时间与上下文

**核心聚合：**

1. **`Questionnaire`（问卷模板）** - 聚合根
   - 字段：Code（唯一标识）、Title、Version、Questions、Pages、JumpLogic
   - 职责：封装问卷结构的完整性约束，支持版本演进

2. **`AnswerSheet`（答卷实例）** - 聚合根
   - 字段：AnswerSheetID、QuestionnaireCode、TesteeRef、FillerRef、AnswerItems、Status、SubmittedAt
   - 职责：记录一次填写行为的完整答案，管理答卷状态（draft/submitted）

**AnswerSheet 完整结构设计：**

```go
// 答卷状态
type AnswerSheetStatus string

const (
    AnswerSheetDraft     AnswerSheetStatus = "draft"     // 草稿（填写中）
    AnswerSheetSubmitted AnswerSheetStatus = "submitted" // 已提交
)

// 填写人类型
type FillerType string

const (
    FillerTypeSelf   FillerType = "self"   // 受试者本人
    FillerTypeParent FillerType = "parent" // 家长代填
    FillerTypeStaff  FillerType = "staff"  // 后台人员协助填写
)

// TesteeRef：受试者引用（值对象）
type TesteeRef struct {
    TesteeID   TesteeID  // 受试者 ID
    Name       string    // 姓名快照（便于报告展示）
    Grade      *string   // 年级快照
    AgeAtTest  *int      // 填写时的年龄（便于常模匹配）
}

// FillerRef：填写人引用（值对象）
type FillerRef struct {
    FillerType FillerType   // 填写人类型
    FillerID   *string      // 填写人 ID（IAM.UserID 或 Staff.StaffID），外部家长可为空
    FillerName *string      // 填写人姓名（外部家长需记录）
}

// AnswerSheet 聚合根
type AnswerSheet struct {
    ID              AnswerSheetID
    QuestionnaireCode QuestionnaireCode
    
    // 受试者与填写人信息
    TesteeRef       TesteeRef   // 受试者引用
    FillerRef       FillerRef   // 填写人引用
    
    // 答案内容
    AnswerItems     []AnswerItem    // 题目答案列表
    
    // 状态与时间
    Status          AnswerSheetStatus
    CreatedAt       time.Time
    SubmittedAt     *time.Time
    
    // 冗余字段（便于查询）
    OrgID           int64           // 组织 ID
    Version         int64           // 乐观锁版本号
}

// AnswerItem：单题答案（实体）
type AnswerItem struct {
    QuestionID   string      // 题目 ID
    QuestionText string      // 题目文本快照
    Value        interface{} // 答案值（单选/多选/填空）
    Score        *float64    // 题目得分（计分后填充）
}
```

**关键设计说明：**

| 设计点 | 说明 | 原因 |
| ------- | ------ | ------ |
| **TesteeRef vs FillerRef** | 受试者≠填写人，分开记录 | 家长代填儿童量表、医生协助患者填写 |
| **FillerType 枚举** | 区分自填、家长代填、后台协助 | 支持统计"代填率"、审计"谁填的" |
| **FillerID 可为空** | 外部家长无 IAM 用户 ID | 入校筛查场景，扫码家长无需注册 |
| **TesteeRef 快照** | 记录填写时的年龄、年级 | 常模匹配需要填写时的年龄，而非当前年龄 |
| **AnswerItem 快照** | 记录题目文本 | 问卷版本演进后，历史答卷仍可正确展示 |
| **不存储业务来源** | 不包含 Scenario、PlanID、ScreeningID | 业务来源由 Assessment 管理，保持 AnswerSheet 纯粹 |

**存储策略：**

- **存储引擎**：MongoDB
- **原因**：答案结构不固定（单选/多选/填空/矩阵题），文档型数据库更灵活
- **索引设计**：
  - `{testee_ref.testee_id: 1, created_at: -1}` - 查询某人的答卷历史
  - `{questionnaire_id: 1, status: 1}` - 统计问卷提交率
  - `{org_id: 1, created_at: -1}` - 按组织查询答卷

**适用场景：**

| 场景类型 | 典型用例 | 是否需要量表 |
| --------- | --------- | ------------ |
| 纯问卷模式 | 客户满意度调查、用户反馈收集、市场调研 | ❌ |
| 量表模式 | 心理量表测评、行为评定、认知评估 | ✅ |

**数据存储策略：**

- Questionnaire：MongoDB（文档型，支持复杂结构与版本演进）
- AnswerSheet：MongoDB（文档型，答案结构不固定）

#### 🧬 1.3.2 scale 子域（量表解读）

**核心定位：** 专业量表的规则库与算法封装

**职责清单：**

- ✅ **量表结构定义**：因子、维度、测量指标
- ✅ **计分规则配置**：题目权重、反向计分、因子聚合
- ✅ **常模与风险区间管理**：定义正常/边缘/异常分数界限
- ✅ **解读规则与结论模板**：不同分数区间对应的文案与建议
- ✅ **评估算法封装**：通过策略模式 + 职责链实现（详见 12-02）
- ❌ **不存储具体测评记录**：不记录"谁在何时完成了哪个量表"
- ❌ **不管理测评执行流程**：不关心测评的发起、查询与结果推送

**核心聚合：**

1. **`MedicalScale`（量表定义）** - 聚合根
   - 字段：MedicalScaleID、Name、Version、Factors、InterpretationRules、NormTable
   - 职责：封装量表的完整语义，确保评分规则与解读逻辑的一致性

**内部实体/值对象：**

- `Factor`（因子/维度）：实体，包含 FactorCode、Name、RelatedQuestions
- `InterpretationRule`（解读规则）：值对象，定义分数区间 → 风险等级 → 结论文案的映射
- `NormTable`（常模表）：值对象，支持按年龄/性别/地区的分层常模

**典型量表示例：**

| 量表类型 | 典型量表 | 因子示例 | 应用场景 |
| --------- | --------- | --------- | --------- |
| 心理量表 | SDS、SAS、SCL-90 | 抑郁、焦虑、人际敏感 | 情绪状态评估 |
| 行为量表 | Conners、CBCL | 多动、对立违抗、社交问题 | 儿童行为问题筛查 |
| 认知量表 | MMSE、MoCA | 定向力、记忆、语言 | 认知功能评估 |

**关键设计点：**

- 量表定义与问卷定义分离：同一份问卷可对应多个量表（如 SCL-90 可拆分为多个维度量表）
- 支持多版本管理：量表常模与解读规则可能因研究进展而更新
- 算法可插拔：通过 Evaluator 接口实现不同量表的评分策略

**数据存储策略：**

- MedicalScale：MySQL（结构化，频繁查询）
- Factors、InterpretationRules：JSON 字段（复杂结构，查询较少）

#### 🎯 1.3.3 assessment 子域（测评案例）

**核心定位：** 一次测评行为的完整生命周期管理，作为桥接聚合连接各子域

**职责清单：**

- ✅ **记录完整测评元信息**：测评对象（Testee）、时间（CreatedAt）、内容（Questionnaire + MedicalScale）、来源（Plan/Screening/Adhoc）
- ✅ **管理测评状态机**：pending → submitted → interpreted → failed
- ✅ **桥接各子域聚合**：通过外键关联 Testee、AnswerSheet、MedicalScale、Plan/Screening
- ✅ **作为统计分析锚点**：支持按人、按时间、按项目聚合查询
- ✅ **发布领域事件**：AssessmentSubmittedEvent、AssessmentInterpretedEvent
- ❌ **不负责具体评分算法**：委托给 scale.Evaluator
- ❌ **不负责数据收集**：委托给 survey.AnswerSheet

**核心聚合：**

1. **`Assessment`（测评案例）** - 聚合根
   - 存储：MySQL
   - 字段：
     - 基础信息：AssessmentID、TesteeID、QuestionnaireCode、AnswerSheetID、MedicalScaleID
     - 来源信息：OriginType (adhoc/plan/screening)、OriginID
     - 状态信息：Status、TotalScore、RiskLevel
     - 时间信息：CreatedAt、SubmittedAt、InterpretedAt
   - 职责：协调各子域聚合完成测评全流程，发布领域事件

**Assessment 完整结构设计：**

```go
// 测评来源类型
type AssessmentOriginType string

const (
    OriginAdhoc      AssessmentOriginType = "adhoc"      // 临时测评（医生开单/用户自助）
    OriginPlan       AssessmentOriginType = "plan"       // 来自测评计划
    OriginScreening  AssessmentOriginType = "screening"  // 来自筛查项目
)

// 测评状态
type AssessmentStatus string

const (
    AssessmentPending    AssessmentStatus = "pending"     // 已创建，等待填写
    AssessmentSubmitted  AssessmentStatus = "submitted"   // 已提交答卷，等待解读
    AssessmentInterpreted AssessmentStatus = "interpreted"// 已完成解读
    AssessmentFailed     AssessmentStatus = "failed"      // 解读失败
    AssessmentCanceled   AssessmentStatus = "canceled"    // 已取消
)

// Assessment 聚合根
type Assessment struct {
    ID                AssessmentID
    
    // 基础关联信息
    TesteeID          TesteeID
    OrgID             int64
    QuestionnaireCode Code                // 使用 Code 作为唯一标识
    MedicalScaleID    *MedicalScaleID     // 量表模式时有值
    AnswerSheetID     *AnswerSheetID      // 提交后有值
    
    // 来源信息（核心：区分业务场景）
    OriginType     AssessmentOriginType // adhoc / plan / screening
    PlanID         *PlanID              // OriginType = plan 时有值
    ScreeningID    *ScreeningProjectID  // OriginType = screening 时有值
    
    // 状态与结果
    Status         AssessmentStatus
    TotalScore     *float64             // 总分（解读后有值）
    RiskLevel      *string              // 风险等级（解读后有值）
    
    // 时间信息
    CreatedAt      time.Time
    CreatedBy      *int64               // Staff.StaffID 或特殊值（自助测评）
    SubmittedAt    *time.Time
    InterpretedAt  *time.Time
    
    // 版本与软删除
    DeletedAt      *time.Time
    Version        int64
}
```

**OriginType 使用场景说明：**

| OriginType | 典型场景 | OriginID 指向 | 统计维度 |
| ----------- | --------- | -------------- | --------- |
| **adhoc** | 医生临时开单、用户自助测评 | 无 | 按人、按时间统计 |
| **plan** | 互联网医院随访、训练中心周期评估 | AssessmentPlan.ID | 计划完成率、趋势分析 |
| **screening** | 学校入学筛查、社区健康普查 | ScreeningProject.ID | 项目覆盖率、风险分布 |

**关键设计说明：**

| 设计点 | 说明 | 原因 |
| ------- | ------ | ------ |
| **OriginType 必填** | 所有测评必须标识来源 | 支持统计分析、追溯测评上下文 |
| **PlanID/ScreeningID 互斥** | 根据 OriginType 决定哪个字段有值 | 避免数据冗余与不一致 |
| **MedicalScaleID 可为空** | 纯问卷模式时为空 | 支持问卷模式与量表模式切换 |
| **AnswerSheetID 可为空** | 创建时为空，提交后填充 | 支持"先创建测评入口，后填写"流程 |
| **RiskLevel 可为空** | 量表模式解读后有值 | 问卷模式无风险评估 |

**状态机流转：**

```text
pending ──Submit()──> submitted ──Interpret()──> interpreted
   ↓                      ↓                           ↓
   └──────────────────────────── Fail() ─────────> failed
                                    ↓
                                canceled
```

**业务来源与统计示例：**

```sql
-- 查询某个计划的完成率
SELECT 
    COUNT(CASE WHEN status = 'interpreted' THEN 1 END) AS completed_count,
    COUNT(*) AS total_count
FROM assessments
WHERE origin_type = 'plan' AND plan_id = ?;

-- 查询某个筛查项目的风险分布
SELECT risk_level, COUNT(*) AS count
FROM assessments
WHERE origin_type = 'screening' AND screening_id = ?
GROUP BY risk_level;

-- 查询某人的所有测评记录（含来源）
SELECT 
    a.id, a.origin_type, a.status, a.total_score,
    p.name AS plan_name,
    sp.name AS screening_name
FROM assessments a
LEFT JOIN assessment_plans p ON a.plan_id = p.id
LEFT JOIN screening_projects sp ON a.screening_id = sp.id
WHERE a.testee_id = ?
ORDER BY a.created_at DESC;
```

2. **`AssessmentScore`（因子得分）** - 实体
   - 存储：MySQL
   - 字段：AssessmentID、FactorCode、RawScore、StandardScore、RiskLevel
   - 职责：便于维度趋势分析（"张三的焦虑因子得分在 3 个月内的变化"）

3. **`InterpretReport`（解读报告）** - 聚合根
   - 存储：MongoDB
   - 字段：ReportID、AssessmentID、TotalScore、RiskLevel、FactorScores、Conclusion、Suggestions
   - 职责：面向用户的最终报告，支持富文本、图表、历史对比

**关键设计点：**

| 设计要点 | 说明 | 实现方式 |
| --------- | ------ | --------- |
| **桥接模式** | 通过外键关联其他聚合，避免聚合间强耦合 | TesteeID, QuestionnaireCode, AnswerSheetID, MedicalScaleID |
| **状态机** | 明确的生命周期状态转换 | Status 字段 + Submit()、Interpret()、Fail() 方法 |
| **来源标识** | 区分不同业务场景的测评 | OriginType (adhoc/plan/screening) + OriginID |
| **事件驱动** | 状态变更发布领域事件 | PublishEvent(AssessmentSubmittedEvent) |
| **读写分离** | Assessment 写入 MySQL，InterpretReport 查询 MongoDB | 提升查询性能与灵活性 |

**状态机流转：**

```text
pending ──Submit()──> submitted ──Evaluate()──> interpreted
   ↓                      ↓                          ↓
   └──────────────────────────── Fail() ─────────> failed
```

**数据流示例（SDS 量表测评）：**

1. 用户扫码进入测评 → 创建 Assessment（status=pending）
2. 用户填写问卷 → 创建 AnswerSheet（status=draft）
3. 用户提交 → AnswerSheet.Submit() + Assessment.Submit() → 发布 AssessmentSubmittedEvent
4. qs-worker 消费事件 → 调用 scale.Evaluator.Evaluate() → 生成 AssessmentScore 与 InterpretReport
5. 更新 Assessment.Status=interpreted → 发布 AssessmentInterpretedEvent
6. 前端轮询或接收推送 → 查询 InterpretReport 展示结果

#### 👥 1.3.4 user 子域（BC内用户视图）

**核心定位：** 问卷&量表 BC 对用户的领域特定视图，与 IAM BC 解耦

**职责清单：**

- ✅ **维护测评相关的用户属性**：年级、标签、组织、角色等
- ✅ **通过 ID 映射与 IAM BC 关联**：IAMUserID、IAMChildID
- ✅ **作为统计分析的"人"维度入口**："张三的所有测评记录"
- ✅ **支持按组织/标签/年级等维度查询**："一年级所有学生的测评完成率"
- ❌ **不负责认证授权**：由 IAM BC 的 AuthN/AuthZ 模块负责
- ❌ **不持有敏感信息**：不存储手机号、密码、身份证号等

**核心聚合：**

1. **`Testee`（受试者）** - 聚合根
   - 字段：TesteeID、IAMUserID、IAMChildID、OrgID、Grade、Tags、CreatedAt
   - 职责：记录"被测人"的领域属性，作为 Assessment 的外键

2. **`Staff`（后台人员）** - 聚合根
   - 字段：StaffID、IAMUserID、OrgID、Role、CreatedAt
   - 职责：记录"执行测评/查看报告"的后台人员信息

**与 IAM BC 的关系：**

```text
IAM BC (统一身份认证)       问卷&量表 BC (业务领域视图)
┌──────────────────┐        ┌──────────────────────┐
│ User             │ ◄──────┤ Testee               │
│ - UserID         │  映射  │ - TesteeID           │
│ - Phone          │        │ - IAMUserID (FK)     │
│ - Password       │        │ - Grade, Tags        │
└──────────────────┘        └──────────────────────┘
        ↓                              ↓
┌──────────────────┐        ┌──────────────────────┐
│ Child            │ ◄──────┤ Testee               │
│ - ChildID        │  映射  │ - TesteeID           │
│ - ParentUserID   │        │ - IAMChildID (FK)    │
│ - Birthdate      │        │ - Grade, Tags        │
└──────────────────┘        └──────────────────────┘
```

**关键设计点：**

- **延迟创建**：Testee 在用户首次参与测评时创建，而非在 IAM 注册时创建
- **ID 映射**：通过 IAMUserID/IAMChildID 关联，避免跨 BC 聚合引用
- **属性冗余**：OrgID、Grade 等属性在本 BC 维护，以减少跨 BC 查询
- **角色分离**：Staff 与 Testee 是不同聚合，角色职责清晰

**数据存储策略：**

- Testee、Staff：MySQL（结构化，频繁查询）

## 2. 子域依赖关系

为避免循环依赖，严格约束如下：

### 2.1 依赖规则

| 子域 | 依赖项 | 说明 |
| ----- | ------- | ------ |
| `survey` | 无 | 基础能力层，不依赖其他子域 |
| `scale` | `survey` | 需要读取 Questionnaire 的 ScoringConfig |
| `assessment` | `survey`、`scale`、`user`、`plan`、`screening` | 作为桥接聚合，协调各子域 |
| `user` | 无 | 独立的用户视图 |
| `plan` | 无 | 独立的计划管理 |
| `screening` | 无 | 独立的筛查项目管理 |

### 2.2 依赖图

```text
               common (值对象、领域事件)
                  ↑
    ┌─────────────┼─────────────┐
    │             │             │
  survey        user      plan  screening
    ↑             ↑         ↑       ↑
    │             └────┬────┴───────┘
    │                  │
  scale                │
    ↑                  │
    └────────assessment┘
```

### 2.3 依赖约束

1. **严禁循环依赖**：任何子域不得反向依赖其下游子域
2. **只读依赖**：下游子域只能读取上游聚合的 ID 和值对象，不得修改上游状态
3. **事件解耦**：跨子域的状态同步通过领域事件实现，避免直接调用

**示例：**

- ✅ 正确：assessment 读取 Testee.TesteeID、MedicalScale.MedicalScaleID
- ❌ 错误：assessment 调用 Testee.UpdateGrade() 修改用户年级
- ✅ 正确：assessment 发布 AssessmentInterpretedEvent，由通知模块订阅后推送消息
- ❌ 错误：assessment 直接调用通知模块的 SendSMS() 方法

## 3. 典型业务流程与数据流

### 3.1 流程一：纯问卷调查（Survey Only）

**场景：** 客户满意度调查，不涉及专业量表评分

```text
1. 管理员创建 Questionnaire（问卷模板）
2. 用户扫码进入 → 创建 AnswerSheet（status=draft）
3. 用户填写答案 → AnswerSheet.AddAnswerItem()
4. 用户提交 → AnswerSheet.Submit() + 校验通过
5. （可选）创建 Assessment 记录（不绑定 MedicalScale）
6. 管理员导出答卷数据进行统计分析
```

**关键点：**

- 不需要 MedicalScale
- 不需要触发评估工作流
- 可选是否创建 Assessment（如需追踪用户填写历史）

### 3.2 流程二：专业量表测评（Survey + Scale + Assessment）

**场景：** 患者扫码填写 SDS（抑郁自评量表）

```text
1. 管理员配置 MedicalScale（SDS 量表定义）
2. 管理员创建 Questionnaire 并绑定 MedicalScale
3. 用户扫码进入 → 创建 Assessment（status=pending, OriginType=adhoc）
4. 用户填写问卷 → 创建 AnswerSheet（status=draft）
5. 用户提交 → AnswerSheet.Submit() + Assessment.Submit()
6. 发布 AssessmentSubmittedEvent → MQ
7. qs-worker 消费事件：
   a. 读取 AnswerSheet 答案
   b. 调用 scale.Evaluator.Evaluate(MedicalScale, AnswerSheet)
   c. 生成 AssessmentScore（各因子得分）
   d. 生成 InterpretReport（解读报告）
   e. 更新 Assessment（status=interpreted, TotalScore, RiskLevel）
8. 发布 AssessmentInterpretedEvent
9. 前端轮询或接收推送 → 查询 InterpretReport 展示结果
```

**关键点：**

- 必须绑定 MedicalScale
- 必须创建 Assessment
- 提交后触发异步评估工作流
- 通过领域事件驱动各环节

### 3.3 流程三：测评计划（Plan-driven Assessment）

**场景：** 互联网医院为术后患者制定随访计划，出院后第 2、4、8、12 周各做一次量表测评

```text
1. 医生创建 AssessmentPlan：

3. 到期触发（Task1 到期）：
   - 创建 Assessment（OriginType=plan, OriginID=PlanID, status=pending）
   - 发送提醒通知（由通知模块负责）
4. 患者点击通知 → 进入填写流程（同流程二）
5. 提交后更新 AssessmentTask.Status=completed
```

**关键点：**

- 计划驱动，自动生成任务
- 通过 OriginType=plan 标识来源
- 支持趋势分析（"张三 4 次测评的抑郁分数变化"）

## 4. 问卷模式 vs 量表模式

### 4.1 模式对比

| 对比维度 | 问卷模式（Survey Only） | 量表模式（Survey + Scale） |
| --------- | ---------------------- | -------------------------- |
| **核心目的** | 数据收集 | 专业评估 |
| **MedicalScale** | ❌ 不绑定 | ✅ 必须绑定 |
| **Assessment** | 可选创建 | 必须创建 |
| **评估工作流** | ❌ 不触发 | ✅ 触发 |
| **InterpretReport** | ❌ 不生成 | ✅ 生成 |
| **AssessmentScore** | ❌ 不记录 | ✅ 记录（便于趋势分析） |
| **适用场景** | 客户满意度、市场调研、用户反馈 | 心理测评、行为评估、认知评估 |
| **数据价值** | 统计分析（频次、分布） | 临床诊断、干预决策、效果评估 |

### 4.2 技术实现差异

#### 问卷模式

```go
// 1. 创建问卷
questionnaire := survey.NewQuestionnaire(title, questions)

// 2. 用户填写
answerSheet := survey.NewAnswerSheet(questionnaire.ID)
answerSheet.AddAnswerItem(questionID, value)
answerSheet.Submit() // 仅校验输入合法性

// 3. 导出数据分析（不生成报告）
```

#### 量表模式

```go
// 1. 配置量表
medicalScale := scale.NewMedicalScale(name, factors, interpretRules)

// 2. 创建问卷并绑定量表
questionnaire := survey.NewQuestionnaire(title, questions)
questionnaire.BindScale(medicalScale.ID)

// 3. 创建测评
assessment := assessment.NewAssessment(testeeID, questionnaire.ID, medicalScale.ID)

// 4. 用户填写
answerSheet := survey.NewAnswerSheet(questionnaire.ID)
answerSheet.AddAnswerItem(questionID, value)
answerSheet.Submit()

// 5. 提交测评
assessment.Submit(answerSheet.ID)
eventBus.Publish(AssessmentSubmittedEvent{AssessmentID: assessment.ID})

// 6. 异步评估（qs-worker）
evaluator := scale.NewEvaluator(medicalScale)
scores := evaluator.Evaluate(answerSheet)
report := assessment.GenerateReport(scores, medicalScale.InterpretRules)
assessment.Interpret(report, scores)
eventBus.Publish(AssessmentInterpretedEvent{AssessmentID: assessment.ID})
```

### 4.3 模式切换

**同一份问卷可同时支持两种模式：**

- 问卷模式：MedicalScaleID = NULL，不创建 Assessment
- 量表模式：MedicalScaleID ≠ NULL，创建 Assessment 并触发评估

**示例：**

- 问卷 A（客户满意度调查）→ 仅问卷模式
- 问卷 B（SDS 自评量表）→ 仅量表模式
- 问卷 C（用户基本信息 + 心理评估）→ 根据 MedicalScaleID 是否绑定，灵活切换

## 5. 跨 BC 集成设计

### 5.1 与 IAM BC 的关系

#### 5.1.1 职责划分

| BC | 职责 | 核心聚合 |
| --- | ------ | --------- |
| **IAM BC** | 统一身份认证、授权、组织管理 | User, Child, Org, Role, Permission |
| **问卷&量表 BC** | 业务领域视图、测评相关属性 | Testee, Staff |

#### 5.1.2 集成方式

***ID 映射***

```go
// IAM BC
type User struct {
    UserID   string
    Phone    string
    Password string
}

type Child struct {
    ChildID      string
    ParentUserID string
    Name         string
    Birthdate    time.Time
}

// 问卷&量表 BC
type Testee struct {
    TesteeID    string       // 本 BC 的主键
    IAMUserID   *string      // 可选：成人用户
    IAMChildID  *string      // 可选：儿童用户
    OrgID       string       // 组织 ID（冗余自 IAM）
    Grade       string       // 年级（本 BC 特有）
    Tags        []string     // 标签（本 BC 特有）
}
```

***认证流程***

```text
1. 用户登录 → IAM BC 验证用户名密码
2. 颁发 JWT Token（包含 UserID、OrgID、Role）
3. 用户访问问卷&量表 BC 接口 → 携带 Token
4. 问卷&量表 BC 验证 Token（调用 IAM BC 或本地解析）
5. 根据 IAM.UserID 查询 Testee 表 → 获取 TesteeID
6. 执行业务逻辑（创建 Assessment、查询报告等）
```

***授权流程***

```text
1. 用户请求查看某报告 → ReportID
2. 查询 InterpretReport → 获取 AssessmentID
3. 查询 Assessment → 获取 TesteeID
4. 校验：Testee.IAMUserID == 当前用户的 UserID？
5. 通过 → 返回报告；否则 → 403 Forbidden
```

#### 5.1.3 数据同步策略

| 场景 | 策略 | 说明 |
| ----- | ------ | ------ |
| **创建 Testee** | 延迟创建 | 首次参与测评时创建，而非 IAM 注册时创建 |
| **同步 OrgID** | 冗余存储 | Testee 表冗余 OrgID，减少跨 BC 查询 |
| **同步 User 信息** | 事件驱动 | IAM BC 发布 UserUpdatedEvent，问卷&量表 BC 订阅并更新 Testee |
| **删除 User** | 逻辑删除 | Testee 不物理删除，通过 Status=deleted 标记 |

---

### 5.2 与上游 BC 的关系

问卷&量表 BC 作为**能力提供方**，为多个上游 BC 提供测评服务。

#### 5.2.1 上游 BC 清单

| 上游 BC | 典型场景 | 集成方式 | 来源标识 |
| --------- | --------- | --------- | --------- |
| **互联网医院 BC** | 患者随访、术后评估 | gRPC / REST API | OriginType=plan + PlanID |
| **行为训练中心 BC** | 学员训练效果评估 | gRPC / REST API | OriginType=plan + PlanID |
| **健康管理 BC** | 慢病管理、定期筛查 | gRPC / REST API | OriginType=plan + PlanID |
| **学校管理 BC** | 入学筛查、期中评估 | gRPC / REST API | OriginType=screening + ScreeningID |

#### 5.2.2 典型集成流程

***场景：互联网医院为患者创建随访计划***

```text
┌──────────────────┐                          ┌──────────────────┐
│ 互联网医院 BC     │                          │ 问卷&量表 BC      │
│ (Hospital BC)    │                          │ (QS BC)          │
```

```go
    // 1. 根据 IAM ID 查询
    testee, err := svc.testeeRepo.FindByIAMID(ctx, req.IAMUserID, req.IAMChildID)
    if err == nil {
        // 已存在，直接返回
        return testee.ID, nil
    }

    testee = user.NewTestee(
        iamUserID:  req.IAMUserID,
        iamChildID: req.IAMChildID,
        orgID:      req.OrgID,
        grade:      req.Grade,
        tags:       req.Tags,
        source:     req.Source,  // "hospital" | "training_center" | "screening"
    )
    
    // 3. 保存（数据库唯一约束：iam_user_id + iam_child_id + org_id）
    err = svc.testeeRepo.Save(ctx, testee)
    if errors.Is(err, ErrDuplicateKey) {
        // 并发创建，重新查询
        testee, _ = svc.testeeRepo.FindByIAMID(ctx, req.IAMUserID, req.IAMChildID)
    }
    
    return testee.ID, nil
}
```

## 8. AnswerSheet 与 Assessment 的职责边界

### 8.1 为什么分离？

***架构原则：内容存储层 vs 业务编排层***

- **AnswerSheet（内容存储层）**：只关心"答了什么题、答案是什么"，是纯粹的数据容器
- **Assessment（业务编排层）**：关心"这次测评从哪来、为什么做、做给谁看"，是业务上下文的载体

**反模式警告：**

❌ **不要在 AnswerSheet 上堆业务字段**：Scenario、PlanID、ScreeningID 等业务来源字段会污染内容存储层

❌ **不要让 AnswerSheet 承担统计职责**：按项目统计、按计划分析应通过 Assessment 维度进行

✅ **正确做法**：业务来源字段全部由 Assessment 管理，AnswerSheet 通过 AssessmentID 关联

### 8.2 职责对比

| 维度 | AnswerSheet（Mongo） | Assessment（MySQL） |
| ----- | --------------------- | ------------------- |
| **核心职责** | 存储答题内容 | 管理测评生命周期 |
| **存储内容** | 题目快照 + 答案 + 得分 | 来源 + 状态 + 关联 + 结果 |
| **业务来源** | ❌ 不关心 | ✅ OriginType + PlanID/ScreeningID |
| **状态管理** | draft/submitted（填写状态） | pending/submitted/interpreted（业务状态） |
| **统计维度** | ❌ 不参与 | ✅ 主入口（按人/按项目/按时间） |
| **查询场景** | 导出答卷、查看历史答案 | 测评列表、完成率、风险分布 |
| **数据结构** | 嵌套文档（灵活） | 关系表（规范） |

### 8.3 关联关系

```text
Assessment (MySQL)
   ↓ answer_sheet_id
AnswerSheet (MongoDB)

关系：1:0..1
- Assessment 创建时：answer_sheet_id = NULL（等待填写）
- 用户提交后：answer_sheet_id = {Mongo ObjectID}
```

**关键约束：**

1. **单向依赖**：Assessment → AnswerSheet，反向不依赖
2. **延迟关联**：AnswerSheet 在提交时才关联到 Assessment
3. **幂等创建**：同一个 Assessment 只能关联一个 AnswerSheet

### 8.4 典型场景示例

#### 场景一：随访计划触发测评

```go

    testeeID:    testeeID,

    originType:  assessment.OriginPlan,   // 来自计划
    planID:      planID,                  // 关联计划 ID

WHERE origin_type = 'plan' AND plan_id = ? AND status = 'interpreted';
```

## 9. 总结与后续文档

### 9.1 核心设计亮点

1. **职责分离**：survey（数据收集）、scale（专业解读）、assessment（生命周期管理）三层分离，各司其职
2. **聚合自治**：每个聚合独立封装业务不变性，通过领域事件松耦合协作
3. **依赖单向**：严格控制子域依赖方向，避免循环依赖
4. **模式灵活**：同一套模型支持问卷模式与量表模式，降低系统复杂度
5. **桥接模式**：Assessment 作为桥接聚合，连接各子域，作为统计分析的锚点
6. **事件驱动**：通过领域事件驱动异步评估、消息推送、统计更新，提升系统解耦与性能
7. **内容与业务分离**：AnswerSheet 存储内容，Assessment 管理业务，存储与统计各取所长

### 10.2 废弃术语警告

**❌ 以下术语不应出现在代码与文档中：**

| 废弃术语 | 原因 | 替代方案 |
| --------- | ------ | --------- |
| **Writer** | 含义模糊，不明确是"创建问卷的人"还是"填写问卷的人" | 使用 **Filler**（填写人）或 **FillerRef**（填写人引用） |
| **Viewer** | 不是实体，是访问控制的运行时概念 | 通过 **CanViewAnswerSheet(Principal, AnswerSheet)** 等策略函数判断 |
| **User**（无前缀） | 不明确指代 IAM.User 还是 Testee 还是 Staff | 始终使用完整前缀：**IAM.User**、**Testee**、**Staff** |
| **答题人** | 与"受试者"和"填写人"混淆 | 使用 **Testee**（受试者）+ **Filler**（填写人） |

| 检查项 | 规范 | 示例 |
| ------- | ------ | ------ |
| **Testee vs User** | 本 BC 统一使用 Testee，涉及 IAM 时用 IAM.User | ✅ Testee.IAMUserID；❌ User.ID |
| **Filler vs Writer** | 统一使用 Filler/FillerRef | ✅ FillerRef.FillerType；❌ Writer.Role |
| **Assessment vs Evaluation** | Assessment 是测评案例，Evaluation 是评估算法 | ✅ Assessment.Status；✅ Evaluator.Evaluate() |
| **Plan vs AssessmentPlan** | 涉及测评计划时使用完整名称 | ✅ AssessmentPlan.PlanID；❌ Plan.ID（除非上下文明确） |
| **Screening vs Project** | 筛查项目统一使用 ScreeningProject | ✅ ScreeningProject.ProjectID；❌ Screening.ID |

### 10.5 领域对象关系术语

| 关系类型 | 术语 | 说明 | 示例 |
| --------- | ------ | ------ | ------ |
| **聚合根** | Aggregate Root | 拥有全局唯一 ID，作为聚合边界的入口 | Questionnaire, AnswerSheet, Assessment |
| **实体** | Entity | 有 ID 但不是聚合根，从属于聚合根 | AssessmentScore, AnswerItem |
| **值对象** | Value Object | 无 ID，通过属性相等性判断，不可变 | TesteeRef, FillerRef, Principal |
| **领域服务** | Domain Service | 跨聚合的领域逻辑，无状态 | Evaluator（评估算法） |
| **应用服务** | Application Service | 编排用例，协调聚合与基础设施 | TesteeAppService, PlanAppService |
| **仓储** | Repository | 聚合的持久化抽象接口 | TesteeRepository, AssessmentRepository |
| **领域事件** | Domain Event | 聚合状态变更的通知 | AssessmentSubmittedEvent |
