# 优雅关闭与资源释放

## 1. 结论

关闭顺序应先停止新流量和新任务，再等待在途工作，最后释放 transport、consumer、后台 runner、数据库和缓存资源。

## 2. 检查项

- HTTP/gRPC 是否停止接收新请求；
- MQ consumer 是否停止拉取并处理在途消息；
- scheduler/relay/reconciler 是否可取消；
- queue、waiter、pubsub 订阅是否退出；
- DB、Redis、Mongo 连接是否只关闭一次；
- Cleanup 顺序是否与依赖方向相反。

## 3. 证据

从各进程应用生命周期、container Cleanup 和模块 `Cleanup()` 追踪，不以接口存在就假定已正确接入。
