# 运行时

本目录回答的不是“某个领域对象是什么”，而是“代码被装配成哪些进程、请求和事件怎样穿过这些进程、进程如何启动与停止”。

qs-server 是一个以模块化单体为业务核心的三进程事件驱动系统：`qs-apiserver` 持有业务规则和业务事实，`collection-server` 承接小程序 BFF 与入口保护，`qs-worker` 消费事件并驱动异步用例。三个可执行进程不等于三个自治微服务。

## 阅读路线

第一次了解运行时，建议按下列顺序阅读：

1. 先读[三进程协作总览](./00-三进程协作总览.md)，建立进程职责和调用方向；
2. 再分别阅读 apiserver、collection 和 worker 的启动与执行模型；
3. 结合 [进程间调用与 gRPC](./04-进程间调用与gRPC.md)理解同步边界；
4. 结合 [IAM 认证与身份链路](./05-IAM认证与身份链路.md)理解身份如何进入业务；
5. 最后阅读后台任务与关闭流程，理解系统如何持续运行和安全退出。

## 运行时事实的优先级

运行时文档很容易随装配和配置变化而过期。发生冲突时按以下顺序判断：

1. `cmd/*` 与 `internal/*/process`：进程入口、启动阶段和关闭回调；
2. `internal/*/container`、`integration`、`transport`：依赖装配和对外能力；
3. `configs/*.yaml` 与 options：当前环境开关和参数；
4. `api/grpc/proto`、`api/rest`、`configs/events.yaml`：跨边界契约；
5. 本目录文档：对上述事实的解释，而不是替代事实源。

配置中的数值只是某个版本、某个环境的运行参数。例如 Submit Gate 的等待时间、gRPC 最大在途数和 worker 并发度都可以调整，不应被描述成永远不变的架构契约。

## 本目录不讲什么

- 领域模型、业务规则和数据库模型进入 `docs/02-业务模块`；
- Redis、MongoDB、MySQL、NSQ 的技术实现进入 `docs/03-基础设施`；
- 接口字段、部署和故障处理进入 `docs/04-接口与运维`；
- 关键业务链路、性能或可靠性深挖进入 `docs/05-专题分析`。

运行时文档只保留理解进程协作所必需的业务语义，并通过源码路径把读者引向更深的专题。

## 维护约定

出现下列变化时，应同步检查本目录：

- 新增或删除 `cmd/` 下的可执行进程；
- 调整 process stage、容器装配、transport 或关闭顺序；
- 新增 gRPC service、事件 consumer、scheduler 或后台 watcher；
- 修改 `202 Accepted`、ACK/NACK、幂等或重试语义；
- 修改 IAM、mTLS、组织范围解析或服务间认证方式。
