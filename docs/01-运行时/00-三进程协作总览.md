# 三进程协作总览

## 1. 结论

`collection-server` 承接前台压力，`qs-apiserver` 持有业务事实，`qs-worker` 只驱动异步用例。两个边界进程都不应绕过 apiserver 直接推进业务状态。

## 2. 调用方向

```text
Client -> collection REST -> apiserver gRPC
Admin  -> apiserver REST
apiserver -> Outbox/MQ -> worker -> apiserver internal gRPC
apiserver/worker -> Redis signal -> collection waiter
```

可靠业务事件与一次性信令是两套语义：前者可重放，后者丢失可由查询或 TTL 兜底。

## 3. 证据

入口在 `cmd/`；事件与信令在 `configs/events.yaml`、`configs/signals.yaml`；worker 处理器在 `internal/worker/handlers`。
