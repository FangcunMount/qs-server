# 项目一句话定位

**本文回答**：`qs-server` 对外应该怎么用一句话讲清楚；在技术分享、项目介绍、简历面试、追问答辩中，如何用不同长度的版本表达同一个项目；哪些说法是准确的，哪些说法容易把项目讲低、讲偏或讲虚。

---

## 1. 最推荐的一句话

> **qs-server 是一个面向心理/医学测评场景的后端系统，通过 Survey、Scale、Evaluation 的领域拆分，以及 collection-server、apiserver、worker 的三进程协作，实现“前台可靠提交答卷、后台异步评估生成报告、读侧统计与安全治理可持续演进”的测评平台。**

这句话里有四个关键词：

| 关键词 | 作用 |
| ------ | ---- |
| 心理/医学测评场景 | 说明业务不是普通表单，而是有专业规则的测评 |
| Survey / Scale / Evaluation | 说明领域拆分，不是 CRUD |
| collection / apiserver / worker | 说明运行时协作，不是单接口系统 |
| 可靠提交 / 异步评估 / 读侧统计 / 安全治理 | 说明工程难点，不是只会写接口 |

## 3. 10 秒版本

> **qs-server 不是普通问卷系统，而是一个测评后端：前台提交答卷，后台按量表规则异步评估，最后生成风险解读报告和统计数据。**

```text
不是问卷 CRUD

```

## 4. 30 秒版本

> **qs-server 是一个面向心理/医学测评场景的 Go 后端系统。它把业务拆成 Survey、Scale、Evaluation 等边界：Survey 负责问卷和答卷，Scale 负责量表规则，Evaluation 负责测评状态、计分、风险和报告。运行时上，前台请求先进入 collection-server 做认证、限流、提交削峰和幂等保护，再通过 gRPC 调 apiserver 保存答卷；答卷提交后通过 Outbox 和 worker 异步推进评估和报告生成。**

适合用于：

- 面试开场。
- 技术分享开头。
- README 顶部定位。
- 对技术同学介绍项目。

## 5. 1 分钟版本

> **qs-server 是一个问卷&量表测评系统的后端项目。它的核心难点不是保存题目，而是把“问卷收集、量表规则、评估执行、报告生成、统计运营、安全授权”这些变化原因不同的能力拆开。**
>
> **业务上，它把 Survey、Scale、Evaluation 拆成三个边界：Survey 管问卷和答卷，Scale 管医学量表、因子和解读规则，Evaluation 管一次测评的状态机、计分、风险和报告。**
>
> **运行时上，它不是单进程直连数据库，而是 collection-server、qs-apiserver、qs-worker 三进程协作：collection-server 保护前台入口，apiserver 处理主业务事实，worker 消费事件执行异步评估。**
>
> **工程上，它用 SubmitQueue、SubmitGuard、Outbox、Redis cache/lock、Backpressure、AuthzSnapshot、读侧统计聚合等机制，解决高峰提交、重复提交、事件可靠出站、权限边界和统计查询性能问题。**

适合用于：

- 简历项目详细介绍。
- 面试“介绍一下你做的项目”。
- 技术分享开场 1 分钟。

## 6. 3 分钟版本

如果面试官愿意听更完整的介绍，可以这样展开：

> **这个项目可以从三个层次讲。**
>
> **第一层是业务建模。它不是普通表单系统，而是心理/医学测评系统。普通问卷只关心题目和答案，但测评系统还要关心量表规则、因子计分、风险等级、解读文案和报告生成。所以我没有把所有东西塞进一个大模块，而是拆成 Survey、Scale、Evaluation：Survey 负责问卷模板和答卷提交，Scale 负责量表规则资产，Evaluation 负责测评执行和报告结果。**
>
> **第二层是运行时链路。前台小程序不会直接打 apiserver，而是先进入 collection-server。collection-server 是前台 BFF 和保护层，它负责 JWT 身份投影、租户范围、监护关系校验、限流、SubmitQueue 削峰、SubmitGuard 幂等，再通过 gRPC 调 apiserver 保存 AnswerSheet。apiserver 是主业务服务，负责持久化、领域服务、Outbox 和内部 gRPC。worker 则消费事件，通过 InternalService 推进答卷计分、创建 Assessment 和 Evaluation Pipeline。**
>
> **第三层是工程治理。答卷提交不是直接同步生成报告，而是“同步保存业务事实，异步生成评估结果”。AnswerSheet 保存和事件 staging 通过 Outbox 绑定在一起，避免 DB 与 MQ 双写不一致；Evaluation pipeline 复杂且可能失败，所以放到 worker 异步执行，并通过 Assessment 状态、报告存储、事件和观测来治理。统计查询也不直接每次扫业务写模型，而是有 Statistics 读侧聚合、BehaviorProjector、SyncService、QueryCache 和 Hotset。安全上则通过 IAMModule 嵌入 IAM SDK，但业务权限判断基于 AuthzSnapshot，而不是直接信任 JWT roles。**
>
> **所以这个项目的价值不在于“做了多少接口”，而在于把一个测评系统里容易混在一起的领域规则、异步链路和工程保护边界拆开，并且让它们可以继续演进。**

> **我重点介绍一下 qs-server，这是我围绕问卷&量表测评场景设计的 Go 后端项目。这个项目不是简单问卷 CRUD，我主要解决了三个问题：第一，如何把问卷、量表规则、评估报告拆成清晰领域边界；第二，如何让用户提交答卷后不阻塞等待报告，而是通过 Outbox 和 worker 异步评估；第三，如何通过 collection-server、限流、SubmitQueue、SubmitGuard、Backpressure、读侧统计和 IAM 授权快照，把系统做成可承载前台流量、可排障、可继续演进的后端架构。**

## 13. 最值得强调的三个亮点

### 13.1 领域边界不是 CRUD 拆包

最值得讲的是：

```text
Survey / Scale / Evaluation 为什么拆开
```

因为这体现了业务建模能力。

一句话讲法：

> **问卷是收集载体，量表是规则资产，评估是执行实例；它们生命周期不同，所以不能塞进一个大模块。**

---

### 13.2 主链路是“事实同步，结果异步”

最值得讲的是：

```text
同步保存 AnswerSheet
异步生成 Evaluation Report
```

因为这体现了时序设计和可靠性设计。

一句话讲法：

> **用户提交时必须先保证答案可靠落库，但报告生成涉及计分、风险和解读，应该通过 Outbox + Worker 异步推进。**

---

### 13.3 collection-server 是前台保护层

最值得讲的是：

```text
collection-server 不只是转发
```

因为这体现了高并发入口保护能力。

一句话讲法：

> **collection-server 是前台 BFF 和保护层，负责认证、限流、提交削峰、幂等和状态查询，避免前台流量直接打穿 apiserver。**

---

## 14. 不建议这样定位

### 14.1 不要说成“问卷系统”

不准确。

“问卷系统”会让人以为只是：

- 创建题目。
- 填写答案。
- 保存提交。

但 qs-server 的重点是：

- 量表规则。
- 因子计分。
- 风险解读。
- 异步评估。
- 报告生成。
- 统计运营。

更准确说法：

```text
问卷&量表测评系统
心理/医学测评后端
```

---

### 14.2 不要说成“微服务系统”

不准确。

当前更准确的是：

```text
多进程协作的模块化后端系统
```

或者：

```text
以 apiserver 为主业务中心，collection-server 和 worker 分别负责前台保护与异步处理的三进程架构
```

不要为了显得高级说成微服务。面试官追问部署、服务发现、独立数据库、CI/CD、治理成本时，很容易穿帮。

---

### 14.3 不要说成“用了 DDD 的项目”

太虚。

DDD 要落到具体边界：

- Survey。
- Scale。
- Evaluation。
- Actor。
- Plan。
- Statistics。

更好的说法：

> **我用 DDD 的思路把问卷收集、量表规则和评估执行拆成不同限界上下文，避免规则、答卷和报告生命周期互相污染。**

---

### 14.4 不要说成“用了 MQ 保证可靠”

不准确。

MQ 只负责消息传输，不解决 DB 与 MQ 双写一致性。

更准确说法：

> **我用 Outbox 把业务事实和待发布事件放进同一个持久化事务，再由 relay 发布到 MQ，避免 AnswerSheet 已保存但 submitted 事件丢失。**

---

### 14.5 不要说成“JWT roles 做权限”

不准确，也有安全风险。

更准确说法：

> **JWT 只负责认证和身份声明，业务授权基于 IAM AuthzSnapshot 的 resource/action 能力判断。**

---

## 15. 状态标签

宣讲时建议保持旧版宣讲文档里的状态标签原则：

| 标签 | 含义 | 讲法 |
| ---- | ---- | ---- |
| 已实现 | 仓库有明确代码和配置支撑 | 可以说“当前已经实现” |
| 待补证据 | 有局部实现，但证据还不完整 | 说“已有基础，正在收敛” |
| 规划改造 | 当前未完成 | 说“下一步计划” |

### 15.1 当前可以讲成“已实现”的能力

| 能力 | 说明 |
| ---- | ---- |
| Survey / Scale / Evaluation 拆分 | 业务模块与 assembler 已存在 |
| collection / apiserver / worker 三进程 | 运行时架构已存在 |
| SubmitQueue | collection 前台提交削峰已存在 |
| SubmitGuard | Redis done marker + in-flight lock 已存在 |
| Mongo/MySQL Outbox | AnswerSheet / Assessment 事件出站基础已存在 |
| Evaluation Pipeline | 评估引擎和 pipeline 已存在 |
| Statistics ReadService / SyncService / BehaviorProjector | 统计读侧能力已存在 |
| IAMModule / AuthzSnapshot | IAM 嵌入与授权快照已存在 |
| Redis/Resilience/Observability 基座 | cache/lock/rate limit/backpressure/status 已存在 |

### 15.2 应谨慎表达的能力

| 能力 | 推荐说法 |
| ---- | -------- |
| 完整生产级事件治理 | 已有 Outbox 基座，仍需补 failed event runbook / replay / DLQ |
| 多实例大规模部署 | 架构具备方向，但仍需压测和部署验证 |
| 完整 ACL | mTLS/ACL seam 已接入，文件加载和策略治理仍需完善 |
| AI 解读 | 只能作为未来增强方向，不是当前主链路 |
| Operating 平台 | 可作为演进方向，不要讲成当前完整能力 |
