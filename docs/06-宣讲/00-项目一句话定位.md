
# 00-项目一句话定位

**本文回答**：`qs-server` 对外应该如何用一句话讲清楚；在技术分享、简历、面试、朋友介绍、业务沟通中，如何用不同长度表达同一个项目；哪些说法准确，哪些说法会把项目讲低、讲偏或讲虚；如何把项目定位从旧的“问卷&量表系统”升级为“多解释模型测评平台”。

## 30 秒结论

最推荐的一句话：

> **qs-server 是一个面向心理、医学和人格测评场景的 Go 后端系统。它不是普通问卷 CRUD，而是一个多解释模型测评平台：Survey 负责作答事实，Interpretation Model 定义统一接入协议，Scale、MBTI、BigFive 等具体模型负责规则表达，Evaluation 作为通用测评执行引擎，按 ModelRef 加载 Provider 执行模型，并产出 EvaluationResult 和 InterpretReport。**

这句话要表达四个判断：

| 判断 | 含义 |
| ---- | ---- |
| 不是普通问卷 CRUD | 项目重点不只是题目和答案保存 |
| 不是单纯医学量表系统 | Scale 是具体模型，不是所有解释能力中心 |
| 是多解释模型测评平台 | Scale、MBTI、BigFive 可作为同级模型接入 |
| Evaluation 是通用执行引擎 | 它管 Assessment / Run / Result / Report 生命周期 |

## 4. 30 秒版本

> **qs-server 是一个面向心理、医学和人格测评场景的 Go 后端系统。它把业务拆成 Survey、Interpretation Model、Concrete Models、Evaluation 等边界：Survey 负责问卷和答卷事实，Interpretation Model 负责 ModelRef、Provider、Context、Registry 等统一接入协议，Scale、MBTI、BigFive 等具体模型负责规则表达，Evaluation 负责 Assessment、EvaluationRun、EvaluationResult、InterpretReport、失败重试和事件。运行时上，前台请求先进入 collection-server 做认证、限流、提交削峰和幂等保护，再通过 gRPC 调 apiserver 保存答卷；答卷提交后通过 Outbox、worker 和 internal gRPC 异步推进 Assessment、Interpretation 和 Report。**

## 5. 1 分钟版本

> **qs-server 是一个多解释模型测评后端系统，面向心理、医学和人格测评场景。它的核心难点不是保存题目，而是把“作答事实、解释模型规则、测评执行、报告生成、统计运营、安全授权”这些变化原因不同的能力拆开。**
>
> **业务上，它不再简单讲成 Survey / Scale / Evaluation 三段式，而是升级为 Survey / Interpretation Model / Concrete Models / Evaluation：Survey 管问卷和答卷事实；Interpretation Model 管 ModelRef、Provider、Context、Registry 这些接入协议；Scale、MBTI、BigFive 等 Concrete Models 管具体规则；Evaluation 管一次测评的 Assessment、EvaluationRun、EvaluationResult、InterpretReport、失败重试和事件出站。**
>
> **运行时上，它不是单进程直连数据库，而是 collection-server、qs-apiserver、qs-worker 三进程协作：collection-server 保护前台入口，apiserver 处理主业务事实，worker 消费事件异步推进测评执行。**
>
> **工程上，它用 SubmitQueue、SubmitGuard、Outbox、Redis cache/lock、Backpressure、AuthzSnapshot、读侧统计聚合、Metrics 和 Governance endpoint，解决高峰提交、重复提交、事件可靠出站、权限边界、统计查询和排障观测问题。**

适合用于：

- 简历项目详细介绍。
- 面试“介绍一下你做的项目”。
- 技术分享开场 1 分钟。

## 6. 3 分钟版本

如果面试官愿意听更完整的介绍，可以这样展开：

> **这个项目可以从三个层次讲。**
>
> **第一层是业务建模。它不是普通表单系统，而是心理、医学和人格测评系统。普通问卷只关心题目和答案，但测评系统还要关心解释模型、规则版本、结果结构、报告快照和统计口径。早期可以用 Survey / Scale / Evaluation 来讲，但支持 MBTI 后，Scale 不能继续作为所有解释能力中心，所以我把主线升级成 Survey / Interpretation Model / Concrete Models / Evaluation。Survey 管作答事实；Interpretation Model 管 ModelRef、Provider、Context、Registry；Scale、MBTI、BigFive 管具体规则；Evaluation 管 Assessment、EvaluationRun、EvaluationResult、InterpretReport 和失败重试。**
>
> **第二层是运行时链路。前台小程序不会直接打 apiserver，而是先进入 collection-server。collection-server 是前台 BFF 和保护层，它负责 JWT 身份投影、TenantDomain/OrgScope、监护关系校验、限流、SubmitQueue 削峰、SubmitGuard 幂等，再通过 gRPC 调 apiserver 保存 AnswerSheet。apiserver 是主业务服务，负责持久化、领域服务、Outbox 和内部 gRPC。worker 则消费事件，通过 internal gRPC 回调 apiserver，推进 Assessment 创建、解释模型执行和报告生成。**
>
> **第三层是工程治理。答卷提交不是直接同步生成报告，而是“同步保存作答事实，异步推进测评执行”。AnswerSheet 保存和事件 staging 通过 Outbox 绑定在一起，避免 DB 与 MQ 双写不一致；Evaluation 执行链路复杂且可能失败，所以放到 worker 事件链路中异步推进，并通过 Assessment 状态、EvaluationRun、Result、Report、事件和观测来治理。统计查询也不直接每次扫业务写模型，而是通过 Statistics 读侧聚合、BehaviorProjector、SyncService、QueryCache 和 Hotset 支撑。安全上则通过 IAMModule 嵌入 IAM SDK，但业务权限判断基于 AuthzSnapshot，而不是直接信任 JWT roles。**
>
> **所以这个项目的价值不在于接口数量，而在于把一个测评系统里容易混在一起的作答事实、模型规则、执行生命周期、报告事实和工程保护边界拆开，并且让它们可以继续向 MBTI、BigFive 和更多解释模型演进。**

## 13. 最值得强调的三个亮点

### 13.1 领域边界不是 CRUD 拆包

最值得讲的是：

```text
Survey / Interpretation Model / Concrete Models / Evaluation 为什么拆开
```

因为这体现了业务建模能力。

一句话讲法：

> **问卷是作答载体，解释模型是规则资产，Evaluation 是执行实例；它们生命周期不同，所以不能塞进一个大模块。**

---

### 13.2 主链路是“事实同步，执行异步”

最值得讲的是：

```text
同步保存 AnswerSheet
异步推进 Assessment / Interpretation / Report
```

因为这体现了时序设计和可靠性设计。

一句话讲法：

> **用户提交时必须先保证答案可靠落库，但模型执行和报告生成涉及 Provider、Context、Result、Report 和事件，应通过 Outbox + Worker 异步推进。**

---

### 13.3 collection-server 是前台保护层

最值得讲的是：

```text
collection-server 不只是转发
```

因为这体现了高并发入口保护能力。

一句话讲法：

> **collection-server 是前台 BFF 和保护层，负责认证、限流、提交削峰、幂等和状态查询，避免前台流量直接打穿 apiserver。**

## 14. 不建议这样定位

### 14.1 不要说成“问卷系统”

不准确。

“问卷系统”会让人以为只是：

- 创建题目。
- 填写答案。
- 保存提交。

但 qs-server 的重点是：

- 解释模型。
- 模型版本。
- 异步测评执行。
- EvaluationResult。
- InterpretReport。
- 统计运营。
- 安全授权。
- 观测治理。

更准确说法：

```text
多解释模型测评系统
心理/医学/人格测评后端
```

---

### 14.2 不要说成“量表系统”

不准确。

“量表系统”会让人以为 Scale 是所有解释能力的中心。

更准确说法：

```text
Scale 是当前具体医学量表模型；
MBTI、BigFive 等未来模型与 Scale 同级；
Interpretation Model 才是接入抽象。
```

---

### 14.3 不要说成“微服务系统”

不准确。

当前更准确的是：

```text
多进程协作的模块化后端系统
```

或者：

```text
以 apiserver 为主业务中心，collection-server 和 worker 分别负责前台保护与异步处理的三进程架构
```

不要为了显得高级说成微服务。面试官追问部署、服务发现、独立数据库、独立发布、服务治理成本时，很容易穿帮。

---

### 14.4 不要说成“用了 DDD 的项目”

太虚。

DDD 要落到具体边界：

- Survey。
- Interpretation Model。
- Concrete Models。
- Evaluation。
- Actor。
- Plan。
- Statistics。

更好的说法：

> **我用 DDD 的思路把作答事实、解释模型规则、测评执行生命周期和报告事实拆成不同限界上下文，避免规则、答卷和报告生命周期互相污染。**

---

### 14.5 不要说成“用了 MQ 保证可靠”

不准确。

MQ 只负责消息传输，不解决 DB 与 MQ 双写一致性。

更准确说法：

> **我用 Outbox 把业务事实和待发布事件放进同一个持久化事务，再由 relay 发布到 MQ，避免 AnswerSheet 已保存但 submitted 事件丢失。**

---

### 14.6 不要说成“JWT roles 做权限”

不准确，也有安全风险。

更准确说法：

> **JWT 只负责认证和身份声明，业务授权基于 IAM AuthzSnapshot 的 resource/action 能力判断。**

---

## 15. 状态标签

宣讲时建议保持状态标签原则：

| 标签 | 含义 | 讲法 |
| ---- | ---- | ---- |
| 已实现 | 仓库有明确代码和配置支撑 | 可以说“当前已经实现” |
| 待补证据 | 有局部实现，但证据还不完整 | 说“已有基础，正在收敛” |
| 规划改造 | 当前未完成 | 说“下一步计划” |

### 15.1 当前可以讲成“已实现”的能力

| 能力 | 说明 |
| ---- | ---- |
| Survey / Scale / Evaluation 拆分 | 业务模块与 assembler 已存在 |
| collection / apiserver / worker 三进程 | 运行时架构已存在 |
| SubmitQueue | collection 前台提交削峰已存在 |
| SubmitGuard | Redis done marker + in-flight lock 已存在 |
| Mongo/MySQL Outbox | AnswerSheet / Assessment 事件出站基础已存在 |
| Evaluation pipeline 基座 | 评估执行与报告生成基础已存在 |
| Statistics ReadService / SyncService / BehaviorProjector | 统计读侧能力已存在 |
| IAMModule / AuthzSnapshot | IAM 嵌入与授权快照已存在 |
| Redis / Resilience / Observability 基座 | cache / lock / rate limit / backpressure / status 已存在 |

### 15.2 应谨慎表达的能力

| 能力 | 推荐说法 |
| ---- | -------- |
| Interpretation Model 代码完全落地 | 文档与设计已完成，代码以当前事实为准 |
| MBTIProvider | 下一阶段计划接入，用于验证多解释模型平台化 |
| 完整生产级事件治理 | 已有 Outbox 基座，仍需补 failed event runbook / replay / DLQ |
| 多实例大规模部署 | 架构具备方向，但仍需压测和部署验证 |
| 完整 ACL | mTLS / ACL seam 已接入，文件加载和策略治理仍需完善 |
| AI 解读 | 只能作为未来增强方向，不是当前主链路 |
| Operating 平台 | 可作为演进方向，不要讲成当前完整能力 |

## 18. 推荐最终背诵版本

如果只背一段，建议背这一段：

> **qs-server 是一个面向心理、医学和人格测评场景的 Go 后端系统。它不是普通问卷 CRUD，而是把作答事实、解释模型规则、测评执行生命周期和报告事实拆开：Survey 负责问卷和答卷，Interpretation Model 负责统一接入协议，Scale、MBTI 等模型负责具体规则，Evaluation 负责 Assessment、EvaluationRun、EvaluationResult 和 InterpretReport。运行时上用 collection-server 保护前台入口，apiserver 处理主业务事实，worker 消费事件异步推进执行；工程上用 SubmitQueue、SubmitGuard、Outbox、Redis Lock、Backpressure、读侧统计聚合和 IAM AuthzSnapshot 解决提交高峰、重复提交、事件可靠出站、报告异步生成、统计查询和权限边界问题。**

这段可以作为所有宣讲的开头。
