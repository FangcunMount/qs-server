## 1. 分享目标

这场 30 分钟分享不是为了把所有代码细节讲完，而是让听众形成一个清晰认知：

> **qs-server 是一个面向心理/医学测评场景的 Go 后端系统，它通过 DDD 边界拆分、三进程协作、Outbox 事件可靠出站、高并发保护、IAM 安全控制面和统计读侧聚合，把“答卷提交”稳定推进为“评估报告和运营统计”。**

听众听完后应该记住 5 件事：

```text
1. 它不是普通问卷 CRUD，而是测评系统。
2. Survey / Scale / Evaluation 是核心领域边界。
3. collection / apiserver / worker 是运行时三进程协作。
4. 答卷同步提交，评估异步执行，Outbox 保证关键事件可靠出站。
5. 高并发、安全、统计和工程质量都有明确治理边界。
```

## 4. 0:00 - 2:30 项目一句话定位

### 4.1 目标

让听众立刻明白：

```text
这是测评系统
不是普通问卷 CRUD
技术重点是边界、异步、可靠性和治理
```

### 4.2 可直接口述

> **qs-server 是一个面向心理/医学测评场景的 Go 后端系统。它不是普通问卷 CRUD，而是把问卷收集、量表规则和评估报告拆成 Survey、Scale、Evaluation 三个边界；运行时上用 collection-server 保护前台入口，apiserver 处理主业务事实，worker 消费事件异步推进评估；工程上用 SubmitQueue、SubmitGuard、Outbox、Redis Lock、Backpressure、读侧统计聚合和 IAM AuthzSnapshot 解决提交高峰、重复提交、事件可靠出站、报告异步生成、统计查询和权限边界问题。**

### 4.3 可以展示的关键词

```text
测评系统
Survey / Scale / Evaluation
collection / apiserver / worker
同步提交 / 异步评估
Outbox / NSQ / Worker
Resilience / IAM / Statistics
```

### 4.4 过渡语

> **接下来我先不讲技术名词，先讲业务问题：为什么这个系统不能简单做成一个问卷 CRUD。**

## 5. 2:30 - 5:00 业务背景与问题

### 5.1 目标

说明业务复杂度来自哪里。

### 5.2 可直接口述

> **在心理/医学测评场景里，用户填写的不是普通表单，而是一份需要被量表规则解释的测评数据。系统要根据答案计算分数、判断风险、生成报告，还要支持医生和机构做统计运营。**
>
> **所以这个项目的难点不是“怎么保存问卷答案”，而是：问卷收集、量表规则、评估执行、报告生成、统计查询和权限控制这些变化原因都不同；前台提交要快，但评估生成可能慢；答卷保存成功后，后续事件又不能丢。**

### 5.3 业务问题表

| 业务问题 | 技术后果 |
| -------- | -------- |
| 问卷、量表、评估容易混在一起 | 需要 DDD 边界 |
| 提交要快，报告生成慢 | 需要异步评估 |
| 前台流量不可控 | 需要 collection 保护层 |
| 事件不能丢 | 需要 Outbox |
| 统计不能实时扫大表 | 需要读侧聚合 |
| 权限和隐私复杂 | 需要 IAM/AuthzSnapshot |

### 5.4 过渡语

> **针对这些问题，系统运行时不是一个单进程，而是拆成了三个职责不同的进程。**

## 6. 5:00 - 8:30 三进程架构

### 6.1 目标

讲清楚运行时骨架。

### 6.2 主图

```mermaid

    Admin["后台 / Operating"]
    MQ["NSQ / MQ"]

    subgraph Collection["collection-server"]

    end

    subgraph API["qs-apiserver"]

        A2["gRPC Server"]
        A3["Domain Modules"]

    end

    subgraph Worker["qs-worker"]

        W2["Event Handler"]

    end

    Client --> C1 --> C2 --> C3 --> A2
    Admin --> A1
    A3 --> A4 --> MQ
    MQ --> W1 --> W2 --> W3 --> A2
```

### 6.3 可直接口述

> **我把运行时讲成三进程协作，但它不是完整微服务。collection-server 是前台 BFF 和保护层，负责认证、限流、SubmitQueue 削峰、SubmitGuard 幂等和状态查询；qs-apiserver 是主业务中心，负责 Survey、Scale、Evaluation、Actor、Plan、Statistics 等领域模块，以及 MySQL/Mongo 持久化、Outbox、REST/gRPC 和调度；qs-worker 是异步执行器，消费事件后通过 internal gRPC 回调 apiserver 推进计分、创建 Assessment 和报告生成。**

### 6.4 重点句

> **collection 保护入口，apiserver 保存事实，worker 推进异步。**

### 6.5 不要讲错

不要说：

```text
这是三个微服务
```

更准确：

```text
这是以 apiserver 为主业务中心的三进程协作架构。
```

### 6.6 过渡语

> **运行时讲清楚之后，再看业务模型。为什么 apiserver 里面不是一个大问卷模块，而是拆成多个限界上下文？**

## 7. 8:30 - 12:30 DDD 与限界上下文

### 7.1 目标

说明 DDD 是为了解决变化原因，不是堆术语。

### 7.2 主图

```mermaid
flowchart LR
    Actor["Actor<br/>主体"]
    Plan["Plan<br/>任务编排"]
    Survey["Survey<br/>采集事实"]
    Scale["Scale<br/>规则权威"]
    Evaluation["Evaluation<br/>测评产出"]
    Statistics["Statistics<br/>读侧聚合"]

    Actor --> Survey
    Plan --> Survey
    Survey --> Evaluation
    Scale --> Evaluation
    Evaluation --> Statistics
    Actor --> Statistics
    Plan --> Statistics
```

### 7.3 可直接口述

> **这个项目里 DDD 最重要的是限界上下文，不是聚合名字。测评业务里，问卷、量表和评估结果很容易混在一起，但它们其实是三个不同问题：问卷是收集载体，关心题目、选项、答卷和答案校验；量表是规则资产，关心因子、计分规则、风险阈值和解读文案；评估是执行实例，关心一次 Assessment 的状态流转、分数、报告和失败重试。**
>
> **所以我把它拆成 Survey、Scale、Evaluation 三个核心上下文。Actor 解决业务参与者，比如受试者、医生、操作员；Plan 解决长期测评计划和任务编排；Statistics 解决读侧统计和运营查询。**

### 7.4 重点句

```text
Survey 管“填什么”
Scale 管“怎么算和怎么解释”
Evaluation 管“这一次测评执行后的结果”
```

### 7.5 面试加分句

> **我判断限界上下文时，不是按数据库表拆，也不是按接口路径拆，而是看变化原因。**

### 7.6 过渡语

> **有了这些领域边界后，最核心的一条链路就是：用户提交答卷后，系统如何异步生成报告。**

## 8. 12:30 - 18:00 异步评估链路

### 8.1 目标

这是分享的核心段落。要讲清楚：

```text
同步保存答卷事实
异步推进评估结果
worker 只是驱动器
apiserver 承载状态机和 pipeline
```

### 8.2 主图

```mermaid
sequenceDiagram
    participant Client as Client
    participant Collection as collection-server
    participant API as qs-apiserver
    participant Outbox as Outbox Relay
    participant MQ as NSQ/MQ
    participant Worker as qs-worker
    participant Pipeline as Evaluation Pipeline

    Client->>Collection: Submit AnswerSheet
    Collection->>API: SaveAnswerSheet via gRPC
    API->>API: Save AnswerSheet + stage answersheet.submitted
    Outbox->>MQ: publish answersheet.submitted
    MQ->>Worker: consume
    Worker->>API: CalculateAnswerSheetScore
    Worker->>API: CreateAssessmentFromAnswerSheet
    API->>API: stage assessment.submitted
    Outbox->>MQ: publish assessment.submitted
    MQ->>Worker: consume
    Worker->>API: EvaluateAssessment
    API->>Pipeline: Validation -> FactorScore -> Risk -> Interpretation
    Pipeline-->>API: Score / Report
```

### 8.3 可直接口述

> **异步评估链路可以理解为“同步保存答卷事实，异步推进测评结果”。前台提交先经过 collection-server 的限流、SubmitQueue 和 SubmitGuard，然后通过 gRPC 调 apiserver 保存 AnswerSheet。apiserver 保存答卷时，会把 AnswerSheet 和 `answersheet.submitted` 事件通过 Outbox 同事务落库，避免 DB 与 MQ 双写不一致。**
>
> **Outbox relay 把事件发布到 MQ 后，qs-worker 消费事件并通过 internal gRPC 回调 apiserver。第一跳会先计算答卷分，再创建 Assessment；第二跳通过 `assessment.submitted` 触发 `EvaluateAssessment`，进入 Evaluation Pipeline。Pipeline 负责校验输入、计算因子分、判断风险等级、生成解读报告和通知等待者。**
>
> **这套设计的关键是：worker 只是异步驱动器，真正的状态机、分数、报告和 Outbox 仍然在 apiserver；事件投递不是 exactly-once，而是通过 Outbox、Ack/Nack、锁、唯一约束和状态机共同实现可靠推进和业务幂等。**

### 8.4 Pipeline 快速讲法

| Step | 讲法 |
| ---- | ---- |
| Validation | 校验评估上下文 |
| FactorScore | 按量表因子和答案算分 |
| RiskLevel | 匹配风险等级并保存结构化分数 |
| Interpretation | 生成解读报告 |
| WaiterNotify | 通知等待报告的请求 |

### 8.5 过渡语

> **刚才反复提到了 Outbox 和 MQ。下一段单独讲：为什么用了 NSQ 之后还需要 Outbox。**

## 9. 18:00 - 21:30 事件与 Outbox

### 9.1 目标

讲清楚：

```text
EventCatalog 管契约
Outbox 管可靠出站
NSQ 管消息传输
Worker 管消费处理
业务状态机管幂等
```

### 9.2 主图

```mermaid
flowchart LR
    App["Application Service"]
    Catalog["EventCatalog<br/>configs/events.yaml"]
    Publisher["RoutingPublisher"]
    Outbox["Outbox Store"]
    Relay["Outbox Relay"]
    NSQ["NSQ Topic / Channel"]
    Worker["Worker Dispatcher / Handler"]

    Catalog --> Publisher
    App -->|best_effort| Publisher
    App -->|durable_outbox| Outbox --> Relay --> Publisher
    Publisher --> NSQ --> Worker
```

### 9.3 可直接口述

> **我不会说“用了 NSQ 就可靠”。NSQ 负责的是消息从一个进程投递到另一个进程，但它不解决业务数据库 commit 和 MQ publish 的原子性。比如 AnswerSheet 保存成功后，进程还没 publish NSQ 就崩溃，MQ 根本不知道这条事件。**
>
> **所以关键事件走 Outbox：业务事实和待发布事件同事务保存，relay 后台 claim due events，publish 成功就 mark published，失败就 mark failed 并设置下一次重试时间。NSQ 是传输层，worker 是消费层，消费端还要靠锁、唯一约束、状态机和 checkpoint 保证幂等。**

### 9.4 重点句

> **Outbox 是 producer-side reliability，NSQ 是 transport，worker handler 是 consumer-side processing。**

### 9.5 不要讲错

不要说：

```text
Outbox 保证 exactly-once
```

更准确：

```text
Outbox 保证可靠出站；consumer 仍需幂等。
```

### 9.6 过渡语

> **事件链路解决的是异步可靠性；但高峰流量来了，还要解决系统怎么不被打穿。**

## 10. 21:30 - 24:30 高并发治理

### 10.1 目标

让听众知道高并发不是只加限流。

### 10.2 主图

```mermaid
flowchart LR
    Client["Client"]
    Rate["RateLimit"]
    Queue["SubmitQueue"]
    Guard["SubmitGuard"]
    Inflight["gRPC max-inflight"]
    BP["Backpressure"]
    Lock["LockLease"]
    Worker["Worker concurrency"]

    Client --> Rate --> Queue --> Guard --> Inflight --> BP
    Lock --> Queue
    Lock --> Worker
```

### 10.3 可直接口述

> **qs-server 的高并发治理是分层做的。前台请求先进 collection-server，先按 submit、query、wait-report 做入口限流；答卷提交再进入 SubmitQueue，把突发请求削成固定 worker 并发；重复提交通过 SubmitGuard 的 done marker 和 in-flight lock 抑制；collection 到 apiserver 的 gRPC 还有 max-inflight 控制；apiserver 内部对 MySQL、Mongo、IAM 用 Backpressure 限制 in-flight 操作数；worker 消费 MQ 时通过 worker.concurrency 控制并发，并用 duplicate suppression 防止重复事件并发处理。**
>
> **我不会把这套东西讲成一个万能高并发框架。RateLimit、Queue、Backpressure、Lock、Idempotency 语义不同，所以具体能力分别实现，但通过 resilienceplane 统一观测 outcome。**

### 10.4 重点句

> **高并发治理的目标不是让所有请求都成功，而是压力超过系统承载时，能明确知道它被挡在哪一层。**

### 10.5 过渡语

> **除了流量治理，测评系统还有一个敏感问题：谁能看谁的答卷和报告？这就进入 IAM 和安全。**

## 11. 24:30 - 27:00 IAM 与安全

### 11.1 目标

讲清楚：

```text
IAM 是身份和授权真值
JWT roles 不是权限真值
AuthzSnapshot 判断 capability
```

### 11.2 主图

```mermaid
flowchart LR
    Token["JWT / Service Token"]
    Principal["Principal<br/>谁在调用"]
    Tenant["TenantScope<br/>在哪个组织"]
    Snapshot["AuthzSnapshot<br/>有哪些 resource/action"]
    Decision["CapabilityDecision<br/>能不能做"]
    Handler["Business Handler"]

    Token --> Principal --> Tenant --> Snapshot --> Decision --> Handler
```

### 11.3 可直接口述

> **qs-server 不自己实现完整 IAM，而是通过 IAMModule 嵌入 IAM SDK。HTTP 和 gRPC 都先用 IAM TokenVerifier 验证 token，投影成 Principal；再用 TenantScope 确定当前 tenant/org；需要业务授权时，通过 IAM GetAuthorizationSnapshot 拉取当前用户在当前 domain 下的 resource/action 权限，再用 CapabilityDecision 判断能不能读取问卷、管理量表、读取答卷或触发评估。**
>
> **我不会直接用 JWT roles 做业务权限，因为 token roles 可能滞后，也不能完整表达 resource/action。本地 Operator roles 只是 IAM roles 的投影，用于展示和协作查询，也不是权限真值。服务间调用则通过 ServiceAuth bearer token 和可选 mTLS identity match 表达 ServiceIdentity。**

### 11.4 重点句

```text
JWT 证明“你是谁”
TenantScope 证明“你在哪个组织”
AuthzSnapshot 判断“你能做什么”
```

## 13. 29:00 - 30:00 总结

### 13.1 可直接口述

> **总结一下，qs-server 的价值不是接口数量，而是边界和链路。业务上，它把问卷收集、量表规则、评估报告拆成 Survey、Scale、Evaluation；运行时上，它用 collection-server、apiserver、worker 三进程隔离前台入口、主业务事实和异步执行；链路上，它用 Outbox 和 NSQ 推进异步评估；工程上，它用 RateLimit、SubmitQueue、SubmitGuard、Backpressure、LockLease、IAM AuthzSnapshot 和 Statistics 读侧聚合来治理高并发、安全和运营查询。**
>
> **如果后续继续演进，我会优先补强 Outbox failed/replay、Evaluation failed retry、统计治理、压测报告和 operating 运维平台，而不是过早拆微服务或盲目堆新功能。**

## 14. 常见时间失控处理

### 14.1 如果 DDD 段讲太久

压缩：

```text
Survey 管采集，Scale 管规则，Evaluation 管结果。
详细聚合不展开，后面异步链路会自然串起来。
```

### 14.2 如果异步链路讲太久

```text
AnswerSheet saved -> Outbox -> worker -> Assessment -> Pipeline -> Report
```

## 16. 面试版压缩脚本：5 分钟

如果面试只给 5 分钟：

> **我介绍一个最有代表性的项目 qs-server。它是面向心理/医学测评场景的 Go 后端，不是普通问卷 CRUD。业务上我把它拆成 Survey、Scale、Evaluation：Survey 管问卷和答卷，Scale 管量表规则，Evaluation 管评估状态、得分和报告。**
>
> **运行时上，它是 collection-server、qs-apiserver、qs-worker 三进程协作。collection 是前台 BFF 和保护层，负责限流、SubmitQueue、SubmitGuard 和状态查询；apiserver 是主业务中心，负责领域模型、MySQL/Mongo 持久化、Outbox、REST/gRPC 和调度；worker 消费事件，通过 internal gRPC 回调 apiserver 推进异步评估。**
>
> **主链路是：用户提交答卷后，系统同步保存 AnswerSheet，并通过 Outbox 写入 `answersheet.submitted`；worker 消费后创建 Assessment，再通过 `assessment.submitted` 触发 Evaluation Pipeline，生成 Score 和 Report。这里我不会说 MQ 天然可靠，而是用 Outbox 解决 DB 与 MQ 双写一致性，用 worker Ack/Nack 和业务幂等处理重复消费。**
>
> **高并发上，我做了分层保护：入口限流、SubmitQueue 削峰、SubmitGuard 防重复、gRPC max-inflight、apiserver Backpressure、Redis LockLease、worker concurrency。安全上通过 IAMModule 接入 IAM，JWT 做认证，AuthzSnapshot 做业务授权。工程质量上有 Go tests、lint/security、Swagger/OpenAPI 对比和 docs hygiene 校验。**

## 17. 面试版压缩脚本：2 分钟

> **qs-server 是一个问卷&量表测评后端，核心不是保存问卷答案，而是把答卷提交稳定推进为评估报告和统计数据。**
>
> **业务上我拆了 Survey、Scale、Evaluation 三个边界：Survey 管问卷和答卷，Scale 管量表规则，Evaluation 管 Assessment、Score 和 Report。运行时上是三进程：collection-server 保护前台入口，apiserver 处理主业务事实，worker 消费事件推进异步评估。**
>
> **主链路是同步提交 AnswerSheet、异步生成 Report。AnswerSheet 保存时会把 `answersheet.submitted` 写入 Outbox，relay 发布到 NSQ，worker 消费后通过 internal gRPC 回到 apiserver 创建 Assessment 并执行 Evaluation Pipeline。**
>
> **工程上，collection 有 RateLimit、SubmitQueue、SubmitGuard，apiserver 有 Backpressure 和 Outbox，worker 有 concurrency 和 duplicate suppression；安全上通过 IAM AuthzSnapshot 做能力判断，不直接用 JWT roles。**

## 18. 不能说过头的点

| 不要说 | 应该说 |
| ------ | ------ |
| 这是微服务系统 | 这是以 apiserver 为主业务中心的三进程协作架构 |
| MQ 保证可靠 | Outbox 保证 producer-side reliable publish，MQ 负责传输 |
| 事件 exactly-once | 至少一次投递 + 业务幂等 |
| SubmitQueue 保证不丢 | SubmitQueue 是内存削峰，事实可靠性来自 durable submit |
| JWT roles 做权限 | JWT 做认证，AuthzSnapshot 做授权 |
| 已经压测 1000 QPS | 有容量档位建议，QPS 需要压测报告支撑 |
| 完整 ACL 已落地 | 有 service identity/mTLS/ACL seam，完整策略仍需完善 |
