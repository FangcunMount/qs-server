# 高频问答与追问脚本

**本文回答**：这篇文档把最常被追问的 8 个问题整理成可直接开口的答法，目标不是背稿，而是让每个问题都能先给短答，再按需要展开，并且始终能回链到真值文档和代码证据。

## 问答路径图

```mermaid
flowchart LR
    q["追问"] --> short["15 秒答法"]
    short --> expand["展开一层"]
    expand --> boundary["别说过头"]
    boundary --> evidence["证据位"]
```

## 使用方式

- 先背每题的 `15 秒答法`，把它练到 15-30 秒能说完。
- 只有在对方继续追问时，再展开第二段，不要一上来把全部细节倒出来。
- 每题的 `证据位` 至少要熟一条，这样说到关键判断时能立刻回链。

## 答题模板

- `15 秒答法`：先给一句能立住场子的结论。
- `展开`：只在对方继续追问时再补第二层。
- `别说过头`：这里专门约束哪些话不能包装成既成事实。
- `证据位`：至少熟一条，确保讲法始终能回链。
- `常见追问`：提前准备下一轮切口，不临场现编。

## Q1：一句话介绍 `qs-server`

- **15 秒答法**：这是一个围绕问卷/量表测评场景的后端系统，`qs-apiserver` 是主业务中心，前面有 `collection-server` 做 BFF，后面有 `qs-worker` 作为异步驱动器推进评估和报告生成。
- **展开**：它的难点不在表单提交，而在于把 `survey`、`scale`、`evaluation` 拆成稳定边界，再通过 `answersheet.submitted` 和 `assessment.submitted` 两个事件把同步提交和异步评估串起来。
- **别说过头**：不要讲成“问卷 CRUD 系统加异步任务”，也不要讲成“已经拆成微服务”。
- **证据位**：[01-项目定位与受众画像.md](./01-项目定位与受众画像.md)、[02-qs-server one-pager.md](./02-qs-server%20one-pager.md)
- **常见追问**：为什么不是一个问卷 CRUD 系统？为什么不是微服务？

## Q2：提交答卷之后，链路怎么走

- **15 秒答法**：先走 `collection-server` 接入与可选 SubmitQueue，再经 gRPC 到 `qs-apiserver` 保存答卷；答卷、幂等记录和 `answersheet.submitted` outbox 会在同一个 Mongo 事务里写入，真正出站由 apiserver relay 补发，后面的计分和建测评再交给 worker 驱动。
- **展开**：`collection-server` 负责鉴权、监护、限流和接入保护；真正的答卷聚合创建在 `apiserver` 的 survey 应用服务里。答卷保存成功后，worker 第一跳先算答卷分，再创建并提交测评。
- **别说过头**：不要讲成“提交时就同步算完测评”，也不要把这套 durable submit 口径外推成“所有事件都已统一可靠出站”。
- **证据位**：[03-主链路 1：提交答卷.md](./03-主链路%201：提交答卷.md)
- **常见追问**：为什么会返回 `202`？答卷落库和发事件是不是事务一致？

## Q3：为什么提交答卷同步完成，但评估放到异步

- **15 秒答法**：因为答卷提交的目标是尽快拿到主状态，评估链路则可能牵涉量表、问卷、计分、解读和报告，应该从请求线程里拆出去。
- **展开**：同步阶段只负责“接住请求 + 落卷 + 发起异步链路”，异步阶段再由 worker 调 internal gRPC 驱动测评和评估引擎。这样可以控制提交延迟，也方便做重试和扩展。
- **别说过头**：不要讲成“用了 MQ 一致性就天然解决了”，也不要把重试能力说成已经完全标准化。
- **证据位**：[04-主链路 2：异步评估流水线.md](./04-主链路%202：异步评估流水线.md)、[06-关键决策卡.md](./06-关键决策卡.md)
- **常见追问**：那一致性怎么保证？失败怎么补偿？

## Q4：为什么 `survey / scale / evaluation` 要拆

- **15 秒答法**：因为“采集答卷”“定义量表规则”“产生测评结果”是三类不同变化源，揉在一起模型会很快失真。
- **展开**：`survey` 关注问卷和答卷，`scale` 关注量表与因子规则，`evaluation` 关注测评状态机、得分和报告。拆开后，各域只背自己的不变量，跨域通过引用值对象和事件协作。
- **别说过头**：不要讲成“为了 DDD 而 DDD”，也不要顺手扩写成“量表多版本治理已经完全闭环”。
- **证据位**：[05-DDD 领域地图与模块协作.md](./05-DDD%20领域地图与模块协作.md)、[06-关键决策卡.md](./06-关键决策卡.md)
- **常见追问**：为什么不直接在答卷里塞评分结果？为什么量表不直接挂在问卷模块里？

## Q5：worker 并发和 `events.yaml` 配置到底怎么生效

- **15 秒答法**：当前真正控制 NSQ `MaxInFlight` 的是 `worker.concurrency`；多实例共享同一个 backlog 的关键是相同的 `worker.service-name`。
- **展开**：worker 在创建 NSQ consumer 时使用的是 `worker.concurrency`，订阅 channel 则来自 `worker.service-name`。因此“同 channel 扩 worker”靠的是副本数增加和相同 service-name，而不是在 `events.yaml` 里再配一套 per-topic 并发。
- **别说过头**：不要把旧的 per-topic consumer 心智再讲成当前 schema，也不要把 RabbitMQ 分支讲成现网真值。
- **证据位**：[07-工程治理与证据.md](./07-工程治理与证据.md)、[../05-专题分析/02-异步评估链路：从答卷提交到报告生成.md](../05-专题分析/02-异步评估链路：从答卷提交到报告生成.md)
- **常见追问**：那 `events.yaml` 为什么还保留这个字段？RabbitMQ 分支又怎样？

## Q6：`AssessmentScore` 和 `InterpretReport` 为什么分开存

- **15 秒答法**：因为它们服务的读取模式不同。结构化得分适合 MySQL 做查询和趋势分析，报告内容适合 MongoDB 存文档化解读和建议。
- **展开**：`AssessmentScore` 是面向查询和统计的结构化结果，`InterpretReport` 是面向报告展示的文档视图，而且 `InterpretReport.ID == AssessmentID`，能直接按测评查询。
- **别说过头**：不要讲成“所有报告型内容都应该进 Mongo”，也不要讲成“报告可以完全无损地从分数实时重建”。
- **证据位**：[../02-业务模块/03-evaluation.md](../02-业务模块/03-evaluation.md)、[04-主链路 2：异步评估流水线.md](./04-主链路%202：异步评估流水线.md)
- **常见追问**：为什么不全放 MySQL？为什么报告不直接从分数实时拼？

## Q7：幂等是怎么做的，哪一段最弱

- **15 秒答法**：当前幂等是分层做的，不是一句“MQ 保证”。显式带 `idempotency_key` 的答卷提交已经有 durable 幂等和 outbox；创建测评这一步仍然是第二层最强保护。
- **展开**：BFF 侧 `request_id` 只负责单实例内存态排队状态；apiserver 对显式 `idempotency_key` 的提交会把答卷、幂等记录和 `answersheet.submitted` outbox 同事务写入；worker 第一跳仍有答卷 Redis 锁；创建测评有按 `answer_sheet_id` 的预查和 MySQL 唯一约束；`EvaluateAssessment` 又受 `submitted` 状态约束。
- **别说过头**：不要讲成“所有提交天然 durable 幂等”，也不要把 `answersheet.submitted` 的 outbox 外推到所有事件。
- **证据位**：[03-主链路 1：提交答卷.md](./03-主链路%201：提交答卷.md)、[04-主链路 2：异步评估流水线.md](./04-主链路%202：异步评估流水线.md)
- **常见追问**：如果客户端重试提交怎么办？如果消息重复投递怎么办？

## Q8：当前系统最大的工程风险和下一步改造是什么

- **15 秒答法**：这条链最大的已知风险已经从“答卷提交缺 durable 幂等 / outbox”转移到了“durable 幂等仍是显式 `idempotency_key` opt-in，而且 outbox 还没有覆盖全部事件系统”。
- **展开**：答卷提交已经补齐 durable 幂等和 outbox/relay，评估主链关键事件也已按 MySQL / Mongo 边界 outbox 化；但没有显式 `idempotency_key` 的提交仍主要依赖下游第二层保护，`task.*`、`questionnaire.changed`、`scale.changed` 等事件仍保持 best-effort publish。
- **别说过头**：不要把这轮改造讲成“全事件系统都 outbox 化了”，也不要讲成“客户端不传业务键也已经天然强幂等”。
- **证据位**：[03-主链路 1：提交答卷.md](./03-主链路%201：提交答卷.md)、[07-工程治理与证据.md](./07-工程治理与证据.md)
- **常见追问**：如果现在就要上线更大流量，你会先补哪一块？
