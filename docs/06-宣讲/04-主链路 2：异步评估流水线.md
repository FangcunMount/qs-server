# 主链路 2：异步评估流水线

**本文回答**：这篇文档负责把“答卷提交之后的异步评估流水线”讲清楚，重点说明两跳事件如何衔接、worker 和 `apiserver` 各自负责什么、`AssessmentID` 在链路里扮演什么角色，以及当前幂等、重试和状态机应该怎么如实表述。

## 30 秒讲法

异步评估链路可以压缩成一句话：

`answersheet.submitted -> qs-worker -> internal gRPC -> assessment.submitted -> qs-worker -> EvaluateAssessment -> pipeline -> assessment.interpreted / report.generated`

这里的关键不是“用了 MQ”，而是**同步提交和重计算评估被明确拆开**，并且测评状态机、得分和报告都还在 `qs-apiserver` 的领域模型里，worker 只做订阅、驱动和 Ack/Nack。

## 重点速查

如果只看一屏，先看下面这张表：

| 维度 | 结论 |
| ---- | ---- |
| 两跳异步 | 第一跳从 `answersheet.submitted` 到建测评，第二跳从 `assessment.submitted` 到真正评估 |
| 角色分工 | worker 负责订阅、分发和回调；真正的状态机、得分、报告和事件发布仍在 `qs-apiserver` |
| 主键语义 | `AssessmentID` 是第二跳及后续得分、报告、状态查询的主键 |
| 幂等来源 | 不是“MQ 天然保证”，而是 Redis 锁、业务键唯一约束和状态机约束叠加 |
| 最该谨慎讲的点 | 不要把 `events.yaml` 旧 schema 讲成现状；当前真正需要谨慎讲的是“哪些事件已经 outbox 化，哪些还没有” |
| 失败口径 | 第一跳和第二跳失败都会回到 worker 错误处理；无量表不是失败，而是显式跳过评估 |

## AssessmentID 的业务语义

- `assessment.submitted` 事件以 `AssessmentID` 作为聚合主键和后续评估入口。证据见 [events.go](../../internal/apiserver/domain/evaluation/assessment/events.go)。
- `InterpretReport` 与 `Assessment` 是 1:1，报告 ID 与 `AssessmentID` 一致。证据见 [report.go](../../internal/apiserver/domain/evaluation/report/report.go)。
- `AssessmentScore` 以 `assessment_id` 作为查询主键，支撑得分查询与趋势分析。证据见 [score_query_service.go](../../internal/apiserver/application/evaluation/assessment/score_query_service.go) 和 [po.go](../../internal/apiserver/infra/mysql/evaluation/po.go)。

## 保证矩阵

| 主题 | 当前能讲成已保证 | 当前不能讲过头 |
| ---- | ---------------- | -------------- |
| 两跳分工 | 建测评和真正评估已明确拆成两跳事件链 | 不能讲成“一个 worker handler 就完成整条评估” |
| 建测评幂等 | 按答卷唯一建测评已有锁、预查和唯一约束 | 不能外推成“所有异步步骤都天然幂等” |
| 状态机保护 | `EvaluateAssessment` 只接受 `submitted`，重放到已成功状态不会再次成功 | 不能讲成“重复消息对任何阶段都完全无影响” |
| 并发与重试 | 失败会走 `Nack()`，`worker.concurrency` 有真实生效路径 | 不能把具体重试次数或旧版 per-topic consumer 并发字段讲成现成事实 |

## Pipeline 五步应该怎么讲

1. **Validation**：检查评估上下文是否齐全，保证后续步骤不在脏数据上继续。
2. **FactorScore**：按答卷和量表规则聚合因子分。
3. **RiskLevel**：计算风险等级，并保存 `AssessmentScore`。
4. **Interpretation**：构建 `InterpretReport`，保存报告内容。
5. **WaiterNotify**：只负责本地 waiter 通知；`assessment.interpreted`、`report.generated` 的可靠出站绑定在报告成功落 Mongo 时的 outbox。

证据见 [service.go](../../internal/apiserver/application/evaluation/engine/service.go) 和 [pipeline](../../internal/apiserver/application/evaluation/engine/pipeline)。

## 失败路径

- 第一跳失败：答卷计分或建测评失败，handler 返回 error，worker 侧进入 `Nack()` 路径。证据见 [answersheet_handler.go](../../internal/worker/handlers/answersheet_handler.go)。
- 第二跳失败：`EvaluateAssessment` 失败时返回 `failed` 状态，worker 同样按错误处理。证据见 [assessment_handler.go](../../internal/worker/handlers/assessment_handler.go)。
- 量表或答卷缺失：`engine.Service.Evaluate` 会把测评标记为失败，并走 `assessment.failed` 相关路径。证据见 [service.go](../../internal/apiserver/application/evaluation/engine/service.go)。
- 无量表：不是错误，而是显式跳过评估。

## 宣讲时建议强调

- worker 不持有主业务模型，它只是**异步驱动器**；真正的状态机和流水线都在 `apiserver`。
- 这条链的幂等不是“MQ 保证”，而是**锁 + 业务键唯一约束 + 状态机约束**共同实现。
- 当前影响 `NSQ MaxInFlight` 的是 `worker.concurrency`；`events.yaml` 里也已经不再存在旧的 per-topic consumer 并发 schema。

## 别说过头

- 不要把旧版 per-topic consumer 并发字段讲成真实生效并发。
- 不要把具体重试次数讲成固定数字，除非你能同时拿出现行 MQ 实现证据。
- 不要把 worker 讲成“拥有评估引擎”；更准确的说法是它驱动 `apiserver` 内的评估引擎执行。
