# 业务背景与问题

**本文回答**：`qs-server` 这个项目到底解决什么业务问题；为什么它不是普通问卷 CRUD；如果从业务方、前台用户、医生/机构、运营后台、技术团队几个角度看，系统分别要解决哪些痛点；对外宣讲时应该如何把“业务背景 → 核心问题 → 技术方案”讲成一条自然的逻辑链。

---

## 1. 一句话背景

> **在心理/医学测评场景中，用户填写的不是普通表单，而是能触发量表计分、风险判断、专业解读、报告生成和运营统计的一次“测评事件”。**

所以 `qs-server` 要解决的不是：

```text
把问卷答案存起来
```

而是：

```text
把一次答卷提交，稳定推进为一次可解释、可追踪、可统计、可治理的测评结果。
```

---

## 2. 业务场景是什么

这个系统面向的是问卷&量表测评场景。

典型流程是：

```text
机构配置测评量表
  -> 用户/家长/受试者填写问卷
  -> 系统保存答卷
  -> 系统按量表规则计算分数
  -> 系统判断风险等级
  -> 系统生成解读报告
  -> 医生/机构查看报告和统计
  -> 后续形成测评计划、跟踪、复测和运营分析
```

这和普通问卷系统的区别在于：

| 普通问卷 | 测评系统 |
| -------- | -------- |
| 关注题目和答案 | 关注答案背后的规则解释 |
| 提交后保存即可 | 提交后还要计分、风险、报告 |
| 报表通常简单统计 | 统计涉及受试者、医生、计划、风险、趋势 |
| 题目变化影响较小 | 量表规则变化会影响专业解释 |
| 权限相对简单 | 涉及机构、医生、监护关系、受试者隐私 |
| 同步返回即可 | 评估和报告适合异步执行 |

一句话：

> **普通问卷的核心是“收集”，测评系统的核心是“收集之后如何解释”。**

---

## 3. 为什么不是普通问卷 CRUD

如果只是普通问卷 CRUD，系统只需要：

```text
Questionnaire
Question
Option
AnswerSheet
```

再加几个接口：

```text
创建问卷
发布问卷
提交答案
查看结果
```

但 qs-server 的真实问题复杂得多。

### 3.1 需要量表规则

测评不是单纯看答案，而是要按量表规则解释答案。

例如：

- 哪些题属于哪个因子。
- 单题如何计分。
- 反向题如何处理。
- 因子分如何合成。
- 总分如何对应风险等级。
- 不同分数区间对应什么解读文案。

所以系统需要 `Scale` 边界，而不是只有 `Questionnaire`。

### 3.2 需要评估执行

用户提交答案之后，系统要创建一次评估实例：

```text
Assessment
```

Assessment 不是 AnswerSheet。

AnswerSheet 表示：

```text
用户提交了什么答案
```

Assessment 表示：

```text
系统基于这份答卷和某套量表规则，执行了一次测评
```

所以系统需要 `Evaluation` 边界。

### 3.3 需要报告生成

报告不是简单把答案展示出来，而是要包含：

- 分数。
- 因子。
- 风险等级。
- 解读内容。
- 建议。
- 可能的后续跟踪。

这意味着报告生成是一个独立链路，不是答卷表的一个字段。

### 3.4 需要运营统计

机构和医生不只看单份报告，还要看：

- 总测评数。
- 高风险数。
- 计划完成率。
- 医生负责的受试者数量。
- 某量表使用情况。
- 不同时间段趋势。
- 测评入口转化。

这意味着系统需要读侧统计聚合，而不是每次实时扫答卷表。

## 4. 参与角色

### 4.1 前台用户 / 家长 / 受试者

他们关心：

- 能不能顺利打开问卷。
- 能不能顺利提交答卷。
- 重复点击是否会出问题。
- 报告什么时候出来。
- 如果报告未生成，能否等待或稍后查询。

他们不关心：

- 后台如何建模。
- Outbox 怎么实现。
- worker 怎么消费。
- Redis lock 怎么抢。

系统要给他们稳定体验：

```text
提交被受理
状态可查询
报告可等待
失败可解释
```

---

### 4.2 医生 / 咨询师

他们关心：

- 哪些受试者完成了测评。
- 报告结果是什么。
- 风险等级如何。
- 是否需要跟进。
- 某个入口/计划下的完成情况。
- 自己负责对象的统计情况。

系统要给他们稳定视图：

```text
报告可信
统计可查
对象范围正确
权限边界清楚
```

---

### 4.3 机构 / 运营人员

他们关心：

- 量表如何配置。
- 问卷如何发布。
- 计划如何安排。
- 哪些任务未完成。
- 哪些人高风险。
- 统计数据是否准确。
- 系统是否稳定。

系统要给他们：

```text
配置能力
运营视图
统计报表
治理入口
```

---

### 4.4 技术团队 / 运维

他们关心：

- 提交高峰会不会打穿系统。
- 事件丢了怎么办。
- 报告没生成怎么查。
- Redis / MQ / IAM 不可用时系统如何表现。
- 权限问题怎么定位。
- 统计不准怎么修复。
- 怎么扩容。

系统要给他们：

```text
可观测
可重试
可降级
可排障
可演进
```

---

## 5. 业务问题一：问卷、量表、评估容易混在一起

最常见的误区是把系统建成：

```text
Questionnaire
  questions
  scoring rules
  report
  assessment status
```

也就是把问卷、量表规则、评估结果揉在一起。

这样短期写代码快，但长期会出问题：

| 混在一起的内容 | 问题 |
| -------------- | ---- |
| 问卷题目 + 量表规则 | 修改题目会影响医学解释 |
| 答卷 + Assessment | 提交成功和评估完成混淆 |
| Scale + Report | 规则定义和执行结果混淆 |
| 计分规则 + 答案校验 | 新题型和新计分方式互相影响 |
| 统计查询 + 写模型 | 报表需求反向污染业务表结构 |

所以 qs-server 的第一类业务问题是：

> **如何把收集载体、专业规则和评估结果拆开。**

对应设计：

```text
Survey
Scale
Evaluation
```

讲法：

> **Survey 管“填什么”，Scale 管“怎么算和怎么解释”，Evaluation 管“这一次测评执行后的结果”。**

---

## 6. 业务问题二：用户提交要快，但报告生成可能慢

用户点击提交时，期待的是：

```text
我提交成功了吗？
```

而不是：

```text
请等系统算完所有因子、风险、解读、报告、统计、通知。
```

但报告生成可能涉及：

- 加载问卷。
- 加载量表。
- 答案校验。
- 答卷持久化。
- 因子计分。
- 风险判断。
- 解读匹配。
- 报告持久化。
- 后续事件。
- 统计投影。

如果全部同步做，提交接口会变慢、变脆弱。

所以 qs-server 的第二类业务问题是：

> **如何让用户提交答卷时得到快速、确定的反馈，同时让复杂评估在后台可靠完成。**

对应设计：

```text
同步提交 AnswerSheet
异步执行 Evaluation
```

讲法：

> **提交阶段只承诺“答案可靠保存”，评估阶段异步承诺“报告最终生成或失败可查”。**

---

## 7. 业务问题三：前台流量不可控

前台用户可能：

- 同时提交。
- 重复点击。
- 网络失败后重试。
- 长时间等待报告。
- 高频刷新报告。
- 多个小程序入口同时访问。

如果前台直接打 apiserver，会让主业务服务同时承受：

- 前台提交高峰。
- 后台管理请求。
- internal gRPC。
- worker 回调。
- 统计查询。
- 调度任务。

所以 qs-server 的第三类业务问题是：

> **如何把前台流量挡在主业务服务前面，并按 submit / query / wait-report 分别保护。**

对应设计：

```text
collection-server
  -> RateLimit
  -> SubmitQueue
  -> SubmitGuard
  -> gRPC max-inflight
  -> submit-status
```

讲法：

> **collection-server 是前台 BFF 和保护层，不只是转发网关。**

---

## 8. 业务问题四：事件不能丢

答卷保存成功后，如果后续事件丢了，会出现：

```text
用户提交成功
但报告永远不生成
```

这在测评系统中非常严重，因为用户和医生看到的是：

```text
系统收到了答卷，但没有结果
```

所以 qs-server 的第四类业务问题是：

> **如何保证业务事实落库之后，对应的异步事件可靠出站。**

错误做法：

```text
保存 AnswerSheet 成功
直接 publish MQ
```

因为 DB 和 MQ 不是一个事务。

对应设计：

```text
Outbox
```

讲法：

> **Outbox 不是替代 MQ，而是解决 DB 与 MQ 双写不一致：业务事实和待发布事件同事务保存，再由 relay 可靠发布。**

---

## 9. 业务问题五：后台统计不能实时扫所有业务表

机构和医生要看统计：

- 总览。
- 医生统计。
- 入口统计。
- 问卷统计。
- 计划统计。
- 趋势。
- 高风险。
- 完成率。

这些统计如果每次实时从 Survey / Evaluation / Actor / Plan 中 join 和 group by，会导致：

- 查询慢。
- 压主库。
- 口径散。
- 难修复。
- 难缓存。
- 难解释统计不准。

所以 qs-server 的第五类业务问题是：

> **如何让统计查询稳定、口径集中、可重建、可缓存。**

对应设计：

```text
Statistics ReadService
BehaviorProjector
SyncService
QueryCache
Hotset
```

讲法：

> **写模型回答业务事实，统计读模型回答运营视角。**

---

## 10. 业务问题六：权限和隐私边界复杂

测评系统涉及敏感数据：

- 儿童/受试者信息。
- 心理/医学答卷。
- 风险等级。
- 解读报告。
- 医生/机构范围。
- 监护关系。

所以不能简单用：

```text
登录了就能看
```

也不能简单用：

```text
JWT roles 里有 admin 就放行
```

系统需要回答：

- 当前用户是谁？
- 属于哪个 tenant/org？
- 是否是某个受试者的监护人？
- 是否是机构内有效 operator？
- 是否有某个 resource/action 的 capability？
- 服务间调用是否可信？
- 本地角色是否只是投影？

所以 qs-server 的第六类业务问题是：

> **如何把身份、租户范围、业务授权、服务身份和本地角色投影分开。**

对应设计：

```text
IAMModule
Principal
TenantScope
AuthzSnapshot
CapabilityDecision
ServiceIdentity
OperatorRoleProjection
```

讲法：

> **JWT 只解决“你是谁”，AuthzSnapshot 才解决“你能做什么”。**

---

## 11. 业务问题七：系统要能演进

测评系统不是一次性项目，它会继续发展：

- 新量表。
- 新题型。
- 新报告模板。
- 新通知渠道。
- 新统计口径。
- 新用户端。
- 新运营平台。
- 未来可能加入 AI 解读。

如果前期边界不清，后续每加一个能力都会改一大片。

所以 qs-server 的第七类业务问题是：

> **如何让系统在业务继续扩展时，不被早期错误边界锁死。**

对应设计：

- DDD 限界上下文。
- 三进程协作。
- Outbox 事件链路。
- 读侧统计聚合。
- IAM SDK 边界。
- Runtime/Observability/Governance 分层。

讲法：

> **这个项目不是为了做一版功能，而是为了给测评业务后续演进留出稳定边界。**

---

## 12. 业务问题到技术方案的映射

| 业务问题 | 技术方案 |
| -------- | -------- |
| 问卷、量表、评估混在一起 | Survey / Scale / Evaluation 拆分 |
| 提交要快，评估可能慢 | 同步提交 AnswerSheet，异步 Evaluation |
| 前台流量不可控 | collection-server 保护层 |
| 重复提交 | SubmitGuard + durable idempotency |
| 提交高峰 | RateLimit + SubmitQueue |
| 事件不能丢 | Outbox |
| 报告生成慢 | worker + Evaluation Pipeline |
| 统计查询复杂 | ReadModel + Statistics ReadService |
| 统计口径变化 | SyncService + repair window |
| 行为事件难归因 | BehaviorProjector + pending retry |
| 权限和隐私复杂 | IAMModule + AuthzSnapshot + TenantScope |
| 服务间调用安全 | ServiceAuth + mTLS / ACL seam |
| 后续演进 | 模块化边界 + runtime governance |

## 15. 面试中的问题链

面试时可以更短：

```text
这个项目主要解决三个问题：

第一，业务建模问题。
问卷、量表、评估报告不能混在一起，所以我拆了 Survey、Scale、Evaluation。

第二，链路可靠性问题。
答卷提交要快，但报告生成慢，所以我用 collection-server 做入口保护，用 Outbox + worker 异步推进评估。

第三，运营和安全问题。
后台统计不能每次实时扫业务表，所以做了读侧统计聚合；权限也不是简单 JWT roles，而是基于 IAM AuthzSnapshot 做能力判断。
```

这个版本适合回答：

- “你这个项目有什么难点？”
- “你的项目体现了什么架构能力？”
- “你做过 DDD 或异步系统吗？”

## 16. 不要这样讲业务背景

### 16.1 不要说“这是一个问卷系统”

太低。

容易让听众以为只是：

- 表单。
- 增删改查。
- 保存答案。

更准确：

```text
问卷&量表测评系统
心理/医学测评后端
```

---

### 16.2 不要一开始就讲技术名词

不要开场就说：

```text
我用了 DDD、Outbox、Redis、gRPC、NSQ、Backpressure
```

这会让人感觉你在堆技术。

应该先说业务问题：

```text
为什么需要拆边界
为什么要异步
为什么不能丢事件
为什么统计不能实时算
```

技术方案自然跟出来。

---

### 16.3 不要把规划能力讲成现状

例如：

- 完整 ACL。
- 完整 operating 平台。
- 大规模多实例生产验证。
- AI 解读主链路。
- 完整 DLQ/replay 平台。

这些可以说成：

```text
下一阶段演进方向
```

不要说成当前已完成能力。

## 18. 本篇最终背诵版

> **这个项目的业务背景是心理/医学测评。用户提交的不是普通问卷答案，而是一份需要被量表规则解释的测评数据。系统要根据答卷计算分数、判断风险、生成报告，还要支持医生和机构做统计运营。**
>
> **所以它的难点不是 CRUD，而是边界和链路：问卷收集、量表规则、评估执行、报告生成、统计查询和权限控制的变化原因都不同；前台提交要快，但评估生成可能慢；答卷保存成功后，异步事件又不能丢。**
>
> **因此我把系统设计成 Survey、Scale、Evaluation 等限界上下文，并用 collection-server 保护前台入口，用 apiserver 保存主业务事实，用 worker 异步推进评估，用 Outbox 保证事件可靠出站，再用 Statistics 读侧聚合和 IAM AuthzSnapshot 支撑后台统计与安全授权。**
