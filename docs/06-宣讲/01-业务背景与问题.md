# 01-业务背景与问题

**本文回答**：`qs-server` 这个项目到底解决什么业务问题；为什么它不是普通问卷 CRUD，也不只是医学量表系统；为什么项目表达要从“问卷&量表测评系统”升级为“多解释模型测评平台”；如果从前台用户、医生/机构、运营后台、技术团队几个角度看，系统分别要解决哪些痛点；对外宣讲时如何把“业务背景 → 核心问题 → 技术方案”讲成一条自然的逻辑链。

---

## 30 秒结论

`qs-server` 面向的是心理、医学和人格测评场景。

这类系统的核心不是：

```text
把问卷答案存起来
```

而是：

```text
把一次答卷提交，稳定推进为一次可解释、可追踪、可统计、可治理的测评结果和解读报告。
```

因此，它不能只讲成普通问卷系统，也不能继续只讲成医学量表系统。

新版业务主线应该是：

```text
Survey
    负责问卷、题目、提交规格、答卷事实

Interpretation Model
    负责 ModelRef / Provider / Context / Registry 等统一接入协议

Concrete Models
    负责具体解释模型规则，例如 Scale / MBTI / BigFive

Evaluation
    负责 Assessment / EvaluationRun / EvaluationResult / InterpretReport / 失败重试 / 事件
```

一句话概括：

> **普通问卷的核心是“收集”，测评系统的核心是“收集之后如何解释、如何生成报告、如何追踪执行、如何统计运营、如何保护隐私”。**

---

## 1. 一句话背景

> **在心理、医学和人格测评场景中，用户填写的不是普通表单，而是一份会被解释模型处理的测评数据。系统要基于不同模型规则生成结构化结果和解读报告，并支撑医生、机构、运营后台做查询、统计、追踪和治理。**

所以 `qs-server` 的业务目标不是：

```text
问卷配置 + 答案提交 + 简单查询
```

而是：

```text
答卷可靠提交
  -> 测评执行可追踪
  -> 解释模型可扩展
  -> 报告生成可恢复
  -> 统计查询可治理
  -> 权限边界可审计
```

---

## 2. 业务场景是什么

这个系统面向的是测评业务。

典型流程是：

```text
机构配置测评模型
  -> 绑定问卷
  -> 用户 / 家长 / 受试者填写问卷
  -> 系统保存答卷
  -> 系统基于 ModelRef 找到具体解释模型
  -> Provider 加载 Context 并执行模型
  -> 系统保存 EvaluationResult
  -> 系统生成 InterpretReport
  -> 医生 / 机构查看报告和统计
  -> 后续形成测评计划、跟踪、复测和运营分析
```

这里的“测评模型”可以是：

```text
医学量表 Scale
MBTI
BigFive
职业兴趣测评
未来的 AI 增强解读模型
```

因此，旧的“问卷&量表测评系统”说法已经不够准确。

更准确的说法是：

```text
多解释模型测评平台
```

---

## 3. 和普通问卷系统的区别

| 普通问卷 | 多解释模型测评系统 |
| -------- | ------------------ |
| 关注题目和答案 | 关注答案背后的模型解释 |
| 提交后保存即可 | 提交后还要执行模型、生成结果和报告 |
| 报表通常是简单统计 | 统计涉及模型、受试者、医生、计划、报告、趋势 |
| 权限相对简单 | 涉及机构、医生、监护关系、受试者隐私和报告访问 |
| 结果通常是答案明细 | 结果是 EvaluationResult + InterpretReport |
| 通常同步完成 | 测评执行和报告生成适合异步推进 |
| 模型扩展有限 | 需要支持 Scale / MBTI / BigFive 等模型扩展 |

一句话：

> **普通问卷回答“用户填了什么”，测评系统还要回答“这些答案意味着什么”。**

---

## 4. 为什么不是普通问卷 CRUD

如果只是普通问卷 CRUD，系统只需要：

```text
Questionnaire
Question
Option
AnswerSheet
```

再加几个接口：

```text
创建问卷
发布问卷
提交答案
查看答案
```

但 `qs-server` 的真实问题复杂得多。

---

### 4.1 需要解释模型规则

测评不是单纯看答案，而是要按模型规则解释答案。

以 Scale 为例，系统要关心：

- 哪些题属于哪个因子。
- 单题如何计分。
- 反向题如何处理。
- 因子分如何合成。
- 总分如何对应风险等级。
- 不同分数区间对应什么解读文案。

以 MBTI 为例，系统要关心：

- 哪些题映射到哪个维度。
- 如何计算 E/I、S/N、T/F、J/P 倾向。
- 如何组合出 TypeCode。
- 不同 TypeCode 对应什么 TypeProfile。
- 报告模板如何生成展示内容。

所以系统需要的不是一个万能 Scale，而是：

```text
Interpretation Model 抽象
+
Concrete Models 具体规则
```

---

### 4.2 需要测评执行实例

用户提交答案之后，系统要创建一次测评执行：

```text
Assessment
```

Assessment 不是 AnswerSheet。

AnswerSheet 表示：

```text
用户提交了什么答案
```

Assessment 表示：

```text
系统基于这份答卷和某个 ModelRef，创建了一次可追踪的测评执行
```

所以系统需要 `Evaluation` 边界。

它负责：

```text
Assessment
EvaluationRun
EvaluationResult
InterpretReport
失败重试
事件出站
```

---

### 4.3 需要解释模型接入协议

如果 Evaluation 直接依赖 Scale，就会变成：

```text
AnswerSheet + MedicalScale -> Report
```

这在只支持医学量表时可行。

但支持 MBTI、BigFive 后，Evaluation 不应该不断增加：

```text
if model_type == scale
if model_type == mbti
if model_type == bigfive
```

更合理的是：

```text
ModelRef
  -> InterpretationRegistry.Resolve(model_type)
  -> Provider.LoadContext
  -> Provider.Evaluate
  -> EvaluationResult
  -> InterpretReport
```

这就是 `Interpretation Model` 的业务意义。

---

### 4.4 需要报告生成

报告不是简单把答案展示出来。

报告通常包含：

- 结构化分数。
- 因子或维度。
- 风险等级或类型编码。
- 解读内容。
- 建议。
- 图表。
- 报告快照。
- 模型版本引用。

这意味着报告生成是独立链路，不是答卷表的一个字段。

---

### 4.5 需要运营统计

机构和医生不只看单份报告，还要看：

- 总测评数。
- 高风险数。
- 计划完成率。
- 医生负责的受试者数量。
- 某模型使用情况。
- MBTI TypeCode 分布。
- 不同时间段趋势。
- 测评入口转化。

这意味着系统需要读侧统计聚合，而不是每次实时扫答卷表或 Assessment 表。

## 5. 参与角色

### 5.1 前台用户 / 家长 / 受试者

他们关心：

- 能不能顺利打开问卷。
- 能不能顺利提交答卷。
- 重复点击是否会出问题。
- 报告什么时候出来。
- 如果报告未生成，能否等待或稍后查询。

他们不关心：

- 后台如何建模。
- Outbox 怎么实现。
- Worker 怎么消费。
- Redis lock 怎么抢。
- Provider 怎么加载 Context。

系统要给他们稳定体验：

```text
提交被受理
状态可查询
报告可等待
失败可解释
```

---

### 5.2 医生 / 咨询师

他们关心：

- 哪些受试者完成了测评。
- 报告结果是什么。
- 风险等级或类型结果如何。
- 是否需要跟进。
- 某个入口 / 计划下的完成情况。
- 自己负责对象的统计情况。

系统要给他们稳定视图：

```text
报告可信
统计可查
对象范围正确
权限边界清楚
```

---

### 5.3 机构 / 运营人员

他们关心：

- 测评模型如何配置。
- 问卷如何发布。
- 模型如何绑定问卷。
- 计划如何安排。
- 哪些任务未完成。
- 哪些人高风险。
- MBTI 等模型分布情况如何。
- 统计数据是否准确。
- 系统是否稳定。

系统要给他们：

```text
配置能力
运营视图
统计报表
治理入口
```

---

### 5.4 技术团队 / 运维

他们关心：

- 提交高峰会不会打穿系统。
- 事件丢了怎么办。
- 报告没生成怎么查。
- Provider 执行失败怎么定位。
- Redis / MQ / IAM 不可用时系统如何表现。
- 权限问题怎么定位。
- 统计不准怎么修复。
- 怎么扩容。

系统要给他们：

```text
可观测
可重试
可降级
可排障
可演进
```

## 6. 业务问题一：问卷、模型规则、测评执行容易混在一起

最常见的误区是把系统建成：

```text
Questionnaire
  questions
  scoring rules
  report
  assessment status
```

也就是把问卷、规则、执行结果揉在一起。

这样短期写代码快，但长期会出问题：

| 混在一起的内容 | 问题 |
| -------------- | ---- |
| 问卷题目 + 解释模型规则 | 修改题型可能影响模型执行 |
| AnswerSheet + Assessment | 提交成功和测评完成混淆 |
| Scale + MBTI | MedicalScale 被 TypeCode / TypeProfile 污染 |
| 规则定义 + 报告事实 | 模型发布和历史报告追溯混淆 |
| 统计查询 + 写模型 | 报表需求反向污染业务表结构 |

所以第一类业务问题是：

> **如何把作答事实、解释模型规则、测评执行生命周期和报告事实拆开。**

对应设计：

```text
Survey
Interpretation Model
Concrete Models
Evaluation
```

讲法：

> **Survey 管“用户填了什么”，Interpretation Model 管“模型如何统一接入”，Concrete Models 管“具体规则是什么”，Evaluation 管“这一次测评如何执行、失败、重试和生成报告”。**

---

## 7. 业务问题二：Scale 不能继续作为所有解释能力中心

旧讲法是：

```text
Survey 管填什么
Scale 管怎么算和怎么解释
Evaluation 管结果
```

这个说法在只支持医学量表时够用。

但支持 MBTI 后会有问题。

Scale 的自然语义是：

```text
MedicalScale
Factor
ScoringSpec
RiskLevel
InterpretationRule
```

MBTI 的自然语义是：

```text
Dimension
Preference
TypeCode
TypeProfile
ReportTemplate
```

如果把 MBTI 塞进 Scale，就会出现：

```text
Factor 被滥用为 Dimension；
RiskLevel 被滥用为 TypeCode；
InterpretationRule 被滥用为 TypeProfile；
MedicalScale 变成万能解释模型。
```

所以第二类业务问题是：

> **如何让 Scale、MBTI、BigFive 作为同级具体模型接入系统，而不是互相污染。**

对应设计：

```text
ScaleProvider implements InterpretationProvider
MBTIProvider implements InterpretationProvider
BigFiveProvider implements InterpretationProvider
```

讲法：

> **Scale 是医学量表解释模型，MBTI 是另一个解释模型；二者应通过统一 Provider 协议同级接入 Evaluation。**

---

## 8. 业务问题三：用户提交要快，但报告生成可能慢

用户点击提交时，期待的是：

```text
我提交成功了吗？
```

而不是：

```text
请等系统算完所有因子、类型、风险、解读、报告、统计、通知。
```

但报告生成可能涉及：

- 加载问卷。
- 保存答卷。
- 创建 Assessment。
- 加载 ModelRef。
- 解析 Provider。
- 加载 Context。
- 执行解释模型。
- 保存 EvaluationResult。
- 生成 InterpretReport。
- 发布事件。
- 统计投影。
- wait-report 唤醒。

如果全部同步做，提交接口会变慢、变脆弱。

所以第三类业务问题是：

> **如何让用户提交答卷时得到快速、确定的反馈，同时让复杂测评执行在后台可靠完成。**

对应设计：

```text
同步提交 AnswerSheet
异步推进 Assessment / Interpretation / Report
```

讲法：

> **提交阶段只承诺“答案可靠保存”，测评执行阶段异步承诺“报告最终生成或失败可查”。**

---

## 9. 业务问题四：前台流量不可控

前台用户可能：

- 同时提交。
- 重复点击。
- 网络失败后重试。
- 长时间等待报告。
- 高频刷新报告。
- 多个小程序入口同时访问。

如果前台直接打 apiserver，会让主业务服务同时承受：

- 前台提交高峰。
- 后台管理请求。
- internal gRPC。
- worker 回调。
- 统计查询。
- 调度任务。

所以第四类业务问题是：

> **如何把前台流量挡在主业务服务前面，并按 submit / query / wait-report 分别保护。**

对应设计：

```text
collection-server
  -> RateLimit
  -> SubmitQueue
  -> SubmitGuard
  -> gRPC max-inflight
  -> submit-status / wait-report
```

讲法：

> **collection-server 是前台 BFF 和保护层，不只是转发网关。**

---

## 10. 业务问题五：事件不能丢

答卷保存成功后，如果后续事件丢了，会出现：

```text
用户提交成功
但 Assessment 永远不创建
报告永远不生成
```

这在测评系统中非常严重，因为用户和医生看到的是：

```text
系统收到了答卷，但没有结果
```

所以第五类业务问题是：

> **如何保证业务事实落库之后，对应的异步事件可靠出站。**

错误做法：

```text
保存 AnswerSheet 成功
直接 publish MQ
```

因为 DB 和 MQ 不是一个事务。

对应设计：

```text
Outbox
```

讲法：

> **Outbox 不是替代 MQ，而是解决 DB 与 MQ 双写不一致：业务事实和待发布事件同事务保存，再由 relay 可靠发布。**

---

## 11. 业务问题六：后台统计不能实时扫所有业务表

机构和医生要看统计：

- 总览。
- 医生统计。
- 入口统计。
- 问卷统计。
- 模型统计。
- 计划统计。
- 趋势。
- 高风险。
- 完成率。
- MBTI TypeCode 分布。

这些统计如果每次实时从 Survey / Evaluation / Actor / Plan / Model 中 join 和 group by，会导致：

- 查询慢。
- 压主库。
- 口径散。
- 难修复。
- 难缓存。
- 难解释统计不准。

所以第六类业务问题是：

> **如何让统计查询稳定、口径集中、可重建、可缓存。**

对应设计：

```text
Statistics ReadService
BehaviorProjector
SyncService
QueryCache
Hotset
```

讲法：

> **写模型回答业务事实，统计读模型回答运营视角。**

---

## 12. 业务问题七：权限和隐私边界复杂

测评系统涉及敏感数据：

- 儿童 / 受试者信息。
- 心理 / 医学答卷。
- 风险等级。
- MBTI 等人格类型结果。
- 解读报告。
- 医生 / 机构范围。
- 监护关系。

所以不能简单用：

```text
登录了就能看
```

也不能简单用：

```text
JWT roles 里有 admin 就放行
```

系统需要回答：

- 当前用户是谁？
- 属于哪个 tenant / org？
- 是否是某个受试者的监护人？
- 是否是机构内有效 operator？
- 是否有某个 resource / action 的 capability？
- 服务间调用是否可信？
- 本地角色是否只是投影？
- 能管理模型规则，是否能查看用户报告？

所以第七类业务问题是：

> **如何把身份、TenantDomain/OrgScope、业务授权、服务身份、本地角色投影、模型管理权限和报告访问权限分开。**

对应设计：

```text
IAMModule
Principal
OrgScope
AuthzSnapshot
CapabilityDecision
ServiceIdentity
OperatorRoleProjection
```

讲法：

> **JWT 只解决“你是谁”，AuthzSnapshot 才解决“你能做什么”。**

---

## 13. 业务问题八：系统要能继续演进

测评系统不是一次性项目，它会继续发展：

- 新量表。
- 新题型。
- 新解释模型。
- 新报告模板。
- 新通知渠道。
- 新统计口径。
- 新用户端。
- 新运营平台。
- 未来可能加入 AI 解读。

如果前期边界不清，后续每加一个能力都会改一大片。

所以第八类业务问题是：

> **如何让系统在业务继续扩展时，不被早期错误边界锁死。**

对应设计：

- DDD 限界上下文。
- Interpretation Model 抽象。
- 三进程协作。
- Outbox 事件链路。
- 读侧统计聚合。
- IAM SDK 边界。
- Runtime / Observability / Governance 分层。

讲法：

> **这个项目不是为了做一版功能，而是为了给测评业务后续从 Scale 扩展到 MBTI、BigFive 和更多模型留出稳定边界。**

---

## 14. 业务问题到技术方案的映射

| 业务问题 | 技术方案 |
| -------- | -------- |
| 问卷、模型规则、评估混在一起 | Survey / Interpretation Model / Concrete Models / Evaluation 拆分 |
| Scale 不能承载所有解释能力 | Provider 抽象，Scale / MBTI / BigFive 同级接入 |
| 提交要快，报告可能慢 | 同步提交 AnswerSheet，异步测评执行 |
| 前台流量不可控 | collection-server 保护层 |
| 重复提交 | SubmitGuard + durable idempotency |
| 提交高峰 | RateLimit + SubmitQueue |
| 事件不能丢 | Outbox |
| 报告生成慢 | worker + Evaluation Engine |
| 统计查询复杂 | ReadModel + Statistics ReadService |
| 统计口径变化 | SyncService + repair window |
| 行为事件难归因 | BehaviorProjector + pending retry |
| 权限和隐私复杂 | IAMModule + AuthzSnapshot + OrgScope |
| 服务间调用安全 | ServiceAuth + mTLS / ACL seam |
| 新解释模型接入 | ModelRef / Provider / Context / Registry |
| 执行慢或失败难排查 | Metrics + Governance endpoint + Runbook |
| 后续演进 | 模块化边界 + runtime governance |

## 15. 对外讲解时的业务背景开场

推荐这样开场：

> **这个项目的业务背景是心理、医学和人格测评。用户提交的不是普通问卷答案，而是一份需要被解释模型处理的测评数据。系统要基于 Scale、MBTI、BigFive 等不同模型生成结构化结果和解读报告，同时后台还要按机构、医生、计划、受试者和模型类型做统计运营。**
>
> **所以这个项目的难点不在于“怎么保存问卷答案”，而在于：如何把作答事实、解释模型规则、测评执行生命周期、报告事实、统计查询和安全权限这些变化原因不同的能力拆开，并且在高峰提交和异步执行场景下保证链路可靠。**

这段是本篇最适合背诵的版本。

## 17. 面试中的问题链

面试时可以更短：

```text
这个项目主要解决四个问题：

第一，业务建模问题。
问卷、解释模型规则、测评执行和报告事实不能混在一起，所以我拆了 Survey、Interpretation Model、Concrete Models、Evaluation。

第二，链路可靠性问题。
答卷提交要快，但测评执行和报告生成慢，所以我用 collection-server 做入口保护，用 Outbox + worker 异步推进 Assessment / Interpretation / Report。

第三，扩展性问题。
Scale 是医学量表模型，MBTI、BigFive 与它同级，所以通过 ModelRef / Provider / Context / Registry 接入 Evaluation。

第四，运营和安全问题。
后台统计不能每次实时扫业务表，所以做读侧统计聚合；权限也不是简单 JWT roles，而是基于 IAM AuthzSnapshot 做能力判断。
```

这个版本适合回答：

- “你这个项目有什么难点？”
- “你的项目体现了什么架构能力？”
- “你做过 DDD 或异步系统吗？”
- “你怎么支持未来新增 MBTI / BigFive 这类模型？”

## 18. 不要这样讲业务背景

### 18.1 不要说“这是一个问卷系统”

太低。

容易让听众以为只是：

- 表单。
- 增删改查。
- 保存答案。

更准确：

```text
心理、医学和人格测评后端
多解释模型测评平台
```

---

### 18.2 不要说“这是一个量表系统”

不够准确。

容易让听众以为：

```text
Scale 是所有解释能力中心
```

更准确：

```text
Scale 是医学量表模型；
MBTI、BigFive 与 Scale 同级；
Interpretation Model 才是接入抽象。
```

---

### 18.3 不要一开始就讲技术名词

不要开场就说：

```text
我用了 DDD、Outbox、Redis、gRPC、NSQ、Backpressure
```

这会让人感觉你在堆技术。

应该先说业务问题：

```text
为什么需要拆边界
为什么要异步
为什么不能丢事件
为什么统计不能实时算
为什么权限不能只看 JWT roles
为什么新模型不能污染旧模型
```

技术方案自然跟出来。

---

### 18.4 不要把规划能力讲成现状

例如：

- MBTIProvider 完整落地。
- 完整 ACL。
- 完整 operating 平台。
- 大规模多实例生产验证。
- AI 解读主链路。
- 完整 DLQ/replay 平台。

这些可以说成：

```text
下一阶段演进方向
```

不要说成当前已完成能力。

## 20. 本篇最终背诵版

> **这个项目的业务背景是心理、医学和人格测评。用户提交的不是普通问卷答案，而是一份需要被解释模型处理的测评数据。系统要根据 Scale、MBTI、BigFive 等不同模型生成结构化结果和解读报告，还要支持医生和机构做统计运营。**
>
> **所以它的难点不是 CRUD，而是边界和链路：作答事实、解释模型规则、测评执行、报告生成、统计查询和权限控制的变化原因都不同；前台提交要快，但测评执行和报告生成可能慢；答卷保存成功后，异步事件又不能丢；后续还要支持新解释模型扩展。**
>
> **因此我把系统设计成 Survey、Interpretation Model、Concrete Models、Evaluation 等限界上下文，并用 collection-server 保护前台入口，用 apiserver 保存主业务事实，用 worker 异步推进 Assessment / Interpretation / Report，用 Outbox 保证事件可靠出站，再用 Statistics 读侧聚合和 IAM AuthzSnapshot 支撑后台统计与安全授权。**
