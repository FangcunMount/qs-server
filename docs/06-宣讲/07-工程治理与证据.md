# 工程治理与证据

**本文回答**：这篇文档负责把 `qs-server` 已经落地的工程治理手段和可核验证据收拢成一页，重点说明哪些保护措施已经真实生效、哪些地方证据还不够硬、哪些风险必须主动承认，而不是把“工程化”讲成空泛口号。

## 30 秒结论

如果只看一屏，先看下面这张表：

| 维度 | 结论 |
| ---- | ---- |
| 最该讲的四类证据 | 接入削峰、消费侧并发、文档/契约校验链、主动承认的风险边界 |
| 已落地保护 | SubmitQueue、保护层与读侧治理、状态机约束、`worker.concurrency` 生效路径、`docs-verify` 校验链 |
| 待补证据 | 具体消息重试次数和退避策略，当前不能讲成固定数字 |
| 规划改造 | 出站一致性 outbox 和答卷持久业务幂等仍未完成 |
| 最适合的讲法 | 先说“哪些已经靠代码兜住”，再说“哪些地方我们明确知道还没闭环” |
| 最忌讳的讲法 | 不要把没有统一证据的重试、压测、SLO、最终一致性讲成既成事实 |

## 先讲结论

`qs-server` 的工程治理亮点，不是“组件很多”，而是**有真实生效的保护手段**，并且仓库里已经有一部分可核验的工程证据。最值得讲的四类证据是：接入侧削峰、消费侧并发、文档与契约校验链、以及当前明确承认的风险边界。

## 重点速查

- 先讲已实现：SubmitQueue、并发控制、状态机约束、文档校验链。
- 再讲待补证据：别把消息重试次数说成固定配置承诺。
- 最后讲规划改造：outbox 和持久业务幂等是当前最值得主动承认的缺口。

## 主动讲的 3 点

1. `collection-server` 的 SubmitQueue 不是概念图，当前代码已经能返回 `202 / 429`，它是接入侧真实生效的保护手段。
2. worker 并发我不是按 yaml 字面理解，而是核对到了 `worker.concurrency -> MaxInFlight` 的真实生效路径。
3. 文档和契约已经不是纯说明稿，现行文档现在会经过 `docs-hygiene` 和 `docs-verify` 这条校验链。

## 被追问再展开的 3 点

1. 状态机保护：`Assessment` 和评估引擎如何阻止重复成功执行。
2. 观测与错误：当前最硬的证据仍然是结构化日志、handler 错误返回和 `Nack` 路径。
3. 风险边界：outbox、持久业务幂等、重试次数证据不足这三块为什么必须主动承认。

## 你可以怎么讲工程治理

1. **接入侧**：我没有把“异步”只理解成 MQ。用户请求先有 BFF 保护层，必要时再进 SubmitQueue 做进程内削峰。
2. **消费侧**：我没有把 `events.yaml` 里的所有字段都当成真相，而是核对了 `worker.concurrency -> MaxInFlight` 的真实生效路径。
3. **文档侧**：我把文档从“设计稿”往“可验证资产”推进了，至少现行文档已经有链接、锚点、章节编号检查，并并入 `docs-verify`。
4. **风险侧**：我不会把没有 outbox、没有持久业务幂等的地方包装成“已经保证一致性”。

## 观测与错误口径

- **日志**：当前最硬的证据还是结构化日志和 handler / pipeline 的错误返回路径。证据见 [assessment_handler.go](../../internal/worker/handlers/assessment_handler.go)、[answersheet_handler.go](../../internal/worker/handlers/answersheet_handler.go)。
- **错误分层**：worker 失败走 `Nack`，pipeline 任一步失败中断链路，测评可进入 `failed` 状态。证据见 [../05-专题分析/02-异步评估链路：从答卷提交到报告生成.md](../05-专题分析/02-异步评估链路：从答卷提交到报告生成.md)。
- **别说过头**：仓库里目前没有成体系的压测报告、仪表盘截图或统一 SLO 文档，所以容量数字和告警覆盖面只能谨慎讲。

## 宣讲时值得主动补的一句

工程治理最加分的点，不是把所有问题都解决了，而是**知道哪些地方已经靠代码兜住了，哪些地方还停留在规划改造**。这个仓库现在已经具备前者，也保留了后者。
