# 关键决策卡

**本文回答**：这篇文档把 `qs-server` 最核心的三张架构决策卡收拢到一起，方便在公开介绍或深入追问时，快速说明“为什么这样设计、没有选什么、代价是什么、证据在哪里”。

## 30 秒结论

如果只看一屏，先看下面这张表：

| 决策 | 核心回答 | 主要代价 |
| ---- | -------- | -------- |
| 为什么 `survey / scale / evaluation` 分离 | 为了把采集、规则、结果三类变化源拆开，避免模型失真 | 跨域协作成本更高，阅读门槛高于简单 CRUD |
| 为什么答卷同步提交，但评估走异步 | 为了控制提交延迟，把重计算和报告生成从请求线程里拆出去 | 必须正视幂等、重试、状态查询和最终一致性问题 |
| 为什么主业务集中在 `apiserver` | 为了维持单一写模型，避免 BFF、worker 和主服务各自维护不同业务规则 | `apiserver` 会成为架构重心，内部边界必须讲得很清楚 |

## 决策卡 1：为什么 `survey / scale / evaluation` 分离

- **15 秒回答**：因为采集、规则、结果是三类不同变化源，必须拆开，模型才不会在一个大模块里失真。
- **问题**：为什么不做一个“大问卷模块”，把题目、规则、答卷、测评结果全放一起？
- **方案**：把采集、规则、评估拆成三个核心上下文：`survey`、`scale`、`evaluation`。
- **为什么**：问卷结构、量表规则、测评结果的变化频率和业务职责不同。答卷提交追求稳定接入，量表追求规则权威，测评追求状态机和结果产出。
- **设计模式**：Survey 使用校验策略和 durable submit；Scale 使用计分策略和值对象；Evaluation 使用状态机、职责链和 Report Builder。
- **替代方案**：一个统一模块。没有选，因为模型会同时背“采集”“规则”“结果”三类责任，边界会很快失真。
- **代价**：跨域要通过引用值对象、事件和 internal gRPC 协作，阅读门槛比 CRUD 项目高。
- **别说过头**：不要讲成“已经拆成微服务”，也不要顺手把量表版本治理扩写成已完全闭环。
- **证据**：[../05-专题分析/01-测评业务模型：survey、scale、evaluation 为什么分离.md](../05-专题分析/01-测评业务模型：survey、scale、evaluation%20为什么分离.md)、[05-DDD 领域地图与模块协作.md](./05-DDD%20领域地图与模块协作.md)

## 决策卡 2：为什么答卷同步提交，但评估走异步

- **15 秒回答**：因为提交追求稳定受理，评估链路太重，应该从请求线程里拆出去，但主状态仍要留在主服务里。
- **问题**：为什么不在用户提交答卷时直接同步算完测评和报告？
- **方案**：同步阶段只做接入校验、落主状态和发事件，计分、建测评、评估和报告走 `MQ + worker + internal gRPC + pipeline`。
- **为什么**：这样可以把请求线程从重计算链里解放出来，让提交延迟可控，也便于把评估链路做成可重试、可扩展的后台流程。
- **设计模式**：Outbox 处理出站一致性，worker adapter 调用 internal gRPC，Evaluation pipeline 用职责链执行评估。
- **替代方案**：全部同步。没有选，因为一旦量表、规则、报告构建变重，用户请求会直接背后端抖动和数据库放大。
- **代价**：系统必须面对幂等、重试、状态查询和最终一致性问题，不能再用“一个事务搞定一切”的心智模型。
- **别说过头**：不要把“用了 MQ”讲成一致性已经天然解决，也不要把重试能力说成全都标准化了。
- **证据**：[../05-专题分析/02-异步评估链路：从答卷提交到报告生成.md](../05-专题分析/02-异步评估链路：从答卷提交到报告生成.md)、[03-主链路 1：提交答卷.md](./03-主链路%201：提交答卷.md)、[04-主链路 2：异步评估流水线.md](./04-主链路%202：异步评估流水线.md)

## 决策卡 3：为什么主业务集中在 `apiserver`，`collection` 是 BFF，`worker` 只做驱动

- **15 秒回答**：因为要维持单一写模型，BFF 和 worker 可以参与接入和驱动，但不应该各自维护一套主业务状态。
- **问题**：为什么不让 `collection-server` 或 `qs-worker` 自己各自维护一套主业务逻辑？
- **方案**：以 `qs-apiserver` 作为领域模型和主业务中心；`collection-server` 负责前台接入与保护层；`qs-worker` 负责订阅消息并调用 internal gRPC。
- **为什么**：这样能保持单一写模型，避免 BFF、worker 和主服务各自演化出不同版本的业务规则。
- **设计模式**：collection 使用 rate-limit/queue/idempotency guard，worker 使用 adapter + handler registry，apiserver 保持 application service 和 domain model 权威。
- **替代方案**：把更多业务下沉到 worker 或 collection。没有选，因为一旦状态机和规则分散，幂等、查询和契约维护成本都会明显上升。
- **代价**：`apiserver` 会成为架构重心，必须把模块边界、internal 接口和容器装配做清楚。
- **别说过头**：不要讲成 `collection` 和 `worker` “完全没有业务逻辑”，更准确的说法是它们不拥有主业务写真值。
- **证据**：[../README.md](../README.md)、[../00-总览/02-代码组织与边界.md](../00-总览/02-代码组织与边界.md)、[../01-运行时](../01-运行时/)

## 宣讲时建议强调

- 三张卡不要背成抽象原则，要始终回到真实链路：答卷提交、异步评估、主业务中心。
- 每张卡都可以顺带讲一个代价，听众通常更愿意相信“知道自己付出了什么代价”的设计。
