# qs-server 系统技术简报

## 1. 一句话定位

qs-server 是把问卷作答、测评模型、评估执行和解释报告拆成独立事实边界，并用可靠异步链支撑多测评模型的后端系统。

## 2. 为什么不是普通问卷 CRUD

系统需要同时处理：

- 问卷和模型版本变化时，历史评估仍可追溯；
- 突发提交不压垮主业务存储；
- 评估失败、报告失败可以独立重试；
- 医学量表、人格类型、行为能力等模型共享执行框架又保留各自语义；
- 前台能等待结果，但临时连接失败不丢业务事实；
- 统计查询不把复杂聚合压力推回写侧。

## 3. 三进程分工

```text
collection-server：前台保护与 BFF
qs-apiserver：业务事实与组合根
qs-worker：可靠事件的异步驱动
```

这是职责拆分，不是三个独立业务中心。

## 4. 领域主线

```text
Survey：作答事实
ModelCatalog：可发布模型资产
Evaluation：执行与 Outcome
Interpretation：报告生命周期
Actor / Plan / Statistics：参与者、编排与读侧支撑
```

关键价值在于每个边界都能回答“谁拥有状态、谁能修改、失败如何恢复”。

## 5. 工程机制

- UoW + Outbox：业务写入和可靠事件意图一致提交；
- MQ + worker：异步解耦与可重试驱动；
- Redis signal：降低在线等待延迟，丢失时回查事实；
- RateLimit + Queue + Guard + Backpressure：分层治理突发与重复；
- cache capability：降低高频读压力，并保留失效、预热和降级边界；
- projection + checkpoint + reconcile：支撑可恢复统计读侧。
