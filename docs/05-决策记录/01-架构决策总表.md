# 架构决策总表

## 1. 三进程但单一业务事实中心

结论：collection 承接前台保护，worker 驱动异步任务，apiserver 持有业务状态机与持久化。

约束：collection/worker 不直接写主业务 repository。证据见 [系统地图](../00-总览/03-系统地图.md) 与三个进程入口。

重新评估条件：某个边界进程开始独立拥有数据、事务和业务生命周期。

## 2. 同步保存答卷，异步执行评估与报告

结论：同步链只保证 AnswerSheet 事实被接受；Evaluation 和 Interpretation 由可靠事件推进。

约束：客户端必须处理 accepted/processing/final 等状态，不能把提交成功等同于报告完成。证据见 [核心业务链路](../00-总览/05-核心业务链路.md) 和 `configs/events.yaml`。

重新评估条件：业务需要同步强一致返回评估结果，且容量与失败语义允许。

## 3. 可编辑模型与运行时发布快照分离

结论：草稿用于 authoring，已发布快照用于运行时；运行链不能读取可变草稿。

约束：发布、重复发布、从已发布版本派生新草稿和绑定都必须保持显式状态语义。证据见 [ModelCatalog](../02-业务模块/20-model-catalog/README.md)。

重新评估条件：能用同一物理存储结构证明不可变版本、原子发布和运行隔离，且迁移收益明确。

## 4. Evaluation 与 Interpretation 分离

结论：Evaluation 提交机制中立 Outcome；Interpretation 独立生成、重试、存储和查询报告。

约束：Interpretation 只读 Outcome，不推进 Evaluation 状态；Evaluation 不拥有报告成品。证据见两个模块的 `domain/doc.go` 与 canonical 文档。

重新评估条件：报告完全退化为无状态同步格式化，且不再需要独立重试、持久化和查询生命周期。

## 5. Outbox、MQ、Signal 分工

结论：Outbox 保证可靠出站，MQ 负责异步传输/消费，Redis signal 只做临时唤醒。

约束：signal 丢失必须可由查询、TTL 或重试兜底；可靠业务事实不得只发 Pub/Sub。证据见 [Event](../03-基础设施/event/README.md)、`configs/events.yaml` 和 `configs/signals.yaml`。

重新评估条件：底层传输提供不同但同等可证明的投递与恢复语义。

## 6. Statistics 作为可恢复读侧

结论：统计通过投影、扫描、checkpoint 与 reconcile 构建，允许明确的最终一致性。

约束：读侧不能反向拥有主业务事实；可重建也不代表存储可直接删除。证据见 [Statistics](../02-业务模块/70-statistics/README.md)。

重新评估条件：查询可以直接从主事实高效、稳定生成且不再需要独立口径和缓存。
