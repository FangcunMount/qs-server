# 代码组织与边界

## 1. 结论

先按进程找入口，再按 `domain -> application -> port/infra -> transport -> container` 找责任链。不要从旧文档中的包名推断当前模块。

## 2. 目录地图

| 目录 | 责任 |
| --- | --- |
| `cmd/` | 三个可执行进程入口 |
| `internal/apiserver/domain/` | 领域模型、规则和领域事件 |
| `internal/apiserver/application/` | 用例编排、事务、repository/port 协作 |
| `internal/apiserver/port/` | 跨层接口与读模型契约 |
| `internal/apiserver/infra/` | MySQL、Mongo、Redis、外部能力适配 |
| `internal/apiserver/transport/` | REST 与 gRPC 入站适配 |
| `internal/apiserver/container/` | 组合根、模块装配与生命周期 |
| `internal/collection-server/` | 前台 BFF 与保护链 |
| `internal/worker/` | MQ 消费和异步驱动 |
| `internal/pkg/`、`pkg/` | 跨进程基础能力与公开基础包 |

## 3. 当前模块语言

`modelcatalog` 是当前代码模块名；“Assessment Model”是业务展示名。`interpretation` 拥有报告生成和报告生命周期，`evaluation` 只拥有机制中立的执行与 Outcome。具体边界分别见 [ModelCatalog](../02-业务模块/20-model-catalog/README.md)、[Evaluation](../02-业务模块/30-evaluation/README.md) 和 [Interpretation](../02-业务模块/40-interpretation/README.md)。

## 4. 架构护栏

`internal/pkg/architecture` 中的测试约束缓存、事件、数据访问、韧性所有权、旧路径和退役表名。结构性重构时应把这些测试和模块定向测试一起作为事实证据。
