# 源码事实矩阵

**本文回答**：`qs-server` 文档体系中哪些关键事实已经能从源码、机器契约或配置中得到证明；每个事实应归属到哪篇文档维护；哪些事实容易被误写、重复写或讲成规划。本文是后续文档重建和评审的事实索引。

> 本文不是业务说明书，也不是完整源码目录。它的作用是给文档维护者提供一张“事实 -> 证据 -> 归属文档”的矩阵。任何文档如果和本文、源码、契约冲突，以源码和机器契约为准。

## 使用方式

### 读者如何使用

| 场景 | 用法 |
| ---- | ---- |
| 第一次读仓库 | 先读 `01-系统地图.md`、`02-代码组织与边界.md`、`03-核心业务链路.md`，再用本文核对证据 |
| 重写文档 | 先找本文对应事实，确认归属文档和源码锚点 |
| 发现文档冲突 | 用本文定位事实源，再决定保留哪一处讲法 |
| 新增能力 | 在对应矩阵中补一行“事实 + 状态 + 锚点 + 归属文档” |
| 宣讲材料 | 宣讲层只能引用本文已标为“已实现”的事实，不能自己造事实 |

### 状态标签

| 状态 | 含义 |
| ---- | ---- |
| 已实现 | 源码、配置或机器契约能直接证明 |
| 待补证据 | 方向存在，但本文未完成源码级确认，不能讲成标准能力 |
| 规划改造 | 当前不属于已实现能力，只能作为后续方向 |
| 历史资料 | 只在 `_archive` 或旧设计里存在，不作为现行事实 |

## 19. 维护流程

### 新增业务能力

```text
领域模型 / 应用服务
  -> 契约或配置
  -> 测试
  -> 对应业务模块文档
  -> 如影响主链路，更新 03-核心业务链路.md
  -> 如新增事实，更新本文
  -> make docs-hygiene
```

### 新增事件

```text
事件类型 / payload
  -> configs/events.yaml
  -> publisher / outbox
  -> worker handler registry
  -> handler 测试
  -> docs/03-基础设施/event/
  -> 本文事件矩阵
  -> make docs-hygiene
```

### 新增 REST/gRPC 接口

```text
handler / service
  -> OpenAPI 或 proto
  -> client / registry
  -> contract tests
  -> docs/04-接口与运维/
  -> 本文接口矩阵
  -> make docs-verify
```

### 新增基础设施能力

```text
port / adapter / config
  -> runtime/container wiring
  -> tests
  -> docs/03-基础设施/对应 plane
  -> 本文对应矩阵
  -> make docs-hygiene
```
