# 代码组织与边界

本文档说明仓库当前如何按运行时、模块和基础设施组织代码，避免继续按照历史设计稿理解目录。

## 职责

当前代码组织有两个维度：

- 运行时维度：`cmd/*` 和 `internal/{apiserver,collection-server,worker}`
- 模块维度：主要集中在 `internal/apiserver/{domain,application,infra,interface,container}`

仓库关键目录：

```text
cmd/
  qs-apiserver/
  collection-server/
  qs-worker/

internal/
  apiserver/
    application/
    domain/
    infra/
    interface/
    container/
  collection-server/
    application/
    infra/
    interface/
    container/
  worker/
    application/
    handlers/
    infra/
    container/
  pkg/
```

## 核心边界

### `apiserver`

`apiserver` 是主业务边界，职责包括：

- 组装领域模块
- 暴露后台 REST API
- 暴露对 `collection-server` 和 `worker` 使用的 gRPC / internal gRPC
- 管理 MySQL、MongoDB、Redis、事件发布器

### `collection-server`

`collection-server` 是收集端边界，职责包括：

- 对前端暴露更轻量的 REST 接口
- 做 JWT 身份解析、监护关系校验、请求排队和限流
- 通过 gRPC 调用 `apiserver`

不负责：

- 独立持久化业务主数据
- 独立实现领域模块

### `worker`

`worker` 是后台异步边界，职责包括：

- 根据 [configs/events.yaml](../../configs/events.yaml) 订阅 Topic 和事件
- 将事件分发给处理器
- 通过内部 gRPC 回调 `apiserver` 完成评估、统计、标签等后台动作

不负责：

- 自己维护完整业务仓储
- 独立定义第二套业务流程

## 误区修正

- 仓库当前不存在 `internal/domain/*` 这一层级，旧文档中以该结构为中心的描述已经失效。
- `collection-server` 与 `apiserver` 不是对等双主服务，前者依赖后者。
- `worker` 与业务模块不是一一对应关系，它更接近“事件处理运行时”。
- `api/rest/*.yaml` 才是当前仓库内的 OpenAPI 契约落点，不应继续引用旧路径说明。
