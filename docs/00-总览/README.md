# 总览（00）

**本文回答**：`00-总览` 这一组文档负责先把 `qs-server` 的整体地图、代码边界、主业务链路和本地开发入口讲清楚，让读者在进入运行时、业务模块和专题之前先建立统一坐标系。

## 30 秒结论

如果只看一屏，先看下面这张表：

| 维度 | 结论 |
| ---- | ---- |
| 本组作用 | 先建立系统地图、代码边界、主链路和本地跑通入口，不直接展开模块内部细节 |
| 阅读顺序 | 最顺的顺序是 `系统地图 -> 代码边界 -> 核心业务链路 -> 本地开发与配置约定` |
| 最重要的认识 | `qs-server` 不是单体进程，而是三进程协作系统；主状态、异步链路、代码边界都要先在这一组里建立整体认识 |
| 真值边界 | 这里讲“全局骨架”；更细的运行时、业务模块、基础设施与专题分析要去各自目录继续下钻 |
| 使用方式 | 第一次读仓库先看本组；排障或改造时，把这里当导航页回跳到更细分的文档组 |

## 重点速查（继续往下读前先记这几条）

1. **地图先于细节**：先看三进程、模块分布和主链路，再进入某一个具体 BC 或某个基础设施机制。  
2. **代码边界很关键**：`02-代码组织与边界` 是避免把 `internal/pkg`、`component-base`、接口层与领域层混读的关键入口。  
3. **主链路要先抓住**：`03-核心业务链路` 把事件名、RPC、代码入口串起来，是后续专题和模块文档的共同骨架。  
4. **本地开发不是附录**：`04-本地开发与配置约定` 负责把 `make`、yaml、环境变量和基础设施自检讲清楚，适合第一次落地运行仓库时对照。  

本目录四篇按 **地图 → 代码边界 → 业务链路 → 本地跑起来** 顺序阅读最顺；与根目录 [docs/README.md](../README.md) 的「阅读顺序」一致。
