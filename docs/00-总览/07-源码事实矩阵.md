# 源码事实矩阵

> 本文不是源码目录清单，而是 qs-server 的事实索引：每一条重要架构结论都应能回到代码、机器契约、配置或可执行测试。它同时记录当前实现的边界，避免文档把设计目标、局部能力或历史方案误写成已经完成的系统能力。

## 1. 三十秒结论

qs-server 的当前源码可以支持以下总体判断：

1. qs-server 是以模块化单体为业务核心、由 `qs-apiserver`、`collection-server` 和 `qs-worker` 三个进程协作运行的事件驱动系统，不是按业务模块独立部署的微服务系统。
2. 七个业务模块的正式清单由组合根注册表定义：Survey、ModelCatalog、Evaluation、Interpretation、Actor、Plan、Statistics；进程只是部署和运行边界，不重新拥有这些领域规则。
3. 当前可靠提交语义是：collection-server 只有在 AnswerSheet 与 Outbox 已经事务化持久化后才返回 `202 Accepted` 和 `answersheet_id`；Worker 是正常创建或恢复 Assessment 的唯一入口。
4. 事件、信号、缓存和数据库承担不同责任：持久业务事实落入 MySQL 或 MongoDB，可靠跨阶段驱动使用 Outbox，Redis Pub/Sub 只负责可丢失唤醒，缓存只提供读性能和运行保护。
5. 源码中存在不少“已经具备局部构件，但不能宣称完整闭环”的能力。本矩阵把这些负面事实单独列出，后续文档与项目宣讲必须遵守。

## 2. 这份矩阵解决什么问题

重建文档时，最危险的不是漏写一个目录，而是把不同性质的信息混在一起：

- 把架构意图写成当前实现；
- 把一个接口或结构体存在，写成完整业务能力已经闭环；
- 把运行时优化写成业务事实；
- 把历史设计继续当作当前链路；
- 把测试工具或压测脚本存在，写成生产容量已经得到证明。

因此，本文给每项事实提供六个要素：

| 要素 | 含义 |
| --- | --- |
| 事实编号 | 后续文档讨论、审校和追踪时使用的稳定编号 |
| 可写入结论 | 当前文档可以直接采用的表述 |
| 状态 | 区分已证实事实、当前约束、设计目标和待验证事项 |
| 首要事实源 | 发生冲突时优先查看的源码、机器契约或配置 |
| 交叉证据 | 用于防止只看一个文件就得出错误结论 |
| 易误述边界 | 明确这条事实不代表什么 |

### 2.1 状态标签

| 标签 | 使用规则 |
| --- | --- |
| **已证实** | 当前检出代码中存在完整、相互一致的实现或契约证据 |
| **当前约束** | 当前实现确实如此，但它可能是兼容、遗留或尚未演进完成的边界 |
| **设计目标** | 团队认同且会指导后续实现，但不能写成当前已经完成 |
| **待运行验证** | 静态代码和测试入口存在，仍需在真实依赖或目标环境中验证 |

### 2.2 证据优先级

发生冲突时，按以下顺序判断：

1. 当前运行源码和组合根；
2. 机器可读契约与配置，例如 OpenAPI、Proto、`events.yaml`、`signals.yaml`；
3. 可执行架构测试、契约测试和集成测试；
4. `docs/00-总览` 至 `docs/05-专题分析` 的当前文档；
5. `docs/06-宣讲` 的讲解材料；
6. `docs/_archive` 的历史材料。

归档文档只能解释设计演进，不能覆盖当前源码事实。

## 11. 维护流程

### 11.1 新增或修改架构结论

当代码变更影响本文中的事实时，按以下顺序处理：

1. 先修改源码、机器契约和对应测试；
2. 判断是更新既有事实，还是新增事实编号；
3. 更新本矩阵的结论、状态、证据和误述边界；
4. 更新拥有解释责任的总览、运行时、业务模块或基础设施文档；
5. 如果旧说法仍有历史价值，将其移入归档并说明被替代原因；
6. 运行文档门禁，并人工沿源码路径复核一次。

### 11.2 发现文档和源码冲突

不要直接选择看起来更合理的一方。应当：

```text
发现冲突
  -> 找到组合根或运行调用链
  -> 核对 OpenAPI、Proto、事件或信号目录
  -> 查看架构测试、契约测试和集成测试
  -> 判断是代码缺陷、文档漂移还是未完成设计目标
  -> 修正事实矩阵
  -> 再修正解释性文档
```

如果无法通过静态证据判断运行结果，将状态标记为“待运行验证”，而不是猜测。

### 11.3 什么变化必须复核本文

- 新增、删除或重命名业务模块；
- 调整三进程职责、启动阶段或通信方向；
- 修改 AnswerSheet 提交和 202 语义；
- 修改核心事件的名称、可靠性级别或消费者；
- 修改 Published Model、EvaluationOutcome、ReportInput 或 Report 的版本边界；
- 调整 MySQL、MongoDB、Redis 的事实所有权；
- 引入新的缓存层、降级策略、限流或重试治理；
- 改变 IAM、Actor、Filler、Testee 的身份关系；
- 改变 Plan、Task 和趋势统计规则；
- 对外宣称新的容量、可靠性或安全能力。

## 12. 验证入口

完成矩阵或解释性文档修改后，至少执行：


不同事实还应补充定向验证：

| 事实范围 | 优先验证方式 |
| --- | --- |

## 13. 最终使用原则

本文的价值不在于表格数量，而在于形成三条稳定习惯：

1. **先分清事实、约束、目标和待验证事项，再开始写架构结论。**
2. **任何跨模块结论，都至少回到组合根、机器契约或一条可执行测试链。**
3. **能证明局部机制存在，不等于能证明端到端能力已经闭环。**

后续每完成一篇业务模块或技术专题文档，都应反向检查本文：如果产生了新的稳定事实，就补充矩阵；如果发现旧结论已经失效，就同步修正，而不是让事实矩阵再次变成静态目录。
