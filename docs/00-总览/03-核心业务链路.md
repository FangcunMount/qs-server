# 核心业务链路

**本文回答**：从「前台提交」到「异步测评与报告」等**最关键**的几条链路，各自经过哪些进程、触发哪些**事件类型字符串**、对应哪些 **RPC** 与**代码入口**。细节以源码与 [configs/events.yaml](../../configs/events.yaml) 为准。

---

## 30 秒结论

如果只看一屏，先记住下面这张表：

| 编号 | 链路 | 典型触发 | 关键事件（`events.yaml`） | 备注 |
| ---- | ---- | -------- | ------------------------- | ---- |
| A | 前台提交答卷 | REST → collection | `answersheet.submitted` | 可经 SubmitQueue 异步落库 |
| B | 答卷 → 测评创建 | worker 消费 `answersheet.submitted` | **`assessment.submitted`**（由 **apiserver** 在测评 **Submit** 成功后通过 **MySQL outbox** 发布；无量表或未自动提交时可能不出现） | internal：`CalculateAnswerSheetScore` → `CreateAssessmentFromAnswerSheet` |
| C | 测评评估与报告 | worker 消费 `assessment.submitted` | `assessment.interpreted`、`assessment.failed`、`report.generated` 等（以 [events.yaml](../../configs/events.yaml) 为准） | `EvaluateAssessment` + evaluation pipeline；其中 `assessment.failed` 走 MySQL outbox，成功事件走 Mongo report-save outbox |
| D | 报告后处理 | worker | `report.generated` | 如打标签（`TagTestee`） |
| E | 计划与统计 | 定时/内部调用 + worker | `task.opened` / `task.completed` / `task.expired` / `task.canceled` | 计划异步副作用已收口到 `task.*`；统计同步落 MySQL 等 |

**Verify**：事件名、Topic、`handler` 字段必须与 [configs/events.yaml](../../configs/events.yaml) 一致；RPC 名与 [internal/apiserver/interface/grpc/proto/](../../internal/apiserver/interface/grpc/proto/) 一致。

## 1. 端到端时序（主干）

下列时序覆盖 A～D 的主干步骤；省略重试、锁与部分条件分支。

```mermaid
sequenceDiagram
    participant C as Client
    participant Col as collection-server
    participant API as qs-apiserver
    participant MQ as MQ
    participant W as qs-worker

    C->>Col: REST 提交答卷
    Col->>Col: JWT / 监护 / 排队策略
    Col->>API: gRPC AnswerSheetService.SaveAnswerSheet
    API->>API: Survey：持久化答卷
    API->>API: Mongo durable submit + Mongo outbox
    API-->>Col: 响应
    Col-->>C: HTTP 响应
    API->>MQ: answersheet.submitted（relay）

    MQ-->>W: answersheet.submitted
    W->>API: InternalService.CalculateAnswerSheetScore
    W->>API: InternalService.CreateAssessmentFromAnswerSheet
    API->>API: MySQL assessment save + outbox
    API->>MQ: assessment.submitted（relay；无量表等则可能无）

    MQ-->>W: assessment.submitted
    W->>API: InternalService.EvaluateAssessment
    API->>API: Evaluation pipeline
    API->>API: Mongo report save + outbox
    API->>MQ: assessment.interpreted, report.generated（relay）

    MQ-->>W: report.generated
    W->>API: InternalService.TagTestee 等
```

## 2. 链路 A：前台提交答卷

**目的**：尽快校验身份与业务前置条件，将答卷交给 apiserver 持久化并发出 `answersheet.submitted`。

**步骤（概要）**：

1. collection REST 接请求（路径以 [api/rest/collection.yaml](../../api/rest/collection.yaml) 为准）。若客户端提供 `idempotency_key`，BFF 会原样透传给 apiserver。
2. JWT、监护关系、受试者等校验（见 application 层）。
3. 若启用 [SubmitQueue](../../internal/collection-server/application/answersheet/submit_queue.go)，请求在 **collection 进程内** 进入带缓冲 channel，由 **本进程内 goroutine worker** 调用 `submitFunc`（即下游 gRPC）；与 **qs-worker 进程无关**。未启用队列则同步直调 gRPC。
4. apiserver `SaveAnswerSheet` 在一个 Mongo 事务里同时写入 **答卷 + durable idempotency 记录 + `answersheet.submitted` outbox**；真正出站由 apiserver 内部 relay 异步补发。

**当前幂等现状**：

- `collection` 的 `SubmitQueue` 仍会基于 `request_id` 维护**单实例、内存态** `queued / processing / done / failed` 状态，适合 `202` 后查询状态，但**不跨实例**，也不再承担 durable 幂等语义。锚点：[submit_queue.go](../../internal/collection-server/application/answersheet/submit_queue.go)。
- 调用方显式传 `idempotency_key` 时，apiserver 会把 **业务幂等记录**与答卷、outbox 同事务落到 Mongo；相同 key 的重复请求不会重复创建答卷，也不会重复写出站事件。锚点：[submission_service.go](../../internal/apiserver/application/survey/answersheet/submission_service.go)、[durable_submit.go](../../internal/apiserver/infra/mongo/answersheet/durable_submit.go)。
- `answersheet.submitted` 已改成 outbox/relay 模式：保存成功后事件不会丢在“写库后 direct publish 失败”的窗口，真正出站由 apiserver relay 负责补发。锚点：[durable_submit.go](../../internal/apiserver/infra/mongo/answersheet/durable_submit.go)、[process/runtime_bootstrap.go](../../internal/apiserver/process/runtime_bootstrap.go)。

**锚点**：

- [internal/collection-server/transport/rest/handler/answersheet_handler.go](../../internal/collection-server/transport/rest/handler/answersheet_handler.go)
- [internal/collection-server/application/answersheet/submission_service.go](../../internal/collection-server/application/answersheet/submission_service.go)
- [internal/apiserver/transport/grpc/service/answersheet.go](../../internal/apiserver/transport/grpc/service/answersheet.go)
- Proto：`answersheet.AnswerSheetService/SaveAnswerSheet`

## 3. 链路 B：答卷提交后 → 计分与创建测评

**目的**：worker 消费 `answersheet.submitted`，避免重复处理，并顺序调用 internal 服务完成计分与 Assessment 创建。

**步骤（概要）**：

1. `events.yaml` 中该事件绑定 `answersheet_submitted_handler` 等（以文件为准）。
2. Handler 内加锁/幂等后调用 **CalculateAnswerSheetScore**、**CreateAssessmentFromAnswerSheet**。
3. 成功后进入 assessment 生命周期（后续事件名查 `events.yaml`）。

**当前幂等现状**：

- worker 第一跳会先对同一答卷拿 Redis 分布式锁，避免并发/重放下重复创建测评。锚点：[answersheet_handler.go](../../internal/worker/handlers/answersheet_handler.go)。
- `CreateAssessmentFromAnswerSheet` 会先按 `answer_sheet_id` 查询已有测评；若并发下仍撞库，MySQL `assessment.answer_sheet_id` 的唯一约束会兜底并返回已有结果。锚点：[internal.go](../../internal/apiserver/transport/grpc/service/internal.go)、[po.go](../../internal/apiserver/infra/mysql/evaluation/po.go)。

**锚点**：

- [internal/worker/handlers/answersheet_handler.go](../../internal/worker/handlers/answersheet_handler.go)
- [internal/apiserver/transport/grpc/service/internal.go](../../internal/apiserver/transport/grpc/service/internal.go)
- [internal/apiserver/interface/grpc/proto/internalapi/internal.proto](../../internal/apiserver/interface/grpc/proto/internalapi/internal.proto)

**发布端说明**：**`assessment.submitted` 由 apiserver**（测评领域在 **Submit** / **Retry** 等路径上挂事件）发布，并通过 **MySQL outbox relay** 最终投递；**qs-worker 只消费**，不替代 apiserver 写模型。锚点示例：[assessment.go](../../internal/apiserver/domain/evaluation/assessment/assessment.go)。

## 4. 链路 C：异步评估与报告

**目的**：对已提交的测评执行 evaluation engine 流水线（校验、因子分、风险、解读等），并发布 `assessment.interpreted`、`report.generated` 等。

**步骤（概要）**：

1. worker 消费 **`assessment.submitted`**（无量表等情况下链路可能提前结束，以代码分支为准）。
2. 调用 **EvaluateAssessment**。
3. 引擎内阶段参见 [pipeline/chain.go](../../internal/apiserver/application/evaluation/engine/pipeline/chain.go) 等。
4. 保存报告成功后，一次性把 `assessment.interpreted` 与 `report.generated` 写入 Mongo outbox，再由 apiserver relay 补发。

**锚点**：

- [internal/worker/handlers/assessment_handler.go](../../internal/worker/handlers/assessment_handler.go)
- [internal/apiserver/application/evaluation/engine/service.go](../../internal/apiserver/application/evaluation/engine/service.go)

## 7. 边界与注意事项

- **worker 不是第二写库入口**：持久化与领域规则仍在 **apiserver** 内完成；worker 通过 **internal gRPC** 驱动。
- **事件路由**以 `events.yaml` 为单一事实来源；handler 名字符串与代码注册须一致。
- **统计**多为「Redis 等缓存/预聚合 + MySQL 落盘」组合，勿假设所有指标来自单次 SQL 扫明细。
- 与前台身份链路的交叉见 [01-运行时/05-IAM认证与身份链路.md](../01-运行时/05-IAM认证与身份链路.md)。
