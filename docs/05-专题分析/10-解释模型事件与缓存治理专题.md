# 10-解释模型事件与缓存治理专题

**本文回答**：在 qs-server 从“医学量表系统”演进为“多解释模型测评平台”后，为什么事件、缓存、统计、安全和观测都需要围绕 Interpretation Model 重新对齐；如何区分规则变化事件与一次测评执行事件；`report.changed`、`interpretation.completed`、`report.generated` 分别表达什么；Redis StaticList、Context cache、WarmupTarget、Statistics ReadModel、Metrics、Governance endpoint 应如何协作；以及为什么规则变化不能默认触发历史测评重算。

---

## 30 秒结论

解释模型扩展后，系统中会同时出现两类变化：

```text
规则变化
    例如 Scale / MBTI / BigFive 模型发布、归档、规则调整、目录变化。

一次执行变化
    例如某次 Assessment 完成解释、解释失败、报告生成。
```

这两类变化必须严格区分。

| 类型 | 示例事件 | 含义 | 典型后续动作 |
| ---- | -------- | ---- | ------------ |
| 规则变化事件 | `report.changed`、`scale.changed`、`mbti-model.published` | 模型规则或目录发生变化 | 缓存失效、Context cache 失效、列表重建、读模型刷新、治理预热 |
| 一次执行事件 | `interpretation.completed`、`interpretation.failed`、`report.generated` | 某次 Assessment 的执行阶段变化 | 报告生成、统计投影、通知、失败处理、wait-report 唤醒 |

核心判断：

> **事件系统只表达阶段事实，不表达具体模型算法；缓存系统只保存可回源派生数据，不成为规则事实源；统计系统只保存查询投影，不成为执行事实源；治理系统只做观察和受控操作，不默认重算历史测评。**

一句话概括：

```text
规则事实在具体模型模块；
执行事实在 Evaluation；
报告事实在 InterpretReport；
统计事实在 Statistics ReadModel；
缓存是可回源派生状态；
指标用低基数模型维度；
Governance endpoint 做可观测和受控治理。
```

---

## 1. 为什么解释模型扩展会影响基础设施

当系统只支持 Scale 时，很多基础设施能力可以围绕“量表”举例：

```text
scale.changed
ScaleListCache
GenerateScaleQRCode
MedicalScale read model
FactorScore statistics
```

但当系统引入 MBTI 后，如果基础设施仍然只围绕 Scale 表达，就会出现问题：

```text
MBTI 模型发布后，ScaleListCache 无法表达 MBTIModelListCache；
MBTI Context 需要缓存，但不能塞进 Scale cache；
MBTI TypeCode 分布属于统计口径，但不是 Scale RiskLevel；
MBTI 报告访问权限不同于 Scale 管理权限；
MBTI 执行耗时需要观测，但不能用 model_code 做 metrics label。
```

因此，解释模型扩展不是只影响业务 domain。

它会横向影响：

- Event。
- Data Access。
- Redis cache。
- Warmup / Hotset。
- Statistics ReadModel。
- Security capability。
- Observability metrics。
- Governance endpoint。
- Runbook。

---

## 2. 规则变化事件 vs 一次执行事件

这是本文最重要的边界。

### 2.1 规则变化事件

规则变化事件表达：

```text
某个解释模型的规则资产或模型目录发生变化。
```

示例：

```text
scale.changed
report.changed
mbti-model.published
mbti-model.archived
bigfive-model.published
bigfive-model.archived
```

这些事件通常用于：

- 发布态模型列表缓存失效。
- Provider Context cache 失效。
- StaticList 重建。
- WarmupTarget 预热。
- 后台读模型刷新。
- operating 页面更新。

它们不表示某次 Assessment 已经完成。

它们也不应默认触发历史报告重算。

---

### 2.2 一次执行事件

一次执行事件表达：

```text
某次 Assessment 的执行状态发生变化。
```

示例：

```text
answersheet.submitted
assessment.created
assessment.completed
interpretation.completed
interpretation.failed
assessment.failed
report.generated
```

这些事件通常用于：

- 创建 Assessment。
- 推进 Evaluation 执行。
- 执行具体 Provider。
- 生成报告。
- 统计投影。
- 通知等待报告的请求。
- 失败处理和补偿入口。

它们通常应走 durable outbox。

---

### 2.3 两类事件不能混用

错误做法：

```text
report.changed
  -> 直接触发所有历史 Assessment 重算
```

错误做法：

```text
interpretation.completed
  -> 用来表达 MBTI 规则发布成功
```

正确边界：

```text
report.changed
    表达规则或目录变化。

interpretation.completed
    表达某次解释执行完成。

report.generated
    表达某份报告事实已经保存。
```

---

## 3. 新主事件链路

多解释模型后的主链路应稳定为：

```text
answersheet.submitted
  -> assessment.created
  -> assessment.completed
  -> interpretation.completed / interpretation.failed
  -> report.generated
```

### 3.1 `answersheet.submitted`

语义：

```text
答卷事实已保存。
```

典型来源：

```text
Survey / AnswerSheet durable submit
```

典型后续：

```text
CreateAssessmentFromAnswerSheet
```

---

### 3.2 `assessment.created`

语义：

```text
已基于 AnswerSheet 创建一次 Assessment。
```

典型后续：

```text
CompleteAssessment
```

---

### 3.3 `assessment.completed`

语义：

```text
Evaluation 层的测评执行准备阶段完成，可以进入具体解释模型执行。
```

注意：

```text
assessment.completed 不等于报告已生成。
```

---

### 3.4 `interpretation.completed`

语义：

```text
具体 Interpretation Provider 已完成解释，并产生 EvaluationResult。
```

例如：

```text
ScaleProvider 完成因子计分和风险解释；
MBTIProvider 完成维度偏好和 TypeCode 解释。
```

典型后续：

```text
GenerateReportFromInterpretation
```

---

### 3.5 `interpretation.failed`

语义：

```text
具体 Provider 执行失败。
```

常见原因：

```text
provider_not_found
context_load_failed
rule_invalid
questionnaire_mismatch
evaluate_failed
```

典型后续：

- 记录失败状态。
- 增加失败统计。
- 进入补偿或重试入口。
- 暴露给 Governance endpoint。

---

### 3.6 `report.generated`

语义：

```text
InterpretReport 已可靠保存。
```

注意：

```text
interpretation.completed 不等于 report.generated。
```

Provider 可以执行成功，但报告保存仍可能失败。

---

## 4. Event Catalog 治理原则

解释模型扩展后，事件目录必须承担两个职责：

```text
统一事件命名；
统一事件 delivery class。
```

### 4.1 命名原则

事件名应表达业务事实，而不是实现动作。

推荐：

```text
assessment.created
assessment.completed
interpretation.completed
interpretation.failed
report.generated
report.changed
```

不推荐：

```text
scale.factor_scored
mbti.type_calculated
provider.executed
report.build_done
```

后者要么过于模型细节化，要么过于技术实现化。

### 4.2 Delivery class

推荐区分：

| Event | Delivery | 原因 |
| ----- | -------- | ---- |
| `answersheet.submitted` | durable_outbox | 答卷提交后必须创建 Assessment |
| `assessment.created` | durable_outbox | 测评创建后必须推进执行 |
| `assessment.completed` | durable_outbox | 测评执行阶段完成后必须进入解释执行 |
| `interpretation.completed` | durable_outbox | 解释完成后必须生成报告 |
| `interpretation.failed` | durable_outbox | 失败必须可追踪和补偿 |
| `report.generated` | durable_outbox | 报告生成后必须投影、通知或统计 |
| `report.changed` | best_effort | 规则变化主要触发缓存和读模型刷新 |

---

## 5. Redis 缓存治理

解释模型扩展后，Redis 不能再只围绕 ScaleListCache 设计。

需要区分：

```text
StaticList
ObjectCache
QueryCache
ContextCache
Hotset
WarmupTarget
```

### 5.1 StaticList：发布态模型列表

StaticList 适合缓存：

```text
PublishedScaleListCache
MBTIModelListCache
BigFiveModelListCache
PublishedInterpretationModelListCache
```

它缓存的是“列表视图”，不是完整规则事实。

适合字段：

```text
ModelType
ModelCode
ModelVersion
DisplayName
QuestionnaireRef
Status
PublishedAt
Summary
```

不适合字段：

```text
完整 DimensionRule
完整 TypeProfile 正文
完整 ReportTemplate
EvaluationResult
InterpretReport
```

---

### 5.2 ContextCache：Provider 执行上下文

Provider 执行需要上下文。

ScaleContext 可能包含：

```text
MedicalScale snapshot
Factor rules
Scoring specs
Interpretation rules
```

MBTIContext 可能包含：

```text
MBTIModel snapshot
Dimension rules
Question mappings
Type profiles
Report template
```

ContextCache 可以提升执行性能，但必须可回源。

```text
Context cache 不是规则事实源。
```

### 5.3 QueryCache：统计查询结果

QueryCache 适合缓存：

```text
interpretation model distribution
mbti type distribution
mbti dimension distribution
assessment overview
report overview
```

它缓存的是查询结果，不是统计事实源。

统计事实源应是 Statistics ReadModel。

---

## 6. WarmupTarget 与 Hotset 治理

新增解释模型后，需要扩展 warmup target。

推荐 target：

```text
static.interpretation_model:{modelType}:{modelCode}:{modelVersion}
static.interpretation_model_list:published
static.mbti_model_list:published
query.mbti_type_distribution:{scope}
query.interpretation_model_distribution:{scope}
```

### 6.1 模型发布后的 warmup

模型发布后可以执行：

```text
report.changed
  -> WarmInterpretationModelList
  -> WarmMBTIModelList
  -> WarmInterpretationModelContext
```

目标：

- 降低模型发布后的首次访问延迟。
- 让后台模型选择页读取更稳定。
- 提前构建 Provider 执行需要的只读 Context。

不应该：

- 修改业务主数据。
- 重算历史 Assessment。
- 修改 EvaluationResult。
- 修改 InterpretReport。

---

### 6.2 Hotset

Hotset 可以用于识别高频访问对象，例如：

```text
热门 questionnaire
热门 scale
热门 mbti model
高频 statistics query
```

但 Hotset 只用于治理缓存，不决定业务事实。

---

## 7. Data Access 治理

解释模型扩展后，数据事实源必须清晰。

| 数据 | 事实源 |
| ---- | ------ |
| MedicalScale 规则 | Scale repository |
| MBTIModel 规则 | MBTI repository |
| TypeProfile 文案 | MBTIModel / TypeProfile |
| Assessment 状态 | Evaluation repository |
| EvaluationResult | Evaluation result repository |
| InterpretReport | Evaluation report repository |
| MBTI TypeCode 分布 | Statistics ReadModel |
| 发布态模型列表缓存 | Redis StaticList，可回源 |
| Provider Context | Redis ContextCache，可回源 |

关键原则：

```text
规则事实、执行事实、报告事实、统计事实、缓存派生状态必须分开。
```

---

## 8. Statistics ReadModel 治理

新增 MBTI 后，统计读模型需要支持新的统计维度。

### 8.1 推荐统计口径

```text
assessment_completed_count by model_type
interpretation_completed_count by model_type
interpretation_failed_count by model_type / reason
report_generated_count by model_type
mbti_type_distribution
mbti_dimension_distribution
```

### 8.2 推荐 read model

```text
statistics_interpretation_model_daily
statistics_mbti_type_daily
statistics_mbti_dimension_daily
```

### 8.3 不应保存完整规则

错误做法：

```text
statistics_mbti_type_daily 保存完整 TypeProfile 正文
```

正确做法：

```text
statistics_mbti_type_daily 保存 type_code + count；
TypeProfile 文案回源 MBTIModel / TypeProfile。
```

---

## 9. Security 治理

解释模型扩展后，权限要分成两类。

### 9.1 模型规则权限

```text
read_interpretation_models
manage_interpretation_models
```

面向：

```text
Scale / MBTI / BigFive 规则创建、编辑、发布、归档、查看
```

### 9.2 报告访问权限

```text
read_interpretation_reports
```

面向：

```text
用户报告访问，例如 Scale 报告、MBTI 报告、BigFive 报告
```

### 9.3 关键边界

```text
能管理 MBTI 模型，不等于能查看用户 MBTI 报告；
能查看用户报告，不等于能修改 MBTI 模型规则。
```

这两个能力必须在 IAM resource/action 和 QS capability 中拆开。

---

## 10. Observability 治理

解释模型指标可以使用低基数模型维度。

推荐：

```text
qs_interpretation_execution_total{model_type,provider,phase,result}
qs_interpretation_execution_duration_seconds{model_type,provider,phase,result}
qs_interpretation_failure_total{model_type,provider,phase,reason}
qs_evaluation_lifecycle_total{stage,result}
```

示例：

```text
qs_interpretation_execution_duration_seconds{model_type="mbti",provider="mbti",phase="evaluate",result="success"}
```

禁止：

```text
model_code
model_version
assessment_id
answer_sheet_id
report_id
type_code
```

说明：

- `model_type=mbti` 可以作为 metrics label。
- `model_code=MBTI_STANDARD` 不建议作为 metrics label。
- `type_code=INTJ` 不应进入 Prometheus label。
- TypeCode 分布属于 Statistics ReadModel。

---

## 11. Governance endpoint 治理

Governance endpoint 应能回答：

```text
模型列表缓存是否可用？
MBTI Context cache 是否命中？
某类模型的 interpretation queue 是否积压？
某类模型最近失败率是否升高？
某个 warmup target 上次执行是否失败？
```

### 11.1 推荐状态维度

```text
family
profile
target_kind
model_type
provider
status
last_run_at
last_error_code
```

### 11.2 缓存状态示例

```text
static.interpretation_model_list
static.mbti_model_list
static.interpretation_model
query.mbti_type_distribution
query.interpretation_model_distribution
```

### 11.3 队列状态示例

```text
model_type=scale
model_type=mbti
model_type=bigfive

pending_count
inflight_count
failed_count
last_success_at
last_failure_at
```

### 11.4 drill-down 边界

Governance drill-down 可以包含：

```text
model_code
model_version
assessment_id
report_id
trace_id
```

但这些字段不应进入 Prometheus label。

## 12. 规则变化为什么不能默认重算历史

这是多解释模型平台必须坚持的边界。

### 12.1 历史报告需要可追溯

一份报告应能说明：

```text
当时使用了哪个 ModelRef；
当时的规则版本是什么；
当时的 EvaluationResult 是什么；
当时生成的 InterpretReport 是什么。
```

如果规则变化后自动重算历史报告，会破坏历史可追溯性。

### 12.2 重算是业务动作，不是缓存动作

规则变化可以触发：

- 缓存失效。
- 列表重建。
- Context cache 预热。
- 读模型刷新。

但重算历史 Assessment 会改变执行结果和报告事实。

这必须建模为：

```text
ReEvaluationJob
RepairJob
BackfillJob
```

并且需要：

- operator 权限。
- 审计。
- 范围控制。
- dry-run。
- 幂等。
- 失败恢复。

## 13. 新增解释模型的治理检查清单

以 MBTI 为例，新增前应检查：

| 检查项 | 是否完成 |
| ------ | -------- |
| ModelRef 是否包含 model_type/code/version | ☐ |
| Provider 是否注册到 Registry | ☐ |
| Context 是否可回源 | ☐ |
| 规则变化事件是否定义 | ☐ |
| 一次执行事件是否复用通用链路 | ☐ |
| MBTIModel repository / document 是否定义 | ☐ |
| MBTIModelListCache 是否定义 | ☐ |
| Context cache 是否定义 | ☐ |
| WarmupTarget 是否注册 | ☐ |
| TypeCode / Dimension ReadModel 是否定义 | ☐ |
| read/manage model capability 是否定义 | ☐ |
| read report capability 是否定义 | ☐ |
| metrics 是否使用低基数 label | ☐ |
| governance drill-down 是否支持 model_type | ☐ |
| 历史重算是否显式建模 | ☐ |

## 14. 设计收益

### 14.1 横切能力可复用

一旦事件、缓存、统计、安全、观测都围绕 Interpretation Model 抽象，就可以复用到：

- Scale。
- MBTI。
- BigFive。
- 职业兴趣测评。
- AI 增强解读。

### 14.2 模型规则和执行事实不混淆

规则变化不会误伤历史报告。

执行失败也不会污染模型规则。

### 14.3 运维定位更清楚

报告不生成时可以判断：

```text
是事件没出？
是 worker 没消费？
是 Provider 失败？
是 Context cache miss？
是 Report save 失败？
是 Statistics projector 延迟？
```

---

## 15. 设计代价

| 代价 | 表现 |
| ---- | ---- |
| 文档复杂度增加 | 必须讲清规则变化和执行事件 |
| 治理对象增加 | cache target、warmup target、metrics、status 都要有模型维度 |
| 测试矩阵增加 | 每个模型都要测 Provider、Context、Event、Cache、Statistics |
| 排障路径变长 | 需要跨 event / worker / evaluation / provider / report |
| 版本治理更严格 | ModelRef / RuleSnapshotRef 必须稳定 |

这些代价是多模型平台的必要成本。

---

## 16. 设计不变量

1. 规则变化事件和一次执行事件必须区分。
2. `report.changed` 不等于 `interpretation.completed`。
3. `interpretation.completed` 不等于 `report.generated`。
4. 规则变化不默认触发历史 Assessment 重算。
5. StaticList 只保存发布态列表摘要。
6. Context cache 必须可回源。
7. Statistics ReadModel 不保存完整规则正文。
8. Metrics 使用低基数 `model_type`，不使用 `model_code`。
9. Governance summary 不产生高基数指标。
10. Governance action 必须受控、审计和有 SOP。
11. Provider 不直接发事件。
12. Provider 不直接保存 Report 主事实。
13. Worker 不直接执行具体模型算法。
14. Evaluation 保存执行事实。
15. 具体模型模块保存规则事实。

---

## 17. 常见误区

### 17.1 “模型规则变化后，历史报告就该自动更新”

不应默认如此。历史报告需要版本可追溯，重算必须显式建模。

### 17.2 “MBTI TypeCode 分布可以放 Prometheus label”

不建议。TypeCode 分布是业务统计，应该进入 Statistics ReadModel。

### 17.3 “StaticList 可以缓存完整 MBTI 规则”

不对。StaticList 只缓存列表摘要，完整规则应回源 MBTI repository 或 Context cache。

### 17.4 “Provider 可以顺手发 report.generated”

不建议。Provider 输出结构化结果，报告事实由 Evaluation 保存，并由 Outbox 可靠出站。

### 17.5 “Governance endpoint 可以顺手修数据”

不应该。Status 是只读，repair/replay/action 必须单独受控。
