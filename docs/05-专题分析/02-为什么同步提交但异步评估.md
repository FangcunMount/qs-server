# 为什么同步提交但异步评估

**本文回答**：为什么 qs-server 的答卷提交链路要“同步保存答卷”，但评估、计分、解读、报告生成要“异步执行”；为什么不能让前台提交请求一直等到报告生成；这个设计如何与 collection-server、SubmitQueue、SubmitGuard、Outbox、Worker、Internal gRPC、Evaluation Pipeline 协作。

---

## 30 秒结论

| 阶段 | 同步 / 异步 | 目标 |
| ---- | ----------- | ---- |
| 前台接收提交 | 同步 | 快速校验请求、认证、监护关系、限流、受理 |
| collection SubmitQueue | 异步受理 | 在 collection-server 内削峰，返回 `202 + request_id` |
| apiserver 保存 AnswerSheet | 同步 | 确定答卷事实已经持久化 |
| AnswerSheet submitted event | 异步 | 将“答卷已提交”传递给评估后台 |
| Worker 处理事件 | 异步 | 计分、创建 Assessment、触发 Evaluation |
| Evaluation Pipeline | 异步 | 计算因子、风险、解读、报告、结果落库 |
| 前台查报告 | 异步查询 / 长轮询 | 通过 status / wait-report 感知报告是否完成 |

一句话概括：

> **同步提交保证“答案不会丢”，异步评估保证“报告生成不拖垮提交体验”。**

更准确地说：qs-server 的提交不是完全同步，也不是完全异步，而是分成两段：

```text
前台请求 -> 同步/排队保存 AnswerSheet
AnswerSheet saved -> 异步执行 Evaluation
```

这是一种**事实先落库，结果后计算**的设计。

---

## 1. 这个问题本质上是在分离两类 SLA

答卷提交和评估报告的 SLA 完全不同。

### 1.1 提交 SLA

用户点击“提交”时，系统必须尽快给出确定反馈：

- 请求是否被接收。
- 答案是否合法。
- 用户是否有权限为该受试者提交。
- 答卷是否已经保存。
- 如果被排队，request_id 是什么。
- 如果队列满，是否返回 429。
- 如果重复提交，是否复用已有结果或提示进行中。

这类 SLA 的核心是：

```text
快、确定、可重试、不能丢答案
```

### 1.2 评估 SLA

评估报告生成涉及：

- 加载量表。
- 加载答卷。
- 加载问卷。
- 计算答卷分数。
- 创建 Assessment。
- 执行因子计分。
- 判定风险等级。
- 匹配解读规则。
- 生成报告。
- 写 MySQL / Mongo / Outbox。
- 通知等待报告的请求。

这类 SLA 的核心是：

```text
可最终完成、可失败记录、可重试、可观测
```

它不适合卡在前台提交请求里。

## 2. 当前链路总图

```mermaid
sequenceDiagram
    participant Client as Client
    participant Collection as collection-server
    participant Queue as SubmitQueue
    participant APIGRPC as apiserver gRPC
    participant Survey as Survey/AnswerSheet
    participant Outbox as Outbox/Event
    participant Worker as qs-worker
    participant Internal as InternalService
    participant Eval as Evaluation Pipeline

    Client->>Collection: POST /api/v1/answersheets
    Collection->>Queue: Enqueue(request_id)
    Queue-->>Client: 202 accepted + request_id

    Queue->>Collection: worker submitWithGuard
    Collection->>APIGRPC: SaveAnswerSheet
    APIGRPC->>Survey: validate + durable save AnswerSheet
    Survey->>Outbox: stage answersheet.submitted
    APIGRPC-->>Collection: answerSheetID
    Collection->>Queue: status=done

    Outbox-->>Worker: answersheet.submitted
    Worker->>Internal: CalculateAnswerSheetScore
    Internal->>Survey: CalculateAndSave
    Worker->>Internal: CreateAssessmentFromAnswerSheet
    Internal->>Eval: Create Assessment / Evaluate
    Eval->>Eval: factor score + risk + interpretation + report
```

注意：客户端拿到 `202 accepted` 并不代表评估完成；它只代表提交请求已被 collection 受理。后续可以用 `submit-status` 或 `wait-report` 查询。

## 3. 为什么 AnswerSheet 保存必须同步确认

异步评估的前提是：必须先有一个可靠的业务事实。

这个事实就是：

```text
AnswerSheet 已经保存
```

如果连 AnswerSheet 保存都异步，前台就会面临几个无法接受的问题：

| 问题 | 后果 |
| ---- | ---- |
| 用户提交后无法知道答案是否保存 | 用户反复提交 |
| 网络断开后无法确认状态 | 客户端无法重试 |
| 评估失败时不知道是没保存还是没生成报告 | 排障困难 |
| 后台 worker 可能处理不存在的答卷 | 异步链路复杂化 |
| 幂等 key 无法稳定落地 | 重复提交难治理 |

所以系统必须先把 AnswerSheet 作为事实源持久化，然后才谈异步评估。

---

## 4. 为什么评估不能同步卡在提交请求里

如果把评估也同步执行，提交链路会变成：

```text
POST /answersheets
  -> 保存答卷
  -> 计算答卷分数
  -> 创建 Assessment
  -> 加载 Scale / Questionnaire / AnswerSheet
  -> 执行 Evaluation Pipeline
  -> 写 Score
  -> 写 Report
  -> 写 Outbox
  -> 返回报告
```

这会带来明显风险。

### 4.1 延迟不可控

评估链路包含多个外部和下游依赖：

- MySQL。
- MongoDB。
- Redis cache。
- 量表规则读取。
- 报告文档写入。
- Outbox。
- Waiter notification。

任意一个依赖慢，都拖慢用户提交请求。

### 4.2 失败面扩大

同步提交原本只需要处理：

- 参数错误。
- 权限错误。
- 答案校验错误。
- 存储错误。

同步评估后，还要处理：

- 量表找不到。
- 问卷版本不匹配。
- 计分规则错误。
- 解读规则错误。
- 报告保存失败。
- 评估状态机失败。
- Waiter notification 失败。

这会让“提交答卷”变成一个巨大的事务。

### 4.3 高峰下容易雪崩

前台提交是高峰入口。如果每个提交请求都同步跑完整评估，会把：

- collection goroutine。
- apiserver gRPC。
- MySQL connection。
- Mongo session。
- Redis。
- Evaluation CPU。

全部绑定在一次用户请求里。

当提交量上来时，系统没有削峰空间。

### 4.4 重试语义混乱

如果同步评估中途失败，用户重试时到底应该：

- 重新保存 AnswerSheet？
- 复用 AnswerSheet？
- 重新创建 Assessment？
- 继续上次 Evaluation？
- 重新生成 Report？

这些需要状态机和幂等治理。与其在提交请求里处理，不如把它们收口到 Evaluation 异步流程。

## 5. collection-server 的角色：先削峰，再同步保存

collection-server 并不是把一切都异步丢出去。它承担的是前台保护层。

### 5.1 SubmitQueue 负责前台削峰

SubmitQueue 是 collection-server 进程内 bounded queue：

```text
jobs chan submitJob
statuses requestID -> queued / processing / done / failed
workerPool
```

入队成功：

```text
queue_accepted
status=queued
返回 202 + request_id
```

队列满：

```text
queue_full
返回 ErrQueueFull / HTTP 429
```

这让前台入口具备削峰能力。

### 5.2 SubmitQueue 不是业务最终异步化

SubmitQueue 只是 collection 进程内削峰。队列 worker 仍然会调用：

```text
submitWithGuard
  -> submitSync
  -> apiserver SaveAnswerSheet
```

也就是说，SubmitQueue 不替代 AnswerSheet durable save，它只是把“前台请求线程”与“提交处理 worker”解耦。

### 5.3 为什么 SubmitQueue 返回 202

因为请求已经被 collection 接收，但还没有保证 AnswerSheet 已保存。

所以正确语义是：

```text
accepted
```

而不是：

```text
created
```

前台要通过 request_id 查询状态。

---

## 6. SubmitGuard 的角色：跨实例幂等与进行中抑制

collection 的 SubmitQueue 是进程内的。多实例时，仅靠 requestID 本地状态不够。

所以后面还有 SubmitGuard：

```text
done marker
+
in-flight lease
```

### 6.1 done marker

如果同一个 idempotency key 已经完成：

```text
返回 already submitted + answerSheetID
```

### 6.2 in-flight lease

如果同一个 key 正在提交：

```text
ResourceExhausted: submit already in progress
```

### 6.3 为什么它不负责评估

SubmitGuard 保护的是：

```text
答卷提交
```

不是：

```text
评估执行
```

评估幂等应由 Assessment 状态机、唯一约束、worker duplicate suppression 和 Evaluation 自身重试策略负责。

---

## 7. Survey 的同步职责：校验并保存 AnswerSheet

apiserver 的 Survey SubmissionService 做的是提交事实确认：

1. 校验 DTO。
2. 加载问卷。
3. 构建答案值对象。
4. 批量校验答案。
5. 创建 AnswerSheet。
6. 保存 AnswerSheet。
7. 返回 AnswerSheetResult。

这个链路的价值在于：

```text
用户提交的答案被可靠保存
```

不是：

```text
用户报告已经生成
```

这就是同步边界。

---

## 8. Event / Outbox 的角色：连接同步事实和异步处理

AnswerSheet 保存之后，需要通知后续 Evaluation。

直接同步调用 Evaluation 不合适，所以需要事件：

```text
answersheet.submitted
```

更进一步，事件不能只是 best-effort fire-and-forget，否则可能出现：

```text
AnswerSheet 已保存
但事件丢了
Evaluation 永远不执行
```

因此需要 Outbox：

```text
保存 AnswerSheet
  -> stage submitted event
  -> relay publish
  -> worker consume
```

这正是“同步提交但异步评估”的可靠连接点。

---

## 9. Worker 的角色：事件驱动执行后续处理

worker 消费 `answersheet.submitted` 后执行两步：

### 9.1 计算答卷分数

调用 InternalService：

```text
CalculateAnswerSheetScore
```

该步骤属于 Survey 域的答卷计分能力，处理 AnswerSheet 自身的分数。

### 9.2 创建 Assessment

调用 InternalService：

```text
CreateAssessmentFromAnswerSheet
```

这一步进入 Evaluation 边界：基于 AnswerSheet 和量表上下文创建评估实例。

后续 Evaluation 可以继续执行：

```text
EvaluateAssessment
```

计算因子、风险、解读和报告。

---

## 10. Evaluation Pipeline 的角色：慢任务和复杂任务后移

Evaluation Pipeline 包含：

- ValidationHandler。
- FactorScoreHandler。
- RiskLevelHandler。
- InterpretationHandler。
- WaiterNotifyHandler。

它涉及：

- Scale factor scorer。
- Interpreter。
- DefaultProvider。
- Assessment result writer。
- Assessment score writer。
- InterpretReport writer。
- Report durable saver。
- Waiter registry。

这是一条复杂链路，适合异步执行。

### 10.1 为什么适合异步

因为它具备以下特征：

| 特征 | 说明 |
| ---- | ---- |
| 多步骤 | 计分、风险、解读、报告 |
| 多存储 | MySQL + Mongo |
| 可重试 | Evaluation 状态可以记录失败 |
| 可观察 | worker / outbox / report status 可查 |
| 非即时交互必需 | 用户提交后可以等待报告 |
| 复杂度高 | 不适合塞进提交事务 |

---

## 11. 用户体验如何保证

异步评估不等于用户体验差。关键是设计正确的交互协议。

### 11.1 提交阶段

前台收到：

```text
202 accepted
request_id
status=queued
```

或：

```text
429 queue full / rate limited
```

或：

```text
validation error
permission error
```

### 11.2 提交状态查询

通过：

```text
GET /answersheets/submit-status?request_id=...
```

查看：

- queued。
- processing。
- done。
- failed。

### 11.3 报告等待

通过：

```text
GET /assessments/:id/wait-report
```

或轮询报告接口，感知报告是否已完成。

### 11.4 体验上的真实承诺

系统对用户的承诺应该是：

```text
你的答卷已被接收/保存，报告正在生成。
```

而不是：

```text
提交按钮点完，报告必须同步返回。
```

---

## 12. 为什么不是“全部异步”

一种极端方案是：前台提交后只写入 MQ，连 AnswerSheet 都不立即保存。

这个方案不适合当前系统。

### 12.1 缺少提交事实

如果只是 MQ 消息：

- MQ ack 了不代表业务保存。
- 用户无法查询 AnswerSheet。
- 重试语义依赖消息系统。
- 前端状态难维护。

### 12.2 不利于幂等

AnswerSheet durable submit 和 idempotency key 更适合在 apiserver 事务中确认。

### 12.3 不利于审计和排障

答卷提交是核心业务事实，应有明确持久化记录，而不是先进入临时队列。

所以当前设计不是“全部异步”，而是：

```text
业务事实同步落库
计算结果异步生成
```

---

## 13. 为什么不是“全部同步”

另一个极端方案是：答卷提交、评估、报告全部同步。

这个方案的问题前面已经说过，总结如下：

| 问题 | 后果 |
| ---- | ---- |
| 提交延迟高 | 用户体验差 |
| 故障面大 | 任一环节失败导致提交失败 |
| 高峰承载差 | 无法削峰 |
| 重试复杂 | 多个副作用难幂等 |
| 模块耦合重 | Survey/Scale/Evaluation 混在一起 |
| 运维困难 | 无法区分提交失败和评估失败 |

所以当前采用中间方案：

```text
同步保存事实
异步计算结果
```

---

## 14. 这个设计的工程收益

### 14.1 抗峰值能力更强

collection SubmitQueue 可以削峰；Evaluation 后移到 worker 事件链路，可以通过 worker concurrency 和 MQ backlog 控制吞吐。

### 14.2 故障隔离更清楚

| 故障 | 影响 |
| ---- | ---- |
| 答案校验失败 | 提交失败 |
| AnswerSheet 保存失败 | 提交失败 |
| Outbox publish 延迟 | 报告延迟 |
| Worker 失败 | 报告延迟或评估失败 |
| Evaluation pipeline 失败 | Assessment failed，可重试 |
| Report 保存失败 | 评估失败，可排障 |

### 14.3 更容易扩展

未来可增加：

- 重试策略。
- DLQ。
- 报告生成优化。
- 多 worker。
- 分事件类型扩容。
- wait-report 优化。
- 报告生成通知。

这些都不需要改变答卷提交主协议。

---

## 15. 这个设计的代价

### 15.1 用户看到的是最终一致

答卷提交成功后，报告不是立即可用。

需要前端支持：

- accepted。
- pending。
- processing。
- report ready。
- failed/retry。

### 15.2 排障链路更长

报告没生成时要查：

1. AnswerSheet 是否保存。
2. submitted event 是否 staged。
3. relay 是否 publish。
4. worker 是否 consume。
5. duplicate gate 是否 skip。
6. CalculateAnswerSheetScore 是否成功。
7. CreateAssessmentFromAnswerSheet 是否成功。
8. Evaluation pipeline 是否成功。
9. Report 是否保存。

### 15.3 需要更多状态和观测

必须有：

- submit-status。
- outbox status。
- worker logs。
- assessment status。
- wait-report。
- resilience metrics。
- event metrics。

否则异步链路会变成黑盒。

---

## 16. 关键设计不变量

后续演进应坚持：

1. AnswerSheet 保存是提交事实边界。
2. 提交成功不等于评估完成。
3. Evaluation 不阻塞前台提交。
4. 评估失败不能回滚已提交答卷。
5. 重复提交优先通过 idempotency key 和 SubmitGuard 治理。
6. worker 重复事件通过 duplicate suppression 和业务幂等兜底。
7. 报告等待通过 status/wait-report，而不是同步提交等待。
8. Outbox 是 AnswerSheet 事实与 Evaluation 异步链路之间的可靠桥梁。
9. collection SubmitQueue 只是进程内削峰，不是 durable MQ。
10. 如果未来要增强可靠性，应增强 Outbox/worker/retry，而不是把评估塞回提交请求。

## 17. 常见误区

### 17.1 “异步评估说明提交不可靠”

不是。提交可靠性来自 AnswerSheet durable save；评估异步只是后续处理异步。

### 17.2 “返回 202 就代表答卷已保存”

不一定。202 只代表 collection 已受理入队。保存完成要看 submit-status done。

### 17.3 “报告慢就应该把评估改成同步”

不应该。报告慢应排查 worker、outbox、pipeline、DB/Mongo，而不是把慢任务放回前台请求。

### 17.4 “SubmitQueue 就是 MQ”

不是。它是 collection 进程内 memory channel，进程退出不 drain。

### 17.5 “评估失败应该让答卷提交失败”

不应该。答卷已经是事实，评估失败应记录在 Assessment/Evaluation 状态里。

## 18. 替代方案分析

### 18.1 全同步

```text
Submit -> Save AnswerSheet -> Evaluate -> Report -> Return
```

优点：

- 前端逻辑简单。
- 一次请求拿结果。

缺点：

- 高延迟。
- 高失败面。
- 高峰下容易雪崩。
- 重试复杂。
- 模块耦合。

结论：不适合当前系统。

### 18.2 全异步

```text
Submit -> MQ -> Save AnswerSheet -> Evaluate
```

优点：

- 前台极快。
- MQ 削峰能力强。

缺点：

- 答卷事实不确定。
- 幂等更复杂。
- 用户状态查询困难。
- 提交可靠性依赖消息系统。
- 前台无法确定保存结果。

结论：不适合答卷这种核心业务事实。

### 18.3 当前方案

```text
Submit accepted
  -> Save AnswerSheet durably
  -> Outbox event
  -> Worker Evaluation
```

优点：

- 事实可靠。
- 体验可控。
- 高峰可削。
- 评估可重试。
- 模块边界清楚。

缺点：

- 最终一致。
- 需要状态查询和观测。
- 排障链路较长。

结论：当前复杂度下更合理。
