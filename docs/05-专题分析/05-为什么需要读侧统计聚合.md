# 为什么需要读侧统计聚合

**本文回答**：为什么 qs-server 不能在每次统计查询时都从 Survey / Evaluation / Actor / Plan 的业务写模型实时 join 和聚合，而需要单独的 Statistics 读侧聚合；读侧统计聚合解决了哪些查询性能、口径稳定、跨模块统计、事件投影、缓存治理和运维补偿问题；它带来了哪些代价和约束。

---

## 30 秒结论

Statistics 读侧聚合不是“为了快一点加的缓存”，而是为了把复杂跨模块统计从主写模型中剥离出来，形成可查询、可重建、可修复、可缓存、可观测的读侧模型。

| 问题 | 如果实时查写模型 | 使用读侧统计聚合 |
| ---- | ---------------- | ---------------- |
| 首页 overview | 多表/多集合 join，慢且不稳定 | 读聚合表/读模型，稳定 |
| 医生/咨询师统计 | 每次实时归因，逻辑复杂 | 行为投影提前归因 |
| Entry / Testee 统计 | 需要跨 Actor/Plan/Evaluation | 读侧统一口径 |
| 日/周/月趋势 | 每次 group by 大表 | 定期同步 daily read model |
| 问卷批量统计 | 每次扫 AnswerSheet | 批量查询 + 必要时补 AnswerSheetReader |
| 统计口径变化 | 历史数据难修复 | SyncService 支持重建 |
| 高并发查询 | 直接压主业务库 | QueryCache + read model |
| 事件归因延迟 | 无法解释 pending | BehaviorProjector + pending retry |

一句话概括：

> **写模型回答“业务事实是什么”，读侧统计聚合回答“面向报表和运营视角，如何高效、稳定、可修复地看这些事实”。**

---

## 1. 为什么统计不能每次实时从业务写模型算

统计查询通常不是简单的“查一条数据”。

它可能需要同时看：

- 机构。
- 医生/咨询师。
- 受试者。
- 测评入口。
- 答卷。
- Assessment。
- Report。
- Plan task。
- 行为事件。
- 时间窗口。
- 风险等级。
- 问卷/量表。
- 完成率。
- 未完成数。
- 趋势序列。

如果每次请求都实时 join / scan / group by，会带来几个问题：

| 问题 | 说明 |
| ---- | ---- |
| 查询路径太长 | 从多个业务模块拼数据 |
| 影响主写库 | 报表查询容易拖慢写入事务 |
| 口径难统一 | 每个接口可能写一套统计 SQL |
| 趋势查询昂贵 | 日期维度 group by 成本高 |
| 权限和范围复杂 | org / clinician / entry / testee 多重 scope |
| 无法修复历史口径 | 改统计规则后旧数据难重算 |
| 难缓存 | 不同查询维度组合过多 |
| 排障困难 | “统计不准”不知道错在查询、归因还是数据源 |

所以需要一个独立的 Statistics 读侧聚合层。

---

## 2. 写模型和读模型的职责不同

### 2.1 写模型负责业务事实

例如：

| 模块 | 写模型事实 |
| ---- | ---------- |
| Survey | Questionnaire、AnswerSheet |
| Scale | MedicalScale、Factor、InterpretRule |
| Evaluation | Assessment、Score、Report |
| Actor | Testee、Clinician、Operator、AssessmentEntry |
| Plan | EvaluationPlan、PlanTask |

这些写模型的首要职责是保证业务不变量，而不是满足复杂报表查询。

### 2.2 读侧统计负责查询口径

Statistics 读侧聚合负责：

- 系统总览。
- 问卷统计。
- 计划统计。
- 受试者统计。
- 医生/咨询师统计。
- Entry 统计。
- 行为旅程统计。
- 周期性统计。
- 趋势统计。
- 批量问卷统计。

它的职责是面向查询和运营视角，提供稳定的统计结果。

## 4. ReadService 为什么存在

ReadService 统一承接统计查询：

- `GetOverview`。
- `ListClinicianStatistics`。
- `GetClinicianStatistics`。
- `ListAssessmentEntryStatistics`。
- `GetAssessmentEntryStatistics`。
- `GetCurrentClinicianStatistics`。
- `ListCurrentClinicianEntryStatistics`。
- `GetCurrentClinicianTesteeSummary`。
- `GetQuestionnaireBatchStatistics`。

它还组合了：

- `StatisticsReadModel`。
- `AnswerSheetReader`。
- `statisticscache.Cache`。
- `HotsetRecorder`。
- overviewQuery。
- clinicianStatsQuery。
- entryStatsQuery。
- questionnaireBatchQuery。
- statisticsCacheHelper。

这说明读侧不是简单 repository，而是一个专门的查询编排层。

---

## 5. 为什么需要 QueryCache 和 Hotset

统计查询有两个特点：

1. 读多写少。
2. 相同 org + filter + preset 的查询可能被反复访问。

所以适合使用 QueryCache。

### 5.1 QueryCache 解决什么

| 问题 | 作用 |
| ---- | ---- |
| overview 重复查询 | 缓存结果 |
| dashboard 高频刷新 | 降低 MySQL 压力 |
| 查询窗口固定 | cache key 可稳定 |
| 统计数据延迟可接受 | 读侧最终一致 |

### 5.2 Hotset 解决什么

Hotset 记录热点查询目标：

- 哪些 org 常被访问。
- 哪些 overview preset 热。
- 哪些问卷 code 热。
- 哪些 plan 热。

这可以支持：

- warmup。
- cache governance。
- operating 只读观察。
- 后续容量优化。

### 5.3 为什么不是普通 ObjectCache

统计查询不是简单对象读取，而是带 filter/time range/preset 的 query result。

所以它应该走 QueryCache，而不是把统计结果塞进对象缓存。

---

## 6. 为什么需要 BehaviorProjector

很多统计不是单条业务表能直接给出的。

例如：

- 一个医生打开了哪个 entry？
- 哪个受试者从 intake 到 assessment 完成经历了哪些步骤？
- 某个行为事件是否能归因到 clinician / entry / testee？
- 行为事件来的时候，关联对象还没准备好怎么办？

BehaviorProjector 解决的是：

```text
把分散的行为事件投影成统计旅程 / 行为足迹 / 分析 checkpoint
```

### 6.1 ProjectBehaviorEvent

ProjectBehaviorEvent 做：

1. 开启事务。
2. 尝试写 analytics projector checkpoint。
3. 如果事件已处理，则直接返回。
4. 路由事件到具体投影逻辑。
5. 如果无法立即归因，则写 pending retry。
6. 标记 checkpoint 为 completed 或 pending。

这带来两个关键能力：

- 幂等投影。
- pending 归因重试。

### 6.2 Pending Reconcile

如果事件暂时无法归因，系统不会直接丢弃，而是进入 pending retry。

后续 `ReconcilePendingBehaviorEvents` 会：

1. 取 due pending rows。
2. 解码事件。
3. 重新 project。
4. 成功后删除 pending。
5. 仍 pending 则 reschedule。
6. 失败则记录 attempt 和 last error。

这就是为什么统计不应只靠实时 join。行为归因本身就是一个过程，需要状态和补偿。

---

## 7. 为什么需要 SyncService 重建读模型

统计口径会变，历史数据也可能需要修复。

例如：

- 新增统计维度。
- 修复完成率算法。
- 补齐历史 plan 数据。
- 修复某个 org 的 daily 统计。
- 重新生成 org snapshot。
- 统计事件投影出现延迟后补偿。

SyncService 的职责是：

```text
应用层负责编排锁、窗口和事务；
infra writer 负责具体重建 SQL。
```

它支持：

- `SyncDailyStatistics`。
- `SyncOrgSnapshotStatistics`。
- `SyncPlanStatistics`。

### 7.1 为什么需要锁

统计重建可能是重操作，如果多个实例同时重建同一 org/window，会造成：

- 重复写入。
- 锁等待。
- 统计抖动。
- DB 压力放大。

SyncService 使用 Redis lock lease：

```text
statistics:daily:{org}:{start}:{end}
statistics:org_snapshot:{org}:{day}
statistics:plan:{org}:{day}
```

抢不到锁则返回 busy，避免并发重建。

### 7.2 为什么要 repair window

默认 repair window 是 7 天。

如果没有窗口控制，每次 daily sync 都扫全量历史，成本不可控。

窗口让系统在“补偿最近变化”和“控制成本”之间折中。

---

## 8. 为什么不能只靠实时查询 + 缓存

有人可能会说：

```text
每次实时算，然后缓存结果，不就行了吗？
```

不够。

### 8.1 缓存不能解决口径重建

如果算法变了，缓存只能失效，不能修复历史读模型。

### 8.2 缓存不能解决行为归因

行为事件可能需要 pending、checkpoint、retry。这不是缓存能解决的问题。

### 8.3 缓存不能降低首次查询成本

首次 miss 仍然要跑复杂实时聚合，可能拖慢接口。

### 8.4 缓存不能解释数据不准

统计不准时，需要知道是：

- 源数据缺失。
- 事件未投影。
- pending 未 reconcile。
- 同步未运行。
- cache 未失效。
- read model 旧。

单纯缓存没有这些状态。

所以需要：

```text
read model + sync + projector + cache
```

而不是只有 cache。

---

## 9. 读侧统计聚合和业务模块的关系

读侧统计不是独立造事实，它依赖业务事实。

| 来源模块 | 统计使用方式 |
| -------- | ------------ |
| Survey | AnswerSheet、Questionnaire、提交时间、答案 |
| Evaluation | Assessment、Score、Report、风险等级 |
| Actor | Testee、Clinician、Entry、Operator |
| Plan | Plan、Task、完成状态 |
| Event | 行为事件、生命周期事件 |
| Security | org/user scope |
| Redis | QueryCache、Hotset、LockLease |

Statistics 不应该反向修改这些业务模块的聚合。

---

## 10. 读侧统计聚合的查询边界

Statistics 可以做：

- 查询聚合表。
- 查询读模型。
- 查询 AnswerSheetReader 作为辅助。
- 使用 QueryCache。
- 记录 Hotset。
- 按 org/filter/preset 归一化查询。
- 分页返回 clinician/entry statistics。

Statistics 不应该做：

- 修改 AnswerSheet。
- 修改 Assessment。
- 修改 PlanTask。
- 修改 Scale 规则。
- 直接替代业务模块状态机。
- 在查询接口中偷偷重建全量统计。

---

## 11. 为什么 Statistics 要有独立 infra writer

SyncService 的注释已经明确：

```text
application 只负责锁、时间窗口和事务编排；
统计表重建 SQL 属于 infra writer。
```

这条边界很重要。

原因：

- SQL 重建是存储细节。
- 应用层只表达“重建哪个 org/window”。
- 事务和锁是应用编排。
- 具体表结构和 SQL 优化留在 infra。

这符合分层原则，也避免 application 层堆大量 SQL。

---

## 12. 为什么统计聚合需要 MySQLLimiter

统计查询和重建可能非常重。

如果不加下游保护：

- overview 高并发刷新会打 MySQL。
- list clinician/entry 可能变慢。
- sync 重建可能和业务写入抢连接。
- report/dashboard 会影响主链路。

StatisticsModule 注入 MySQLLimiter，说明统计读写也必须参与 backpressure 治理。

---

## 13. 为什么统计读模型仍然需要 AnswerSheetReader

有些查询，尤其是问卷批量统计，可能需要补充 AnswerSheet 信息。

ReadService 中的 questionnaireBatchQuery 同时持有：

```text
readModel
answerSheetRead
```

这说明读侧聚合不是完全脱离业务源，而是：

```text
优先使用读模型；
必要时通过受控 reader 补充源数据。
```

关键是不能让每个接口自由散查业务库。

---

## 14. 这个设计带来的收益

### 14.1 查询性能稳定

统计接口可以读取为查询优化过的 read model，而不是实时 join 多个业务表。

### 14.2 统计口径集中

overview、clinician、entry、questionnaire batch 等查询集中在 ReadService 和 ReadModel，避免口径散落。

### 14.3 可修复

口径变更或数据异常后，可以通过 SyncService 重建。

### 14.4 可观测

可以观察：

- cache hit/miss。
- hotset。
- sync lock busy。
- pending behavior events。
- projector checkpoint。
- sync 日志。
- governance status。

### 14.5 可扩展

未来新增统计维度时，可以：

- 增加 read model。
- 增加 projector。
- 增加 sync writer。
- 增加 cache key。
- 不必改写业务写模型。

---

## 15. 这个设计的代价

| 代价 | 说明 |
| ---- | ---- |
| 最终一致 | 统计可能滞后于业务写入 |
| 系统复杂度上升 | projector、sync、cache、read model 都要维护 |
| 排障路径更长 | 要查源数据、投影、同步、缓存 |
| 需要补偿任务 | pending reconcile、daily sync、plan sync |
| 需要治理缓存 | version token、warmup、hotset |
| 需要容量规划 | 统计查询和重建都可能压 MySQL |

这些代价是读侧统计聚合必须付出的成本。

---

## 16. 常见问题场景

### 16.1 统计数比实际少

可能原因：

- 业务事件还没投影。
- pending behavior event 未 reconcile。
- daily sync 未运行。
- cache 返回旧值。
- read model 重建窗口不覆盖目标日期。
- org scope 过滤错误。

### 16.2 统计趋势缺日期

可能原因：

- daily read model 没生成。
- 查询窗口 from/to 错误。
- fillMissingDailyCounts 只对有效窗口补 0。
- 时区导致日期边界偏移。

### 16.3 医生统计不准

可能原因：

- 行为事件未正确归因 clinician。
- entry 与 clinician 关系变更。
- care relationship transfer 未投影。
- pending attribution 未完成。

### 16.4 查询很慢

可能原因：

- QueryCache miss。
- read model SQL 慢。
- MySQL backpressure 已满。
- org 数据量过大。
- 没有命中合适索引。
- 统计接口误用实时源表。

## 17. 替代方案分析

### 17.1 每次实时 join 计算

优点：

- 数据最新。
- 不需要同步任务。

缺点：

- 查询慢。
- 压主库。
- 口径散落。
- 不能修复历史口径。
- 高并发下不可控。

结论：不适合 dashboard / operating / 管理后台统计。

### 17.2 只加 Redis 缓存

优点：

- 对热点查询有效。
- 实现相对简单。

缺点：

- 首次查询仍慢。
- 无法重建历史。
- 无法处理行为归因。
- 缓存失效和统计口径仍复杂。

结论：缓存只能辅助，不能替代读模型。

### 17.3 离线数仓

优点：

- 分析能力强。
- 不影响业务库。

缺点：

- 架构成本高。
- 实时性弱。
- 对当前中小规模系统过重。
- 需要额外数据管道。

结论：未来可以演进，但当前不必一开始就上数仓。

### 17.4 当前方案：业务库内读侧聚合 + 投影 + 同步 + 缓存

优点：

- 工程复杂度可控。
- 查询性能较稳定。
- 可重建、可修复。
- 与业务系统耦合适中。
- 能支持 operating/dashboard。

缺点：

- 最终一致。
- 需要维护同步和投影。

结论：符合当前 qs-server 阶段。

---

## 18. 设计不变量

后续演进应坚持：

1. 写模型不为统计查询牺牲业务不变量。
2. 统计查询不直接散查多个业务模块。
3. 复杂统计优先进入 ReadModel。
4. 高频统计查询必须支持 QueryCache。
5. 统计重建必须有锁和窗口。
6. 行为投影必须有 checkpoint 和 pending retry。
7. 统计口径变化必须支持重建。
8. 统计服务不反向修改 Survey/Evaluation/Actor/Plan 聚合。
9. 统计查询必须带 org scope。
10. 统计性能问题优先查 read model/cache/backpressure，而不是直接加 DB 连接。

---

## 19. 常见误区

### 19.1 “统计就是几条 SQL”

不是。统计包含读模型、同步、行为投影、缓存、锁、补偿和治理。

### 19.2 “统计一定要实时”

不一定。大多数运营统计允许短暂延迟，但必须口径稳定、可解释。

### 19.3 “加缓存就不用读模型”

不对。缓存不能解决首次查询、历史重建和行为归因。

### 19.4 “统计不准就是查询 SQL 错”

不一定。可能是事件投影、pending reconcile、sync window 或 cache 版本问题。

### 19.5 “读侧聚合会污染领域模型”

不会，只要它不反向修改写模型聚合，并通过 port/read model 协作。
